# 01. 챗봇과 싸우지 않기

## AI와 말씨름하는 순간

AI를 써 본 사람이라면 비슷한 경험이 있을 것입니다. 어떤 날은 내 말을 기가 막히게 알아듣습니다. 대충 던진 요청에서도 의도를 잡아내고, 코드도 만들고, 설명도 해줍니다. 그런데 어떤 날은 같은 말을 몇 번이나 반복해도 이상한 방향으로 갑니다. 분명히 말했는데 놓치고, 방금 고친 내용을 다시 되돌리고, 하지 말라고 한 일을 또 합니다.

이때 우리는 보통 프롬프트를 고칩니다. 더 자세히 말해 봅니다. 역할을 부여합니다. 예시를 붙입니다. 금지 사항을 추가합니다. 그래도 잘 안 되면 더 길게 씁니다. 어느 순간부터는 AI를 쓰는 것이 아니라 AI와 말씨름을 하는 느낌이 듭니다.

## 챗봇은 답하고, 에이전트는 일한다

프롬프트는 중요합니다. 좋은 지시는 나쁜 지시보다 낫습니다. 하지만 긴 작업에서는 더 좋은 문장만으로 충분하지 않습니다. 단발성 질문에서는 프롬프트의 품질이 크게 먹힙니다. 그러나 에이전트에게 실제 작업을 맡기는 순간 문제의 성격이 바뀝니다.

챗봇은 답을 냅니다. 에이전트는 일을 합니다.

파일을 읽고, 판단하고, 수정하고, 명령을 실행하고, 결과를 생성합니다. 이 과정이 한두 번이면 별문제가 없어 보입니다. 하지만 작업이 길어질수록 초반의 작은 오해가 다음 판단의 근거가 됩니다. 잘못된 가정이 계획으로 굳고, 그 계획이 파일 수정으로 이어지고, 그 수정 결과가 다시 검증 대상이 됩니다. 처음에는 작은 흠이었는데, 몇 단계 뒤에는 어디서부터 어긋났는지 찾기 어려워집니다.

중요한 것은 한 번의 답변 품질이 아닙니다. 실패가 누적되고 증폭되는 방식을 통제하는 일입니다.

## Harness는 말이 아니라 환경이다

AI를 잘 쓰는 사람과 그렇지 못한 사람의 차이도 여기서 갈립니다. 좋은 질문을 한 번 던지는 능력만으로는 부족합니다. 무엇을 읽게 할지, 무엇을 못 하게 막을지, 어디서 검증할지, 상태를 어디에 남길지 정해야 합니다. 챗봇과 싸우지 않으려면 말투를 더 세게 고치는 것이 아니라, AI가 일하는 환경을 바꿔야 합니다.

좋은 결과는 한 번의 멋진 프롬프트에서 나오지 않습니다. 반복 가능한 규칙, 선별된 컨텍스트, 위험을 막는 권한 경계, 결정론적 검증, 세션 밖에 남는 상태 기록이 함께 작동할 때 나옵니다. 이 전체 구조를 `Harness`라고 부릅니다.

`Harness`는 프롬프트를 버리자는 말이 아닙니다. 컨텍스트를 덜 중요하게 보자는 말도 아닙니다. 프롬프트와 컨텍스트가 흩어진 요령으로 남지 않게, 실제 작업 환경 안에 고정하자는 말입니다.

예를 들어 "기존 코드를 잘 재사용해줘"라는 지시는 무력합니다. 에이전트가 관련 파일을 보지 못하면 아무 의미가 없습니다. "테스트를 잘 확인해줘"라는 당부도 소용없습니다. 실제 테스트를 실행하고, 실패하면 멈추고, 실패 메시지를 기준으로 다시 고치도록 시스템이 강제해야 합니다. "위험한 명령은 조심해줘"라고 타이를 것이 아니라, 권한과 승인 규칙으로 원천 차단해야 합니다.

프롬프트는 지시입니다. 컨텍스트는 지시가 기대는 정보입니다. Harness는 그 지시와 정보가 실제 작업으로 이어질 때의 환경입니다.

이 차이를 이해하면 AI 도구를 보는 관점도 달라집니다. "어떤 모델이 더 똑똑한가"라는 질문보다는, 그 모델이 무엇을 볼 수 있는지, 어떤 도구를 쓸 수 있는지, 어디서 멈추게 해야 하는지, 어디서 검증을 통과해야 하는지, 실패하면 어떤 상태로 돌아가는지가 더 중요해집니다.

좋은 도구는 대화를 잘하는 도구가 아니라, 일을 안전하게 반복할 수 있게 만드는 도구입니다. 좋은 사용자는 AI에게 더 화려한 주문을 외우는 사람이 아니라, AI가 길을 잃지 않도록 환경을 설계하는 사람입니다.

이 글의 제목을 `Harness 잘 사용하기`로 잡은 이유도 여기에 있습니다. 이제 더 이상 챗봇과 말씨름하며 더 좋은 질문을 찾는 것만으로는 충분하지 않습니다. 에이전트가 일할 수 있는 환경을 만들어야 합니다.
