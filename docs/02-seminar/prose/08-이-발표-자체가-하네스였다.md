# 08. 이 발표 자체가 Harness였다

## 08-1. 결과물보다 과정을 봐야 한다

00장에서 제가 이런 말을 했습니다. 이 발표 자료 또한 전부 Harness로 만들었습니다. 이제 그 말을 다시 꺼낼 차례입니다.

여기서 중요한 것은 "AI가 만들었다"는 자랑이 아닙니다. AI가 HTML을 만든 사실도 흥미롭지만 핵심은 아닙니다. 중요한 것은 무엇을 AI에게 맡겼고, 무엇을 사람이 끝까지 붙들었는지입니다.

원본 발표도 89장의 발표 자료를 만드는 과정에서 마크다운, `CLAUDE.md`, 규칙 문서, 검증 프로세스를 핵심 생산물로 둡니다.[^08-page77] 특별한 마법이 아니라, 사람이 설계한 경계와 책임이 하네스였다는 점이 중요합니다.

## 08-2. source에서 prose로

가장 먼저 한 일은 source 경계를 정하는 것이었습니다. 어떤 자료가 1차 source인지, 어떤 자료가 보조 source인지, 어떤 자료는 이번 글에 쓰지 않을지 나눴습니다. 이 구분이 없으면 나중에 "이 주장은 어디서 왔는가"라는 질문에 답할 수 없습니다.

OpenAI Harness engineering 글은 에이전트에게 "a map"을 주라고 말합니다.[^08-openai-map] source 경계를 정한다는 건 자료를 몰아서 던지는 게 아니라, 어디를 먼저 볼지 흐름을 고정하는 일입니다.

source는 곧바로 slide가 되지 않았습니다. 먼저 prose가 되었습니다. prose는 발표 원고라기보다 논리의 기준선입니다. 어떤 주장이 중심이고, 어떤 주장이 보조인지, 어떤 순서로 독자를 데려갈지 여기서 정합니다.

slide는 prose를 압축할 수는 있지만 벗어날 수 없습니다. notes는 prose를 구어체로 풀 수는 있지만 논리 순서를 마음대로 바꿀 수 없습니다. 이 기준선이 있어야 뒤의 산출물이 대화창의 분위기에 끌려가지 않습니다.

이 단계에서 사람이 맡은 것은 논지의 흐름, 주장 강도, 한국어 문체, source 누락 방지였습니다. 에이전트에게 맡긴 것은 초안 정리와 변형, 반복 검토, 구조화 보조였습니다.

## 08-3. outline과 manifest가 계획을 들고 갔다

prose가 있다고 곧바로 HTML을 만들지 않았습니다. 먼저 outline을 만들었습니다.

outline은 목차가 아닙니다. 어떤 장면을 slide로 만들지, 어떤 source section을 참조할지, 어떤 시각 구조가 필요한지 정하는 승인 게이트입니다. 여기서 방향이 틀리면 HTML을 아무리 잘 만들어도 비용만 커집니다.

manifest는 상태를 들고 갔습니다. 어떤 본문이 어떤 slide에 담겼는지, 무엇이 완료되었고 무엇이 pending인지, 어떤 파일이 기준인지, 어떤 검증을 통과했는지 기록했습니다.

작업이 길어질수록 이런 파일이 중요해집니다. 세션이 바뀌어도 작업이 이어지는 것은 기억력이 좋아서가 아닙니다. 상태를 파일에 남겨 두었기 때문입니다.

수집층/계획층/실행층은 폴더 구조로도 고정했습니다. 실제로는 `research/`, `sections/`, `decks/`가 서로 다른 역할을 담당했고, 그 경계가 제작 과정의 실행 규칙이었습니다.[^08-page78]

검토와 수정은 `pending.md` 같은 피드백 흐름을 거쳐 반복되었고, 최종 산출은 PDF로 묶였습니다.[^08-page81]

## 08-4. design contract가 시각 결정을 고정했다

발표 자료는 내용만으로 완성되지 않습니다. 시각 구조도 기준이 필요합니다. 그래서 design contract를 먼저 고정했습니다.

디자인 계약에는 톤, 색상 팔레트, 타이포그래피, 여백, layout family, placeholder 규칙, 금지 표현, 과한 장식의 기준이 들어갑니다.
`Pretendard`와 720pt × 405pt, 명사형 위주 톤, 불필요한 장식 축소 같은 규칙은 이런 계약 안에서 고정됐고, 이 덕분에 deck의 톤이 매 slide마다 갈라지지 않았습니다.[^08-page79]

OpenAI는 agent-first 저장소에서 목표가 `agent legibility`, 즉 에이전트가 저장소 자체에서 비즈니스 도메인을 읽고 추론할 수 있게 만드는 것이라고 설명합니다.[^08-openai-legibility] design contract도 같은 방향입니다. 사람 취향을 즉흥 피드백으로 남기지 않고, 재현 가능한 규칙으로 바꾼다는 점에서 같습니다.

좋은 design contract는 에이전트의 모든 자유를 막으려고 만든 문서가 아닙니다. 중요한 선택만 규칙으로 강하게 남기고, 나머지는 반복 사용에서 검증되게 만드는 문서입니다.

## 08-5. 생성과 검증을 같은 손에 쥐지 않았다

생성자가 자기 결과를 스스로 통과시키게 두면 그럴듯한 오류가 오래 살아남습니다. 그래서 생성과 검증을 분리했습니다.

slides-grab 기준으로 Plan → Design → Export를 세 단계로 나눴고, Plan과 Export 사이에 사람이 승인했습니다.[^08-page80]
생성 직후에는 slide contract, 한국어 점검, deck consistency 점검, source 매핑 검증, 디자인 계약 위반 검출이 이어졌습니다. 사람 검토와 자동 점검이 서로 다른 층에서 같은 방향을 봤습니다.

Codex app의 Skills 설명도 같은 방향입니다. Skills는 instructions, resources, scripts를 묶어 팀의 방식대로 작업을 안정적으로 끝내게 하는 장치입니다.[^08-codex-skills] 검증도 일회성 대화가 아니라 반복 가능한 절차로 묶여야 합니다.

이 원칙은 코드와 같습니다. 에이전트가 코드를 생성했다면 같은 에이전트의 "괜찮습니다"만 믿으면 안 됩니다. 테스트, lint, type check, 리뷰어, 다른 모델, 사람 검토가 분리되어야 합니다.

이 발표 제작에서도 중요한 것은 "AI가 만들었다"가 아니라 "AI가 만든 것이 어떤 검증 게이트를 거쳤는가"였습니다. 하네스는 생성 능력을 과시하는 장치가 아니라, 생성된 것을 믿을 수 있게 만드는 구조입니다.

## 08-6. 계층을 섞지 않았다

이 제작 과정은 층을 섞지 않았습니다.

source는 근거의 경계를, prose는 주장 순서를, outline은 장면의 논리를, design은 시각 규칙을, validation은 실패 조건을 담당합니다.

문제가 생기면 계층별로 돌아갑니다. 문장이 흔들리면 prose를 다시 보고, slide 구성이 흐트러지면 outline을 조정합니다. 시각 규칙이 흔들리면 design contract를 확인하고, 근거가 빠지면 source로 돌아갑니다. 막연하게 "다시 잘해 봐"라고 말하지 않아도 되는 이유가 여기입니다.

## 08-7. 이 제작 과정이 보여 준 것

이 제작 과정이 보여 준 증거는 단순합니다. source를 먼저 문장으로 정돈하고, prose로 논리 축을 세우고, outline으로 장면을 나눴고, design contract로 시각을 고정하고, 검증으로 다시 돌려봤습니다.

코드든 문서든 기획이든 디자인이든 구조 자체의 리듬은 같습니다. 원하는 것을 먼저 글로 정하고, 규칙을 문서로 고정하고, 산출물을 사람이 검증합니다. 이 발표 제작 과정 자체가 그 작은 사례였습니다.
