# 프롬프트를 넘어서: 에이전트를 움직이는 기술, Harness

## 시작하며: 이제는 문장보다 구조다

AI 코딩 도구가 처음 주목받았을 때 사람들의 관심은 한곳에 몰렸습니다. 어떤 문장을 써야 모델이 원하는 결과를 내놓는가. `Prompt Engineering`이 중심이던 시기입니다. 곧 질문은 넓어졌습니다. 한 문장만 잘 쓴다고 끝나지 않고, 어떤 파일과 어떤 배경지식을 함께 보여주느냐가 성능을 갈랐기 때문입니다. 여기서 `Context Engineering`이라는 말이 자연스럽게 따라붙었습니다.

하지만 에이전트가 여러 단계를 거쳐 오래 일하기 시작하자, 문제는 다시 바깥으로 밀려났습니다. 이제 핵심은 “무엇을 말할 것인가”, “무엇을 보여줄 것인가”에서 멈추지 않습니다. 더 중요한 질문은 이것입니다. 에이전트가 어떤 환경 위에서 움직이고, 어떤 제약 안에서 일하며, 어떤 검증을 통과해야 하느냐. 이 질문을 붙드는 일이 `Harness Engineering`입니다.

## Prompt, Context, Harness는 질문이 다르다

이 셋은 비슷해 보이지만, 실제로는 전혀 다른 층위의 문제를 다룹니다.

| 층위 | 핵심 질문 | 초점 |
| --- | --- | --- |
| `Prompt` | 어떻게 말할 것인가 | 명령의 품질 |
| `Context` | 무엇을 보여줄 것인가 | 정보의 품질 |
| `Harness` | 무엇을 통제하고 방지할 것인가 | 시스템의 안정성 |

처음에는 문장을 다듬는 일이 중요했습니다. 그다음에는 어떤 정보를 밀어 넣을지 고르는 일이 중요해졌습니다. 하지만 장시간 자율 작업으로 넘어오면 그것만으로는 버티지 못합니다. 무엇을 못 하게 막을지, 어디서 멈추게 할지, 어떤 문을 통과해야 다음 단계로 넘어갈지까지 설계해야 하기 때문입니다.

그래서 `Harness Engineering`은 프롬프트를 밀어내는 개념이 아닙니다. 프롬프트와 컨텍스트를 더 큰 구조 안에 배치하는 상위 개념입니다. 좋은 하네스는 좋은 프롬프트를 버리지 않습니다. 좋은 컨텍스트도 포기하지 않습니다. 다만 그 둘을 그냥 풀어놓지 않습니다.

## 컨텍스트만으로는 오래 못 간다

여기서 `Context Engineering`을 “더 많이 넣는 기술”로 이해하면 곧바로 길을 잃습니다. 좋은 컨텍스트는 넓은 컨텍스트가 아니라 선별된 컨텍스트입니다. 필요한 파일과 규칙만 남기고, 지금 태스크와 무관한 인터페이스와 소음을 덜어내야 판단이 또렷해집니다.

그런데 그것만으로는 충분하지 않습니다. 에이전트는 한 번 답하고 끝나는 존재가 아니라 탐색하고, 실행하고, 검증하고, 다시 시도하는 루프를 돕니다. 이 루프에서는 작은 실패 하나도 다음 단계의 입력으로 다시 들어갑니다. 단일 응답이 그럴듯하다는 사실만으로 전체 작업의 신뢰성을 설명할 수 없는 이유가 여기에 있습니다.

병목도 바로 여기서 드러납니다. 긴 작업에서 문제를 갈라놓는 것은 모델의 재치만이 아닙니다. 권한 경계, 상태 기록, 검증 루프, 실행 흔적, 오류 복구, 인간 개입 지점 같은 바깥 구조가 훨씬 더 크게 작동합니다. 하네스가 필요한 이유는 결국 모델이 부족해서가 아니라, 에이전트 작업이 루프이기 때문입니다.

## Agent = Model + Harness

이 관점을 가장 짧게 요약하면 흔히 이 한 줄로 정리합니다.

> **`Agent = Model + Harness`**

이 문장의 뜻은 단순합니다. 모델은 추론과 생성을 담당하는 엔진에 가깝지만, 실제 에이전트의 성격과 성능은 그 엔진 바깥에서 크게 갈린다는 뜻입니다. 무엇을 볼 수 있는지, 어떤 도구를 쓸 수 있는지, 어디서 멈추는지, 잘못됐을 때 무엇으로 걸러내는지는 모두 하네스 쪽 문제입니다.

같은 모델을 써도 결과가 달라지는 이유도 여기에 있습니다. 프롬프트를 조금 더 잘 썼느냐보다, 어떤 실행 환경 위에 그 모델을 올려놓았느냐가 더 크게 작동하는 순간이 분명히 존재합니다. 에이전트를 논할 때 모델만 보고 끝내면 중요한 절반을 놓치게 됩니다.

## 하네스의 5대 해부학

하네스를 손에 잡히는 말로 설명하려면 다섯 블록으로 나누는 편이 가장 명확합니다.

| 블록 | 하는 일 | 구현 예시 |
| --- | --- | --- |
| `Guardrails` | 위험한 행동을 막고 권한 경계를 고정한다 | 승인 규칙, 샌드박스, 위험 명령 차단 |
| `Specification` | 작업 범위와 기준선을 흔들리지 않게 고정한다 | 작업 명세, 계획 문서, 규칙 파일 |
| `Verification` | 결과를 결정론적으로 걸러낸다 | `lint`, `type check`, `test`, 리뷰 게이트 |
| `State Management` | 세션이 바뀌어도 상태가 증발하지 않게 만든다 | 파일, `Git` 이력, 진행 기록 |
| `Observability` | 루프가 어디서 흔들리는지 보이게 만든다 | 실행 로그, 행동 trace, checkpoint |

이 다섯 블록이 필요한 이유는 단순합니다. 하네스를 그저 “중요한 것”이라고 부르면 아무것도 설계되지 않습니다. 무엇이 손을 막고, 무엇이 결과를 걸러내고, 무엇이 상태를 세션 밖에 남길지까지 이름 붙여야 구조가 보입니다. 그때 하네스는 태도가 아니라 운영 도면이 됩니다.

특히 `State Management`와 `Observability`는 장시간 작업에서 무게가 커집니다. 기억을 대화 안에만 쌓아 두면 세션이 끊기는 순간 맥락도 함께 사라집니다. 반대로 실행 흔적이 없으면 루프가 어디서 어긋났는지 읽을 수도 없습니다. 많이 기억하는 것보다, 기억이 썩지 않게 남기는 편이 더 중요합니다.

## 기능 블록과 도구 레이어는 다르다

여기서 한 번 더 분리해야 할 것이 있습니다. `Guardrails`, `Specification`, `Verification`, `State Management`, `Observability`는 하네스의 기능 블록입니다. 반면 `CLAUDE.md`, `AGENTS.md`, `Skills`, `Hooks`, `MCP`, `Plugins`는 그 블록을 실제 작업 환경에 배치하는 도구 레이어입니다.

이 둘을 섞어 생각하면 구조가 금세 흐려집니다. 먼저 어떤 기능이 필요한지 정하고, 그다음 그 기능을 어떤 파일과 도구에 올릴지 정해야 합니다. 하네스는 도구 목록이 아니라 배치 설계라는 말이 여기서 나옵니다.

## `CLAUDE.md`는 기준선을 고정한다

`CLAUDE.md`나 `AGENTS.md` 같은 규칙 파일은 프로젝트가 매번 다시 토론할 필요가 없는 기준을 붙들어 두는 층입니다. 프로젝트 공통 규칙, 로컬 보강 규칙, 사용자 전역 규칙처럼 높이가 다른 상시 지침을 나눠 두면, 에이전트는 매 세션 같은 출발선을 다시 협상할 필요가 없습니다. 코딩 스타일, 금지 명령, 승인 기준, 문서 어투, 리뷰 원칙처럼 세션마다 재설명하면 낭비만 커지는 규칙은 여기 고정하는 편이 낫습니다.

중요한 것은 모든 것을 이 파일 하나에 밀어 넣지 않는 일입니다. 기준선을 세우는 데 필요한 내용과 특정 작업에서만 쓰는 절차가 한데 뭉치면, 규칙 파일은 기준이 아니라 소음으로 바뀝니다. 상시 지침층은 늘 읽혀야 하는 것만 붙들고, 나머지는 `Skills`나 별도 참조 문서로 내려보내는 편이 낫습니다. 그래야 하네스가 두꺼워지는 대신 또렷해집니다.

## `Skills`, `Hooks`, `MCP`는 같은 말이 아니다

이 구분도 중요합니다. `Skills`는 필요할 때 불러오는 작업 지식에 가깝습니다. 특정 종류의 일을 할 때만 꺼내 쓰는 절차와 매뉴얼은 여기 두는 편이 낫습니다. 반대로 `Hooks`는 우회하면 안 되는 기계적 관문입니다. 위험한 명령 차단, 포맷 강제, 검증 자동화처럼 반드시 걸려야 하는 문은 이쪽에 놓입니다.

`MCP`와 `Plugins`는 또 다른 층에 있습니다. 둘은 하네스 전체와 같은 말이 아니라, 외부 도구와 데이터, 기능 묶음을 연결하는 수단에 가깝습니다. 요점은 간단합니다. 도구 이름을 곧 하네스로 오해하지 말아야 한다는 것입니다. 하네스는 이 자산들을 어디에, 어떤 책임으로 배치할지 정하는 쪽에 더 가깝습니다.

그래서 하네스 설계의 핵심은 규칙을 더 보태는 데 있지 않습니다. 규칙을 맞는 높이에 올려놓는 데 있습니다. 프로젝트 전체를 묶는 원칙은 상시 규칙 파일에, 반복되는 업무 절차는 `Skills`에, 우회 불가능한 검증은 `Hooks`와 테스트 게이트에, 외부 시스템 연결은 `MCP`나 `Plugins`에 두는 식입니다. 층이 어긋나는 순간 규칙은 기준이 아니라 소음이 됩니다.

## 정리: 프롬프트와 컨텍스트는 하네스 안에서 제자리를 찾는다

프롬프트는 여전히 중요합니다. 컨텍스트도 여전히 중요합니다. 다만 둘 다 하네스 바깥에서 따로 노는 요소가 아닙니다. 하네스가 어떤 구조를 세우느냐에 따라 프롬프트는 제자리를 찾고, 컨텍스트는 소음이 아니라 작동하는 정보가 됩니다.

좋은 하네스는 더 많이 붙이는 구조가 아닙니다. 필요한 파일, 필요한 도구, 필요한 규칙만 남겨 두는 구조입니다. 어디서 손을 막고, 어디서 결과를 걸러내고, 무엇을 세션 밖에 남길지 분명할수록 에이전트는 오래 버팁니다. `Harness Engineering`이 다루는 것은 결국 모델의 문장력이 아니라, 그 문장이 실제 작업으로 굴러가게 만드는 바깥 구조입니다.
