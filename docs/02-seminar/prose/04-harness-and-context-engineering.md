# Harness와 Context Engineering

## 00. 왜 Prompt만으로는 더 이상 충분하지 않은가

Section 02가 tool genealogy를 보여 주었다면, 이 섹션은 그 계보가 결국 어디로 수렴하는지를 이론적으로 정리한다. 처음에는 `Prompt Engineering`이 중심이었다. 모델에게 무엇을 말할지, 어떤 문장을 주어야 원하는 결과를 얻을 수 있을지가 핵심이었다. 곧이어 무엇을 보여 줄 것인가가 더 중요해졌다. 하지만 긴 작업과 도구 사용이 결합된 순간, 프롬프트와 컨텍스트만으로는 실패를 설명할 수 없게 된다. 그래서 지금 더 앞에 놓이는 것은 에이전트가 어떤 환경에서 일하는가, 즉 `Harness`다.

## 01. Prompt, Context, Harness는 서로 다른 층이다

`Prompt`는 의도를 압축해 전달하는 층이다. 무엇을 하려는지, 어떤 결과를 원하는지, 어떤 제약을 따를지를 문장으로 정리하는 일이다. `Context`는 판단의 근거를 배치하는 층이다. 관련 문서, 코드, 예시, 과거 상태, 외부 자료처럼 모델이 보고 추론해야 할 재료를 어떤 순서와 밀도로 넣을지 결정한다. `Harness`는 그 위를 감싸는 운영 층이다. 어떤 `Prompt`와 어떤 `Context`가 들어갈지 고르고, 어떤 도구와 권한이 열릴지 제한하고, 결과가 행동으로 이어지기 전에 검증과 승인을 거치게 만들고, 작업 상태를 세션 바깥에 유지하며, 전체 루프를 관측 가능하게 만든다.

그래서 `Context Engineering`은 `Harness Engineering`의 일부이지 동의어가 아니다. 컨텍스트를 잘 넣는 것만으로는 충분하지 않고, 그 컨텍스트가 어떤 실행 구조 안에서 소비되는지까지 설계해야 한다.

## 02. Agent = Model + Harness

이 문제를 설명하기 위해 발표는 `Agent = Model + Harness`라는 공식을 제시한다. 모델은 텍스트를 생성하고 판단을 수행하는 엔진이지만, 실제 에이전트의 행동은 모델 바깥의 구조에 의해 결정된다. 무엇을 읽을 수 있는지, 어떤 도구를 쓸 수 있는지, 어디까지 자율적으로 행동할 수 있는지, 어떤 결과가 통과되고 어떤 행동이 차단되는지, 긴 작업 동안 무엇을 기억하고 무엇을 버릴지, 실패했을 때 어떻게 복구할지까지를 정하는 나머지 전부가 `Harness`다.

원본 세미나가 정리한 에이전트 루프인 `Gather Context → Take Action → Verify Work → Repeat`도 이 관점에서 다시 읽을 수 있다. 컨텍스트 수집 단계에는 잡음과 과잉 정보가 유입될 수 있고, 행동 단계에는 권한 남용과 잘못된 도구 사용이 끼어들 수 있으며, 검증 단계가 부실하면 환각과 오작동이 다음 루프의 사실처럼 굳어진다. `Harness Engineering`은 이 루프의 각 단계에서 실패가 증폭되지 않도록 구조를 설계하는 작업이다.

## 03. 하네스의 다섯 기능 블록

실무 용어로 압축하면 다섯 개의 기능 블록이 남는다. `Guardrails`는 위험한 행동과 권한을 제한하는 층이다. `Specification`은 의도와 작업 분해를 선명하게 만드는 층이다. `Verification`은 결과를 통과시키기 전에 점검하는 층이다. `State Management`는 세션 바깥에 기억을 유지하는 층이다. `Observability`는 에이전트가 실제로 무엇을 했는지 보이게 만드는 층이다.

중요한 것은 이 다섯 블록이 하네스의 기능적 요구사항이라는 점이다. 다시 말해 이것들은 “하네스가 무엇을 해야 하는가”를 설명한다. 반면 `CLAUDE.md`, `AGENTS.md`, `Skills`, `Hooks`, `MCP`, `Plugins`, 혹은 `OpenCode`의 agent config처럼 도구마다 다른 구현 수단은 “그 요구사항을 어떤 층에 어떻게 구현하는가”를 설명한다. 기능과 구현을 같은 수준에서 다루기 시작하면 하네스는 금방 비대해지고 설명 불가능해진다.

## 04. 실행 레이어를 분리해서 이해해야 한다

이 구분을 가장 이해하기 쉽게 보여 주는 층이 `CLAUDE.md`와 `AGENTS.md` 같은 규칙 파일이다. 이것들은 모든 세션에서 항상 함께 읽히는 프로젝트 수준의 기준선으로 적합하다. 반면 `Skills`는 특정 작업에 필요한 지식과 절차를 `SKILL.md` 형태로 묶어 두고, 요청과 설명이 맞을 때만 불러오는 온디맨드 자산이다. `Hooks`는 특정 이벤트에 반응해 자동 실행되는 결정론적 장치다. `MCP`는 모델과 외부 세계를 연결하는 통신 계층이고, `Plugins`는 이런 요소들을 팀 자산으로 묶어 배포하는 포장 레이어다.

이렇게 보면 발표가 말하는 `Right Altitude`도 더 선명해진다. 너무 낮은 지시는 모든 예외를 고정해 brittle해지고, 너무 높은 지시는 “잘 해줘” 수준의 모호한 주문이 되어 세션마다 다른 해석을 낳는다. 좋은 하네스는 더 많은 것을 쌓는 구조가 아니라, 필요한 것을 적절한 층에 배치하는 구조다.

## 05. 결론: Harness Engineer라는 새로운 역할

결국 이 섹션의 핵심은 분명하다. `Prompt Engineering`은 모델에게 일을 시키는 기술이고, `Context Engineering`은 모델이 판단할 재료를 설계하는 기술이며, `Harness Engineering`은 그 판단과 행동이 반복 가능한 시스템 안에서 안전하게 일어나도록 만드는 기술이다. 발표가 궁극적으로 말하는 바는, AI 시대의 경쟁력이 더 좋은 문장 하나를 쓰는 능력에만 있지 않다는 점이다. 진짜 차이는 모델이 일하는 환경을 얼마나 정교하게 설계하고, 얼마나 얇고 명확한 구조로 유지하며, 얼마나 검증 가능하게 운영하느냐에서 벌어진다.

그래서 개발자의 역할도 `Harness Engineer`라는 표현으로 수렴한다. 이제 중요한 것은 코드를 직접 얼마나 많이 타이핑하느냐보다, 에이전트가 잘 일할 수 있는 환경을 얼마나 명확하게 설계하고 결과의 품질을 얼마나 엄격하게 감별하느냐에 있다. 동시에 좋은 하네스는 영구히 비대해지는 구조가 아니다. `Build to Delete`라는 원칙이 말하듯, 오늘의 모델 한계를 보완하기 위해 만든 우회 구조는 내일 더 나은 모델 앞에서 조금씩 지워질 수 있어야 한다.
