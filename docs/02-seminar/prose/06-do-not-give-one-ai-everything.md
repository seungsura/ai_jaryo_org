# 06. 하나의 AI에게 다 맡기지 않는다

## 병렬화보다 먼저 분해다

하나의 에이전트에게 모든 것을 맡기면 편해 보입니다. 하나의 창에서 계획하고, 파일을 읽고, 구현하고, 테스트하고, 리뷰까지 끝내면 흐름이 단순해 보입니다. 하지만 긴 작업에서는 이 단순함이 곧 약점이 됩니다.

첫째, 컨텍스트가 비대해집니다. 탐색, 구현, 로그, 실패한 명령, 리뷰 의견, 문서 초안이 같은 창에 쌓입니다. 에이전트가 더 많이 보는 것이 아니라, 더 많은 잡음 속에서 판단하게 됩니다.

둘째, 역할이 섞입니다. 계획한 에이전트가 구현하고, 구현한 에이전트가 자기 결과를 리뷰하고, 리뷰한 에이전트가 다시 테스트 결과를 해석합니다. 자기 답을 자기가 채점하는 구조입니다. 여기에는 blind spot이 남습니다.

셋째, 신뢰 경계가 흐려집니다. 조사만 하면 되는 작업에 쓰기 권한이 열리고, 민감한 정보를 볼 필요가 없는 작업이 전체 컨텍스트를 받습니다. 이 구조에서는 작은 실수가 권한 문제로 번질 수 있습니다.

멀티 에이전트는 이 문제를 줄이기 위한 구조입니다. 더 많은 AI를 붙이는 묘기가 아닙니다. 본질은 병렬화가 아니라 분해입니다. 컨텍스트를 쪼개고, 역할을 쪼개고, 검증 책임을 쪼개는 일입니다.

질문은 이것입니다. 메인 컨텍스트에 무엇을 남기고, 무엇을 밖으로 밀어내며, 누가 누구를 검증하게 만들 것인가.

## 하나의 에이전트는 하나의 역할을 맡아야 한다

멀티 에이전트가 실패하는 가장 흔한 방식은 이름만 늘어나는 것입니다. `planner`, `builder`, `reviewer`, `tester`라는 이름은 붙었지만 모두 같은 정보를 보고, 같은 권한을 갖고, 같은 기준으로 움직이면 실제로는 한 에이전트를 여러 번 부른 것과 다르지 않습니다.

역할이 의미를 가지려면 세 가지가 달라야 합니다.

- 보는 정보
- 할 수 있는 행동
- 통과시켜야 하는 기준

조사자는 읽고 요약해야 합니다. 구현자는 승인된 계획 안에서 파일을 고쳐야 합니다. 리뷰어는 구현자의 의도를 그대로 믿지 않고 요구사항과 테스트 기준으로 봐야 합니다. 테스트 담당자는 코드가 마음에 드는지가 아니라 실패와 회귀를 찾아야 합니다.

이렇게 나누면 에이전트 수가 늘어나는 것이 아니라 책임의 표면이 선명해집니다. 누구에게 어떤 일을 맡겼는지, 어떤 결과만 메인으로 가져올지, 어떤 검증을 통과해야 다음 단계로 갈지 보이기 시작합니다.

## Sub-Agent는 중간 작업을 격리한다

가장 자주 쓰는 기본형은 `Sub-Agent`입니다. 메인 에이전트가 특정 작업을 맡기고, 격리된 워커가 처리한 뒤, 요약과 근거만 돌려주는 구조입니다.

로그를 길게 읽어야 할 때, 특정 디렉터리의 파일을 훑어야 할 때, 여러 문서에서 관련 규칙을 찾아야 할 때, 테스트 실패 원인을 좁혀야 할 때 유용합니다. 서브 에이전트가 많은 파일을 읽어도 메인 컨텍스트는 오염되지 않습니다. 메인은 전체 탐색 과정을 들고 있는 것이 아니라, 결론과 근거만 받습니다.

여기서 중요한 원칙은 최소 권한입니다. 서브 에이전트에게 전체 저장소와 전체 대화를 통째로 넘기지 않습니다. 그 작업에 필요한 파일, 질문, 출력 형식, 금지 행동만 줍니다. 가능하면 읽기 전용으로 시작하고, 쓰기 권한은 정말 필요할 때만 열어야 합니다.

좋은 `Sub-Agent` 지시는 대개 이 정도면 충분합니다. 무엇을 찾을지, 어디까지 읽을지, 무엇은 보지 말아야 할지, 어떤 형식으로 돌려줄지, 확신이 낮은 부분을 어떻게 표시할지 정합니다. 이 작은 경계가 메인 세션을 지킵니다.

## Orchestrator는 작업을 나누고 수렴시킨다

범위가 더 커지면 `Orchestrator`가 필요합니다. 하나의 큰 목표를 하위 작업으로 나누고, 여러 실행자에게 배정하고, 결과를 모아 다시 통합하는 구조입니다.

대규모 리팩터링, 문서 전체 개편, 여러 모듈에 걸친 테스트 보강, 마이그레이션 준비처럼 넓은 작업에 잘 맞습니다. 하위 작업끼리 깊게 토론할 필요는 없지만, 전체 방향과 수렴 지점은 하나로 유지해야 할 때 쓰기 좋습니다.

`Orchestrator`의 품질은 실행자의 수가 아니라 분해의 품질에서 갈립니다. 작업 경계가 흐리면 실행자들은 서로 같은 파일을 건드리고, 다른 기준으로 판단하고, 나중에 통합 비용을 폭발시킵니다. 반대로 경계가 선명하면 각 실행자는 좁은 문제를 깊게 처리하고, 메인은 결정과 통합과 검증에 집중할 수 있습니다.

그래서 `Orchestrator`는 단순한 관리자 역할이 아닙니다. 어떤 일을 나눌 수 있고, 어떤 일은 나누면 안 되는지 판단하는 역할입니다.

## 생성자와 평가자를 같은 손에 쥐지 않는다

중요한 작업에서는 생성자와 평가자를 분리해야 합니다. 이 원칙은 코드에서도 문서에서도 같습니다.

에이전트는 자기 결과를 "완료"라고 말하는 데 능숙합니다. 보고는 그럴듯하고, 설명도 매끈합니다. 하지만 그 결과가 정말 요구사항을 만족하는지는 다른 관점에서 봐야 합니다. 자기 계획, 자기 구현, 자기 리뷰, 자기 승인으로 이어지는 구조는 약합니다.

그래서 `GAN-Style` 패턴이 나옵니다. 한쪽은 만들고, 다른 한쪽은 루브릭으로 평가합니다. 평가자는 감상평을 쓰는 사람이 아닙니다. 다시 작업 가능한 피드백을 줘야 합니다. 예를 들어 "별로다"가 아니라 "모바일에서 버튼이 겹친다", "스펙의 필드명이 맞지 않는다", "빈 케이스 테스트가 없다", "권한 변경이 승인 없이 지나갔다"처럼 수정 가능한 문장이어야 합니다.

이 발표를 만드는 과정에서도 같은 원칙을 썼습니다. source를 정리하는 손, prose를 쓰는 손, outline을 잡는 손, HTML을 생성하는 손, 검증하는 손을 같은 책임으로 묶지 않았습니다. 중요한 것은 에이전트가 여러 명이었다는 사실이 아니라, 생성과 검증이 한 손에 갇히지 않았다는 사실입니다.

## Agent Team은 토론이 필요할 때만 쓴다

`Sub-Agent`와 `Agent Team`은 다릅니다. `Sub-Agent`는 메인이 작업을 던지고 결과를 받는 구조입니다. 대부분의 파일 탐색, 로그 분석, 단일 검증은 이것으로 충분합니다.

`Agent Team`은 여러 독립 세션이 서로 대화하며 수렴하는 구조입니다. 연구, 전략 문서, 아키텍처 검토, 제품 기획처럼 관점의 충돌과 조정이 결과 품질을 좌우할 때 의미가 있습니다. `Researcher`, `Architect`, `Reviewer`, `Editor`, `Coordinator`처럼 각자 다른 관점을 두고 토론하게 만들 수 있습니다.

하지만 팀이라고 늘 좋은 것은 아닙니다. 에이전트 수가 늘면 대기 시간과 통합 비용도 늘어납니다. 세 명 안팎에서는 관점 분리가 이득을 줄 수 있지만, 너무 커지면 회의만 길어지고 결과는 흐려집니다.

그래서 기본값은 작아야 합니다. 단순한 심부름은 `Sub-Agent`로 충분합니다. 양방향 토론이 필요한 경우에만 `Agent Team`을 씁니다. 멀티 에이전트의 성숙도는 얼마나 많이 띄웠는지가 아니라, 언제 띄우지 않을지 아는 데서 드러납니다.

## 패턴보다 경계가 중요하다

패턴 이름은 도구일 뿐입니다. 중요한 것은 경계입니다.

각 에이전트는 자기 일에 필요한 것만 받아야 합니다. 역할과 금지 행동을 먼저 알아야 합니다. 출력 형식은 나중에 사람이 읽기 좋게만 만들 것이 아니라, 다음 단계가 다시 사용할 수 있게 구조화되어야 합니다. 생성자는 평가자와 분리되어야 합니다. 재시도 횟수와 중단 조건도 있어야 합니다.

이 관점에서 `SOLID` 같은 오래된 원칙도 다시 읽을 수 있습니다. 단일 책임은 클래스만의 문제가 아닙니다. 에이전트와 도구도 하나의 책임을 가져야 합니다. 인터페이스 분리는 메서드만의 문제가 아닙니다. 에이전트에게 불필요한 도구와 파일 권한을 노출하지 않는 일입니다. 의존성 역전은 코드 구조만의 문제가 아닙니다. 하드코딩된 프롬프트에 기대는 대신, 컨텍스트와 규칙과 검증을 파일로 밖에 두는 일입니다.

새로운 이름을 붙였을 뿐, 원리는 낯설지 않습니다. 좋은 소프트웨어 설계가 책임을 나누고 결합을 줄이듯, 좋은 하네스 설계도 역할과 컨텍스트와 검증을 나눕니다.

## 정리

하나의 AI에게 모든 것을 맡기면 처음에는 빠릅니다. 하지만 작업이 길어질수록 컨텍스트는 흐려지고, 역할은 섞이고, 자기 검증의 blind spot은 커집니다.

멀티 에이전트의 핵심은 더 많은 AI가 아닙니다. 더 좁은 역할의 AI 여럿입니다. 컨텍스트를 쪼개고, 역할을 쪼개고, 검증을 쪼개야 합니다. 메인 세션은 모든 중간 기억을 떠안는 곳이 아니라, 결정과 통합과 승인만 남기는 곳이어야 합니다.

다음 장에서는 이 원칙을 실제 작업 환경에 어떻게 고정할지 보겠습니다. 결국 하네스는 말로 아는 원칙이 아니라 파일, 명령어, 게이트, 이슈, worktree로 남아야 합니다.
