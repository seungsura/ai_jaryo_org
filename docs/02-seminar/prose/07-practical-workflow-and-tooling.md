# 실전 워크플로우와 도구 세팅

## 00. 이 섹션의 주제는 도구가 아니라 운영 구조다

이 섹션의 핵심은 역할의 이동이다. 이제 중요한 사람은 “도구를 많이 아는 사람”이 아니라 “에이전트가 일할 구조를 설계하는 사람”이다. 따라서 실전 세팅의 출발점도 “무엇을 더 붙일까”보다 “무엇을 어떤 층에 둘까”에 가깝다. 원리를 이해하지 않은 채 `Skill`과 `plugin`만 가져다 쓰면 이상 동작이 생겨도 원인을 설명할 수 없고, 모든 것을 직접 세팅하려 들면 결국 만든 사람만 이해하는 사적 시스템이 되기 쉽다.

## 01. 좋은 workflow는 도구를 더하는 기술보다 도구를 줄이는 기술이다

`prompt-context-harness-1-15.md`와 `claude-code-mastery-playbook.md`가 공통으로 강조하는 것도 같은 지점이다. 에이전트는 도구가 많을수록 똑똑해지는 것이 아니라, 현재 과업과 무관한 도구 정의와 유휴 `MCP` 서버가 쌓일수록 컨텍스트 예산을 낭비하고 판단 품질이 흔들린다. 좋은 운영은 도구를 늘리는 기술보다, 작업별로 필요한 인터페이스만 남기는 tool curation의 기술에 더 가깝다.

이 원칙은 `OpenCode`의 agent docs처럼 permission을 명령 단위로 조정하는 구조에서도 확인된다. `plan` 에이전트를 읽기와 분석 중심으로 제한하고, 다른 agent에만 더 넓은 권한을 주는 방식은 단순한 편의 기능이 아니라 tool hygiene 자체를 하네스에 박아 넣는 방식이다.

## 02. 계획과 실행을 분리해야 blind spot이 줄어든다

발표는 에이전트가 가장 자주 실패하는 지점을 “계획 없이 바로 쓰기 시작하는 순간”으로 본다. 그래서 먼저 읽기 중심의 계획 단계에서 문제를 구조화하고, 그다음 다른 눈으로 계획을 비평한 뒤, 마지막에만 쓰기 권한을 열어 실행하는 순서를 강조한다. 세미나의 `Plan-Critic-Build`나 `Claude Code`의 `Plan Mode`, `OpenCode`의 `plan` agent가 모두 같은 원리를 구현한다. 핵심은 Planner와 Critic을 같은 컨텍스트에 두지 않는 것이다. 자기 계획을 자기 비평하게 하면 blind spot이 그대로 남기 쉽기 때문이다.

## 03. 검증 루프와 권한 제어는 workflow의 일부여야 한다

실전에서 계획 분리는 검증 루프와 붙어야 의미가 있다. AI의 생성은 확률적이지만, `lint`, `type check`, `test`, 승인 게이트는 결정론적이어야 한다. 즉 에이전트는 초안을 만들고 수정안을 제안할 수는 있지만, 기계적으로 검증 가능한 규칙을 건너뛸 수는 없다. 발표가 검증 루프를 별도 command나 단계로 고정해 보여 주는 이유도 여기에 있다. 중요한 것은 “에이전트가 잘하겠지”가 아니라, 실패를 빨리 드러내고 다시 태울 수 있는 구조를 만들어 두는 일이다.

이때 세션 수준의 제어 장치도 중요하다. `Approval`, `Auto-accept`, `Plan Mode`, sandbox 같은 장치는 단순 편의 기능이 아니라 자율성의 범위를 조절하는 runtime control이다. Anthropic이 2025년 10월 20일 공개한 sandboxing 글 역시 permission prompts를 줄이되 filesystem과 network를 분리해 더 안전한 자율성을 확보하는 방향을 설명한다. 자율성과 안전은 대립하는 것이 아니라, 더 좋은 경계 설계를 통해 함께 올라가는 값으로 읽는 편이 정확하다.

또 하나의 실전 원칙은 단일 모델의 판단을 절대화하지 않는 것이다. 구조나 아키텍처처럼 되돌리기 어려운 결정은 `Codex` 같은 다른 모델에게 교차 리뷰를 요청하고, 레이아웃이나 가독성처럼 다른 시각이 유익한 질문은 또 다른 모델에 물어 합의와 불일치를 신호로 읽는 편이 낫다. 중요한 것은 한 모델을 더 오래 설득하는 일이 아니라, 서로 다른 blind spot을 검증 장치로 활용하는 일이다.

## 04. 장기 작업은 세션이 아니라 외부 아티팩트에 상태를 남겨야 한다

이런 논리를 실제 작업 환경에 옮기면 다중 세션, `git worktree`, 이슈 기반 운영이 등장한다. 메인 작업 창은 현재 이슈의 구현과 수정, 즉 핵심 판단을 담당하고, 조사 창은 코드베이스 탐색과 문서 검색 같은 읽기 중심 작업을 맡는다. 감시 창은 테스트와 빌드 로그를 추적하고, 별도 창은 이슈와 `PR` 흐름을 관리한다. 여기서 중요한 것은 병렬성 자체가 아니라, 메인 컨텍스트를 오염시키지 않는 것이다.

`git worktree`가 반복해서 등장하는 이유도 같다. 세션마다 독립된 작업 디렉토리를 두면 파일 충돌을 줄일 수 있고, 브랜치와 컨텍스트가 자연스럽게 분리된다. 한 세션이 한 이슈를 담당하고, 한 이슈가 한 브랜치와 연결되며, 그 단위로 계획, 구현, 검증, `PR`이 이어지면 컨텍스트 드리프트를 구조적으로 줄일 수 있기 때문이다. 결국 좋은 workflow는 상태를 대화 안에만 남기지 않고, 규칙과 상태를 파일과 이슈와 검증 로그에 적층하는 구조다.

## 05. 결론: workflow는 Harness를 일상 운영으로 번역한 것이다

정리하면 이 섹션의 메시지는 간단하다. 실전에서 중요한 것은 더 강한 모델 하나를 붙잡는 일이 아니라, 계획과 실행을 분리하고, 검증을 자동 순환시키고, 도구를 줄이고, 세션을 물리적으로 분리하고, 이슈 단위로 상태를 외부화하는 구조를 만드는 일이다. 좋은 워크플로우는 에이전트를 더 자유롭게 풀어놓는 체계가 아니라, `Context`, 권한, 검증, 상태를 설계하는 하네스를 일상 운영으로 번역한 체계다.
