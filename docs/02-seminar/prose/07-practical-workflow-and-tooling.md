# 실전 워크플로우와 도구 세팅

## 시작하며: 원리를 잊어도 버티는 구조가 필요하다

실전에서 먼저 피해야 할 막다른 길은 둘입니다. 하나는 모든 것을 직접 세팅하느라 구조보다 손기술에 매달리는 길입니다. 다른 하나는 원리도 모른 채 남이 만든 `Skills`와 플러그인만 주워 붙이는 길입니다. 전자는 유지되지 않고, 후자는 설명되지 않습니다. 이상 동작이 나와도 왜 그런지 읽지 못하고, 세션이 길어지면 기준은 흐려지고, 팀에는 방식이 아니라 블랙박스만 남습니다.

그래서 좋은 워크플로우의 목표는 새 도구를 더 붙이는 일이 아닙니다. 원리를 시스템 기본값으로 만드는 일입니다. 사람이 매번 “이번에는 계획부터 보고, 이번에는 테스트를 돌리고, 이번에는 상태를 파일에 남겨야지”라고 다짐하는 방식으로는 오래 버티지 못합니다. 좋은 구조는 사람이 잠시 잊어도, command와 gate와 파일이 그 원칙을 먼저 강제합니다.

앞 장에서 본 멀티 에이전트 패턴이 역할 분해의 설계도였다면, 이번 장은 그 설계도를 운영 문법으로 굳히는 이야기입니다. 무엇을 command로 만들고, 무엇을 우회할 수 없는 문으로 고정하고, 무엇을 세션 밖으로 빼고, 무엇을 물리적으로 격리할지를 다룹니다.

## 실전 워크플로우는 네 동사로 굴러간다: `Write`, `Select`, `Compress`, `Isolate`

긴 작업을 버티는 workflow는 의외로 소박한 네 동사로 요약됩니다. 규칙과 기준을 밖에 써 두는 `Write`, 지금 단계에 필요한 도구와 정보만 고르는 `Select`, 세션 안의 상태를 바깥 아티팩트로 눌러 담는 `Compress`, 그리고 중간 노동과 위험한 실행을 분리하는 `Isolate`입니다. 거창한 프레임워크보다 먼저 이 네 가지가 서야 합니다.

이 네 동사는 결국 같은 문제를 겨냥합니다. 메인 컨텍스트를 보호하는 일입니다. 작업이 길어질수록 세션 안에는 오래된 지시, 끝난 시도, 중복 로그, 잠정 가정이 눌어붙습니다. 그 상태에서 더 많은 도구와 더 많은 기억을 밀어 넣으면 생산성이 아니라 혼탁함이 커집니다. 그래서 workflow는 기억을 더 쌓는 기술이 아니라, 무엇을 밖에 쓰고 무엇을 안으로 덜 들일지 정하는 기술이 됩니다.

이제부터 나오는 command, gate, `worktree`, 이슈 시스템은 전부 이 네 동사의 구현물입니다. 이름은 달라도 목적은 같습니다. 메인 세션이 판단과 통합에 집중하게 만들고, 탐색과 실패 흔적과 반복 규칙은 다른 층에 가두는 것.

## command가 먼저이고 tool은 그 뒤다

`OMC(Oh My Claude Code)` 같은 preset pack을 사례로 드는 이유는 툴 이름을 소개하려는 게 아닙니다. 중요한 것은 사람이 매번 원리를 다시 조립하지 않아도, command와 routing이 먼저 역할과 순서를 강제한다는 점입니다. 여기서는 역할 분리된 `29+` 에이전트, 모델 라우팅, `Plan-Critic-Build` 명령 `/ralplan`, 검증 루프 `/ultraqa`, 최대 `5` worker 병렬, `worktree` 기반 병렬 운영이 한 묶음으로 등장합니다.

핵심은 숫자나 고유명사가 아닙니다. 사용자가 작업의 성격만 말해도 시스템이 먼저 모드와 책임과 검증 순서를 잡아 준다는 사실입니다. 이 차이는 생각보다 큽니다. 어떤 날은 계획부터 보고, 어떤 날은 곧바로 실행하고, 검증은 바쁠 때만 생략하는 식으로 흐르면 workflow는 남지 않고 기분만 남습니다. 반대로 원리가 command로 내려가면 좋은 습관은 기억이 아니라 기본값이 됩니다.

이런 preset pack이 보여 주는 `30~50%↓` 같은 수치도 같은 방식으로 읽어야 합니다. 중요한 것은 숫자의 과시가 아닙니다. 역할과 모델과 검증을 먼저 구조화했을 때 토큰과 판단 비용이 함께 줄어드는 방향을 보여 준다는 점입니다. 결국 도구의 힘은 기능 목록보다 원리를 얼마나 먼저 강제하느냐에서 나옵니다.

## 구현 전에 계획을 떼어 놓는다: `Plan-Critic-Build`

실전에서 가장 먼저 고정해야 할 문법은 구현 전에 계획을 떼어 놓는 일입니다. `Plan-Critic-Build`라는 이름은 중요하지 않습니다. 순서가 중요합니다. 먼저 계획을 세우고, 그 계획을 비평하고, 합의가 끝난 뒤에만 실행으로 넘어갑니다. `/ralplan`이 상징하는 것도 바로 이 분리입니다.

이 원칙은 익숙한 `Explore -> Plan -> Code -> Commit` 흐름과도 이어집니다. 먼저 저장소 구조와 스타일을 읽고, 그 다음 계획을 제안하고, 그 계획을 기준으로 구현하고, 마지막에 리뷰와 검증을 거칩니다. 여기서 빠지면 안 되는 것이 `success criteria`입니다. 무엇을 끝으로 볼 것인지, 어떤 테스트와 어떤 검증을 통과해야 하는지 먼저 못 박아야 합니다. 그래야 실행이 속도 경쟁이 아니라 기준 수렴이 됩니다.

계획을 따로 세우는 이유는 단순합니다. 코드 수정은 되돌릴 수 있어도, 부실한 전제 위에서 시작한 실행은 로그와 상태와 후속 판단까지 함께 오염시킵니다. 긴 작업일수록 이 초기 오염의 비용이 훨씬 큽니다. 그래서 좋은 workflow는 구현 능력을 먼저 키우지 않습니다. 구현 전에 멈출 수 있는 구조부터 만듭니다.

## 좋은 워크플로우는 pass/fail보다 gate와 `Observability`를 먼저 세운다

`lint`, `type check`, 테스트, 리뷰, 승인, `Sandbox`, `Hooks`는 편의 기능이 아닙니다. 우회할 수 없는 문입니다. 여기서 중요한 것은 에이전트에게 재량을 더 주는 일이 아니라, 재량을 줄여야 하는 구간을 분명히 고정하는 일입니다. `Approval`, `Auto-accept`, `Plan Mode`를 위험도에 맞춰 조절하고, `PreToolUse`와 `PostToolUse` 같은 `Hooks`로 파괴적 명령과 품질 검증을 강제하는 이유도 여기 있습니다.

하지만 gate만으로는 충분하지 않습니다. 성공과 실패만 찍는 구조는 긴 작업에서 너무 많은 것을 놓칩니다. 실전에는 `Observability`가 필요합니다. 어떤 command가 실행됐는지, 어떤 테스트가 통과했는지, 어디서 대기 상태에 들어갔는지, 어떤 로그가 남았는지, 왜 재시도가 일어났는지를 봐야 합니다. pass/fail은 결과를 말하지만, `Observability`는 경로를 드러냅니다. 경로가 보이지 않으면 drift의 기원을 추적할 수 없습니다.

이 구도는 결국 결정론적 제어와 확률적 제어의 분리입니다. 기계가 확인할 수 있는 것은 기계에게 맡깁니다. AI는 설계 대안, 초안, 우선순위 판단 같은 확률적 영역에 집중합니다. 둘을 섞으면 기계는 애매해지고 AI는 방종해집니다. 좋은 workflow는 이 둘을 한 시스템 안에 넣되, 같은 층에 두지는 않습니다.

## 컨텍스트를 덜어내고 반복 규칙을 자산으로 올린다

긴 세션이 무너지는 이유는 정보가 부족해서가 아닙니다. 오래 쌓인 기억이 점점 잡음으로 바뀌기 때문입니다. 그래서 `/context`, `/compact`, `/clear`는 편의 명령이 아니라 운영 명령입니다. 지금 무엇이 얼마나 쌓였는지 보고, 이어갈 일만 압축해 남기고, 다른 문제로 넘어갈 때는 과감히 지워야 합니다. 세션을 비우지 못하면 작업은 이어지는 것이 아니라 끌려갑니다.

반복되는 지침도 대화 안에 묻어 두면 안 됩니다. 매번 같은 설명을 타이핑하고 있다면 이미 세션 바깥으로 나가 있어야 할 지식이라는 뜻입니다. 반복 지시는 `Skills`로 올리고, 우회하면 안 되는 규칙은 `Hooks`로 고정하고, 자주 쓰는 루프는 `Custom Commands`로 묶어야 합니다. 개인 메모에 머물면 요령으로 끝나지만, 저장소와 팀 설정으로 올라가면 표준이 됩니다.

이것이 상태 외부화의 실제 의미입니다. 세션 안의 기억을 믿지 말고, 파일 시스템과 `Git` 이력과 command 체계로 분산 저장하라는 뜻입니다. 세션은 뜨겁지만 휘발하고, 아티팩트는 느리지만 남습니다. 긴 작업을 버티게 하는 쪽은 늘 후자입니다.

## core workflow가 선 뒤에 시야를 교차시킨다

여기까지의 문법이 먼저 서고 나면, 그 다음에야 시야 분산이 의미를 가집니다. 단일 모델에 모든 판단을 몰아주지 않고 `Claude`, `Codex`, `Gemini`를 교차 투입하는 이유는 어느 모델이 더 우월해서가 아닙니다. 훈련 분포와 `blind spot`이 다르기 때문입니다. 한 모델의 자신감은 자주 유창함에 가깝고, 여러 모델의 불일치는 더 자주 경고 신호에 가깝습니다.

그래서 `/ask codex`, `/ask gemini` 같은 교차 질의는 부가 장식이 아니라 실전적인 안전장치가 됩니다. `Codex`는 되돌리기 어려운 구조와 아키텍처 결정을 다시 읽게 하고, `Gemini`는 레이아웃과 가독성처럼 다른 시각의 피드백을 보탭니다. 이때 중요한 것은 다수결이 아닙니다. `합의`, `불일치`, `불확실`을 분리하는 일입니다. 그래야 무엇이 이미 안정됐고 무엇이 다시 검증돼야 하는지 해상도가 올라갑니다.

대부분의 경우 이것은 `Agent Teams`라기보다 독립 평가자 풀에 가깝습니다. 굳이 팀 토론까지 가지 않아도, 서로 다른 눈으로 같은 결과를 읽게 하는 것만으로도 self-review의 착시는 크게 줄어듭니다.

## 메인 컨텍스트를 지키는 물리적 운영: `cmux`와 `Git Worktree`

논리적 경계를 세웠다면, 이제 그 경계를 작업 환경에 새겨야 합니다. 하나의 창에 모든 탐색, 구현, 로그, 이슈 관리, `PR` 확인을 몰아넣으면 메인 컨텍스트는 금방 흐려집니다. 반대로 메인 작업, 조사와 탐색, 감시, 이슈와 `PR` 관리를 분리하면 탐색은 옆 창에서 소모되고 결과만 메인으로 돌아옵니다. `cmux`가 유용한 이유가 바로 여기에 있습니다.

`Git Worktree`는 이 분리를 파일 시스템 수준에서 고정합니다. 창마다 독립 작업 디렉터리를 두면 충돌 없이 여러 세션을 병렬로 굴릴 수 있고, 한쪽의 중간 상태가 다른 쪽의 기준을 더럽히지 않습니다. 여기서도 포인트는 병렬성 자체가 아닙니다. 탐색은 탐색 자리에서 끝내고, 메인 세션은 결정과 통합과 검증에 집중하게 만드는 경계입니다.

실전 workflow가 멀티태스킹의 묘기로 보일 때가 많지만, 실제 핵심은 그 반대입니다. 메인 컨텍스트를 얼마나 덜 망가뜨리느냐입니다. 물리적 분리는 생산성 기술이기 전에 컨텍스트 방어 기술입니다.

## 세션이 아니라 이슈가 상태를 들고 간다

긴 작업에서 가장 위험한 착각 하나가 있습니다. “지금 이 대화창이 모든 상태를 기억하고 있다”는 믿음입니다. 실제로는 그 반대입니다. 채팅은 빨리 달아오르지만 빨리 휘발합니다. 그래서 상태 저장소는 세션 안이 아니라 세션 밖에 있어야 합니다. 이슈 시스템이 중요한 이유도 바로 여기 있습니다.

`이슈 1개 = 에이전트 세션 1개` 같은 규칙은 형식주의가 아닙니다. 이슈가 작업의 목적과 기준과 증거를 붙들고, `Worktree` 생성과 세션 시작, `PR` 생성과 검증 완료가 한 줄로 이어지게 만듭니다. 이슈 생성, 증거 첨부, 검증, 완료 판정의 체인은 에이전트를 더 똑똑하게 만드는 장치가 아닙니다. 완료 기준과 흔적을 바깥에 남겨 drift를 막는 장치입니다.

상태를 세션 안에만 쌓아 두면 창이 닫히는 순간 기준도 함께 사라집니다. 반대로 이슈와 `PR`, 파일, `Git` 이력에 남겨 두면 세션이 바뀌어도 작업은 이어집니다. 실전에서 신뢰할 것은 대화의 열기보다 바깥 아티팩트의 지속성입니다.

## 결론: 좋은 workflow는 생성 능력이 아니라 운영 구조를 설계한다

이 장의 결론은 분명합니다. 실전 워크플로우의 중심은 더 많은 도구를 붙이는 데 있지 않습니다. command로 원리를 고정하고, 계획과 실행을 분리하고, gate와 `Observability`를 세우고, 상태를 세션 밖으로 밀어내고, 모델의 시야를 교차시키고, `cmux`와 `Git Worktree`로 경계를 물리적으로 고정하고, 이슈와 `PR`에 기준을 남기는 데 있습니다.

에이전트가 강해질수록 더 중요한 것은 생성 능력 자체가 아닙니다. 무엇을 어디서 검증할지, 어떤 흔적 위에 다음 판단을 올릴지, 메인 컨텍스트를 어떻게 보호할지를 설계하는 일입니다. 결국 좋은 workflow는 사람의 기억력을 돕는 장치가 아닙니다. 사람이 잠시 잊어도 원리가 먼저 작동하게 만드는 구조입니다.
