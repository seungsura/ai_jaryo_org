## 오늘의 목적지

### 슬라이드에 적을 내용

- `챗봇과 싸우지 않고`
- `AI와 함께 일하는 방법`
- `그것이 Harness`
- `이 발표 자료 또한 전부 Harness로 만들었습니다`

### 이 슬라이드에서 할 말

오늘 이야기할 것은 AI에게 일을 통째로 맡기는 법이 아닙니다. 챗봇에게 더 길게 말하고, 더 세게 설득하고, 원하는 답이 나올 때까지 싸우는 법도 아닙니다. 키보드 앞에서 모든 것을 직접 붙들고 버티는 방식도 아닙니다.

제가 말하고 싶은 것은 AI가 일할 환경을 설계하고, 그 안에서 함께 일하는 방법입니다. 이 발표 자료도 마찬가지입니다. 손으로 한 장씩 밀어붙여 만든 발표가 아닙니다. source를 정하고, 규칙을 세우고, 검증을 분리하고, 그 안에서 AI가 일하게 만든 결과입니다.

그래서 오늘 말할 Harness는 추상적인 방법론이 아닙니다. 이 발표를 만드는 데도 실제로 쓴 작업 방식입니다.

## 00-3. 추상화의 역사: 직접 하던 일은 계속 줄어들었다

### 00-3-1. 기계어에서 어셈블리, 그리고 C로

#### 슬라이드에 적을 내용

```jsx
10111000 00000001 00000000 00000000 00000000
// 기계어
```

```jsx
MOV EAX, 1
// 어셈블리
```

```jsx
printf("Hello\n");
// 고급 언어 (C, Pascal)
```

- `컴파일러가 사람보다 효율적일 리 없다`
- `진짜 프로그래머는 Pascal 같은 걸 쓰지 않는다`

#### 이 슬라이드에서 할 말

개발의 역사는 직접 하던 일을 더 높은 수준의 표현으로 넘겨 온 역사입니다. 기계어를 쓰다 어셈블리로, 다시 C와 Pascal로 넘어갈 때마다 반발은 늘 같았습니다.

컴파일러를 믿지 못하는 불안도 있었지만, 핵심은 자부심이었습니다. 하드웨어를 직접 제어해야 '진짜 개발자'라는 감각 때문입니다.

하지만 추상화는 개발자를 약하게 만들지 않았습니다. 비트와 레지스터 대신 더 큰 논리를 다루게 했습니다. 직접 통제하는 단위를 높여 거대한 시스템을 짓게 만들었을 뿐입니다.

Dijkstra는 1972년 Turing Lecture에서 추상화의 목적은 모호해지는 것이 아니라 "a new semantic level"을 만들어 더 정확해지는 데 있다고 말했습니다.[^00-dijkstra] 우리가 기계어에서 C로 올라온 것도 정확성을 포기한 것이 아니라, 정확하게 다룰 수 있는 단위를 바꾼 일이었습니다.

### 00-3-3. C에서 Java로

#### 슬라이드에 적을 내용

```jsx
Point* p = (Point*)malloc(sizeof(Point));
p->x = 10;
p->y = 20;
printf("Point: %d, %d\n", p->x, p->y);
free(p);
```

```jsx
Point p = new Point(10, 20);
System.out.println("Point: " + p.x + ", " + p.y);
p = null;
```

- `메모리를 직접 관리하지 않으면 진짜 개발자가 아니다`

#### 이 슬라이드에서 할 말

수동 메모리 관리에서 `GC`로 넘어갈 때도 같은 말이 나왔습니다. 포인터와 메모리를 직접 다룰 줄 알아야 진짜 개발자라는 반발이 있었습니다.

하지만 `GC`는 개발자를 게으르게 만든 것이 아니라, 해제 누락과 이중 해제 같은 오류를 줄이고 더 높은 수준의 설계에 집중할 수 있게 했습니다. 직접 관리하던 일이 런타임으로 넘어간 것입니다.

### 00-3-4. Java에서 Python으로

#### 슬라이드에 적을 내용

```jsx
public class HelloWorld {
    public static void main(String[] args) {
        System.out.println("Hello, World!");
    }
}
```

```jsx
print("Hello, World!")
```

- `장난감 언어로 무슨 개발을 하냐`

#### 이 슬라이드에서 할 말

Python도 처음부터 지금 같은 지위를 가진 언어가 아니었습니다. 정적 타입 없이 인터프리터로 동작하는 언어는 실무에 쓰기 어려운 스크립트처럼 취급받았습니다.

그런데 오늘날 Python은 AI와 데이터 분야의 사실상 표준 언어가 되었습니다. 추상화는 처음에는 가벼워 보입니다. 시간이 지나면 그 가벼움이 더 큰 일을 가능하게 만들기도 합니다.

### 00-3-5. 그리고 AI 개발

#### 슬라이드에 적을 내용

```jsx
이 함수를 리팩터링하고 테스트 코드를 작성해줘.
```

- `자연어로 시키는 건 진짜 개발이 아니다`

#### 이 슬라이드에서 할 말

지금 AI 개발을 향한 반응도 이 흐름에서 크게 벗어나지 않습니다. 자연어로 코드를 만들고, 에이전트가 파일을 수정하고, 테스트를 돌리는 모습을 보면 불편함이 생깁니다. 직접 하지 않으면 진짜 개발이 아니라는 감각이 다시 올라옵니다.

하지만 이번에도 핵심은 개발이 사라진다는 말이 아닙니다. 개발의 추상화 수준이 올라간다는 말입니다.

### 00-3-6. 위임의 방향

#### 슬라이드에 적을 내용

| 시대 | 전환 | 위임한 것 |
| --- | --- | --- |
| 1950s | 기계어 -> 초기 고급 언어 | 이진 코드를 직접 작성하는 부담 |
| 1960s | 어셈블리 -> C | 레지스터와 명령어 최적화 |
| 1990s | C/C++ -> Java | 메모리 해제 |
| 2000s | 언어 -> 프레임워크 | 반복되는 보일러플레이트 |
| 2010s | 온프레미스 -> 클라우드 | 서버 운영과 확장 |
| 2020s | 직접 코딩 -> AI 에이전트 | 코드 작성의 상당 부분 |

`직접 하는 일은 줄고, 설계하는 일은 늘어난다`

#### 이 슬라이드에서 할 말

매번 같은 일이 벌어졌습니다. 직접 하던 일은 줄었고, 더 큰 구조를 설계하는 일이 실력의 중심으로 올라왔습니다.

AI agent도 이 흐름 위에 있습니다. 사람이 덜 타이핑할 수는 있습니다. 하지만 무엇을 만들지, 어떤 기준으로 끝났다고 볼지, 어디서 멈춰야 하는지 판단해야 합니다.

이제 중요한 질문은 `코드를 직접 쓰느냐`가 아닙니다. `AI에게 무엇을 위임하고, 무엇을 사람이 붙들 것인가`입니다.

## 00-4. 숫자로 보는 변화

### 슬라이드에 적을 내용

- `25%`
- `95% 이상 AI codebase`
- `5개월`
- `100만 줄`
- `3개월 이상`

### 이 슬라이드에서 할 말

이 변화는 분위기나 예측만의 이야기가 아닙니다.

YC W25, 그러니까 Y Combinator 2025 참가팀 중 약 25%의 스타트업이 코드베이스의 95% 이상을 AI로 만들었다는 공개 보도가 나왔습니다.[^00-yc] OpenAI Codex 팀은 5개월 동안 수동 코드를 한 줄도 직접 작성하지 않고 100만 줄 규모의 프로덕트를 완성했습니다.[^00-openai-harness] Meta 엔지니어는 직접 코드를 타이핑하지 않은 지 3개월이 넘었다고 밝혔습니다.[^00-kakao-meta]

다만 이 숫자들은 성격이 다릅니다. YC와 OpenAI는 공개 출처로 확인되는 사례이고, 3개월 사례는 카카오 발표 원문에서 가져온 내부 관찰에 가까운 신호입니다. 그래서 특정 회사의 숫자를 일반화하면 안 됩니다. 그래도 흐름은 분명합니다. 이 숫자는 위기감을 만들기 위한 장치이기도 합니다. 하지만 겁을 주기 위한 숫자로만 쓰면 금방 약해집니다. 이 숫자를 `사람이 필요 없어졌다`는 근거로 읽으면 안 됩니다. 오히려 반대입니다.

생성은 빨라졌고, 그래서 방향 설정과 검증과 환경 설계가 더 중요해졌습니다. 숫자가 보여주는 것은 개발자의 소멸이 아니라 역할의 이동입니다.

## 00-5. 문서가 코드다

### 슬라이드에 적을 내용

- `README가 아니라 실행 규칙`
- `CLAUDE.md`
- `AGENTS.md`
- `Cursor Rules`
- `copilot-instructions.md`

### 이 슬라이드에서 할 말

AI와 함께 일하는 환경에서는 문서의 역할도 바뀝니다.

예전 문서는 사람이 읽는 참고 자료에 가까웠습니다. README, 컨벤션 문서, 아키텍처 노트는 중요했지만, 실제 실행 경로와는 떨어져 있는 경우가 많았습니다.

AI coding agent 환경에서는 이야기가 달라집니다. `CLAUDE.md`, `AGENTS.md`, `Cursor Rules`, `copilot-instructions.md`는 이름은 다르지만 역할은 비슷합니다. 에이전트가 매 세션 읽고 따라야 할 운영 규칙입니다.

코딩 스타일, 금지 명령, 테스트 방법, 리뷰 기준, 문서 어투, 승인 조건이 파일 안에 들어갑니다. 문서가 실행 경로 바깥의 설명이 아니라, AI가 일하는 환경의 일부가 됩니다.

이 지점이 Harness로 이어지는 첫 번째 다리입니다.

## 00-6. 개발자의 역할은 환경 설계로 이동한다

### 슬라이드에 적을 내용

- `타이핑의 비중은 줄어든다`
- `판단의 밀도는 올라간다`
- `환경을 설계하는 개발자`

### 이 슬라이드에서 할 말

AI가 더 많은 코드를 써 줄수록 개발자의 기본기는 덜 중요해지는 것처럼 보입니다. 실제로는 반대입니다.

사람이 덜 타이핑할 수는 있습니다. 하지만 무엇을 만들지, 어떤 기준으로 끝났다고 볼지, 어떤 결과를 의심해야 할지 판단해야 합니다. AI가 만든 결과가 안전한지, 유지보수 가능한지, 기존 시스템과 잘 맞는지 판단하려면 여전히 시스템을 이해하는 힘이 필요합니다.

줄어든 것은 타이핑의 비중이지, 시스템을 이해하는 능력의 가치가 아닙니다.

OpenAI의 Harness engineering 글은 이 전환을 아주 짧게 요약합니다. "Humans steer. Agents execute."[^00-openai-harness] 사람이 사라진다는 말이 아니라, 사람이 잡아야 할 손잡이가 타이핑에서 방향과 피드백 루프로 옮겨간다는 말입니다.

그래서 질문을 바꿔야 합니다. `코딩은 사라지는가`가 아닙니다. `AI와 어떻게 함께 일할 것인가`입니다. 여기서 다음 챕터, `챗봇과 싸우지 않기`로 넘어갑니다.
