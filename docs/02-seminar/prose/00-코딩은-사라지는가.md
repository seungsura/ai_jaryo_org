## 00-3. 00. 코딩은 사라지는가

### 00-3-1. 기계어에서 어셈블리,C/Pascal (고급언어)로

#### 슬라이드에 적을 내용

```jsx
10111000 00000001 00000000 00000000 00000000
// 기계어
```

```jsx
MOV EAX, 1
// 어셈블리
```

```jsx
printf("Hello\n");
// 고급 언어 (C, Pascal)
```

- `컴파일러가 사람보다 효율적일 리 없다`
- `진짜 프로그래머는 Pascal 같은 걸 쓰지 않는다`

#### 이 슬라이드에서 할 말

처음 기계어에서 어셈블리/C 와 같은 고급언어로 넘어갈 때 이런 반발이 있었다고 합니다. 

“컴파일러가 사람보다 효율적일 리 없다!”

그런데 지금은 컴파일러 안쓰는 언어는 없죠. 모두 고급언어를 사용합니다. 

### 00-3-3. C에서 Java로

#### 슬라이드에 적을 내용

```jsx
Point* p = (Point*)malloc(sizeof(Point));
p->x = 10;
p->y = 20;
printf("Point: %d, %d\n", p->x, p->y);
free(p);
```

```jsx
Point p = new Point(10, 20);
System.out.println("Point: " + p.x + ", " + p.y);
p = null;
```

- `메모리를 직접 관리하지 않으면 진짜 개발자가 아니다`

#### 이 슬라이드에서 할 말

다음에 Java 가 처음 등장했을 때도 부정적인 시각이 있었습니다. “메모리를 직접 관리하지 않으면 진짜 개발자가 아니다” 

그런데 지금 대한민국은 자바 공화국이라고 불릴 정도로  자바의 위상이 높습니다. 

### 00-3-4. Java에서 Python으로

#### 슬라이드에 적을 내용

```jsx
public class HelloWorld {
    public static void main(String[] args) {
        System.out.println("Hello, World!");
    }
}
```

```jsx
print("Hello, World!")
```

- `장난감 언어로 무슨 개발을 하냐`

#### 이 슬라이드에서 할 말

Python도 마찬가지 입니다. 정적 타입 없이 인터프리터로 동작하는 언어는 실무에 쓰기 어려운 장난감언어 취급받았습니다.

그런데 오늘날 Python은 AI와 데이터 분야의 사실상 표준 언어가 되었습니다. 

### 00-3-5. 그리고 AI 개발

#### 슬라이드에 적을 내용


- `자연어로 시키는 건 진짜 개발이 아니다`
- 과연 그럴까?

#### 이 슬라이드에서 할 말

현재 AI 시대에서 이런 말이 있죠. “자연어로 시키는 건 진짜 개발이 아니다” 

### 00-3-6. 추상화의 역사

#### 슬라이드에 적을 내용

| 시대 | 전환 | 위임한 것 |
| --- | --- | --- |
| 1950s | 기계어 -> 초기 고급 언어 | 이진 코드를 직접 작성하는 부담 |
| 1960s | 어셈블리 -> C | 레지스터와 명령어 최적화 |
| 1990s | C/C++ -> Java | 메모리 해제 |
| 2000s | 언어 -> 프레임워크 | 반복되는 보일러플레이트 |
| 2010s | 온프레미스 -> 클라우드 | 서버 운영과 확장 |
| 2020s | 직접 코딩 -> AI 에이전트 | 코드 작성의 상당 부분 |

`직접 하는 일은 줄고, 설계하는 일은 늘어난다`

#### 이 슬라이드에서 할 말

매번 같은 일이 벌어졌습니다. 직접 하던 일은 줄었고, 더 큰 구조를 설계하는 일이 개발 실력의 중심으로 올라왔습니다.

AI 시대도 동일합니다. 역사적으로 추상화의 수준은 점점 올라갔고 사람은 설계하는 일에 더 집중하게 되었습니다. 이제 중요한 것은 `코드를 직접 쓰느냐`가 아닙니다. `AI에게 무엇을 위임하고, 사람은 무엇을 할 것이냐`입니다.

## 00-5. 문서가 코드다

### 슬라이드에 적을 내용

- `README가 아니라 실행 규칙`
- `CLAUDE.md`
- `AGENTS.md`
- `Cursor Rules`
- `copilot-instructions.md`

### 이 슬라이드에서 할 말

AI와 함께 일하는 환경에서는 문서의 역할도 바뀝니다.

예전 문서는 사람이 읽는 참고 자료에 가까웠습니다. README, 컨벤션 문서, 아키텍처 노트는 중요했지만, 실제 실행 경로와는 떨어져 있는 경우가 많았습니다.

AI coding agent 환경에서는 이야기가 달라집니다. `CLAUDE.md`, `AGENTS.md`, `Cursor Rules`, `copilot-instructions.md`는 이름은 다르지만 역할은 비슷합니다. 에이전트가 매 세션 읽고 따라야 할 운영 규칙입니다.

AI 시대에서는 이 문서 자체가 코드다 라고 볼 수 있습니다. 

## 00-6. 개발자의 역할은 환경 설계로 이동한다

### 슬라이드에 적을 내용

- `타이핑의 비중은 줄어든다`
- `판단의 밀도는 올라간다`
- `환경을 설계하는 개발자`
- "Humans steer. Agents execute." - Harness engineering, OpenAI

### 이 슬라이드에서 할 말

AI가 더 많은 코드를 써 줄수록 개발자의 실력이 덜 중요해지는 것처럼 보입니다. 실제로는 반대입니다.

사람이 코드를 덜 타이핑할 수는 있습니다. 하지만 무엇을 만들지, 완료 조건이 무엇인지, 답변을 의심해야 하는지 판단해야 합니다. AI가 만든 결과가 안전한지, 유지보수 가능한지, 기존 시스템과 잘 맞는지 판단하려면 여전히 시스템을 이해하는 힘이 필요합니다. 오히려 시니어의 역할을 수행해야 한다고 봐야 합니다.

줄어든 것은 코드 타이핑의 비중이지, 시스템을 이해하는 능력이 아닙니다.

"Humans steer. Agents execute." - OpenAI의 Harness engineering 리서치

사람이 사라진다는 말이 아니라, 사람은 방향과 피드백을 담당해야한다 라는 말입니다. 

그래서 질문의 결론, `코딩은 사라지는가` 아닙니다.

 `AI와 어떻게 함께 일할 것인가`입니다.
