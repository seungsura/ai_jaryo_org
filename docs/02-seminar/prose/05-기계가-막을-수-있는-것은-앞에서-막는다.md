# 05. 기계가 막을 수 있는 것은 앞에서 막는다

## 05-1. 방향만으로는 부족하다

### 슬라이드에 적을 내용

- `스펙이 있어도 틀릴 수 있다`
- `계획이 좋아도 오해할 수 있다`
- `잘 확인해줘로는 부족하다`

### 이 슬라이드에서 할 말

앞 챕터에서는 방향을 먼저 잡아야 한다고 했습니다. 맞는 말입니다. 하지만 방향을 잡았다는 사실이 결과의 안전을 보장하지는 않습니다.

스펙이 있어도 구현은 틀릴 수 있습니다. 계획이 좋아도 에이전트는 중간에 테스트를 오해하고, 깨진 상태를 정상으로 읽고, 실패한 명령 결과를 그럴듯하게 해석할 수 있습니다.

그래서 AI에게 "잘 확인해줘"라고 말하는 것만으로는 부족합니다. 확인할 수 있는 것은 실제로 확인하게 만들어야 합니다. 검증은 마지막에 붙는 QA가 아닙니다. 에이전트가 넘어설 수 없는 통제선입니다.

Claude Code 문서는 "Give Claude a way to verify its work"를 중요한 실천으로 꼽습니다.[^05-claude-verify] 테스트, 스크린샷, 예상 출력처럼 에이전트가 직접 확인할 수 있는 기준을 줘야 사람이 유일한 피드백 루프가 되지 않습니다.

## 05-2. 말이 아니라 멈춤을 만든다

### 슬라이드에 적을 내용

- `테스트 실패 -> 다음 단계 없음`
- `타입 오류 -> 커밋 없음`
- `위험 명령 -> 승인 앞에서 정지`

### 이 슬라이드에서 할 말

사람에게는 "테스트를 보고 판단해줘"라고 말할 수 있습니다. 하지만 에이전트에게는 말만으로 충분하지 않습니다. 테스트가 실패하면 다음 단계로 못 가게 해야 합니다. 타입 오류가 남아 있으면 커밋할 수 없어야 합니다. 위험한 명령은 실행 전에 멈춰야 합니다.

하네스가 하는 일은 멈출 기준과 통제선을 구조로 고정하는 것입니다.

이것이 우리가 처음부터 붙들고 있는 이야기입니다. AI가 위험한 것이 아니라, 검증 없이 쓰는 방식이 위험합니다. AI가 만든 코드를 믿고 검증 없이 넘기는 장면은 기술 문제가 아니라 작업 환경의 문제입니다.

## 05-3. TDD는 에이전트의 자유를 제한한다

### 슬라이드에 적을 내용

- `실패하는 테스트를 먼저 본다`
- `테스트를 지우지 않는다`
- `assert를 느슨하게 만들지 않는다`
- `구현에 맞춰 테스트를 끼우지 않는다`

### 이 슬라이드에서 할 말

`TDD(Test-Driven Development)`는 AI 시대에 다시 강해집니다. 이유는 단순합니다. 에이전트는 그럴듯하게 동작하는 코드를 너무 자신 있게 만들 수 있기 때문입니다. 테스트가 없으면 틀렸는데도 맞는 것처럼 보이기 쉽습니다.

AI 시대의 `TDD`에서 중요한 것은 AI가 테스트도 알아서 쓰게 하자는 말이 아닙니다. 요구사항의 핵심은 사람이 먼저 잡아야 합니다. 특히 버그 수정, 회귀 방지, 보안 경계, 금전 처리, 권한 처리처럼 틀렸을 때 비용이 큰 영역에서는 사람이 실패하는 테스트를 먼저 만들어야 합니다.

여기서 테스트는 품질 관리 산출물이 아닙니다. 권한 경계입니다. 에이전트가 마음대로 넘어서면 안 되는 선입니다.

## 05-4. 결정론적 제어와 확률적 제어를 분리한다

### 슬라이드에 적을 내용

| 결정론적 제어 | 확률적 제어 |
| --- | --- |
| `lint` | 설계 대안 |
| `type check` | 코드 리뷰 의견 |
| 테스트 | 아키텍처 판단 |
| 위험 명령 차단 | 요약과 우선순위 |

### 이 슬라이드에서 할 말

하네스 설계에서 자주 헷갈리는 지점이 있습니다. AI가 잘할 일과 기계가 확실히 막을 일을 섞어 버리는 것입니다.

AI는 애매한 문제에서 강합니다. 설계 대안을 비교하고, 실패 로그를 읽고, 코드 리뷰 의견을 제안하고, 복잡한 맥락을 요약합니다. 이런 일은 확률적 판단이 필요합니다.

반대로 기계가 더 잘하는 일도 있습니다. 포맷이 맞는지, 타입이 깨졌는지, 테스트가 실패했는지, 금지된 파일을 건드렸는지, 위험 명령을 실행하려는지 같은 일입니다. 이런 것은 AI에게 잘 봐 달라고 부탁할 필요가 없습니다. 실제 명령을 실행하고, 실패하면 멈추면 됩니다.

OpenAI의 agent guide는 guardrail을 "layered defense mechanism"으로 설명합니다.[^05-openai-guardrails] 하나의 장치가 모든 위험을 막는다고 기대하지 않고, 규칙 기반 차단, 모델 기반 판별, 권한, 검증을 겹겹이 놓는다는 뜻입니다.

좋은 하네스는 AI를 믿지 않는 구조가 아닙니다. AI가 잘하는 판단에 집중할 수 있도록, 기계가 막을 수 있는 일을 먼저 치워 주는 구조입니다.

## 05-5. 권한은 작업의 위험도에 맞춰야 한다

### 슬라이드에 적을 내용

| 작업 | 권한 |
| --- | --- |
| 문서 초안, 로그 요약 | 넓은 자율성 |
| 테스트 보강, 작은 리팩터링 | 자동화 + 검증 |
| 아키텍처, DB, 권한 | 계획 + 승인 |
| 삭제, 배포, 결제 | 수동 승인 |

### 이 슬라이드에서 할 말

모든 작업에 같은 자율성을 주면 안 됩니다. 에이전트를 믿을 것인가 말 것인가의 문제가 아닙니다. 어떤 작업에 어느 정도의 자유를 줄 것인가의 문제입니다.

문서 초안, 로그 요약, 작은 표현 정리처럼 실패 비용이 낮고 되돌리기 쉬운 작업은 비교적 넓은 자율성을 줄 수 있습니다. 테스트 보강이나 작은 리팩터링은 제한적인 자동 수락이 가능하지만, 검증 게이트가 같이 있어야 합니다. 아키텍처 변경, DB 마이그레이션, 권한 모델 변경은 계획과 승인이 먼저 필요합니다. 삭제, 배포, 결제, 민감 데이터 접근은 수동 승인을 통과해야 합니다.

`Plan Mode`는 구현을 늦추는 모드가 아니라, 쓰기 전에 전제를 확인하는 모드입니다. `Auto-accept`는 전부 열거나 전부 닫는 스위치가 아니라, 되돌리기 쉬운 작업과 검증 가능한 작업에만 제한적으로 열어야 하는 권한입니다.

## 05-6. 품질을 앞단으로 당기기

### 슬라이드에 적을 내용

- `format`
- `type`
- `test`
- `hook`
- `approval`
- `Keep Quality Left`

### 이 슬라이드에서 할 말

여기서 필요한 원칙이 `품질을 앞단으로 당기기 (Keep Quality Left)`입니다. 품질을 마지막 리뷰에서 한꺼번에 잡으려 하면 너무 늦습니다. 에이전트는 이미 많은 파일을 고쳤고, 그 변경은 서로 얽혀 있고, 사람은 전체 맥락을 다시 읽어야 합니다.

앞에서 막을 수 있는 것은 앞에서 막아야 합니다. 포맷은 formatter가 잡습니다. 타입은 type checker가 잡습니다. 회귀는 테스트가 잡습니다. 위험 명령은 Hook이나 승인 규칙이 잡습니다. 큰 방향은 스펙과 계획이 잡습니다. 사람은 그다음에 남는 비싼 판단을 봅니다.

이 구조는 사람을 빼자는 말이 아닙니다. 오히려 사람의 판단을 더 중요한 곳에 남기는 방식입니다. 사람이 모든 것을 마지막에 검수하는 구조에서는 사소한 오류와 중요한 결정이 같은 무게로 밀려옵니다.

AI가 빠를수록 검증은 뒤로 밀리면 안 됩니다. 더 긴 프롬프트가 아니라 더 이른 게이트. 더 많은 자동 수락이 아니라 위험도에 맞춘 권한. 이 구조가 있어야 긴 작업에서도 결과를 믿을 수 있습니다.
