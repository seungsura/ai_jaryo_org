# 08. 이 발표 자체가 Harness였다

## 08-1. 결과물보다 과정을 봐야 한다

### 슬라이드에 적을 내용

- `00장에서 말했습니다`
- `이 발표 자료 또한 전부 Harness로 만들었습니다`
- `기억하시나요?`

### 이 슬라이드에서 할 말

00장에서 제가 이 말을 했습니다. 이 발표 자료 또한 전부 Harness로 만들었습니다. 기억하시나요?

이제 그 말을 다시 꺼낼 차례입니다. 여기서 중요한 것은 "AI가 만들었다"는 자랑이 아닙니다. 이 발표에서 봐야 할 것은 결과물만이 아니라 결과물이 만들어진 방식입니다.

AI가 HTML을 만들었다는 사실은 흥미롭지만 핵심은 아닙니다. 중요한 것은 AI에게 무엇을 맡기고, 무엇을 맡기지 않았는지입니다. source를 어디까지 믿을지 정하고, prose를 기준 문장으로 세우고, outline으로 장면을 나누고, design contract로 시각 문법을 고정하고, 생성과 검증을 분리했습니다.

앞 챕터들에서 말한 원칙이 여기서는 실제 제작 구조로 들어갔습니다. 방향을 먼저 잡고, 게이트를 세우고, 역할을 나누고, 상태를 파일에 남겼습니다. 이 발표 제작 과정 자체가 작은 하네스였습니다.

자랑하려는 이야기가 아닙니다. 오히려 반대입니다. 특별한 마법 없이도 하네스는 만들 수 있다는 이야기입니다. 대부분은 markdown, 규칙 문서, HTML, 검증 스크립트, manifest 같은 평범한 파일이었습니다.

## 08-2. source에서 prose로

### 슬라이드에 적을 내용

- `source boundary`
- `prose`
- `논리의 기준선`

### 이 슬라이드에서 할 말

가장 먼저 한 일은 source 경계를 정하는 것이었습니다. 어떤 자료가 1차 source인지, 어떤 자료가 보조 source인지, 어떤 자료는 이번 글에 쓰지 않을지 나눴습니다. 이 구분이 없으면 나중에 "이 주장은 어디서 왔는가"라는 질문에 답할 수 없습니다.

source는 곧바로 slide가 되지 않았습니다. 먼저 prose가 되었습니다. prose는 발표 원고라기보다 논리의 기준선입니다. 어떤 주장이 중심이고, 어떤 주장이 보조인지, 어떤 순서로 독자를 데려갈지 여기서 정합니다.

slide는 prose를 압축할 수는 있지만 벗어날 수 없습니다. notes는 prose를 구어체로 풀 수는 있지만 논리 순서를 마음대로 바꿀 수 없습니다. 이 기준선이 있어야 뒤의 산출물이 대화창의 분위기에 끌려가지 않습니다.

이 단계에서 사람이 붙든 것은 논지의 흐름, 주장 강도, 한국어 문체, source 누락 방지였습니다. 에이전트에게 맡긴 것은 초안 정리와 변형, 반복 검토, 구조화 보조였습니다.

## 08-3. outline과 manifest가 계획을 들고 갔다

### 슬라이드에 적을 내용

- `outline`
- `manifest`
- `기억력이 아니라 기록`

### 이 슬라이드에서 할 말

prose가 있다고 곧바로 HTML을 만들지 않았습니다. 먼저 outline을 만들었습니다.

outline은 목차가 아닙니다. 어떤 장면을 slide로 만들지, 어떤 source section을 참조할지, 어떤 시각 구조가 필요한지 정하는 승인 게이트입니다. 여기서 방향이 틀리면 HTML을 아무리 잘 만들어도 비용만 커집니다.

manifest는 상태를 들고 갔습니다. 어떤 본문이 어떤 slide에 담겼는지, 무엇이 완료되었고 무엇이 pending인지, 어떤 파일이 기준인지, 어떤 검증을 통과했는지 기록했습니다.

작업이 길어질수록 이런 파일이 중요해집니다. 세션이 바뀌어도 작업이 이어지는 이유는 기억력이 좋아서가 아닙니다. 기억해야 할 것을 파일에 남겼기 때문입니다.

## 08-4. design contract가 시각 결정을 고정했다

### 슬라이드에 적을 내용

- `tone`
- `palette`
- `typography`
- `layout family`
- `forbidden patterns`

### 이 슬라이드에서 할 말

발표 자료는 내용만으로 완성되지 않습니다. 시각 구조도 기준이 필요합니다. 그래서 design contract를 먼저 고정했습니다.

디자인 계약에는 톤, 색상 팔레트, 타이포그래피, 여백, layout family, placeholder 규칙, 금지 표현, 과한 장식의 기준이 들어갑니다. 이것이 없으면 에이전트는 매 slide마다 새로 디자인합니다. 그러면 결과는 빠르게 나오지만 deck 전체의 호흡은 무너집니다.

좋은 design contract는 에이전트에게 자유를 주지 않는 문서가 아닙니다. 쓸데없는 자유를 줄이고, 중요한 선택에만 에너지를 쓰게 만드는 문서입니다.

이 과정에서 피드백은 한 번의 대화로 사라지지 않았습니다. 반복되는 피드백은 규칙으로 올라갔습니다. "이 표현은 어색하다", "이 위치의 문구는 제거한다", "제목은 줄바꿈되면 안 된다" 같은 판단은 다음 slide에서도 다시 적용될 수 있어야 합니다.

## 08-5. 생성과 검증을 같은 손에 쥐지 않았다

### 슬라이드에 적을 내용

- `HTML 생성`
- `contract check`
- `한국어 점검`
- `deck consistency`
- `source mapping`

### 이 슬라이드에서 할 말

생성자가 자기 결과를 스스로 통과시키게 두면 그럴듯한 오류가 오래 살아남습니다. 그래서 생성과 검증을 분리했습니다.

HTML을 만든 뒤에는 별도의 검증이 따라왔습니다. slide contract를 확인하고, 한국어 문장을 점검하고, deck-level consistency를 보고, source와 주장 매핑을 확인하고, 디자인 계약 위반을 찾았습니다. 사람이 보는 검토와 기계가 보는 검증이 서로 다른 역할을 맡았습니다.

이 원칙은 코드와 같습니다. 에이전트가 코드를 생성했다면 같은 에이전트의 "괜찮습니다"만 믿으면 안 됩니다. 테스트, lint, type check, 리뷰어, 다른 모델, 사람 검토가 분리되어야 합니다.

이 발표 제작에서도 중요한 것은 "AI가 만들었다"가 아니라 "AI가 만든 것을 어떤 문으로 통과시켰는가"였습니다. 하네스는 생성 능력을 과시하는 장치가 아니라, 생성된 것을 믿을 수 있게 만드는 구조입니다.

## 08-6. 계층을 섞지 않았다

### 슬라이드에 적을 내용

| 층 | 사람이 붙든 것 | 에이전트에게 맡긴 것 |
| --- | --- | --- |
| Source | 입력 경계, 근거 층위 | 자료 정리 보조 |
| Prose | 논리 순서, 문체 | 초안 정리와 변형 |
| Outline | 장면 단위 주장 | 구조화 보조 |
| Design | 시각 문법과 제약 | 계약 안에서 생성 |
| Validation | 통과 기준과 최종 판단 | 체크와 수정 제안 |

### 이 슬라이드에서 할 말

이 제작 과정은 계층을 나눴습니다.

층이 나뉘면 문제가 생겼을 때 돌아갈 위치가 보입니다. 문장이 흔들리면 prose로 돌아갑니다. slide 구성이 흔들리면 outline으로 돌아갑니다. 시각 규칙이 흔들리면 design contract로 돌아갑니다. 근거가 비면 source로 돌아갑니다.

이것이 하네스의 장점입니다. 실패했을 때 막연히 "다시 잘해 봐"라고 하지 않습니다. 어느 층이 흔들렸는지 보고, 그 층의 기준을 고칩니다.

## 08-7. 이 제작 과정이 보여 준 것

### 슬라이드에 적을 내용

- `기준을 먼저 세운다`
- `실행 환경에 박는다`
- `산출물을 검증한다`
- `그것이 Harness`

### 이 슬라이드에서 할 말

이 발표와 글이 보여 준 것은 하나입니다. 원하는 것을 먼저 글로 고정하고, 규칙을 문서로 만들고, 결과를 사람이 검증합니다.

코드, 문서, 기획, 디자인은 도메인이 다릅니다. 하지만 AI를 실제 일에 쓰는 원리는 비슷합니다. 기준을 먼저 세우고, 실행 환경에 박고, 산출물을 검증합니다.

그래서 이 챕터의 결론은 단순합니다. 이 발표 제작 과정 자체가 Harness 방식으로 만들어진 사례였습니다. 다음 챕터에서는 이 사례를 개인과 팀의 행동으로 돌려보겠습니다. 그러면 우리는 무엇부터 바꿔야 하는가.
