# 코딩은 어디로 가고 있는가

## 시작하며: 코딩이 사라지는가?

이 글에서 던지는 질문은 “코딩이 사라지는가”가 아닙니다. 더 정확히는 개발자가 직접 붙들고 있어야 하는 일이 어떻게 달라지고 있는가입니다. 손으로 코드를 한 줄씩 타이핑하던 시대에서, 이제는 규칙을 정하고 맥락을 설정하며 결과를 검증하는 일이 개발의 최전선으로 이동하고 있다는 점입니다. 따라서 개발의 중심축이 어디로 옮겨가고 있는지를 살펴보겠습니다.

## 추상화: 우리는 늘 직접 하던 일을 줄여왔다

이러한 변화는 어느 날 갑자기 일어난 일이 아닙니다. 소프트웨어의 역사를 돌이켜 보면 패턴은 언제나 같았습니다. 직접 해야 할 일은 점점 줄어들고, 더 높은 수준의 일을 설계하는 것이 개발자의 역할로 올라왔습니다.

| 시대 | 전환 | 위임한 것 |
|------|------|-----------|
| 1950s | 기계어 → 어셈블리 | 이진 코드 직접 작성 |
| 1960s | 어셈블리 → 고급 언어 (FORTRAN, C) | 레지스터·명령어 최적화를 **컴파일러**에 위임 |
| 1990s | C → Java | 메모리 해제를 **가비지 컬렉터(GC)**에 위임 |
| 2000s | 언어 → 프레임워크 (Spring, Rails, Django) | 반복되는 보일러플레이트를 **프레임워크**에 위임 |
| 2010s | 온프레미스 → 클라우드 (AWS, GCP) | 서버 관리·확장을 **클라우드**에 위임 |
| 2020s | 직접 코딩 → AI 에이전트 | 코딩 자체를 위임, **'환경'을 설계**하는 것이 핵심 역할로 |

그리고 전환이 일어날 때마다, 저항도 함께 있었습니다.

- **기계어 → 어셈블리**: "기계어를 직접 다루지 않으면 진정한 개발이 아니다!" — 1950년대 초기 프로그래머들은 '컴파일러가 만든 코드가 사람보다 효율적일 리 없다'며 고급 언어 도입을 거부했습니다. 이는 단순한 기술적 주장이 아니라 "하드웨어를 직접 제어하는 것이 진짜 전문성"이라는 직업적 자부심에서 비롯된 저항이었습니다.

- **어셈블리 → C**: "진짜 프로그래머는 파스칼 같은 걸 쓰지 않는다!" — 1983년 Ed Post의 에세이 *"Real Programmers Don't Use Pascal"*은 당시 분위기를 잘 보여줍니다. 고급 언어를 쓰는 개발자는 '퀴슈나 먹는 사람들(quiche eaters)'이라는 비하가 유행했고, 하드웨어를 직접 제어하지 않으면 실력 없는 개발자로 취급받았습니다.

- **C → Java**: "메모리를 직접 관리하지 않으면 진짜 개발자가 아니다!" — GC가 등장하자 숙련된 C 개발자들은 "내가 포인터를 직접 다룰 줄 알아야 진짜지"라고 반발했습니다. 하지만 GC는 결국 메모리 해제 버그를 획기적으로 줄이며 대세가 되었습니다.

- **Java → Python**: "장난감 언어로 무슨 개발을 하냐!" — Python이 처음 등장했을 때, 정적 타입 없이 인터프리터로 동작하는 언어는 '실업무에 쓸 수 없는 스크립트' 수준으로 여겨졌습니다. 오늘날 Python은 AI·데이터 분야의 사실상 표준 언어입니다.

그리고 지금, 우리는 AI에게 `"이 함수를 리팩터링하고 테스트를 작성해 줘"`라고 지시합니다. 다음 반발은 이미 들립니다. `"자연어는 진짜 개발이 아니다!"`

높은 추상화로 나아갈 때마다 저항은 반드시 있었지만, 매번 틀렸습니다. 직접 하는 일은 줄었고, 더 큰 구조를 설계하는 일이 그 자리를 채웠습니다. 이번에도 다르지 않을 것입니다.

## 타이핑은 지표가 될 수 없다.

이 변화는 이미 현장에서 강력하게 나타나고 있습니다. YC W25(Y Combinator 2025 참가팀)의 통계에 따르면, 약 25%의 스타트업이 코드베이스의 95% 이상을 AI로 만들었습니다. OpenAI Codex 팀은 5개월간 수동 코드를 한 줄도 직접 작성하지 않고 100만 줄 규모의 프로덕트를 완성했으며, 팀이 주로 한 일은 “에이전트가 유용한 일을 할 수 있도록 환경을 만드는 것”이었다고 밝혔습니다. “직접 타이핑을 얼마나 하는가”는 더 이상 개발자의 역량을 가늠하는 지표가 되지 못합니다.

## 문서가 결과를 결정하는 세계

가장 두드러진 변화는 코드 바깥에 적힌 문서의 역할이 완전히 달라졌다는 사실입니다. `CLAUDE.md`, `Cursor Rules`, `copilot-instructions.md`, `AGENTS.md` 등 이름은 제각각이지만 이들의 역할은 동일하게 AI에게 프로젝트의 규칙과 우선순위를 전달하는 문서입니다.

과거에 이런 문서들은 단순한 참고 자료나 보충 설명 정도에 불과했습니다. 하지만 AI와 함께 일하는 환경에서는 이 문서들이 곧 실제 작업의 방향을 바꿀 수 있는 핵심 지침서가 됩니다. 어떤 코드 컨벤션을 지킬지, 어떤 문맥을 살펴봐야 할지, 금지해야 할 패턴은 무엇인지를 글로 명확히 적어두는 일이 실제 코드 작성 못지않게 중요해졌습니다.

## 개발자의 새로운 역할

결과적으로 개발자의 핵심 역량도 이동하고 있습니다. 한쪽에는 여전히 “코드를 잘 짜는 능력”이 있지만, 다른 한편에는 “문서를 다루는 능력”과 “컨텍스트를 설계하는 능력”이 급속도로 부상하고 있습니다. 여기서 문서를 잘 쓴다는 것은 예쁘게 정리한다는 뜻이 아니라, AI가 따라야 할 명확한 규칙과 기준을 정의하는 역량입니다.

마치 건축가가 직접 벽돌을 쌓기보다 설계도로 전체 구조를 결정하고, 오케스트라 지휘자가 모든 악기를 직접 연주하기보다 전체를 조율하는 것과 같습니다. 앞으로 개발자는 코딩을 보조하는 존재를 넘어, 복잡한 시스템을 지휘하고 관리하는 `Harness Engineer`의 역할로 전환될 것입니다.

## 그럼에도 기본기

물론 그렇다고 해서 소프트웨어의 기본기가 덜 중요해지는 것은 결코 아닙니다. 오히려 정반대입니다. AI가 실무의 많은 부분을 자동화할수록, 엔지니어링의 본질을 깊이 이해하는 사람의 경쟁력은 역설적으로 더 높아집니다. 줄어든 것은 '코드를 타이핑하는 비중'일 뿐, '시스템을 이해하는 능력'의 가치는 변함이 없습니다.

데모 수준의 간단한 앱을 만드는 일은 극단적으로 쉬워졌습니다. 하지만 실제 리얼 서비스로 넘어가면 보안, 동시성 제어, `DB` 스키마 설계, 캐싱 전략 같은 복잡한 문제가 튀어나오기 마련입니다. 이런 문제들은 프롬프트 몇 줄로 그냥 해결되지 않습니다. 무엇을 파고들어야 할지, AI의 답변 중 어느 부분을 의심해야 할지, 어느 구간에 촘촘한 검증 로직을 짜야 할지는 오롯이 개발자의 탄탄한 기본기에 달려 있습니다.

## 결론

정리하자면, 코딩이라는 행위 자체가 종말을 맞이하는 것은 아닙니다. 다만, 손으로 한 줄씩 코드를 치는 비중이 획기적으로 줄어드는 대신 올바른 규칙을 세우고, 필요한 컨텍스트를 제공하며, 최종 산출물을 엄격하게 검증하는 책임이 더욱 무거워지고 있습니다.

우리가 던져야 할 질문은 'AI가 코딩을 대체할 것인가?'가 아니라 '우리 개발자의 일은 어디로 옮겨 가고 있는가?'입니다. 그 대답은 한 가지를 가리킵니다. 코딩의 중심 공간은 손가락 끝에서, 설계와 검증의 영역으로 확실하게 이동하고 있습니다.
