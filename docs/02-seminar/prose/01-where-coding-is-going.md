# 코딩은 어디로 가고 있는가

## 00. 질문을 먼저 다시 세우기

이 섹션의 출발점은 단순하다. 코딩이 사라지는가를 묻기보다, 코딩의 중심이 어디로 이동하는가를 먼저 묻는 편이 정확하다. 예전에는 모델에게 무엇이라고 말할지, 즉 `Prompt`가 중심처럼 보였다. 곧 무엇을 보여 줄지, 즉 `Context`가 더 중요해졌다. 그리고 지금 발표가 더 앞에 놓는 것은 에이전트가 어떤 규칙과 권한과 검증 아래서 일하는지, 즉 `Harness`다. 이후 섹션에서 다룰 tool genealogy와 방법론도 모두 이 이동 위에서 읽어야 의미가 생긴다.

## 01. 추상화의 역사는 늘 직접 하는 일을 줄여 왔다

이 변화는 갑자기 등장한 예외가 아니라 소프트웨어 역사에서 반복되어 온 추상화의 연장선에 가깝다. 한때는 어셈블리를 직접 다루지 않으면 진짜 개발이 아니라는 감각이 있었고, 이후에는 메모리를 손으로 관리하지 않으면 개발이 아니라는 관성이 있었다. 하지만 `C`, `Java`, `Python` 같은 고급 언어와 `GC`, 그리고 클라우드는 그 확신을 반복해서 무너뜨렸다. 직접 하는 일은 줄고, 설계해야 하는 일은 늘어난다는 점에서 지금 AI를 둘러싼 변화도 같은 계열에 있다.

## 02. 이번 변화의 핵심은 타이핑의 독점성이 무너지는 데 있다

오늘의 전환을 가장 잘 설명하는 문장은 직접 타이핑이 개발을 대표하던 위치가 약해지고 있다는 것이다. 이제 중요한 것은 코드를 한 줄씩 직접 입력하는 능력만이 아니라, 어떤 문맥을 제공하고 어떤 검증 구조 안에서 작업을 반복시킬지를 정하는 능력이다. Section 02에서 `GitHub Copilot`부터 `Claude Code`, `Codex`, `OpenCode`까지 이어지는 계보를 다시 정리하는 이유도 여기 있다. 도구 이름은 달라도 결국 모두 타이핑보다 설계와 검증의 비중을 키우는 방향으로 가고 있기 때문이다.

## 03. 코드 바깥의 텍스트와 규칙이 생산 시스템이 되었다

이 변화가 선명하게 보이는 지점 중 하나는 `Markdown` 문서의 위상 변화다. 예전에는 문서가 코드의 보조물이었다. 하지만 AI coding workflow에서는 `CLAUDE.md`, `AGENTS.md`, `.cursorrules` 같은 규칙 파일이 에이전트에게 프로젝트의 규칙과 우선순위를 전달하는 실행 입력이 된다. 즉, 이 문서들은 더 이상 설명서에 머물지 않고 생산 시스템의 일부가 된다. 함수와 클래스만이 아니라 규칙 파일, 상태 문서, 검증 루프도 모두 실질적인 개발 산출물로 올라오는 셈이다.

## 04. 그래서 개발자의 역할은 Harness Engineer 쪽으로 이동한다

발표가 개발자를 오케스트라 지휘자에 비유하는 이유가 여기에 있다. 지휘자는 모든 악기를 직접 연주하지 않는다. 대신 전체가 어떤 구조와 리듬으로 움직여야 하는지를 조율한다. AI coding 시대의 개발자 역시 점점 이쪽 역할에 가까워진다. `Harness Engineer`라는 표현도 같은 방향을 가리킨다. 중요한 것은 직접 타이핑하는 양보다, 에이전트가 어떤 환경 안에서 일하고 어떤 기준으로 멈추고 통과할지를 설계하는 일이다.

## 05. 엄밀함과 기본기는 검증 구조 안에서 더 중요해진다

이 변화 앞에서 자주 나오는 반응은 “AI가 코드를 쓰면 엔지니어링의 엄밀함이 사라지는 것 아닌가”라는 우려다. 하지만 발표의 관점은 반대에 가깝다. 엄밀함은 사라지지 않는다. 다만 위치를 바꾼다. 예전에는 사람이 코드를 한 줄씩 직접 쓰는 과정 안에 엄밀함이 많이 묻어 있었다면, 이제는 어떤 결과를 통과로 볼지, 어떤 위험을 사전에 차단할지, 어떤 테스트와 리뷰와 승인 과정을 거치게 할지 같은 검증 구조 쪽으로 엄밀함이 이동한다. 그래서 기본기는 대체되지 않고, 무엇이 중요한 경계 조건인지 어디서 결과를 의심해야 하는지 판단하게 해 주는 역량으로 더 중요해진다.

## 06. 결론: 코딩의 중심은 손가락에서 설계와 검증으로 이동한다

따라서 이 섹션의 결론은 분명하다. 코딩은 사라지지 않지만, 코딩의 중심은 환경 설계와 검증 쪽으로 이동하고 있다. 원하는 변화를 정확히 서술하고, 그것을 뒷받침할 문맥과 규칙을 설계하고, 결과를 검증 가능한 구조 안에 넣는 능력이 점점 더 중요해진다. 이후 섹션에서 다룰 `Copilot`, `Cursor`, `Claude Code`, `Codex`, `OpenCode`, `Spec`, `Harness`, `Multi-Agent`, 실전 워크플로우도 모두 이 역할 이동 위에서 읽어야 한다.
