# 02. 하네스는 무엇인가

챗봇은 대화창 안에서 답을 주고 끝납니다. 그런데 에이전트는 거기서 멈추지 않습니다. 파일을 읽고, 계획을 세우고, 고치고, 명령을 실행하고, 다시 결과를 확인합니다.

여기서부터 질문이 달라집니다. 좋은 답변을 한 번 받는 문제가 아니라, 루프가 계속 돌아도 같은 기준을 잃지 않게 만드는 문제가 됩니다.

## 02-1. 에이전트는 루프를 돈다

코딩 에이전트의 움직임을 가장 작게 줄이면 `gather context -> take action -> verify -> repeat`입니다. 먼저 관련 파일과 규칙과 로그를 모읍니다. 그다음 실제 변경을 만들고, 테스트나 실행 결과로 확인한 뒤, 그 결과를 다시 다음 판단의 재료로 씁니다.

이 구조는 챗봇과 완전히 다릅니다. 챗봇이 틀리면 다시 물으면 됩니다. 하지만 에이전트의 오해는 파일 수정, 실패한 명령, 꼬인 계획, 깨진 테스트처럼 실제 작업 상태에 남습니다.

그래서 에이전트를 잘 쓴다는 것은 좋은 문장 하나를 입력하는 일이 아닙니다. 에이전트가 여러 번 움직여도 같은 기준을 유지하도록 환경을 잡아 주는 일입니다.

## 02-2. Prompt에서 Context로, 다시 Harness로

처음에는 프롬프트가 중요했습니다. 어떻게 말해야 원하는 답을 얻을 수 있을까. 이것이 `Prompt`의 질문입니다.

그다음에는 컨텍스트가 중요해졌습니다. 모델에게 무엇을 보여줘야 제대로 판단할까. 이것이 `Context`의 질문입니다. 코드베이스, 문서, 로그, 이전 결정이 여기 들어갑니다.

하지만 컨텍스트도 끝이 아닙니다. 많이 보여준다고 에이전트가 끝까지 잘 움직이는 것은 아닙니다. 무엇을 볼 수 있는지, 무엇을 할 수 있는지, 어디서 멈춰야 하는지, 실패하면 무엇을 남겨야 하는지까지 정해야 합니다. 이 전체가 `Harness`의 질문입니다.

## 02-3. Harness는 실행 환경이다

`Harness`는 특정 기능 하나가 아닙니다. 모델이 실제 일을 할 때 붙어 있는 실행 환경 전체입니다. 어떤 파일을 읽을 수 있는지, 어떤 도구를 쓸 수 있는지, 어떤 명령은 승인 없이는 못 하는지, 검증 실패를 만나면 어디서 멈추는지가 모두 여기에 들어갑니다.

하네스가 약하면 에이전트는 더 똑똑해 보여도 오래 버티지 못합니다. 무엇을 봐야 하는지 모르고, 아무 도구나 열고, 실패한 흔적을 최신 사실처럼 다시 읽습니다.

반대로 하네스가 제자리에 있으면 같은 모델도 훨씬 안정적으로 움직입니다. 필요한 파일을 보고, 허용된 행동만 하고, 검증에서 멈추고, 남은 상태를 다음 작업이 이어받을 수 있게 남깁니다.

## 02-4. Agent = Model + Harness

OpenAI의 agent 가이드는 에이전트를 모델 하나로 보지 않고, 모델과 도구와 지시가 결합된 시스템으로 봅니다.[^02-openai-agent] LangChain의 `Agent = Model + Harness`라는 표현도 같은 관점을 짧게 잡아 줍니다.[^02-langchain-harness]

모델은 추론하고 생성합니다. 하지만 에이전트가 실제로 어떻게 움직이는지는 모델 바깥의 환경이 많이 정합니다. 무엇을 볼 수 있는지, 무엇을 실행할 수 있는지, 어떤 검증을 통과해야 하는지, 실패하면 어떻게 복구할지가 행동을 만듭니다.[^02-canonical-039]

그래서 하네스는 도구를 많이 붙이는 문제가 아닙니다. 지금 이 루프에서 어떤 책임이 비어 있는지 찾는 일입니다. 컨텍스트가 비었는지, 권한 경계가 비었는지, 검증이 비었는지, 상태 기록이 비었는지를 먼저 봐야 합니다.

## **문서가 코드다.**

AI와 함께 일하는 환경에서 문서의 역할도 바뀌었습니다. 과거의 문서는 대개 사람이 읽는 참고 자료였습니다. README, 컨벤션 문서, 아키텍처 노트는 중요했지만, 실제 실행 경로와는 떨어져 있는 경우가 많았습니다.

AI 코딩 도구에서는 이야기가 달라집니다. `CLAUDE.md`, `AGENTS.md`, `Cursor Rules`, `copilot-instructions.md` 이름은 모두 다르지만 역할은 동일합니다. 에이전트가 매 세션 읽고 따라야 할 운영 규칙입니다. 코딩 스타일, 금지 명령, 리뷰 기준, 테스트 규칙, 문서 어투, 승인 조건 같은 내용이 이 파일에 들어갑니다.

과거에는 “문서화”가 사후 정리처럼 취급되었습니다. 이제는 문서가 곧 개발의 일부가 됩니다.

## **개발자의 새로운 역할: 컨텍스트를 설계하는 능력**

과거의 개발자는 코드를 잘 작성하고 버그를 잘 고치는 사람이었습니다. 앞으로의 개발자는 에이전트가 코드를 잘 쓸 수 있도록 환경을 설계하고, 결과를 판단하는 사람에 가까워집니다.

건축가는 직접 벽돌을 모두 쌓지 않습니다. 하지만 설계도, 구조, 재료, 안전 기준, 시공 순서를 정의합니다. 오케스트라 지휘자도 모든 악기를 직접 연주하지 않습니다. 대신 어느 악기가 언제 들어오고, 어디서 멈추고, 전체가 어떤 소리를 내야 하는지 조율합니다.

AI 시대의 개발자도 비슷한 방향으로 이동합니다. 직접 구현하는 시간은 줄어들 수 있지만, 다음 능력의 중요성은 더 커집니다.

- 문제를 올바른 단위로 자르는 능력
- 에이전트에게 보여 줄 컨텍스트를 고르는 능력
- 요구사항과 수용 기준을 명확히 쓰는 능력
- 테스트와 검증 게이트를 설계하는 능력
- 위험한 권한과 안전한 권한을 나누는 능력
- 에이전트의 결과를 의심하고 판단하는 능력
- 반복되는 판단 기준을 파일과 규칙으로 외재화하는 능력

이것이 이 글에서 말하는 `Harness Engineer`의 출발점입니다.

## **그래도 기초가 중요하다.**

AI가 코드를 더 많이 써 줄수록 기본기가 덜 중요해질 것처럼 보입니다. 실제로는 반대입니다. AI는 빠르게 초안을 만들 수 있지만, 그 초안이 안전한지, 유지보수 가능한지, 기존 시스템과 잘 맞는지 판단하려면 여전히 깊은 기본기가 필요합니다.

간단한 데모 앱은 매우 쉽게 만들 수 있습니다. 하지만 실제 서비스에서 곧 문제가 터집니다. 인증과 권한, 동시성, 데이터베이스 스키마, 캐싱, 관측 가능성, 장애 복구, 보안, 비용, 배포 파이프라인 같은 문제가 뒤따릅니다. 이런 구간에서 “AI 가 시켰어요” 라고 탓할 수는 없습니다.

오히려 AI가 코드를 많이 만들어 줄수록, 어느 부분을 의심해야 하는지 아는 능력이 더 중요해집니다. 모델이 제안한 아키텍처가 왜 위험한지, 테스트가 무엇을 놓쳤는지, API 설계가 나중에 어떤 병목을 만들지, 보안 경계가 어디서 무너지는지를 파악할 수 있어야 합니다.

줄어든 것은 타이핑의 비중이지, 시스템을 이해하는 능력의 가치가 아닙니다. 개발자의 기본기는 사라지는 것이 아니라 더욱 더 중요하게 여겨집니다.

> **AI가 더 많이 해줄수록 기초 지식을 가진 사람의 경쟁력이 더 올라간다**

코딩은 끝났다 라는 말은 틀렸습니다. 하지만 손으로 한 줄씩 작성하는 비중은 줄고 있습니다. 대신 규칙을 세우고, 컨텍스트를 고르고, 결과를 검증하고, 상태를 남기는 비중이 점점 늘고 있습니다.

그래서 질문을 바꿔야 합니다. “코딩은 사라지는가?”가 아니라 “AI 코딩은 무엇인가?”입니다.


## 02-5. 좋은 하네스는 경계에서 시작한다

좋은 하네스는 `Skills`, `Hooks`, `MCP`, `Plugins`를 많이 붙여서 만들어지는 것이 아닙니다. 어떤 일은 자동으로 해도 되고, 어떤 일은 사람에게 돌아와야 하는지 경계를 선명하게 정할 때 만들어집니다.

컨텍스트도 다 채우는 것이 능력이 아닙니다. 지금 판단에 필요한 신호만 남겨야 합니다. 대화창은 모든 상태를 저장하는 창고가 아닙니다. 다음 루프가 다시 쓸 수 있는 사실만 남겨야 하는 통로에 가깝습니다.

실무에서 흔히 놓치는 순간은 "설정이 다 끝났다"는 착각입니다. 하네스는 한 번에 완성되지 않습니다. 루프가 돌아가면서 빈틈이 드러나고, 그때마다 경계를 보강하면서 좋아집니다.[^02-openai-harness]
