# 02. 하네스는 무엇인가

## 02-1. 에이전트는 루프를 돈다

### 슬라이드에 적을 내용

`gather context -> take action -> verify -> repeat`

### 이 슬라이드에서 할 말

앞 챕터의 결론은 단순했습니다. 에이전트에게 일을 맡기려면 먼저 일할 환경을 만들어야 합니다. 그렇다면 이제 봐야 할 것은 에이전트가 실제로 일하는 방식입니다.

코딩 에이전트는 한 번 답하고 끝나는 챗봇이 아닙니다. 작은 작업 루프를 계속 반복합니다.

OpenAI의 agent guide는 agent를 "independently accomplish tasks on your behalf"하는 시스템으로 정의합니다.[^02-openai-agent] 이 정의에서 중요한 단어는 독립성입니다. 독립적으로 움직이는 순간, 답변 품질보다 루프의 통제가 더 중요해집니다.

`gather context`에서는 관련 파일, 규칙, 로그, 이전 결정, 에러 메시지를 모읍니다. `take action`에서는 파일을 수정하거나 명령을 실행하거나 외부 도구를 호출합니다. `verify`에서는 테스트, 린트, 타입 체크, 실행 로그로 결과를 확인합니다. `repeat`에서는 그 결과를 다음 판단의 재료로 삼아 다시 움직입니다.

루프는 좋은 판단만 반복하지 않습니다. 잘못 읽은 파일, 잘못 세운 계획, 실패한 도구 결과 같은 잡음도 함께 쌓입니다. 그래서 환경을 봐야 합니다. 에이전트를 잘 쓴다는 것은 이 루프를 통제한다는 뜻입니다.

## 02-2. Prompt에서 Context로, 다시 Harness로

### 슬라이드에 적을 내용

- `Prompt`
- `Context`
- `Harness`
- `통제 범위가 넓어진다`

### 이 슬라이드에서 할 말

AI 도구를 다루는 질문은 시간이 지나며 바뀌었습니다.

처음에는 "어떻게 말해야 원하는 답을 얻을까"가 중심이었습니다. 하지만 프롬프트에는 분명한 한계가 있습니다. 아무리 좋은 지시를 내려도 모델이 프로젝트의 다른 파일, 기존 설계 결정, 내부 유틸리티, 팀 컨벤션을 보지 못하면 제대로 활용할 수 없습니다.

그래서 다음 질문은 "무엇을 보여줘야 제대로 판단할까"로 옮겨갔습니다. 코드베이스 인덱싱, 문서 검색, `RAG`, 심볼 검색은 이 질문에 대한 답입니다.

하지만 컨텍스트도 끝이 아닙니다. 좋은 입력만으로는 루프를 통제할 수 없습니다. 에이전트는 컨텍스트를 읽고 답하는 데서 끝나지 않습니다. 그 컨텍스트를 바탕으로 파일을 고치고, 명령을 실행하고, 테스트 결과를 다시 읽고, 다음 행동을 고릅니다.

그래서 질문은 다시 바뀝니다. "무엇을 통제해야 오래 버틸까." 여기서 `Harness`가 등장합니다.

## 02-3. Prompt ⊂ Context ⊂ Harness

### 슬라이드에 적을 내용

`Prompt ⊂ Context ⊂ Harness`

### 이 슬라이드에서 할 말

`Prompt`는 목표, 금지 사항, 출력 형식을 어떻게 말할지 다룹니다. `Context`는 모델이 판단할 때 볼 재료를 고릅니다. `Harness`는 그 지시와 정보가 실제 행동으로 이어질 때의 환경을 설계합니다.

`Harness`는 프롬프트와 컨텍스트를 없애지 않습니다. 오히려 둘이 제자리에서 작동하게 만듭니다. 프롬프트는 한 번의 말입니다. 컨텍스트는 그 지시가 제대로 작동하려면 필요한 정보입니다. `Harness`는 지시와 정보가 실제 작업으로 이어지는 환경입니다.

좋은 `Harness`도 좋은 프롬프트와 좋은 컨텍스트를 필요로 합니다. 모호한 지시는 좋은 환경 위에서도 흔들리고, 잡음 많은 컨텍스트는 좋은 규칙 아래에서도 판단을 흐립니다.

## 02-4. Agent = Model + Harness

### 슬라이드에 적을 내용

`Agent = Model + Harness`

`모델이 아닌 것은 전부 하네스다`

### 이 슬라이드에서 할 말

에이전트를 모델과 동일시하면 문제의 본질을 놓치게 됩니다. 모델은 텍스트를 생성하고 추론하는 엔진일 뿐입니다. 실제 에이전트의 행동 방식은 모델 바깥에서 훨씬 많이 결정됩니다.

OpenAI는 agent의 기본 구성요소를 `Model`, `Tools`, `Instructions`로 나눕니다.[^02-openai-agent] 이 세 가지를 우리 말로 다시 잡으면, 모델은 엔진이고 도구와 지시는 하네스의 일부입니다.

무엇을 볼 수 있는지, 어떤 도구를 쓸 수 있는지, 어떤 권한은 막히는지, 어떤 검증을 통과해야 하는지, 실패하면 어떤 상태로 돌아가는지가 행동을 만듭니다.

LangChain의 Vivek Trivedy는 이 관점을 더 짧게 표현했습니다. 모델이 아닌 것은 전부 하네스입니다.

모델이 힘의 원천이라면, `Harness`는 그 힘이 어디로 흐를지 정하는 장치입니다. 같은 모델을 써도 환경에 따라 결과는 크게 달라집니다. 어떤 환경에서는 에이전트가 관련 파일을 읽고, 계획을 세우고, 테스트를 통과한 뒤 보고합니다. 다른 환경에서는 엉뚱한 파일만 만지다가 자신 있게 완료했다고 말합니다.

모델이 달라서가 아닙니다. 모델을 감싸는 환경이 다르기 때문입니다.

## 02-5. 구성요소, 책임, 도구

### 슬라이드에 적을 내용

| 층 | 질문 |
| --- | --- |
| 구성요소 | 무엇을 할 수 있어야 하는가 |
| 책임 | 무엇을 지켜야 하는가 |
| 도구 | 어디에 고정할 것인가 |

### 이 슬라이드에서 할 말

`Harness`를 한 덩어리로 말하면 금방 막막해집니다. 실제로 설계하려면 세 가지를 나누어 봐야 합니다. 구성요소, 책임, 도구입니다.

구성요소는 하네스가 갖춰야 할 능력입니다. `Context Engineering`, `Tool Orchestration`, `State & Memory`, `Verification Loop`, `Error Recovery`, `Human-in-the-Loop Control` 같은 것들이 여기에 들어갑니다.

책임은 그 능력으로 무엇을 지킬지 정한 기준입니다. 위험한 행동을 막는 `Guardrails`, 작업 범위와 기준선을 고정하는 `Specification`, 결과를 흔들리지 않게 걸러내는 `Verification`, 상태가 사라지지 않게 하는 `State Management`, 루프가 어디서 흔들리는지 보이게 하는 `Observability`입니다.

도구는 그 책임을 실제 작업 환경에 고정하는 수단입니다. `AGENTS.md`, `CLAUDE.md`, `Skills`, `Hooks`, `MCP`, `Plugins`, `Git Worktree`, 테스트 명령, CI, 이슈, PR 체크가 모두 도구가 될 수 있습니다.

## 02-6. 루프마다 필요한 책임이 다르다

### 슬라이드에 적을 내용

| 루프 | 필요한 책임 |
| --- | --- |
| `gather context` | `Specification`, `Context Engineering` |
| `take action` | `Guardrails`, `Tool Orchestration` |
| `verify` | `Verification`, `Error Recovery` |
| `repeat` | `State Management`, `Observability` |

### 이 슬라이드에서 할 말

하네스의 구성요소와 책임은 추상적인 목록이 아닙니다. 에이전트가 실제로 도는 네 단계마다 필요한 장치입니다.

`gather context`에서는 작업 범위, 성공 기준, 금지 사항, 프로젝트 규칙이 흔들리지 않아야 합니다. 많이 넣는 것이 아니라 지금 판단에 필요한 신호를 남겨야 합니다.

`take action`에서는 권한과 도구를 다뤄야 합니다. 읽기, 수정, 삭제, 배포, 외부 API 호출은 같은 위험도를 갖지 않습니다. 어떤 도구를 열고, 어떤 행동은 승인 뒤에만 허용할지 정해야 합니다.

`verify`에서는 "잘 확인해줘"가 아니라 실제 `lint`, `type check`, 테스트, 리뷰 게이트를 통과하게 만들어야 합니다. 실패하면 다시 고칠지, 멈출지, 사람에게 올릴지 정해야 합니다.

`repeat`에서는 상태와 기억을 놓치지 않아야 합니다. 결정, 진행 상황, 실패 원인, 사용자 선호는 파일, 이슈, Git 이력, 구조화된 메모처럼 세션 밖의 기록으로 남아야 합니다.

## 02-7. 도구 이름보다 책임이 먼저다

### 슬라이드에 적을 내용

- `AGENTS.md`
- `Hooks`
- `MCP`
- `Worktree`
- `Test`

`도구 목록이 아니라 책임 배치`

### 이 슬라이드에서 할 말

쉽게 빠지는 길이 있습니다. 원리를 모른 채 `Skills`, `Hooks`, `MCP`, `Plugins`, preset pack만 주워 붙이는 길입니다. 겉보기에는 편하지만 이상 동작이 나오면 어디가 문제인지 읽지 못합니다. 시간이 지나면 팀에는 원리가 아니라 블랙박스만 남습니다.

그래서 도구 이름보다 책임이 먼저입니다. `CLAUDE.md`나 `AGENTS.md`는 프로젝트의 기준선을 고정합니다. `Hooks`는 위험한 명령을 막거나 테스트를 자동 실행하게 만들 수 있습니다. `Git Worktree`는 작업을 격리하고 상태를 나눕니다. `MCP`와 `Plugins`는 외부 도구와 데이터를 연결합니다.

도구와 책임은 1:1로 대응하지 않습니다. 하나의 도구가 여러 책임을 맡을 수 있고, 하나의 책임도 여러 도구로 구현될 수 있습니다. 설계는 도구 이름에서 시작하면 안 됩니다. 먼저 어떤 책임이 비어 있는지 보고, 그 책임을 어떤 파일, 명령어, 게이트, 기록에 배치할지 정합니다.

## 02-8. Harness Builder

### 슬라이드에 적을 내용

- `Harness Builder`
- `에이전트가 유용한 일을 할 수 있도록`
- `환경을 만드는 사람`

### 이 슬라이드에서 할 말

OpenAI Codex 팀은 이 변화를 이렇게 표현했습니다. 우리 팀의 주된 업무는 에이전트가 유용한 일을 할 수 있도록 환경을 만드는 것이 되었다.

원문 표현으로는 "enabling the agents to do useful work"입니다.[^02-openai-harness] 코드를 덜 치게 되었다는 말보다, 에이전트가 쓸 수 있는 구조를 만드는 일이 주 업무가 되었다는 점이 더 중요합니다.

모델이 코드를 대신 써 준다는 말보다 더 중요한 문장입니다. 중심이 작성 속도에서 작업 환경으로 옮겨갔기 때문입니다.

`Harness Builder`는 에이전트가 유용한 일을 할 수 있도록 돕는 환경을 만드는 사람입니다. 저장소 지식을 문서와 규칙으로 정리하고, 테스트와 린터 같은 결정론적 제약을 붙이고, 에이전트가 읽기 좋은 구조를 남기고, 팀의 취향과 원칙을 파일과 게이트에 새깁니다.

이제 다음 챕터에서는 이 루프가 왜 망가지는지 보겠습니다. 긴 작업, 긴 컨텍스트, 검증 없는 수용은 왜 이상한 수정이 누적되는 장면으로 이어질까요.
