# 02. Harness는 환경이다

## 에이전트가 일하는 방식

앞 장의 결론은 단순했습니다. 에이전트에게 일을 맡기려면 먼저 일할 환경을 만들어야 합니다. 그렇다면 이제 봐야 할 것은 에이전트가 실제로 일하는 방식입니다.

코딩 에이전트는 한 번 답하고 끝나는 챗봇이 아닙니다. 작은 작업 루프를 계속 반복합니다. 간단하게 표현하면 다음과 같습니다.

`gather context -> take action -> verify -> repeat`

`gather context`에서는 관련 파일, 규칙, 로그, 이전 결정, 에러 메시지를 모읍니다. `take action`에서는 파일을 수정하거나 명령을 실행하거나 외부 도구를 호출합니다. `verify`에서는 테스트, 린트, 타입 체크, 실행 로그로 결과를 확인합니다. `repeat`에서는 그 결과를 다음 판단의 재료로 삼아 다시 움직입니다.

에이전트는 이 과정을 반복합니다. 사람은 "이 파일을 보고, 이 테스트를 돌리고, 실패하면 다시 고쳐"라고 모든 과정을 일일이 지시하지 않아도 됩니다. 에이전트 스스로 읽고, 고치고, 확인하는 흐름을 이어 갑니다.

그래서 환경을 봐야 합니다. 루프는 좋은 판단만 반복하지 않습니다. 잘못 읽은 파일, 잘못 세운 계획, 실패한 도구 결과 같은 잡음도 함께 쌓입니다. 이를 방지하려면 루프를 통제해야 합니다.

## Prompt, Context, Harness

AI 도구를 다루는 질문은 시간이 지나며 바뀌었습니다. 처음에는 "어떻게 말해야 원하는 답을 얻을까"가 중심이었습니다. 하지만 프롬프트에는 분명한 한계가 있습니다. 아무리 좋은 지시를 내려도 모델이 프로젝트의 다른 파일, 기존 설계 결정, 내부 유틸리티, 팀 컨벤션을 보지 못하면 제대로 활용할 수 없습니다. "기존 코드를 재사용하라"고 써도, 관련 파일이 컨텍스트에 없으면 에이전트는 그 존재를 알지 못합니다.

그래서 다음 질문은 "무엇을 보여줘야 제대로 판단할까"로 옮겨갔습니다. 코드베이스 인덱싱, 문서 검색, `RAG`, 심볼 검색은 이 질문에 대한 답입니다. 모델 앞에 더 긴 프롬프트를 붙이는 대신, 필요한 파일과 규칙과 히스토리를 찾아 컨텍스트로 조립하는 방식입니다.

하지만 컨텍스트도 끝이 아닙니다. 좋은 입력만으로는 루프를 통제할 수 없습니다. 에이전트는 컨텍스트를 읽고 답하는 데서 끝나지 않습니다. 그 컨텍스트를 바탕으로 파일을 고치고, 명령을 실행하고, 테스트 결과를 다시 읽고, 다음 행동을 고릅니다. 이때 도구 호출이 실패할 수 있고, 테스트를 잘못 해석할 수 있고, 위험한 명령을 실행할 수도 있습니다. 그래서 질문은 다시 바뀝니다. "무엇을 통제해야 오래 버틸까." 여기서 `Harness`가 등장합니다.

`Prompt`는 목표, 금지 사항, 출력 형식을 어떻게 말할지 다룹니다. `Context`는 모델이 판단할 때 볼 재료를 고릅니다. `Harness`는 그 지시와 정보가 실제 행동으로 이어질 때의 환경을 설계합니다. 행동을 허용하는 방식, 검증하는 방식, 멈추는 방식, 기억을 남기는 방식이 여기에 들어갑니다.

> **Prompt ⊂ Context ⊂ Harness**

`Harness`는 프롬프트와 컨텍스트를 없애지 않습니다. 오히려 둘이 제자리에서 작동하게 만듭니다. 프롬프트는 한 번의 말입니다. 컨텍스트는 프롬프트가 기대는 정보입니다. `Harness`는 지시와 정보가 실제 작업으로 이어지는 환경입니다.

좋은 `Harness`도 좋은 프롬프트와 좋은 컨텍스트를 필요로 합니다. 모호한 지시는 좋은 환경 위에서도 흔들리고, 잡음 많은 컨텍스트는 좋은 규칙 아래에서도 판단을 흐립니다.

## Agent = Model + Harness

에이전트를 모델과 동일시하면 문제의 본질을 놓치게 됩니다. 모델은 텍스트를 생성하고 추론하는 엔진일 뿐입니다. 하지만 실제 에이전트의 행동 방식은 모델 바깥에서 훨씬 많이 결정됩니다. 무엇을 볼 수 있는지, 어떤 도구를 쓸 수 있는지, 어떤 권한은 막히는지, 어떤 검증을 통과해야 하는지, 실패하면 어떤 상태로 돌아가는지가 행동을 만듭니다.

> **Agent = Model + Harness**

LangChain의 Vivek Trivedy는 이 관점을 더 짧게 표현했습니다. "모델이 아닌 것은 전부 하네스입니다."

모델이 힘의 원천이라면, `Harness`는 그 힘이 어디로 흐를지 정하는 장치입니다. 원래 `harness`는 말을 다루는 마구를 뜻합니다. 힘이 센 말을 원하는 방향으로 움직이게 하려면 고삐, 안장, 재갈, 멈춤 지점이 필요합니다. LLM도 비슷합니다. 모델이 강해질수록 더 많은 일을 할 수 있지만, 그 힘이 곧바로 좋은 작업이 되지는 않습니다.

같은 모델을 써도 환경에 따라 결과는 크게 달라집니다. 어떤 환경에서는 에이전트가 관련 파일을 읽고, 계획을 세우고, 테스트를 통과한 뒤 보고합니다. 다른 환경에서는 엉뚱한 파일만 만지작거리다가 아무 말이나 뱉고는 자신 있게 완료했다고 말합니다. 모델이 달라서가 아닙니다. 모델을 감싸는 환경이 다르기 때문입니다.

이 관점에서 좋은 AI 코딩 도구는 대화를 잘하는 도구에 그치지 않습니다. 에이전트가 오래 일해도 무너지지 않도록 작업 환경을 구성하는 도구입니다. 좋은 사용자는 모델에게 화려한 주문을 외우는 사람이 아니라, 모델이 일할 수 있는 환경을 자기 프로젝트에 맞게 다듬는 사람입니다.

## Harness의 구성요소

`Harness`를 한 덩어리로 말하면 금방 막막해집니다. 실제로 설계하려면 세 가지를 나누어 봐야 합니다. 구성요소, 책임, 도구입니다.

구성요소는 하네스가 갖춰야 할 능력입니다. `Context Engineering`, `Tool Orchestration`, `State & Memory`, `Verification Loop`, `Error Recovery`, `Human-in-the-Loop Control` 같은 것들이 여기에 들어갑니다. 이 이름들은 하네스가 무엇을 할 수 있어야 하는지 가리킵니다.

책임은 그 능력으로 무엇을 지킬지 정한 기준입니다. 위험한 행동을 막는 `Guardrails`, 작업 범위와 기준선을 고정하는 `Specification`, 결과를 흔들리지 않게 걸러내는 `Verification`, 상태가 사라지지 않게 하는 `State Management`, 루프가 어디서 흔들리는지 보이게 하는 `Observability`입니다.

도구는 그 책임을 실제 작업 환경에 고정하는 수단입니다. `AGENTS.md`, `CLAUDE.md`, `Skills`, `Hooks`, `MCP`, `Plugins`, `Git Worktree`, 테스트 명령, CI, 이슈, PR 체크가 모두 도구가 될 수 있습니다.

## 루프를 통제하는 책임과 구성요소

이제 다시 에이전트 루프로 돌아가 보겠습니다. 하네스의 구성요소와 책임은 추상적인 목록이 아닙니다. 에이전트가 실제로 도는 네 단계마다 필요한 장치입니다.

`gather context`에서는 기준점과 정보를 먼저 붙잡아야 합니다. 여기서 필요한 책임은 `Specification`입니다. 작업 범위, 성공 기준, 금지 사항, 프로젝트 규칙이 흔들리지 않아야 합니다. 이를 가능하게 하는 구성요소가 `Context Engineering`입니다. 필요한 파일과 규칙은 남기고, 오래된 가정과 관련 없는 도구 결과는 걷어냅니다. Context Engineering은 많이 넣는 기술이 아니라, 지금 판단에 필요한 신호를 남기는 기술입니다.

`take action`에서는 권한과 도구를 다뤄야 합니다. 여기서 필요한 책임은 `Guardrails`입니다. 읽기, 수정, 삭제, 배포, 외부 API 호출은 같은 위험도를 갖지 않습니다. 이를 다루는 구성요소가 `Tool Orchestration`입니다. 어떤 도구를 열고, 어떤 순서로 호출하고, 어떤 행동은 승인 뒤에만 허용할지 정합니다.

여기에 `Human-in-the-Loop Control`이 붙습니다. 모든 행동을 사람에게 물어보자는 뜻은 아닙니다. 읽기, 검색, 로컬 테스트처럼 되돌리기 쉬운 행동은 에이전트가 빠르게 처리해도 됩니다. 반대로 삭제, 배포, 권한 변경, 큰 설계 변경처럼 한 번 지나가면 비용이 커지는 행동은 승인 없이 넘어가면 안 됩니다. 사람의 개입 지점을 루프 안에 관문으로 넣어 두면, 에이전트는 계속 달리는 대신 필요한 순간 멈춥니다.

`verify`에서는 결과를 흔들리지 않는 기준으로 걸러내야 합니다. 여기서 필요한 책임은 `Verification`입니다. "잘 확인해줘"라고 말하는 것보다 실제 `lint`, `type check`, 테스트, 리뷰 게이트를 통과하게 만드는 편이 덜 흔들립니다. 이를 움직이는 구성요소가 `Verification Loop`와 `Error Recovery`입니다. 실패하면 실패 메시지를 읽고 다시 고치게 할지, 더 진행하지 못하게 멈출지, 사람에게 올릴지 정해야 합니다.

`repeat`에서는 상태와 기억을 놓치지 않아야 합니다. 여기서 필요한 책임은 `State Management`와 `Observability`입니다. 무엇이 결정되었고, 무엇이 실패했고, 다음 루프에서 무엇을 봐야 하는지 남겨야 합니다. 이를 담당하는 구성요소가 `State & Memory`입니다.

모델 호출 자체는 본질적으로 무상태(stateless)입니다. 그래서 매 호출 시마다 이전에 대화를 나누었던 내용을 통째로 다시 넣고 답변을 받는 방식으로 동작합니다. 이런 특성 때문에 새로운 채팅 세션을 열면 모든 기억을 잃어버린 AI와 대화하는 것과 같습니다. 따라서 다음 루프에 남아야 할 결정, 진행 상황, 실패 원인, 사용자 선호는 파일, 이슈, Git 이력, 구조화된 메모처럼 세션 밖의 기록으로 미리 기록하고 다음 실행 때 가져와야 합니다.

## 도구는 책임을 작업 환경에 고정한다

여기서 쉽게 빠지는 두 길이 있습니다. 하나는 모든 것을 손으로 기억하고 매번 직접 조작하는 길입니다. 명령어를 매번 다시 조합하고, 검증 루프를 기억에 의존하고, 세션마다 같은 지시를 다시 입력합니다. 초반에는 통제하는 느낌이 들지만 오래 유지되지 않습니다. 기억은 새고, 기준은 흔들립니다.

다른 하나는 원리를 모른 채 `Skills`, `Hooks`, `MCP`, `Plugins`, preset pack만 주워 붙이는 길입니다. 겉보기에는 편하지만 이상 동작이 나오면 어디가 문제인지 읽지 못합니다. 시간이 지나면 팀에는 원리가 아니라 블랙박스만 남습니다.

그래서 도구 이름보다 책임이 먼저입니다. `CLAUDE.md`나 `AGENTS.md`는 프로젝트의 기준선을 고정합니다. 매 세션 반복해서 설명할 필요가 없는 규칙, 문서 어투, 금지 명령, 테스트 방식, 리뷰 기준을 담을 수 있습니다. 이 파일들은 `Specification`을 맡고, 상황에 따라 약한 `Guardrails`도 됩니다.

`Hooks`는 특정 이벤트에 반응해 위험한 명령을 막거나 테스트를 자동 실행하게 만들 수 있습니다. 이 경우 `Hooks`는 `Guardrails`이면서 `Verification`입니다. `Git Worktree`는 작업을 격리하고 상태를 나눕니다. 그래서 `State Management`와 위험 격리에 기여합니다. `MCP`와 `Plugins`는 외부 도구와 데이터를 연결합니다. 하지만 연결이 많다고 좋은 `Harness`가 되는 것은 아닙니다. 어떤 책임을 맡길지 모르면 도구는 곧 잡음이 됩니다.

도구와 책임은 1:1로 대응하지 않습니다. 하나의 도구가 여러 책임을 맡을 수 있고, 하나의 책임도 여러 도구로 구현될 수 있습니다. 그래서 설계는 도구 이름에서 시작하면 안 됩니다. 먼저 어떤 책임이 비어 있는지 봅니다. 그다음 그 책임을 어떤 파일, 명령어, 게이트, 기록에 배치할지 정합니다.

예를 들어 "테스트를 반드시 통과해야 한다"는 말은 도구 이름이 아니라 `Verification` 책임입니다. 구현은 `pytest`, CI, `PostToolUse` Hook, PR 체크, 리뷰 체크리스트가 나눠 맡을 수 있습니다. "위험한 명령은 막아야 한다"는 말은 `Guardrails` 책임입니다. 구현은 승인 규칙, sandbox, `PreToolUse` Hook, 권한 분리로 나뉩니다. "작업 기준이 세션마다 흔들리면 안 된다"는 말은 `Specification`과 `State Management`의 책임입니다. 구현은 `AGENTS.md`, 이슈, 진행 파일, Git 이력이 맡습니다.

`Harness`를 잘 쓴다는 것은 멋진 도구 목록을 많이 갖는다는 뜻이 아닙니다. 필요한 파일, 필요한 규칙, 필요한 검증, 필요한 권한만 남기는 일입니다. 어디서 손을 막고, 어디서 결과를 걸러내고, 무엇을 세션 밖에 남길지 분명할수록 에이전트는 오래 버팁니다.

## Harness Builder
OpenAI Codex 팀은 이 변화를 이렇게 표현했습니다. "우리 팀의 주된 업무는 에이전트가 유용한 일을 할 수 있도록 환경을 만드는 것이 되었다." 모델이 코드를 대신 써 준다는 말보다 더 중요한 문장입니다. 중심이 작성 속도에서 작업 환경으로 옮겨갔기 때문입니다.

`Harness Builder`는 에이전트가 유용한 일을 할 수 있도록 돕는 환경을 만드는 사람입니다. 저장소 지식을 문서와 규칙으로 정리하고, 테스트와 린터 같은 결정론적 제약을 붙이고, 에이전트가 읽기 좋은 구조를 남기고, 팀의 취향과 원칙을 파일과 게이트에 새깁니다.
