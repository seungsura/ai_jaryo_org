# 02. Harness는 환경이다

## 에이전트가 일하는 방식

에이전트가 일을 하려면 작업 환경을 만들어 주어야 합니다. 그러기 위해선 먼저 에이전트가 실제로 일하는 방식을 이해해야 합니다.

코딩 에이전트는 한 번 답하고 끝나는 챗봇이 아닙니다. 작은 작업 루프를 계속 반복합니다. 간단하게 표현하면 다음과 같습니다.

`gather context -> take action -> verify -> repeat`

`gather context`에서는 관련 파일, 규칙, 로그, 이전 결정, 에러 메시지를 모읍니다. `take action`에서는 파일을 수정하거나 명령을 실행하거나 외부 도구를 호출합니다. `verify`에서는 테스트, 린트, 타입 체크, 실행 로그로 결과를 확인합니다. `repeat`에서는 그 결과를 다음 판단의 재료로 삼아 다시 움직입니다.

에이전트는 이 과정을 반복합니다. 사람은 "이 파일을 보고, 이 테스트를 돌리고, 실패하면 다시 고쳐"라고 모든 과정을 일일히 지시하지 않아도 됩니다. 에이전트 스스로 읽고, 고치고, 확인하는 흐름을 이어 갑니다.

하지만 바로 그 이유 때문에 환경을 봐야 합니다. 루프는 좋은 판단만 반복하지 않습니다. 잘못 읽은 파일, 잘못 세운 계획, 실패한 도구 결과와 같은 잡음이 쌓입니다. 이를 방지하기 위해, 우리는 루프를 통제해야 합니다.

## Prompt, Context, Harness

AI 도구를 다루는 질문은 시간이 지나며 바뀌었습니다. 처음에는 "어떻게 말해야 원하는 답을 얻을까"가 중심이었습니다. 다음에는 "무엇을 보여줘야 제대로 판단할까"가 중요해졌습니다. 시간이 흐르면서 질문은 "무엇을 통제해야 오래 버틸까"로 옮겨갔습니다. 여기서 `Harness`가 등장합니다.

`Prompt`는 목표, 금지 사항, 출력 형식을 어떻게 말할지 다룹니다. `Context`는 모델이 판단할 때 볼 재료를 고릅니다. `Harness`는 그 말과 재료가 실제 행동으로 이어질 때의 환경을 설계합니다. 행동을 허용하는 방식, 검증하는 방식, 멈추는 방식, 기억을 남기는 방식이 여기에 들어갑니다.

관계는 이렇게 잡는 편이 좋습니다.

> **Prompt ⊂ Context ⊂ Harness**

`Harness`는 프롬프트와 컨텍스트를 없애지 않습니다. 오히려 둘이 제자리에서 작동하게 만듭니다. 프롬프트는 한 번의 말입니다. 컨텍스트는 그 말이 기대는 정보입니다. `Harness`는 그 말과 정보가 실제 작업으로 이어지는 환경입니다.

좋은 `Harness`도 좋은 프롬프트와 좋은 컨텍스트를 필요로 합니다. 모호한 지시는 좋은 환경 위에서도 흔들리고, 잡음 많은 컨텍스트는 좋은 규칙 아래에서도 판단을 흐립니다. 차이는 반복성입니다. 좋은 `Harness`는 좋은 지시와 좋은 컨텍스트가 매번 우연히 만들어지기를 기다리지 않습니다. 규칙, 파일, 명령어, 게이트, 기록으로 고정합니다.

여기서 `Context`의 한계도 분명히 봐야 합니다. 프롬프트의 한계는 모델이 보지 못한 것을 알 수 없다는 점이었습니다. 그래서 코드베이스 인덱싱, 문서 검색, `RAG`, 심볼 검색 같은 흐름이 중요해졌습니다. 하지만 좋은 입력만으로는 루프를 통제할 수 없습니다. 에이전트는 컨텍스트를 읽고 끝나는 것이 아니라, 그 컨텍스트를 바탕으로 행동하고, 도구 결과를 다시 읽고, 다음 행동을 고릅니다. 컨텍스트는 판단 재료를 고르고, `Harness`는 그 판단이 행동으로 이어지는 전체 루프를 다룹니다.

## Agent = Model + Harness

에이전트를 모델과 동일시하면 문제의 본질을 놓치게 됩니다. 모델은 텍스트를 생성하고 추론하는 엔진일 뿐입니다. 하지만 실제 에이전트의 행동 방식은 모델 바깥에서 훨씬 많이 결정됩니다. 무엇을 볼 수 있는지, 어떤 도구를 쓸 수 있는지, 어떤 권한은 막히는지, 어떤 검증을 통과해야 하는지, 실패하면 어떤 상태로 돌아가는지가 행동을 만듭니다.

> **Agent = Model + Harness**

모델이 힘의 원천이라면, `Harness`는 그 힘이 어디로 흐를지 정하는 장치입니다. 원래 `harness`는 말을 다루는 마구를 뜻합니다. 힘이 센 말을 원하는 방향으로 움직이게 하려면 고삐, 안장, 재갈, 멈춤 지점이 필요합니다. LLM도 비슷합니다. 모델이 강해질수록 더 많은 일을 할 수 있지만, 그 힘이 곧바로 좋은 작업이 되지는 않습니다.

같은 모델을 써도 환경에 따라 결과는 크게 달라집니다. 어떤 환경에서는 에이전트가 관련 파일을 읽고, 계획을 세우고, 테스트를 통과한 뒤 보고합니다. 다른 환경에서는 엉뚱한 파일만 만지작거리다가 아무 말이나 뱉고는 자신 있게 완료했다고 말합니다. 모델이 달라서가 아닙니다. 모델을 감싸는 환경이 다르기 때문입니다.

이 관점에서 좋은 AI 코딩 도구는 대화를 잘하는 도구에 그치지 않습니다. 에이전트가 오래 일해도 무너지지 않도록 작업 환경을 구성하는 도구입니다. 좋은 사용자는 모델에게 화려한 주문을 외우는 사람이 아니라, 모델이 일할 수 있는 환경을 자기 프로젝트에 맞게 다듬는 사람입니다.

## Harness의 구성요소

`Harness`를 실행 관점에서 보면 몇 가지 능력으로 나눌 수 있습니다. 먼저 `Context Engineering`이 있습니다. 모델이 봐야 할 정보와 버려야 할 정보를 고르는 일입니다. 컨텍스트는 많이 넣는다고 좋아지지 않습니다. 지금 판단에 필요한 신호를 남기고, 오래된 가정과 무관한 자료를 걷어내야 합니다. 필요할 때 필요한 파일을 찾게 하고, 끝난 도구 결과는 압축하거나 버리게 하며, 항상 읽을 기준과 필요할 때만 꺼낼 자료를 나누는 일도 여기에 들어갑니다.

다음은 `Tool Orchestration`입니다. 에이전트가 어떤 도구를 어떤 순서와 권한으로 호출할지 정하는 일입니다. 파일 읽기와 파일 삭제는 같은 종류의 행동이 아닙니다. 테스트 실행과 배포도 같은 위험도를 갖지 않습니다. 도구가 많아지는 순간 중요한 것은 연결 수가 아니라 호출 조건과 권한 경계입니다.

여기에 `State & Memory`가 결합됩니다. 컨텍스트는 상태 저장소가 아닙니다. 매 호출마다 모델에 주입되는 단기 기억에 불과합니다. 모델 호출 자체도 본질적으로 무상태(stateless)입니다. 대화창 안에만 상태를 방치하면 세션이 끝나는 순간 증발하거나, 대화가 길어질수록 컨텍스트를 오염시키는 잡음으로 변질됩니다. `State & Memory`는 상태를 세션 바깥으로 빼내는 설계입니다. 에이전트의 결정, 진행 상황, 실패 원인, 사용자의 선호를 구조화된 메모나 파일, Git 이력 같은 외부 아티팩트에 기록하고, 다음 실행 시 필요한 조각만 다시 컨텍스트로 불러옵니다.

핵심은 단순한 저장이 아니라 생명주기 관리입니다. 실행 중에는 임시로 기록을 남기되, 작업이 끝나면 중복을 덜어내고 유효한 정보만 장기 기억으로 승격시켜야 합니다. 반대로 낡은 정보는 과감히 폐기해야 합니다. 무엇을 남기고 버릴지 엄격하게 통제하지 않으면, 메모리는 곧 거대한 잡음 저장소로 전락합니다.

여기에 `Verification Loop`, `Error Recovery`, `Human-in-the-Loop Control`이 붙습니다. 결과를 테스트와 린트로 확인하고, 실패를 감지하면 재시도하거나 멈추며, 위험한 순간에는 사람의 승인을 받게 만드는 구조입니다. 결국 구성요소는 하나의 질문으로 모입니다. 이 루프를 어디까지 자동화하고, 어디서 멈추게 할 것인가.

## 구성요소, 책임, 도구를 분리한다

이 대목에서 흔히 개념의 혼선이 발생합니다. 구성요소, 책임, 도구를 한데 섞어버리면 결국 껍데기인 도구 목록만 남게 됩니다. 이 셋은 엄격히 분리해서 다뤄야 합니다.

구성요소는 하네스가 갖춰야 할 능력입니다. `Context Engineering`, `Tool Orchestration`, `State & Memory`, `Verification Loop` 같은 이름은 "무엇을 할 수 있는가"에 가깝습니다. 책임은 그 능력으로 무엇을 지킬지 정하는 운영 기준입니다. 도구는 그 책임을 실제 작업 환경에 고정하는 수단입니다.

운영 관점에서 보면 책임은 다섯 가지로 정리할 수 있습니다. 위험한 행동을 막는 `Guardrails`, 작업 범위와 기준선을 고정하는 `Specification`, 결과를 흔들리지 않게 걸러내는 `Verification`, 세션이 바뀌어도 상태가 사라지지 않게 하는 `State Management`, 루프가 어디서 흔들리는지 보이게 하는 `Observability`입니다.

이 책임들은 에이전트 루프의 각 지점에 붙습니다. `gather context`에서는 `Specification`과 `Context Engineering`이 기준을 잡습니다. 무엇을 읽어야 하는지, 무엇은 지금 작업과 상관없는지 정해야 합니다. `take action`에서는 `Guardrails`와 `Tool Orchestration`이 붙습니다. 어떤 도구를 열고, 어떤 명령은 막고, 어떤 행동은 승인 뒤에만 허용할지 정합니다. `verify`에서는 `Verification`이 필요합니다. "잘 확인해줘"라는 말보다 실제 `lint`, `type check`, 테스트, 리뷰 게이트가 덜 흔들립니다. `repeat`에서는 `State Management`와 `Observability`가 중요해집니다. 무엇이 결정되었고, 무엇이 실패했고, 다음 루프에서 무엇을 봐야 하는지 남겨야 합니다.

루프를 통제하기 시작하면 개발자의 임무도 완전히 재정의됩니다. 더 이상 프롬프트를 깎으며 에이전트를 어르고 달래는 사람이 아닙니다. 에이전트가 안전하게 질주할 궤도를 직접 설계해야 합니다. 어느 파일을 반드시 읽게 할지, 어떤 규칙은 우회할 수 없게 막을지, 실패한 테스트 앞에서 어디까지 재시도를 허용할지, 어느 구간에서 사람의 승인을 강제할지 엄격하게 지정해야 합니다.

이것이 00장에서 말한 역할 변화와 이어집니다. 직접 타이핑하는 비중은 줄 수 있습니다. 하지만 기준을 세우고, 컨텍스트를 고르고, 검증 게이트를 만들고, 상태를 남기는 책임은 더 커집니다. `Harness`는 개발자의 일을 없애는 구조가 아니라, 개발자의 판단을 실행 환경 안으로 옮기는 구조입니다.

## 도구는 책임을 작업 환경에 고정한다

여기서 쉽게 빠지는 두 길이 있습니다. 하나는 모든 것을 손으로 기억하고 매번 직접 조작하는 길입니다. 명령어를 매번 다시 조합하고, 검증 루프를 기억에 의존하고, 세션마다 같은 지시를 다시 입력합니다. 초반에는 통제하는 느낌이 들지만 오래 유지되지 않습니다. 기억은 새고, 기준은 흔들립니다.

다른 하나는 원리를 모른 채 `Skills`, `Hooks`, `MCP`, `Plugins`, preset pack만 주워 붙이는 길입니다. 겉보기에는 편하지만 이상 동작이 나오면 어디가 문제인지 읽지 못합니다. 시간이 지나면 팀에는 원리가 아니라 블랙박스만 남습니다.

그래서 도구 이름보다 책임이 먼저입니다. `CLAUDE.md`나 `AGENTS.md`는 프로젝트의 기준선을 고정합니다. 매 세션 반복해서 설명할 필요가 없는 규칙, 문서 어투, 금지 명령, 테스트 방식, 리뷰 기준을 담을 수 있습니다. 이 파일들은 `Specification`을 맡고, 상황에 따라 약한 `Guardrails`도 됩니다.

`Hooks`는 특정 이벤트에 반응해 위험한 명령을 막거나 테스트를 자동 실행하게 만들 수 있습니다. 이 경우 `Hooks`는 `Guardrails`이면서 `Verification`입니다. `Git Worktree`는 작업을 격리하고 상태를 나눕니다. 그래서 `State Management`와 위험 격리에 기여합니다. `MCP`와 `Plugins`는 외부 도구와 데이터를 연결합니다. 하지만 연결이 많다고 좋은 `Harness`가 되는 것은 아닙니다. 어떤 책임을 맡길지 모르면 도구는 곧 잡음이 됩니다.

도구와 책임은 1:1로 대응하지 않습니다. 하나의 도구가 여러 책임을 맡을 수 있고, 하나의 책임도 여러 도구로 구현될 수 있습니다. 그래서 설계 순서는 반대여야 합니다. 먼저 어떤 책임이 비어 있는지 봅니다. 그다음 그 책임을 어떤 파일, 명령어, 게이트, 기록에 배치할지 정합니다.

예를 들어 "테스트를 반드시 통과해야 한다"는 말은 도구 이름이 아니라 `Verification` 책임입니다. 구현은 `pytest`, CI, `PostToolUse` Hook, PR 체크, 리뷰 체크리스트가 나눠 맡을 수 있습니다. "위험한 명령은 막아야 한다"는 말은 `Guardrails` 책임입니다. 구현은 승인 규칙, sandbox, `PreToolUse` Hook, 권한 분리로 나뉩니다. "작업 기준이 세션마다 흔들리면 안 된다"는 말은 `Specification`과 `State Management`의 책임입니다. 구현은 `AGENTS.md`, 이슈, 진행 파일, Git 이력이 맡습니다.

`Harness`를 잘 쓴다는 것은 멋진 도구 목록을 많이 갖는다는 뜻이 아닙니다. 필요한 파일, 필요한 규칙, 필요한 검증, 필요한 권한만 남기는 일입니다. 어디서 손을 막고, 어디서 결과를 걸러내고, 무엇을 세션 밖에 남길지 분명할수록 에이전트는 오래 버팁니다.

결국 `Harness`는 환경입니다. 모델이 똑똑해지는 속도만 기다리는 대신, 그 모델이 우리 작업 안에서 안전하게 반복할 수 있는 환경을 만드는 일입니다. 다음 장에서는 이 환경이 비어 있을 때 루프가 어떤 방식으로 무너지는지 보겠습니다.
