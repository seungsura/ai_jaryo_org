# 왜 Claude Code인가, 그리고 코딩 도구는 왜 Harness로 수렴하는가

## 시작하며: 왜 하필 Claude Code인가

이 글의 출발점은 “왜 `Claude Code`인가”라는 질문입니다. 우리가 수많은 도구 중 `Claude Code`를 선정한 이유는 최근 Anthropic이 보여주는 진화의 방향성이 코딩 에이전트의 발전 과정에서 거의 정답에 가까운 모습을 보여주고 있기 때문입니다. 특히 `Harness`의 관점에서 볼 때 가장 생태계가 탄탄하게 구성되어 있고 대중적인 인지도 역시 높아, 현재 AI 코딩 도구들이 나아가는 방향을 가장 잘 보여주는 대표적인 사례입니다.

물론 단순히 특정 제품의 훌륭한 기능들을 늘어놓으려는 것은 아닙니다. 우리는 단일 제품의 리뷰를 넘어, 왜 최근의 코딩 도구들이 한 방향으로 진화하고 있는지 그 거대한 흐름에 주목해야 합니다. IDE 플러그인, 워크스페이스, 터미널 CLI 기반, 채팅형 UI 등 최신 코딩 도구의 출발점은 저마다 달랐습니다. 그러나 긴 호흡으로 관찰해 보면 거의 모든 최신 코딩 도구는 더 방대한 `Context`를 읽어 들이고, 스스로 툴을 사용하며(tool use), 더 세밀한 권한 제어와 검증 루프를 필연적으로 요구하는 방향으로 변모하고 있습니다. 도구 간의 진정한 경쟁은 단일 모델의 성능을 넘어, `Harness`라는 실행 체계를 누가 더 견고하게 구성하느냐로 이동하고 있습니다. 이러한 관점에서 지난 4년간(2022~2026)의 코딩 도구 진화 궤적을 연대기순으로 짧게 요약해 볼 수 있습니다.

### 코딩 에이전트 패러다임 진화 연대기 (2022-2026)
| 시대 구분 | 시기 | 핵심 질문 | 대표 도구 | 주안점 |
|---|---|---|---|---|
| **Prompt Engineering** | 2022~2024 | "어떤 말을 해야 하나?" | ChatGPT, 초기 Copilot | 자연어 지시문의 품질과 기법 (CoT 등) |
| **Context Engineering** | 2025 중반~ | "어떤 정보를 넣어야 하나?" | Cursor Composer, RAG 파이프라인, KV-cache 최적화 | 전체 코드베이스 및 구조화된 정보의 적절한 주입 |
| **Harness Engineering** | 2026~ | "어떤 시스템을 만들어야 하나?" | Claude Code, Copilot Coding Agent | 권한 제어, 피드백 루프, 검증 자동화, 서브에이전트 |

각 시대가 왜 한계에 부딪혔으며 어떻게 다음 패러다임으로 진화했는지 자세히 살펴보겠습니다.

## GitHub Copilot, 프롬프트(Prompt)의 시대

이러한 흐름의 시작에는 2022년 세상에 등장한 `GitHub Copilot`이 있었습니다. `OpenAI Codex`를 엔진으로 삼은 초창기 `Copilot`은 현재 열려 있는 파일을 바탕으로 다음 코드를 추천해 주는 '스마트한 자동완성'에 가까웠습니다. 이 도구는 “코드는 인간이 쓰고 AI가 거들뿐”이라는 인식을 뒤흔들기 시작했지만, 이 자체만으로는 '어떻게 지시할 것인가'를 고민할 필요가 없는 구조였습니다.

진정한 프롬프트(Prompt)의 시대는 불과 5개월 뒤 공개된 `ChatGPT`와 함께 폭발합니다. 단순한 자동완성을 넘어 기계와 자연어로 대화하며 코드를 생성하고 구조를 짤 수 있게 되면서, 개발자들의 최대 질문은 "이것에게 어떻게 말해야(지시해야) 내가 원하는 코드를 얻어낼 수 있는가"로 바뀌었습니다. 모델이 의도를 정확히 이해하도록 명확한 지시문, 상세한 설명, 양질의 예제를 설계하는 것이 코드의 품질과 생산성을 좌우하는 핵심 역량이 되었습니다.

학계와 업계에서는 한계를 뛰어넘기 위한 수많은 '프롬프트 엔지니어링' 기법을 탄생시켰습니다. 2022년 1월 Google Brain 팀이 발표한 `CoT(Chain-of-Thought)`는 프롬프트에 중간 추론 단계를 예시로 포함시켜 모델이 단계별로 사고하도록 유도하는 기법으로, 수학 추론 기준 수십 퍼센트 포인트의 성능 도약을 이끌었습니다. 같은 해 10월에는 추론(Thought)과 외부 도구 사용(Action)을 번갈아 수행하도록 설계해 현대 에이전트의 원형이 된 `ReAct`가 등장했고, 2023년에는 일직선 추론을 넘어 여러 추론 경로를 트리 형태로 탐색하고 막다른 길에서 되돌아오는(backtracking) `Tree-of-Thought`가 제안되었습니다(단, 하나의 문제에 수십 번의 API 호출이 필요해 비용이 폭발하는 부작용이 따랐습니다). 이 프롬프트 시대의 정점에서는 2024년 3월 Andrew Ng이 `Reflection`, `Tool Use`, `Planning`, `Multi-Agent Collaboration`이라는 4가지 에이전틱 패턴을 체계화하며, "GPT-3.5에 에이전틱 워크플로우를 씌우면 GPT-4 제로샷보다 낫다"는 발견을 통해 점차 모델 그 자체보다 모델을 감싸는 '시스템'의 질이 성패를 가름할 것임을 예고했습니다.

## Cursor, 컨텍스트(Context)의 시대

하지만 프롬프트의 한계는 명확했습니다. 제아무리 좋은 지시를 내려도 AI 모델이 프로젝트 내 다른 위치에 있는 파일이나 설계 철학을 알지 못하면 환각에 가까운 코드를 쏟아냈기 때문입니다. 2023년 3월 등장한 `Cursor`는 이 문제를 가장 먼저 정면으로 다뤘습니다. VS Code를 기반으로 전체 코드베이스를 인덱싱하고 시맨틱 검색으로 관련 파일을 자동으로 끌어오는 방식을 택함으로써, AI 코딩 도구가 '현재 파일 한 장'에서 '프로젝트 전체'로 시각을 확장하는 전환의 교두보 역할을 했습니다. 같은 시기 `Copilot Chat`(2023.11)도 대화형 코드 탐색과 편집을 결합하며 같은 방향으로 이동했습니다.

컨텍스트 시대가 개념적으로 공식화된 것은 2025년 6월입니다. Shopify CEO Tobi Lütke가 "'프롬프트 엔지니어링'보다 '컨텍스트 엔지니어링'이 핵심 역량을 더 잘 설명한다"고 선언하자, Karpathy가 "컨텍스트 윈도우에 다음 단계에 딱 필요한 정보를 채우는 섬세한 기술"이라 화답하면서 업계 어휘가 단번에 전환됩니다. 이 시점부터 `Cursor Composer`, RAG 파이프라인, KV-cache 최적화 등 '무엇을 넣을 것인가'를 정밀하게 설계하는 기법들이 중심 무대에 섰습니다. 그러나 이 시점에도 중심 목표가 여전히 "어떤 정보를 잘 수집해서 보여 줄 것인가"에 머물러 있었다는 점에서, 연속적이고 긴 작업을 통제하는 단단한 구조는 여전히 부재했습니다.

## 에이전트 루프와 Tool Use

진짜 변화와 문제는 에이전트 도구가 스스로 무언가를 ‘실행(Action)’하기 시작하면서 표면화되었습니다. 거의 모든 에이전트 기반 코딩 도구는 `Context 수집 → Action 실행 → 결과 Verify(검증) → 반복`이라는 루프를 쉼 없이 돕니다. 자료를 수집하고 파일을 수정한 뒤, CLI에서 명령을 실행하고, 테스트 결과와 리뷰를 바탕으로 궤도를 이탈하지 않기 위해 안간힘을 씁니다.

루프의 각 단계마다 도사리는 위험 요소 역시 명확합니다. 쓰레기 데이터가 섞이면 소셜 노이즈가 발생하고, 무분별한 실행 권한을 주면 치명적 오류를 발생시킬 리스크가 커집니다. 검증이 약하면 틀린 방향으로 질주하는 것을 막을 수 없습니다. 도구를 제대로 활용하는 문제는 단순한 모델 지능 테스트가 아닙니다. 에이전트의 런타임 루프를 어떤 규칙과 권한 아래에 둘 것인가의 문제입니다. 이 순간이야말로 견고한 `Harness`가 필수불가결한 존재가 되는 시점입니다.

## Claude Code가 대표 사례가 된 이유

이런 흐름 속에서 `Claude Code`가 빈번히 예시로 등장하는 이유는 단순히 퍼포먼스 때문만은 아닙니다. `Claude Code`는 터미널 네이티브 워크플로우(terminal-native workflow), `Plan Mode`, 자동/수동 승인 체계, `CLAUDE.md`, `Skills`, `MCP`, 다양한 플러그인과 `Subagents`까지, `Harness`를 구축하는 요소들을 가장 노골적으로 표면 위에 드러내는 도구이기 때문입니다. “좋은 에이전트란 결국 어떤 환경 위에서 구동해야 하는가”라는 질문에 가장 직관적인 답을 던지고 있습니다.

파일을 분석하고 셸(shell)을 실행하며 결과를 뜯어 살펴보고 궤도를 수정하는 복합적인 전체 흐름을 하나로 묶음으로써, 단순한 기능 확장 보조 도구의 딱지를 떼고 독립적이고 완결된 '엔지니어링 코-워커 시스템'으로 나아갔음을 보여줍니다. 나아가 이러한 진화의 양상은 단순한 우연이 아닙니다. `Codex`와 `OpenCode` 등 주요 이니셔티브들 역시 비슷한 궤도를 그리고 있다는 점이 산업 전체가 `Harness` 중심으로 수렴하고 있음을 재차 증명합니다.

## 결론

결론적으로 최근의 코딩 도구 간 경쟁은 특정 도구 한두 가지의 우열 논쟁으로 귀결되지 않습니다. `GitHub Copilot`이 첫 삽을 뜬 이후, 현대의 코딩 도구는 어떻게 하면 자신들만의 `Harness`를 더 두껍고 단단하게 쌓아 올릴 것인지의 방향으로 치열하게 수렴하는 중입니다. 

이제 우리가 살펴봐야 할 렌즈는 명징합니다. “어떤 AI가 코드를 더 훌륭하게 쓰는가?”를 넘어서기 위해 “어떤 도구가 나의 개발 문맥, 권한, 검증 루프와 워크플로우를 아우르는 강력한 Harness 구조를 제공하는가”를 핵심 잣대로 삼아야 할 때입니다.
