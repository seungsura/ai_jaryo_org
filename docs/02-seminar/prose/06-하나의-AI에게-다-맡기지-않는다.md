# 06. 하나의 AI에게 다 맡기지 않는다

## 06-1. 병렬화보다 먼저 분해다

하나의 창에서 계획하고, 탐색하고, 구현하고, 테스트하고, 리뷰까지 끝내면 편해 보입니다. 하지만 긴 작업에서는 이 편함이 금방 약점이 됩니다. 컨텍스트가 뒤섞이고, 역할이 섞이고, 자기 결과를 자기가 검증하는 구조가 됩니다.

멀티 에이전트의 이점은 수를 늘리는 데 있지 않습니다. 역할을 쪼개고, 각 작업의 맥락을 분리하고, 검증 책임을 나눌 때 생깁니다. Anthropic의 멀티 에이전트 연구도 별도 컨텍스트와 역할 분해를 중요한 장점으로 봅니다.[^06-anthropic-multi] 다만 비용도 커질 수 있으니, 기본값은 많이 붙이는 것이 아니라 필요할 때만 나누는 것입니다.

## 06-2. 역할은 정보, 행동, 기준이 달라야 산다

이름만 `planner`, `builder`, `reviewer`로 나눈다고 역할이 나뉘는 것은 아닙니다. 같은 정보를 보고, 같은 권한을 갖고, 같은 기준으로 움직이면 결국 한 에이전트를 여러 번 부른 것과 크게 다르지 않습니다.

의미 있는 역할 분리는 세 가지가 달라야 합니다. 보는 정보, 할 수 있는 행동, 통과해야 하는 기준입니다. 조사자는 읽고 요약해야 합니다. 구현자는 승인된 계획 안에서 파일을 바꿔야 합니다. 리뷰어는 구현자의 의도를 그대로 믿지 않고 실패 조건과 수용 기준을 확인해야 합니다.

이렇게 나누면 더 많은 AI가 아니라 더 선명한 팀이 됩니다.

## 06-3. Sub-Agent는 중간 작업을 덜어주는 필터다

실무에서 가장 자주 쓰는 패턴은 `Sub-Agent`입니다. 메인은 큰 맥락을 유지하고, 서브 세션은 한 가지 목적의 탐색, 로그 분석, 테스트 원인 추적 같은 좁은 작업만 맡습니다. 작업이 끝나면 결론과 근거만 되돌아옵니다.

중요한 것은 컨텍스트 오염을 줄이는 것입니다. `separate context windows`에서 결과를 압축해 받는 구조라면, 메인 컨텍스트는 모든 탐색 흔적을 들고 있지 않아도 됩니다.[^06-claude-subagent]

최소 권한도 같이 봐야 합니다. 전체 저장소와 전체 대화 기록을 한 번에 넘기기보다, 필요한 파일, 질문, 출력 형식만 전달하는 편이 좋습니다.

## 06-4. Orchestrator는 병렬이 아니라 수렴을 설계한다

범위가 커지면 `Orchestrator`가 필요해집니다. 하나의 큰 목표를 하위 작업으로 나누고, 소유 범위를 정하고, 결과를 다시 합쳐야 하기 때문입니다.

이때 Orchestrator의 품질은 실행자 수가 아니라 분해의 품질에서 나옵니다. 경계가 흐리면 같은 파일을 서로 건드리고, 나중에 통합 비용이 커집니다. 반대로 경계가 선명하면 각 실행자는 좁은 문제를 처리하고, 메인은 판단과 통합에 집중할 수 있습니다.

## 06-5. 생성자와 검증자는 같은 손이 아니다

같은 주체가 스스로 계획하고, 구현하고, 리뷰하고, 승인하면 자기 블라인드 스팟을 놓치기 쉽습니다. 중요한 작업에서는 생성과 평가를 분리해야 합니다. 한쪽은 실제 구현을 맡고, 다른 쪽은 다시 작업할 수 있는 피드백을 줘야 합니다.

좋은 피드백은 감상평이 아닙니다. "괜찮다"가 아니라 "이 필드는 비어 있고, 이 케이스는 빠져 있습니다"처럼 다음 행동을 바꾸는 문장이어야 합니다.

## 06-6. Agent Team은 기본값이 아니라 선택이다

`Sub-Agent`와 `Agent Team`은 목적이 다릅니다. 대부분의 탐색, 로그 정리, 단일 검증은 `Sub-Agent`로 충분합니다. 반대로 관점 충돌이 큰 설계 토론이나 전략 문서처럼 여러 시야의 충돌이 품질을 좌우하는 영역에서는 `Agent Team`이 의미가 있습니다.

그래서 기본값은 작게 두는 편이 좋습니다. 팀처럼 보이는 구조를 만들기 전에, 병렬화가 실제로 값을 만드는지 먼저 봐야 합니다.

## 06-7. 경계가 패턴보다 먼저다

패턴 이름보다 먼저 지켜야 할 것은 경계입니다. 어느 작업에 어떤 정보가 들어가고, 어떤 권한이 열리고, 언제 멈추는지, 어떤 실패 조건을 가지는지가 분명해야 합니다.

`SOLID`를 굳이 가져온다면 단일 책임의 확장판으로 볼 수 있습니다. 하나의 에이전트가 모든 역할을 갖지 않게 나누는 것입니다. 인터페이스 분리도 코드에만 해당하지 않습니다. 도구와 컨텍스트 접근권에도 같은 원리가 적용됩니다.

역할 분해의 목적은 AI를 많이 쓰는 것이 아닙니다. AI가 잘 맞는 지점에 오래 머물게 만드는 일입니다.
