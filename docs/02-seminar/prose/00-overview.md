# 개요

## 00. 이 문서가 풀려는 질문

이 문서는 특정 AI coding tool의 사용기를 정리하는 글이 아니다. 더 앞에 놓인 질문은 코딩이라는 행위 자체가 어디로 이동하고 있는가, 그리고 그 이동 속에서 개발자의 핵심 경쟁력이 무엇으로 다시 정의되고 있는가다. `Claude Code`, `Codex`, `OpenCode`, `Copilot`, `Cursor`를 하나의 계보 안에 놓고 보면, 핵심 차이는 단일 모델의 똑똑함보다 모델이 일하는 환경이 얼마나 잘 설계되어 있는가로 점점 옮겨 간다.

## 01. 중심 thesis: Prompt에서 Harness로

이 문서의 중심 thesis는 분명하다. AI coding tool의 진화는 `Prompt → Context → Harness`로 이동해 왔다. 처음에는 모델에게 무엇을 말할지가 중요해 보였다. 곧이어 무엇을 보여 줄지, 즉 코드베이스와 문서와 상태를 어떤 방식으로 제공할지가 중요해졌다. 이제는 그것만으로도 부족하다. 어떤 도구를 허용할지, 어떤 규칙을 강제할지, 언제 멈추고 무엇을 검증할지, 긴 작업의 상태를 어디에 남길지까지 포함한 전체 실행 환경이 성능과 신뢰성을 가른다.

## 02. 왜 여러 도구를 함께 보는가

이 관점에서 보면 특정 vendor의 기능 목록만 비교하는 방식은 큰 흐름을 놓치기 쉽다. `GitHub Copilot`은 프롬프트 시대의 상징으로 시작했고, `Cursor` 같은 도구는 codebase-level context를 전면화했으며, `Claude Code`, `Codex`, `OpenCode`는 도구 사용, 권한 관리, 검증 루프, `Sub-Agent`, background execution, memory 같은 하네스 요소를 더 두껍게 쌓는 방향으로 이동하고 있다. 이름은 달라도 거의 모든 modern coding tool이 결국 더 많은 `Harness`를 품는 방향으로 수렴한다는 점이 이 문서의 중요한 전제다.

## 03. 문서의 구성

전반부는 추상화의 역사와 tool genealogy를 통해 이 이동을 위치시킨다. 중반부는 `spec-first`, `TDD`, `SDD`, `Harness Engineering`, long context failure, `Multi-Agent` 같은 방법론과 운영 원리를 설명한다. 후반부는 실전 workflow와 이 발표 자체의 제작 사례를 통해, 좋은 하네스가 실제로 어떻게 산출물 품질을 바꾸는지를 보여 준다. 마지막에는 최신 트렌드와 함께, 개발자가 지금 무엇을 외재화하고 무엇을 검증해야 하는지로 결론을 닫는다.

## 04. 이 문서를 읽는 방식

이 문서는 입문 독자에게는 AI coding tool의 큰 흐름을 설명하는 글이어야 하고, 실무 독자에게는 즉시 적용 가능한 운영 원리와 방법론을 남기는 글이어야 한다. 그래서 개념 설명과 실무 밀도를 동시에 유지한다. `Claude Code`는 로컬 자료가 가장 풍부한 대표 사례로 전면에 두지만, 본문 전체의 논지는 특정 제품의 우열보다 `Harness`를 중심에 둔 개발론으로 읽히게 정리한다.

## 05. 한 문장 결론

이 문서가 반복해서 남기려는 메시지는 하나다. 이제 중요한 것은 코드를 직접 얼마나 많이 타이핑하느냐보다, 원하는 결과가 반복 가능하게 나오도록 문맥과 규칙과 검증 루프를 얼마나 선명하게 설계하느냐다. 이후 각 문서는 이 흐름에 맞춰 발표의 아홉 개 섹션을 다시 정렬한다.
