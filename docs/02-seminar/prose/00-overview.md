# Harness를 설계하는 법

> 챗봇과 씨름하는 대신, 에이전트가 일할 구조를 짜는 방법

## 시작하며: 개발의 중심축이 옮겨가고 있다

AI 코딩 도구는 하루가 다르게 좋아집니다. 그런데 진짜 변화는 모델이 더 똑똑해졌다는 사실보다, 개발자가 붙들어야 할 일이 바뀌고 있다는 데 있습니다. 손으로 코드를 밀어 넣는 시간은 줄고, 규칙을 정하고 문맥을 고르고 결과를 검증하는 시간이 빠르게 늘고 있습니다.

이 글은 그 이동을 설명하려고 씁니다. 개발자의 일이 어디로 옮겨 가는지, 무엇을 새로 배워야 하는지, 무엇을 더 엄격하게 통제해야 하는지를 한 흐름으로 묶어 보겠습니다.

## 핵심 메시지: 코딩보다 운영 구조가 앞선다

지금 AI 엔지니어링은 점점 하나의 방향으로 모입니다. `Harness`입니다. 출발점은 제각각이었습니다. 어떤 도구는 자동완성에서 시작했고, 어떤 도구는 채팅형 UI에서 시작했고, 어떤 도구는 터미널 에이전트로 출발했습니다. 하지만 긴 작업을 맡기기 시작하는 순간, 결국 다 같은 문제를 만납니다. 무엇을 읽게 할 것인가, 무엇을 못 하게 막을 것인가, 어디서 검증할 것인가, 상태를 어떻게 남길 것인가. 이 질문에 답하는 구조가 바로 `Harness`입니다.

처음에는 프롬프트가 전부처럼 보였습니다. 곧 컨텍스트가 더 중요해 보였습니다. 그러나 실제로 오래 굴려 보면 프롬프트와 컨텍스트만으로는 버티지 못합니다. 어떤 도구를 열어 둘지, 어떤 규칙을 상시로 강제할지, 세션이 길어질 때 무엇을 파일로 남길지, 실패를 어디서 끊을지 같은 바깥 설계가 결과의 품질을 갈라놓습니다. 경쟁력은 더 빨리 타이핑하는 데서 나오지 않습니다. 원하는 결과를 반복 가능하고 안정적으로 만들 운영 구조를 짜는 데서 나옵니다.

## 앞으로 다룰 이야기

이 글은 특정 제품을 치켜세우는 도구 리뷰가 아닙니다. 왜 최근의 AI 코딩 도구와 방법론이 `Harness` 중심으로 수렴하는지, 그 구조가 실무에서 어떻게 작동하는지 따라가려는 글입니다.

1. 코딩이라는 행위가 어디로 이동하고 있는지, 그리고 왜 최신 도구들이 `Harness` 중심 구조로 모이는지 먼저 봅니다.
2. `spec-first`, `TDD`, `SDD`, `Harness Engineering`, long context failure, `Multi-Agent`, 실전 workflow 같은 운영 원리를 차례로 정리합니다.
3. 마지막으로 이 원리들이 실제 산출물 제작 과정에서 어떻게 고정되는지 사례와 실천 항목으로 묶습니다.

핵심은 단순합니다. AI 시대의 경쟁력은 더 좋은 모델을 고르는 데서 끝나지 않습니다. 더 나은 `Harness`를 설계하고 운영하는 쪽으로 이동하고 있습니다. 이제 그 이유와 방법을 차례대로 보겠습니다.
