## 현재 상태

- `docs/02-seminar/prose`는 이 `target-map.md`만 남긴 상태에서 새 원고를 다시 작성한다.
- 기존 prose 원고와 1:1로 대응시키지 않는다.

- speaker notes는 이번 prose 재작성 범위에서 제외한다.
- 이 파일은 이후 대화와 작업의 handoff memory로 사용한다. context가 줄어들거나 세션이 바뀌어도 이 파일을 먼저 읽고 현재 논지, 판단, 작성 규칙을 복원한다.

## Context Handoff Memory

- 사용자는 이번 원고를 회사 동료에게만 닫힌 내부 발표로 만들고 싶지 않다. 다만 출발점은 AI 활용이 막막한 개발자, 특히 AI agent를 순정 상태로 쓰다가 실망하거나 검증 없이 결과를 넘기는 개발자의 경험이다.
- 사용자가 말하고 싶은 것은 `AI 이렇게 사용하면 망한다`, `AI 이렇게 사용해야 한다`이다. 하지만 단순한 경고나 도구 사용법이 아니라, AI 시대에 개발자가 일하는 방식을 바꾸는 이야기로 확장되어야 한다.
- 현재 핵심 문장은 `AI를 잘 쓰는 개발자는 멋진 프롬프트를 쓰는 사람이 아니라, AI가 일할 환경을 설계하는 사람이다`이다.
- `Harness의 정의`가 목적이 아니다. 목적은 `AI를 실무에서 망하지 않게 쓰는 법`이고, Harness는 그 답이다.
- 사용자는 모든 주제를 말하고 싶어 한다. 이때 범위를 억지로 줄이는 것이 아니라, A -> B -> C 흐름으로 전체를 꿰어야 한다.
  - A: Claude Code 같은 AI agent를 실무에서 안정적으로 쓰는 법
  - B: AI 시대에 개발자가 일하는 방식을 바꾸는 법
  - C: 팀이 AI를 개인 요령이 아니라 반복 가능한 구조로 쓰는 법
- 청중 감정의 순서는 다음과 같다.
  1. `내가 AI를 못 쓰는 게 아니라, 잘못된 방식으로 쓰고 있었구나.`
  2. `그런데 이 방식 전환을 안 하면 진짜 뒤처지겠구나.`
  3. `AI가 위험한 게 아니라, 검증 없이 쓰는 방식이 위험한 거구나.`
  4. `복잡해 보여도 결국 내 작업 환경에 구조를 몇 개 만들면 되는구나.`
- 사용자는 무조건적인 동조를 원하지 않는다. 이후 답변과 편집 판단에서는 범위가 흐려질 때, 기능 소개로 흩어질 때, 설명문이 안전하게 죽을 때 적극적으로 의문을 제기한다.
- 사용자와 대화할 때도, prose markdown을 쓸 때도 번역 말투를 피하고 유창한 한국어 구어체를 우선한다. `칼끝 후보`처럼 영어식 은유를 직역한 듯한 표현은 쓰지 않는다.
- 00장부터 다시 본다. 00장은 역사 강의가 아니라 발표 전체의 목적지를 먼저 박고, 추상화 역사를 그 목적지의 근거로 쓰는 장이어야 한다.

## 이번 재설계의 핵심

- 제목은 `Harness 잘 사용하기`, 부제는 `챗봇과 싸우지 않는 방법`으로 둔다.
- 이 원고는 특정 회사 내부 사례에만 갇히지 않는다. 다만 출발점은 AI 활용이 막막한 개발자의 실제 경험이다.
- 중심 문장은 `AI를 잘 쓰는 개발자는 멋진 프롬프트를 쓰는 사람이 아니라, AI가 일할 환경을 설계하는 사람이다`로 둔다.
- 진짜 말하고 싶은 것은 `Harness의 정의` 자체가 아니라 `AI를 실무에서 망하지 않게 쓰는 법`이다.
- `Harness`는 답이고, 문제의 출발점은 개발자가 느끼는 막막함과 잘못된 사용 습관이다.
- 발표의 적은 AI 자체가 아니다. 구조 없이 순정 AI를 채팅창처럼 쓰면서 실무 결과를 기대하는 방식이다.
- `잘 사용하기`는 A -> B -> C 흐름으로 전개한다.
  - A: Claude Code 같은 AI agent를 실무에서 안정적으로 쓰는 법
  - B: AI 시대에 개발자가 일하는 방식을 바꾸는 법
  - C: 팀이 AI를 개인 요령이 아니라 반복 가능한 구조로 쓰는 법
- `Prompt -> Context -> Harness` 흐름은 제거하지 않는다. 이 흐름은 용어 변천사가 아니라 통제 범위가 넓어지는 과정으로 설명한다.
- `기계어 -> 어셈블리 -> C -> Java -> Python` 흐름도 제거하지 않는다. 다만 본론이 아니라, 개발자가 늘 추상화에 저항했고 결국 작업 단위가 이동했다는 도입 장치로 압축한다.
- 역사와 미국 테크 기업 사례는 배경이다. 본론은 `막막함을 행동 가능한 구조로 바꾸는 방법`이다.

## Slide-First 작성 규칙

- 이제 원고는 slide 기준으로 작성한다. prose chapter를 먼저 길게 쓰고 나중에 slide로 압축하는 방식이 아니라, 각 slide가 무엇을 보여주고 무엇을 말할지 먼저 고정한다.
- 각 slide draft는 반드시 두 층으로 쓴다.
  - `슬라이드에 적을 내용`: 실제 화면에 들어갈 짧은 문구, 제목, 숫자, 도식, 코드 조각, 비교 축.
  - `이 슬라이드에서 할 말`: 발표자가 실제로 말할 내용. speaker notes처럼 과하게 잘게 쪼개지 않되, 그 slide에서 전달해야 할 주장과 연결 문장을 실제 발화에 가깝게 쓴다.
- slide 화면 문구는 `assets/claude-code-seminar-kakao/page-001.png`부터 `page-006.png`처럼 간결하고 크게 읽히는 구조를 기준으로 삼는다.
- slide에는 설명문을 길게 넣지 않는다. 한 slide에는 하나의 주장, 하나의 대비, 하나의 숫자, 하나의 장면만 둔다.
- 단, 00장의 `추상화의 역사: 직접 하던 일은 계속 줄어들었다`는 예외로 둔다. 이 구간은 이미 만들어 둔 자료 자체가 설득의 계단이므로, 한 소제목 안에서 여러 장면을 연속으로 사용한다.
- prose 문장은 slide의 `이 슬라이드에서 할 말`을 자연스러운 기술 블로그 에세이 문장으로 확장한 것이다. 따라서 prose와 slide는 따로 놀면 안 된다.
- 각 slide의 제목은 원문 소제목이나 현재 target-map의 장 역할과 연결되어야 한다. source에 없는 비교 축, label, 사례, 수치를 새로 만들지 않는다.
- 이미지만 흉내 내지 않는다. 카카오 deck의 장점은 슬라이드가 짧다는 것보다, 초반에 목적지를 먼저 박고 이후 모든 장을 그 목적지로 회수한다는 점이다.

## 카카오 목적지 문장 사용 판단

- 카카오 발표식 `이 89장의 슬라이드를 / 한 장도 직접 만들지 않은 방법 / 한 것은 하나 - 하네스 설계 / 이것이 오늘의 목적지` 구조는 그대로 쓰지 않는다.
- 그대로 쓰면 00장의 주제가 `이 발표를 어떻게 만들었는가`로 읽힌다. 그 이야기는 08장의 proof 장에 더 잘 맞는다.
- 대신 구조만 빌린다. 즉 `구체적 결과물 -> 한 일은 하나 -> 오늘의 목적지` 구조를 이번 발표의 문제의식에 맞게 바꾼다.
- 00장 목적지 slide 후보:
  - `AI에게 일을 맡기는 법이 아니다` / `AI가 망하지 않고 일하게 만드는 법` / `그것이 Harness다`
  - `멋진 프롬프트가 아니다` / `AI가 일할 환경을 설계하는 것` / `이것이 오늘의 목적지`
  - `챗봇과 싸우지 않는 방법` / `더 세게 말하는 것이 아니라` / `일할 환경을 바꾸는 것`
- 사용자가 제안한 목적지 문장 후보:
  - `키보드 전사가 되지 않고 AI와 함께 일하는 방법` / `그것이 Harness`
  - `이 00장의 슬라이드를 한 장도 직접 만들지 않은 방법` / `한 것은 하나 - 하네스 설계` / `이것이 오늘의 목적지`
- 위 후보는 의미가 강하지만 그대로 쓰면 두 위험이 있다.
  - `키보드 전사`는 기억에 남지만 약간 밈처럼 튈 수 있으므로, 발표 톤에서 허세나 조롱처럼 들리지 않는지 확인해야 한다.
  - `이 00장의 슬라이드`는 표현이 좁다. 전체 발표의 proof hook으로 쓰려면 `이 발표의 슬라이드` 또는 `이 발표 자료`가 더 자연스럽다.
- 2026-04-25 사용자 판단: `키보드 전사`는 slide에 적기에는 애매하다. slide 문구에서는 제거하고, 필요하면 `키보드 앞에서 모든 것을 직접 붙들고 버티는 방식`처럼 본문 발화에서만 낮게 풀어 쓴다.
- 현재 가장 유력한 절충안:
  - slide: `챗봇과 싸우지 않고` / `AI와 함께 일하는 방법` / `그것이 Harness`
  - sub line: `이 발표 자료 또한 전부 Harness로 만들었습니다`
  - talk: `이 발표도 AI에게 통째로 맡긴 것이 아니라, source와 규칙과 검증을 먼저 세우고 그 안에서 AI가 일하게 만든 결과입니다. 챗봇에게 더 길게 말하고, 키보드 앞에서 모든 것을 직접 붙드는 방식에서 벗어나는 것이 출발점입니다.`
- 08장에서는 카카오 문장과 더 가까운 자기참조 구조를 사용할 수 있다. 예: `이 발표가 만들어진 방식` / `한 것은 하나 - 하네스 설계` / `결과보다 과정을 보자`.

## 00장 Slide-First 초안

00장의 역할은 역사 설명이 아니라 목적지 선공개다. 청중이 처음 받아야 하는 감각은 `AI를 못 쓰는 것이 아니라, 아직 잘못된 방식으로 쓰고 있었구나`이다. 역사, 숫자, 문서가 코드가 되는 흐름은 이 목적지를 설득하는 근거로 뒤따라온다.

### 00-1. 오늘의 목적지

- 슬라이드에 적을 내용:
  - `챗봇과 싸우지 않고`
  - `AI와 함께 일하는 방법`
  - `그것이 Harness`
  - `이 발표 자료 또한 전부 Harness로 만들었습니다`
- 이 슬라이드에서 할 말:
  - 오늘 이야기할 것은 AI에게 일을 통째로 맡기는 법이 아니다.
  - 챗봇에게 더 길게 말하고, 더 세게 설득하고, 원하는 답이 나올 때까지 싸우는 법도 아니다.
  - 키보드 앞에서 모든 것을 직접 붙들고 버티는 방식도 아니다.
  - AI가 일할 환경을 설계하고, 그 안에서 함께 일하는 방법이다.
  - 이 발표 자료도 AI에게 던져서 한 번에 만든 것이 아니라, source를 정하고, 규칙을 세우고, 검증을 분리하고, 그 안에서 AI가 일하게 만든 결과다.
  - 그래서 오늘 말할 Harness는 추상적인 방법론이 아니라, 이 발표를 만드는 데도 실제로 쓴 작업 방식이다.
- 주의:
  - `키보드 전사`는 slide에 쓰지 않는다. 필요하면 본문 발화에서 `키보드 앞에서 모든 것을 직접 붙들고 버티는 방식` 정도로 낮게 푼다.
  - `AI가 만들었습니다`라는 표현만 단독으로 쓰지 않는다. 핵심은 AI가 아니라 Harness다.

### 00-2. 코딩은 사라지는가

- 슬라이드에 적을 내용:
  - `코딩은 사라지는가`
  - `질문을 바꿔야 한다`
  - `무엇을 직접 할 것인가 -> 무엇을 위임하고 통제할 것인가`
- 이 슬라이드에서 할 말:
  - AI 코딩 도구가 빠르게 좋아질수록 같은 질문이 나온다. 이제 코딩은 사라지는가.
  - 하지만 이 질문은 조금 빗나가 있다.
  - 사라지는 것은 개발이 아니라, 개발자가 직접 붙들던 작업 단위다.
  - 중요한 질문은 `내가 무엇을 직접 타이핑할 것인가`가 아니라, `무엇을 AI에게 위임하고, 무엇을 사람이 통제할 것인가`다.

### 00-3. 추상화의 역사: 직접 하던 일은 계속 줄어들었다

- 슬라이드에 적을 내용:
  - 이 구간은 한 장 요약으로 압축하지 않는다.
  - 이미 만들어 둔 `기계어 -> 어셈블리`, `어셈블리 -> C/Pascal`, `C -> Java`, `Java -> Python`, `AI 개발`, `추상화의 역사` 자료를 연속으로 활용한다.
  - 코드 예시와 반발 문구는 설득의 장치이므로 유지한다.
- 이 구간에서 할 말:
  - 개발자는 늘 새로운 추상화 앞에서 비슷하게 반발했다.
  - 하드웨어를 직접 다루지 않으면 개발자가 아니라고 했고, 메모리를 직접 관리하지 않으면 실력이 아니라고 했고, Python 같은 언어는 장난감이라고 했다.
  - 하지만 시간이 지나면 매번 같은 결론이 났다. 직접 하던 일은 줄었고, 더 큰 구조를 설계하는 일이 실력의 중심으로 올라왔다.
  - AI agent도 같은 흐름 위에 있다. 이번에도 핵심은 개발이 사라진다는 말이 아니라, 개발의 추상화 수준이 올라간다는 말이다.

### 00-4. 숫자로 보는 변화

- 슬라이드에 적을 내용:
  - `25%`
  - `95% AI-generated codebase`
  - `0 manually-written code`
  - `reviewed and accepted by engineers`
- 이 슬라이드에서 할 말:
  - 이 변화는 분위기나 예측만의 이야기가 아니다.
  - YC W25에서는 일부 스타트업이 코드베이스 대부분을 AI로 만들었다는 공개 발언이 나왔고, OpenAI의 Harness Engineering 사례에서는 5개월 동안 사람이 직접 코드를 쓰지 않고 agent가 작성한 코드로 내부 제품을 만들었다.
  - Google도 새 코드의 상당 부분이 AI로 생성되고 엔지니어가 검토해 받아들인다고 공개했다.
  - 다만 이 숫자는 `사람이 필요 없어졌다`는 근거가 아니다. 오히려 사람이 무엇을 해야 하는지가 바뀌었다는 근거다.
  - 생성은 빨라졌고, 그래서 방향 설정과 검증과 환경 설계가 더 중요해졌다.
- 주의:
  - `Meta 엔지니어 3개월` 사례는 공개 근거가 약하면 사용하지 않는다.
  - 숫자는 과시가 아니라 역할 이동의 증거로만 쓴다.
- 근거 후보:
  - YC W25 `25%`, `95% AI-generated codebase`: https://techcrunch.com/2025/03/06/a-quarter-of-startups-in-ycs-current-cohort-have-codebases-that-are-almost-entirely-ai-generated/
  - OpenAI Harness Engineering `0 lines of manually-written code`, `five months`, `million lines of code`: https://openai.com/index/harness-engineering/
  - Google `reviewed and accepted by engineers`: https://abc.xyz/2024-q3-earnings-call/

### 00-5. 문서가 코드다

- 슬라이드에 적을 내용:
  - `README가 아니라 실행 규칙`
  - `CLAUDE.md`
  - `AGENTS.md`
  - `Cursor Rules`
  - `copilot-instructions.md`
- 이 슬라이드에서 할 말:
  - AI와 함께 일하는 환경에서는 문서의 역할도 바뀐다.
  - 예전 문서는 사람이 읽는 참고 자료에 가까웠다. 지금은 에이전트가 매 세션 읽고 따르는 운영 규칙이 된다.
  - 코딩 스타일, 금지 명령, 테스트 방법, 리뷰 기준, 문서 어투, 승인 조건이 파일 안에 들어간다.
  - 문서가 실행 경로 바깥의 설명이 아니라, AI가 일하는 환경의 일부가 된다.
  - 이 지점이 Harness로 이어지는 첫 번째 다리다.

### 00-6. 개발자의 역할은 환경 설계로 이동한다

- 슬라이드에 적을 내용:
  - `타이핑의 비중은 줄어든다`
  - `판단의 밀도는 올라간다`
  - `환경을 설계하는 개발자`
- 이 슬라이드에서 할 말:
  - AI가 더 많은 코드를 써 줄수록 개발자의 기본기는 덜 중요해지는 것이 아니라 더 중요해진다.
  - 사람이 덜 타이핑할 수는 있다. 하지만 무엇을 만들지, 어떤 기준으로 끝났다고 볼지, 어떤 결과를 의심해야 할지 판단해야 한다.
  - 개발자의 역할은 코드를 한 줄씩 붙드는 사람에서, AI가 길을 잃지 않도록 환경을 설계하고 결과를 검증하는 사람으로 이동한다.
  - 그래서 다음 질문은 `코딩은 사라지는가`가 아니다.
  - 다음 질문은 `AI와 어떻게 함께 일할 것인가`다. 여기서 01장 `챗봇과 싸우지 않기`로 넘어간다.

## 대화 기록과 판단

- 처음 문제의식은 현재 본문이 부족하다는 말이 아니었다. 사용자가 진짜 말하고 싶은 것이 무엇인지, 어떤 순서와 말로 전달해야 하는지에 대한 메타인지가 부족하다는 문제였다.
- 사용자는 회사 동료에게 `AI 이렇게 사용하면 망한다`, `AI 이렇게 사용해야 한다`를 설명하고 싶어 한다. 다만 원고가 너무 회사 내부 이야기로만 닫히는 것은 원하지 않는다.
- 자주 겪는 실패 장면은 `Claude Code에게 큰 작업을 통째로 맡겼다가 이상한 수정이 누적되는 장면`과 `AI가 만든 코드를 믿고 검증 없이 넘기는 장면`이다. 그러나 원고는 이 두 장면만이 아니라, AI 활용 전반의 막막함과 구조 부재를 다뤄야 한다.
- 사용자는 `docs/01-sources/local-canonical/claude-code-seminar-kakao.md`가 잘 작성된 발표라고 느낀다. 이 발표의 강점은 전체를 다루면서도 초반에 목적지를 먼저 박아두고, 이후의 역사/방법론/도구/사례를 그 목적지로 회수한다는 점이다.
- 따라서 이번 원고도 초반에 목적지를 먼저 보여줘야 한다. 단, 카카오 발표식 `이 89장의 슬라이드를 한 장도 직접 만들지 않은 방법` 문장을 그대로 가져오면 제작 자랑처럼 보일 수 있으므로, 이번 발표의 문제의식에 맞게 바꿔야 한다.
- 후보 목적지 문장은 다음 방향으로 둔다.
  - `이 발표의 주제는 AI로 무언가를 만드는 법이 아니다. AI가 망하지 않고 일할 수 있는 환경을 만드는 법이다. 그것이 Harness다.`
  - `오늘 이야기할 것은 AI에게 일을 맡기는 법이 아니라, AI가 망하지 않고 일하게 만드는 법이다.`
- 청중에게 주고 싶은 감정의 순서는 다음과 같다.
  1. `내가 AI를 못 쓰는 게 아니라, 잘못된 방식으로 쓰고 있었구나.`
  2. `그런데 이 방식 전환을 안 하면 진짜 뒤처지겠구나.`
  3. `AI가 위험한 게 아니라, 검증 없이 쓰는 방식이 위험한 거구나.`
  4. `복잡해 보여도 결국 내 작업 환경에 구조를 몇 개 만들면 되는구나.`
- 이후 원고를 다듬을 때는 사용자 판단에 무조건 동조하지 않는다. 범위가 너무 넓어지거나, 말이 안전한 설명문으로 흐르거나, 좋은 재료가 기능 소개로 흩어질 때 의문을 제기한다.
- 00장은 `AI를 잘 쓰는 개발자는...` 문장으로 바로 시작하는 방향이 우세하다. 막막함과 말씨름의 체감은 01장에서 충분히 받을 수 있으므로, 00장은 목적지를 먼저 제시하는 장으로 둔다.
- 00장의 현재 문제는 첫 90줄이 역사와 코드 예시 중심이라 발표의 목적지가 늦게 나온다는 점이다. 추상화 역사는 유지하되, slide에서는 3-4장면으로 압축하고 prose에서는 목적지 뒤에 근거로 배치한다.
- `Meta 엔지니어가 3개월 넘게 직접 타이핑하지 않았다`류의 사례는 공개 근거가 약하면 빼거나 보수적으로 바꾼다. 00장의 숫자 근거는 YC W25, OpenAI harness engineering, Google AI-generated code처럼 공개 확인 가능한 사례를 우선한다.

## 보강 조사 계획

새 목차와 각 장의 논지가 확정되면 인터넷 조사는 다음 목적에 한정한다.

1. 이미 원고에 있는 주장에 대한 정확한 출처 확인
2. `Prompt -> Context -> Harness` 흐름을 보강하는 대표 문헌 확인
3. 컨텍스트 부패 (`Context Rot`), 중간 누락 (`Lost in the Middle`), 보정된 신뢰 (`Calibrated Trust`), 품질을 앞단으로 당기기 (`Keep Quality Left`) 같은 표현의 근거와 한국어 정착 표현 확인
4. `SDD`, `TDD`, `Plan-Critic-Build`, multi-agent pattern의 최신 사용 사례 확인
5. 비용, 성숙도, 내부 공유 사례처럼 민감하거나 맥락이 필요한 수치는 공개 가능한 범위와 표현을 재검토

외부 자료가 기존 원고의 방향을 바꿔야 할 정도로 강하면 바로 본문을 고치지 않고, 먼저 이 target map 또는 별도 research note에 판단을 남긴다.
