# 멀티 에이전트 활용

## 00. 멀티 에이전트는 수를 늘리는 기술이 아니라 분해의 기술이다

발표는 하나의 강한 에이전트가 모든 문제를 오래 붙들수록 오히려 실패 가능성이 커진다고 본다. 이유는 단순한 성능 부족이 아니라 구조 문제에 가깝다. 긴 세션은 파일, 로그, 이전 판단, 도구 결과가 한 컨텍스트 안에 뒤섞이면서 신호 대 잡음비가 무너지고, 결국 `Context Rot`로 이어진다. 여기에 구현, 조사, 리뷰, 검증 같은 서로 다른 역할까지 한 에이전트에 겹치면 시야가 흐려진다. 그래서 멀티 에이전트의 출발점은 “에이전트 수를 늘리자”가 아니라 “일을 어떤 경계로 나눠야 실패가 줄어드는가”여야 한다.

## 01. 패턴의 핵심은 의존성과 검증 필요성이다

이 관점에서 가장 먼저 세워야 할 기준은 역할 분리다. 코드베이스 구조 조사, 특정 로직 추적, 로그 요약처럼 중간 탐색 과정이 길지만 최종 사실만 필요할 때는 `Sub-Agent`가 자연스럽다. 하나의 목표를 여러 실행 단위로 나누고 메인이 계획과 종합을 맡아야 할 때는 `Orchestrator`가 맞다. 서로 독립적인 초안 생성, 대안 비교, 여러 파일 영역의 동시 탐색처럼 작업 간 의존성이 낮을 때는 `Parallel`이 효과적이다. 생성 결과의 품질을 독립적으로 압박해야 하는 디자인 검수나 장시간 앱 생성에는 `GAN-Style`이 적합하다. 그리고 구현자, 리뷰어, 편집자처럼 독립된 관점을 가진 여러 역할이 서로의 산출물을 직접 비판하고 조정해야 할 때는 `Agent Team`이 맞다.

패턴의 우열은 없다. 핵심은 작업 의존성과 검증 필요성, 컨텍스트 오염 가능성에 따라 구조를 고르는 것이다.

## 02. 가장 중요한 원리는 컨텍스트 격리와 최소 권한이다

실무에서 가장 자주 기본형이 되는 것은 `Sub-Agent`다. 이유는 화려해서가 아니라 비용 대비 효과가 가장 분명하기 때문이다. 메인 세션은 결정, 우선순위 판단, 최종 통합만 맡고, 서브 에이전트는 수십 개 파일 탐색이나 검색, 로그 분석 같은 소모적인 읽기 작업을 자기 컨텍스트 안에서 처리한다. 이렇게 하면 메인 스레드에는 중간 노이즈 대신 요약된 결과만 남는다. 멀티 에이전트의 핵심 가치는 병렬 처리 이전에 컨텍스트 격리이며, 격리 덕분에 메인은 더 오래 선명한 상태를 유지할 수 있다.

여기서 `least privilege`는 별도의 설계 원칙으로 등장한다. 조사용 에이전트에 읽기 전용 도구만 주면 실수로 파일을 바꾸지 않을 뿐 아니라, 그 에이전트의 사고도 탐색과 사실 확인에 집중된다. 리뷰 에이전트에 제한된 셸과 `diff` 확인만 허용하면 구현보다 평가에 초점이 맞춰진다. 수정 담당에게만 `Edit`나 `Write`를 부여하면 책임도 선명해진다. 최소 권한은 보안 원칙이기도 하지만, 멀티 에이전트에서는 품질 원칙이기도 하다.

## 03. output contract가 있어야 멀티 에이전트가 시스템이 된다

좋은 서브 에이전트는 그럴듯한 문장을 길게 쓰는 에이전트가 아니라, 어디서 멈추고 무엇을 반환해야 하는지가 명확한 에이전트다. `Summary`, `Critical Issues`, `Recommendations`, `Approval Status`처럼 결과 형식을 고정하면 메인은 결과를 훨씬 빠르게 해석하고 다음 결정을 내릴 수 있다. 더 중요한 점은, 구조화된 출력이 서브 에이전트에게 명확한 종료 조건을 준다는 사실이다.

이 원칙은 더 큰 패턴에서도 반복된다. `Orchestrator`가 효과적인 이유는 메인이 모든 일을 직접 처리하지 않고 계획과 합성만 담당하기 때문이다. `GAN-Style`이 필요한 이유는 생성자와 평가자를 분리해야 자기평가의 착시를 깨뜨릴 수 있기 때문이다. 2026년 3월 24일 Anthropic이 공개한 `planner`, `generator`, `evaluator` 구조는 이 원리를 장시간 자율 작업에 적용한 대표 사례다.

## 04. anti-pattern은 대개 관찰 가능성과 책임 경계를 무너뜨린다

실패하는 멀티 에이전트 구조도 패턴이 있다. “너는 전문가다” 같은 선언만 적어 놓는 `expert claims only` 패턴은 역할이 아니라 허상을 추가할 뿐이다. 강하게 의존하는 단계를 억지로 잘라 넘기는 `sequential pipeline`은 앞단 맥락을 다음 단계가 다시 복원하느라 오히려 비용만 키운다. 테스트 실행기를 별도 에이전트로 분리해 성공 여부만 받아보는 `hidden test runner`는 디버깅에 필요한 로그와 실행 맥락을 잘라버리므로 검증 루프가 아니라 정보 손실 루프가 된다.

이 세 가지 anti-pattern은 공통으로, 에이전트를 추가하면서 실제로는 관찰 가능성, 책임 경계, 맥락 보존을 더 나쁘게 만든다.

## 05. 결론: 멀티 에이전트는 Harness 분해의 한 형태다

정리하면, 멀티 에이전트의 핵심은 “여러 AI가 함께 일한다”가 아니다. 더 정확히는 컨텍스트를 격리하고, 역할을 좁히고, 권한을 제한하고, 출력 계약을 고정하고, 생성과 평가를 분리하는 설계다. 어떤 문제에서는 `Sub-Agent` 하나면 충분하고, 어떤 문제에서는 `Orchestrator`와 `Parallel`이 맞으며, 어떤 문제에서는 `GAN-Style`이나 `Agent Team`이 필요하다. 그러나 어느 경우든 원칙은 같다. 메인은 결정과 통합에 집중하고, 각 에이전트는 자기 일에 필요한 것만 가지며, 결과는 구조화되어야 하고, 검증은 독립적이어야 한다. 결국 멀티 에이전트의 본질도 더 좋은 `Harness`를 어떻게 분해하고 배치할 것인가의 문제다.
