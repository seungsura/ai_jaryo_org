# 멀티 에이전트 활용 패턴

## 시작하며: 병렬화보다 분해가 먼저다

앞 장에서 본 것은 실패가 어떻게 쌓이는가였습니다. 긴 세션, 흐려지는 기준, 썩어 가는 컨텍스트, 그리고 그럴듯해서 더 위험한 작은 오류들. 그렇다면 이번 장의 질문은 자연스럽습니다. 이 누적 비용을 구조적으로 어떻게 줄일 것인가.

멀티 에이전트는 여기에 대한 답입니다. 하지만 출발점부터 바로잡아야 합니다. 멀티 에이전트는 AI를 더 많이 붙이는 묘기가 아닙니다. 병렬화는 결과일 수 있어도 본질은 아닙니다. 본질은 분해입니다. 무엇을 메인 컨텍스트에 남기고, 무엇을 밖으로 밀어내고, 누가 생성하고 누가 검증할지를 역할 단위로 끊어 내는 설계입니다.

그래서 중요한 것은 패턴 이름을 외우는 일이 아닙니다. 경계를 어디에 그을 것인지 먼저 정하는 일입니다. 같은 작업도 어떤 경우에는 단순 심부름 분리가 맞고, 어떤 경우에는 계획자 하나가 전체를 지휘해야 하며, 어떤 경우에는 독립 평가자를 따로 붙여야 합니다. 멀티 에이전트는 “더 많은 지능”의 문제가 아니라 “더 좁은 책임”의 문제입니다.

## 같은 일을 다섯 방식으로 쪼개기

비교를 위해 하나의 공통 시나리오를 놓고 봅시다. 저장소 안에서 중간 규모 변경 하나를 끝내야 합니다. 관련 파일을 읽고, 계획을 세우고, 구현하고, 테스트와 리뷰를 거쳐, 상태를 이슈와 `PR` 같은 바깥 아티팩트에 남겨야 하는 일입니다. `06`과 `07` 전체가 암시해 온 실전 작업을 한 줄로 압축하면 대체로 이런 모습입니다.

이제 질문은 “어떤 패턴이 더 고급인가”가 아닙니다. 이 한 가지 일을 어떤 식으로 나누는 편이 drift를 덜 만들고, 메인 컨텍스트를 덜 오염시키고, 검증 책임을 더 또렷하게 만드는가입니다. 다섯 패턴은 바로 그 다른 답들입니다.

## 1. `Sub-Agent`: 중간 노동을 메인 컨텍스트 밖으로 격리한다

가장 기본이 되는 패턴은 `Sub-Agent`입니다. 메인이 해야 할 판단은 남기고, 그 판단에 바로 들고 갈 필요는 없는 중간 노동을 떼어 냅니다. 저장소 변경 시나리오라면 대표적인 대상은 코드베이스 탐색, 로그 파싱, 테스트 실패 분류, 관련 파일 인벤토리 같은 일입니다. 토큰은 많이 먹지만, 그 흔적 전체를 메인 세션에 눌어붙게 둘 이유는 없습니다.

이 패턴의 핵심은 속도가 아니라 격리입니다. 서브 에이전트가 파일을 수십 개 읽고 길게 헤매도, 메인은 요약과 결론만 받습니다. 그래서 메인 컨텍스트는 “무엇을 바꿔야 하는가”에 집중하고, 탐색의 먼지는 옆방에서 끝납니다. 긴 작업에서 이 단순한 분리가 생각보다 자주 이깁니다.

여기서 `least privilege`는 권한 모델만 뜻하지 않습니다. 정보 스코핑도 포함합니다. 서브 에이전트에게 전체 대화와 저장소를 통째로 넘기는 것이 아니라, 지금 필요한 파일과 맥락만 줍니다. 읽기 전용처럼 도구 제한을 더하면 사고 가능성은 더 줄어듭니다. 많이 읽게 하되, 넓게 오염시키지 않는 것. `Sub-Agent`의 강점은 바로 여기에 있습니다.

## 2. `Orchestrator`: 계획자가 전체 분해를 쥔다

`Sub-Agent`가 단일 심부름 분리라면, `Orchestrator`는 한 단계 더 큽니다. 저장소 변경 하나를 끝내기 위해 먼저 계획자가 전체 작업을 분해합니다. 어떤 파일군을 먼저 읽을지, 어떤 변경을 독립 단위로 자를지, 어떤 검증을 마지막이 아니라 중간에 둘지를 정합니다. 그 다음에야 여러 실행자에게 일을 나눠 맡깁니다.

이 패턴은 메인이 직접 모든 세부 작업을 붙드는 대신, 계획과 순서와 수렴 지점을 계속 쥐고 있을 때 강합니다. 하위 작업끼리 실시간 토론이 필요하지는 않지만, 큰 목표를 안정적으로 쪼개야 할 때가 그렇습니다. 예를 들어 한쪽은 테스트 기반을 정리하고, 다른 한쪽은 구현 후보를 만들고, 또 다른 한쪽은 관련 문서를 확인할 수 있습니다. 하지만 이 셋이 제멋대로 달리기 시작하면 결과는 쉽게 흩어집니다. 그래서 `Orchestrator`에서는 실행보다 먼저 분해 규칙이 중요합니다.

이 패턴이 강한 이유는 일을 많이 시켜서가 아닙니다. 메인이 세부 실행의 진흙탕에 빠지지 않고도 전체 방향을 유지할 수 있기 때문입니다. 계획자가 없는 병렬은 흩어지고, 실행자가 없는 계획은 문서로 끝납니다. `Orchestrator`는 둘 사이의 빈칸을 메우는 운영 패턴입니다.

## 3. `Parallel`: 같은 목표를 동시에 벌리되 서로를 더럽히지 않는다

`Parallel`은 이름 때문에 늘 속도 패턴처럼 들립니다. 실제로는 간섭 제어 패턴에 가깝습니다. 저장소 변경 시나리오를 다시 가져오면, 관련 구현 사례 탐색, 테스트 실패 원인 후보 정리, 영향 범위 문서 확인처럼 서로 직접 의존하지 않는 탐색은 동시에 벌릴 수 있습니다. 중요한 것은 빨리 끝내는 것이 아니라, 중간 상태가 서로를 오염시키지 않게 경계를 고정하는 일입니다.

좋은 병렬은 “다 같이 달린다”가 아니라 “서로의 미완성 판단을 먹지 않는다”에 가깝습니다. 한쪽의 섣부른 결론이 다른 쪽의 전제가 되기 시작하면 병렬은 금세 잡음 제조기가 됩니다. 그래서 병렬을 쓸 때는 결과를 언제, 어떤 형식으로 다시 합칠지까지 같이 정해야 합니다. 비교 가능한 초안, 분리된 로그, 독립된 요약이 있어야 나중에 메인이 제대로 고를 수 있습니다.

즉 `Parallel`은 언제나 좋은 기본값이 아닙니다. 서로가 서로의 중간 상태를 몰라도 되는 작업에서만 강합니다. 독립성이 낮은데 병렬부터 걸면, 속도보다 불일치가 더 크게 남습니다.

## 4. `GAN-Style`: 생성자와 평가자를 갈라 자기최면을 끊는다

실전에서 자주 놓치는 것은 생성과 평가를 같은 손에 쥐게 두는 일입니다. 초안을 만든 주체가 “충분히 괜찮다”고 말하는 순간, 그럴듯한 오류는 가장 오래 살아남습니다. 그래서 `GAN-Style` 패턴이 필요합니다. 한쪽은 만듭니다. 다른 한쪽은 루브릭으로 깎습니다. 둘이 같은 목표를 보더라도 같은 입장을 가져서는 안 됩니다.

저장소 변경 시나리오에서는 이 구조가 특히 잘 먹힙니다. 한 에이전트가 구현 초안을 만들면, 다른 에이전트는 “테스트를 통과하느냐”만 보는 것이 아니라, 요구를 빠뜨리지 않았는지, 부작용을 만들지 않았는지, 실패 시 어떤 로그와 흔적이 남는지를 따로 봅니다. 이때 평가자는 기분 좋게 의견을 더하는 조언자가 아니라, 재작업 범위를 명확히 잘라 주는 존재여야 합니다.

`planner`, `generator`, `evaluator` 같은 구조가 반복해서 등장하는 이유도 같습니다. self-review에는 늘 blind spot이 남기 때문입니다. 멀티 에이전트가 필요한 진짜 이유 중 하나는 처리량이 아니라 이 blind spot의 구조적 축소입니다. 다음 장에서 다룰 gate와 검증 루프는 바로 이 분리를 일회성 요령이 아니라 운영 문법으로 고정하는 방식입니다.

## 5. `Agent Teams`: 양방향 대화 자체가 일인 경우에만 쓴다

`Agent Teams`는 가장 화려해 보이지만, 그래서 더 자주 오해됩니다. 이 패턴은 서브 에이전트를 많이 붙인 형태가 아닙니다. 여러 독립 세션이 각자 별도 컨텍스트를 가진 채, 서로 직접 메시지를 주고받으며 결론을 만드는 구조입니다. 메인이 모든 handoff를 중계하지 않아도 됩니다. 대신 팀원 사이에 왕복이 생깁니다.

이 패턴이 정당화되는 경우는 분명합니다. 저장소 변경 요청 안에서도 요구가 충돌하거나 판단 기준이 둘 이상일 때입니다. 예를 들어 한쪽은 구현 가능성과 영향 범위를 보고, 다른 한쪽은 검증 전략과 실패 비용을 보고, 또 다른 한쪽은 문서화와 운영 흔적을 봅니다. 이런 경우에는 단방향 심부름보다 양방향 조정이 더 중요해집니다. 서로가 서로의 결론을 다시 밀어보는 과정이 품질을 만듭니다.

하지만 팀이라고 늘 좋은 것은 아닙니다. 인원이 늘어날수록 대기 시간이 작업 시간을 잡아먹고, 대화 자체가 새로운 컨텍스트 비용이 됩니다. 그래서 `Agent Teams`는 상위 패턴이 아니라 예외 패턴에 가깝습니다. 양방향 대화가 결과 품질의 핵심일 때만 정당화됩니다. 그렇지 않다면 대개 `Sub-Agent`, `Orchestrator`, `GAN-Style`의 조합이 더 싸고 더 선명합니다.

## 선택 기준: 패턴 이름보다 handoff 설계가 먼저다

이 다섯 패턴을 비교할 때 가장 먼저 볼 것은 복잡도가 아닙니다. handoff의 성격입니다. 한 번 던지고 요약만 받으면 되는가. 전체 분해를 쥔 계획자가 필요한가. 서로 독립적으로 달려도 되는가. 생성과 평가를 반드시 분리해야 하는가. 아니면 아예 여러 역할이 서로 논쟁해야 하는가. 패턴 선택은 이 질문에 대한 답입니다.

그래서 멀티 에이전트 설계에서 가장 위험한 습관은 패턴을 장식처럼 붙이는 일입니다. 팀이 멋져 보여서 팀을 만들고, 병렬이 빨라 보여서 병렬을 걸면 거의 반드시 과잉 설계가 됩니다. 먼저 정해야 할 것은 작업의 경계, 검증의 독립성, 메인 컨텍스트의 보존 범위입니다. 패턴은 그 뒤에 옵니다.

결국 멀티 에이전트의 힘은 “AI가 많다”는 데 있지 않습니다. 메인이 모든 것을 기억하지 않아도 되게 만들고, 생성과 검증이 서로를 감시하게 만들고, 중간 노동을 밖으로 밀어내며, 역할 경계를 더 좁고 더 정확하게 그을 수 있다는 데 있습니다.

## 결론: 더 많은 AI가 아니라 더 좁은 책임이 필요하다

이 장의 결론은 단순합니다. 멀티 에이전트는 병렬화 기술이 아닙니다. 책임 분해 기술입니다. 하나의 긴 일을 여러 지능에게 나누어 준다는 말보다, 어떤 판단을 어디에 가둘 것인가를 설계한다는 말이 더 정확합니다.

그래서 좋은 패턴은 늘 메인 컨텍스트를 가볍게 하고, 검증을 독립시키고, 중간 상태를 격리합니다. 반대로 나쁜 패턴은 에이전트 수만 늘리고 경계는 흐리게 만듭니다. 이제 다음 장에서 봐야 할 것은 이 패턴을 어떻게 실전 workflow로 굳히느냐입니다. command, gate, `worktree`, 이슈 시스템 같은 운영 장치가 들어오는 이유도 바로 여기에 있습니다. 패턴은 설계 원리이고, workflow는 그 원리를 잊어도 버티게 만드는 기계적 문법입니다.
