# 03. 이렇게 하면 망한다

## 03-1. 루프 속에서 길을 잃는 에이전트

### 슬라이드에 적을 내용

- `95%`
- `20 steps -> 36%`
- `50 steps -> 8%`
- `긴 작업은 성공률을 곱한다`

### 이 슬라이드에서 할 말

에이전트 루프는 강력하지만, 무한정 이어지면 길을 잃기 쉽습니다. 에이전트는 계획하고, 찾고, 고치고, 실행하고, 다시 확인하는 단계를 계속 이어갑니다. 이 연결 고리가 길어질수록 아무리 똑똑한 모델이라도 전체 성공률은 생각보다 빨리 떨어집니다.

예를 들어 한 단계의 성공률을 95%로 잡아도, 스무 단계를 지나면 전체가 끝까지 맞을 확률은 36% 정도까지 내려갑니다. 50단계로 가면 8% 수준입니다.

그래서 긴 호흡의 작업에서는 "이번 답변이 괜찮은가"만 봐서는 부족합니다. 이 작업이 열 단계, 스무 단계 뒤에도 같은 기준으로 움직일 수 있는지를 봐야 합니다.

## 03-2. 작은 오해는 누적된다

### 슬라이드에 적을 내용

- `잘못 읽은 파일`
- `잘못 세운 계획`
- `깨진 테스트의 오해`
- `오래된 대화의 잔상`

### 이 슬라이드에서 할 말

잘못된 파일을 기준으로 계획을 잡거나, 이미 깨진 테스트를 정상으로 읽거나, 오래된 대화 내용을 현재 결정처럼 붙드는 순간부터 작업은 점점 느려집니다.

Claude Code에게 큰 작업을 통째로 맡겼다가 이상한 수정이 누적되는 장면은 여기서 자주 나옵니다. 에이전트가 갑자기 나빠져서가 아닙니다. 작은 오해가 다음 판단의 전제가 되고, 그 전제가 수정으로 남고, 그 수정이 다시 다음 루프의 사실처럼 읽히기 때문입니다.

이때 필요한 것은 일을 어디서 끊을지, 어느 결과를 믿을지, 무엇을 다음 단계로 넘기기 전에 확인할지 정하는 일입니다.

## 03-3. 컨텍스트가 크다고 항상 좋은 것은 아니다

### 슬라이드에 적을 내용

- `more context != better context`
- `압축은 손실을 만든다`
- `중간에 묻힌 정보는 약해진다`

### 이 슬라이드에서 할 말

컨텍스트를 많이 넣으면 안심이 됩니다. 모델이 더 많이 보면 더 잘 판단할 것 같습니다. 실제로 필요한 정보를 못 보는 것보다 보는 편이 낫습니다. 하지만 많이 들어 있는 것과 잘 쓰는 것은 다릅니다.

컨텍스트 창은 무한하지 않습니다. 대화가 길어지면 지시, 파일 내용, 도구 결과, 에러 로그, 이전 답변이 계속 쌓입니다. 최신 도구들은 컨텍스트 최대 용량에 가까워지면 이전 대화를 압축해서 이어 가도록 해줍니다. 압축은 긴 작업을 계속하게 해 주지만, 정보 손실도 만듭니다.

또 하나는 `Lost in the Middle` 문제입니다. 긴 컨텍스트 안에 정보가 들어 있어도 관련 정보가 중간에 묻히면 모델이 제대로 활용하지 못하는 경우가 있습니다. "분명 넣어 줬는데 왜 못 봤지"라는 일이 생기는 이유가 여기에 있습니다.

## 03-4. 컨텍스트는 작업대다

### 슬라이드에 적을 내용

- `필요한 단서`
- `오래된 계획`
- `실패한 명령 결과`
- `철회한 지시`
- `모두 같은 무게로 남는다`

### 이 슬라이드에서 할 말

컨텍스트 안에는 관련 파일만 들어가지 않습니다. 낡은 계획, 실패한 명령 결과, 이미 철회한 지시, 지금 작업과 비슷하지만 다른 문서도 같이 들어옵니다. 창이 커지면 그런 것까지 오래 살아남습니다.

대화창은 기억 장소처럼 보이지만, 사실은 작업대에 가깝습니다. 작업대 위에 모든 것을 계속 올려두면 필요한 도구도 못 찾고, 버린 부품도 다시 집어듭니다. 컨텍스트도 마찬가지입니다. 정리하지 않으면 필요한 정보와 버려야 할 정보가 같은 무게로 남습니다.

쓸 수 있는 만큼 다 채우는 것이 능력이 아닙니다. 무엇을 넣을지보다, 무엇을 빼야 하는지 아는 쪽이 더 중요해질 때가 많습니다.

## 03-5. 대표 실패 패턴 네 가지

### 슬라이드에 적을 내용

| 패턴 | 증상 |
| --- | --- |
| `Poisoning` | 잘못된 정보가 뒤의 결정을 물들인다 |
| `Distraction` | 자료가 많아 핵심이 묻힌다 |
| `Confusion` | 예전 결정과 최신 결정이 섞인다 |
| `Clash` | 서로 다른 지시와 기준이 부딪힌다 |

### 이 슬라이드에서 할 말

긴 세션에서 자주 보이는 망가짐은 크게 네 가지입니다.

`Poisoning`은 처음부터 잘못 들어간 정보가 계속 살아남아 뒤의 결정을 물들이는 경우입니다. `Distraction`은 자료가 너무 많아 핵심이 묻히는 경우입니다. `Confusion`은 예전 결정과 최신 결정이 한 창 안에 같이 남아 어느 쪽이 현재 기준인지 흐려지는 경우입니다. `Clash`는 서로 다른 지시와 기준이 같은 컨텍스트 안에서 부딪히는 경우입니다.

이 네 가지는 보통 같이 옵니다. 오래된 정보가 남아 `Poisoning`이 생기고, 자료가 많아 `Distraction`이 생기고, 그 안에서 기준이 엉키면 `Confusion`과 `Clash`가 이어집니다.

결국 문제는 하나입니다. 컨텍스트가 깨끗한 작업 공간으로 남지 못하고, 오래된 기억과 잡음과 규칙이 뒤섞이는 것입니다.

## 03-6. 현실에서 보이는 증상들

### 슬라이드에 적을 내용

- `AI Slop`
- `Doom Loop`
- `Shadow Agent`

### 이 슬라이드에서 할 말

이런 문제가 쌓이면 겉으로도 티가 납니다.

`AI Slop`은 산출물이 갑자기 조잡해지는 상태입니다. 겉보기에는 그럴듯하지만 기준이 무너져 있습니다. 문장만 많고 핵심은 비어 있거나, 코드가 돌아가는 척하지만 실제 흐름은 깨져 있습니다.

`Doom Loop`는 같은 실수를 다른 말로 반복하는 상태입니다. 테스트를 고치지 못한 채 프롬프트만 바꾸거나, 같은 파일을 계속 만지면서 앞으로 나아가지 못합니다.

`Shadow Agent`는 에이전트가 지금 왜 이 일을 하는지 잊은 채 움직이는 상태입니다. 겉으로는 바쁘게 읽고 쓰고 실행하지만, 실제로는 기준을 놓친 채 관성으로 달립니다.

## 03-7. Context Rot

### 슬라이드에 적을 내용

- `Context Rot`
- `오래된 기준`
- `실패한 흔적`
- `최신 판단처럼 보이는 순간`

### 이 슬라이드에서 할 말

컨텍스트 부패(`Context Rot`)는 이 장면을 잘 설명하는 말입니다. 컨텍스트가 길어진다고 바로 썩는 것은 아닙니다. 문제는 오래된 기준과 실패한 흔적이 계속 남아 있고, 그 흔적이 최신 판단처럼 보이는 순간입니다.

그래서 상태를 대화창 안에만 두면 안 됩니다. 결정, 남은 일, 실패 원인, 현재 기준은 파일, 이슈, 테스트 결과, Git 이력 같은 바깥 기록으로 빼내야 합니다. 그래야 다음 루프가 대화의 잔상에 끌려가지 않고, 지금 남아 있는 사실에서 다시 시작할 수 있습니다.

에이전트가 무너지는 이유는 모델의 능력이 부족해서만은 아닙니다. 끊임없이 도는 루프 속에서 작업의 방향을 잃고, 쏟아지는 단서를 검증하지 않은 채 방치하기 때문입니다.

방향이 흐린 상태에서 루프의 속도만 올리면, 에이전트는 더 빠른 속도로 엉뚱한 곳을 향해 달립니다. 긴 호흡의 작업을 끝까지 가져가려면 먼저 방향을 고정하고, 매 단계를 의심하며 검증할 바닥을 만들어야 합니다.
