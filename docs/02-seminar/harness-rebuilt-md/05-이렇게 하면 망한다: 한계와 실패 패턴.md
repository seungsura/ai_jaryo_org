## 오류의 나비효과

AI 코딩 에이전트를 몇 번만 길게 돌려 봐도 알게 되는 사실이 있습니다. 문제는 한 번의 대형 사고가 아니라, 그럴듯하게 지나간 작은 오류입니다.

에이전트는 읽고, 수정하고, 실행하고, 다시 판단하는 루프를 반복합니다. 이 구조에서는 초반의 작은 오해도 다음 단계의 입력으로 들어갑니다. 잘못된 가정이 계획으로 굳고, 그 계획이 코드 수정으로 이어지고, 그 수정 결과가 다시 검증 대상이 됩니다. 시간이 지날수록 어디서부터 어긋났는지 추적하기 어려워집니다.

긴 루프 작업에서 중요한 것은 한 번의 답변 품질이 아니라, 실패가 누적되고 증폭되는 방식 전체입니다. 

## **긴 작업 사슬이 특히 위험한 이유**

단발성 작업에서는 모델의 영리함이 꽤 잘 먹힙니다. 하지만 작업이 길어질수록 기준이 달라집니다. 길게 작업이 진행될 수록 당장의 답변이 중요한 것이 아니라 “스무 단계 뒤에도 정말 좋은 결과물이 나올까”가 됩니다.

긴 사슬에서는 작은 오류가 비용으로 바뀝니다. 잘못된 파일을 기준으로 계획을 세우고, 이미 깨진 테스트 결과를 정상 상태로 오해하고, 오래된 문맥을 현재 상태처럼 읽기 시작합니다. 이 시점부터는 더 좋은 프롬프트보다 더 좋은 작업 분해가 중요합니다. 어디서 끊고, 어디서 검증하고, 무엇을 다음 단계로 넘기기 전에 고정할지를 먼저 결정해야 합니다.

## **컨텍스트가 길수록 항상 좋은 것은 아니다**

긴 컨텍스트는 분명 강력한 기능입니다. 더 많이 넣으면 더 잘할 것이라고 믿는 것입니다.

## **대표 실패 패턴 네 가지**

긴 세션에서 반복해서 나타나는 실패는 4가지 정도로 분류하고 있습니다.

첫째는 `Poisoning`입니다. 오래된 가정이나 잘못된 메모가 이후 판단 전체를 오염시키는 경우입니다. 처음에 잘못 넣은 정보 하나가 루프 전체를 틀어버립니다.

둘째는 `Distraction`입니다. 관련 정보가 너무 많아 정작 핵심 신호가 묻히는 경우입니다. 에이전트는 필요한 파일을 보고도 우선순위를 잃어버립니다.

셋째는 `Confusion`입니다. 과거 결정과 현재 결정이 한 창 안에서 뒤엉켜, 무엇이 최신 기준인지 분간하지 못하는 경우입니다. 길게 이어진 세션에서 특히 자주 보입니다.

넷째는 `Clash`입니다. 서로 다른 지시와 기준이 같은 컨텍스트 안에 섞여 충돌하는 경우입니다. 예를 들어 오래된 규칙과 새 규칙이 함께 남아 있을 때 이런 일이 벌어집니다.

이 네 가지는 이름이 다를 뿐, 결국 하나의 문제를 가리킵니다. 기억과 잡음과 규칙이 하나의 컨텍스트 안에서 썩고 있다는 뜻입니다.

## **현실에서 보이는 증상들**

`AI Slop`은 산출물이 급격히 조잡해지는 상태를 가리킵니다. 그럴듯해 보이지만 품질 기준이 빠르게 무너집니다.

`Doom Loop`는 같은 실수를 다른 표현으로 반복하는 상태입니다. 테스트를 고치지 못한 채 프롬프트만 바꾸거나, 같은 파일을 돌려 만지면서 진행이 없는 경우가 여기에 가깝습니다.

`Shadow Agent`는 지금 왜 이 작업을 하고 있는지, 어떤 기준을 따라야 하는지를 잊어버린 채 움직이는 상태입니다. 겉으로는 계속 일하고 있지만 실제로는 방향 감각이 무너진 상태라고 보는 편이 맞습니다.

이 증상들을 보면 흔히 “오늘 모델이 이상하다”고 말하고 넘어갑니다. 하지만 중요한 것은 구조입니다. 어떤 검증이 비어 있었는지, 어떤 상태가 파일로 밖에 남지 않았는지, 어디서 루프를 끊었어야 했는지를 먼저 봐야 합니다.

## **Context Rot: 길어진 기억은 조용히 썩는다**

긴 세션의 진짜 문제는 단순한 길이가 아닙니다. 시간이 지나면서 초반의 의도와 제약이 점점 희미해진다는 점입니다. 처음에는 분명했던 규칙이 뒤로 갈수록 약해지고, 중간에 생긴 가정이 사실처럼 굳고, 오래된 도구 결과가 아직도 유효한 정보처럼 남습니다.

이 상태를 흔히 `Context Rot`이라고 부릅니다. 세션을 오래 이어 가는 것이 곧 숙련은 아닙니다. 오히려 잘하는 팀일수록 끊을 타이밍을 더 잘 압니다. 현재 결론과 다음 행동만 남기고 새 세션으로 넘어갈 줄 알아야 합니다.

상태를 대화창 안에만 두면 결국 썩습니다. 파일, 이슈, 진행 기록, 테스트 결과, Git 이력처럼 바깥의 아티팩트에 남겨야 기억이 살아남습니다.

## **신뢰는 조율되어야 한다**

AI를 믿느냐 마느냐는 감정 문제가 아닙니다. 어떤 작업에 어느 정도 자율성을 줄 것인지 조절하는 운영 문제입니다. 이것을 `Calibrated Trust`라고 볼 수 있습니다.

초안 작성, 반복적인 리팩터링, 구조 탐색처럼 비용이 낮은 구간에는 비교적 넓은 자율성을 줄 수 있습니다. 반대로 배포, 삭제, 민감 데이터 접근, 최신 API 사용, 외부 인용, 보안 관련 코드처럼 틀렸을 때 비용이 큰 구간에는 더 강한 검증이 필요합니다. 이 차이를 구조로 고정하지 않으면, 큰 사고가 발생하는 것입니다. 

에이전트가 똑똑한 것과, 그 결과를 그대로 믿어도 된다는 것은 전혀 다른 문제입니다.

## **결정 제어와 확률 제어를 분리하라**

하네스 설계에서 중요한 원칙은 기계가 확실히 판단할 수 있는 것과 AI가 유연하게 판단해야 하는 것을 분리하는 일입니다.

- 결정론적 제어: 린트, 타입 체크, 테스트, 포맷, 위험 명령 차단, 파일 권한, 승인 게이트
- 확률적 제어: 설계 대안 제안, 코드 리뷰 의견, 아키텍처 판단, 요약, 우선순위 평가

기계가 확인할 수 있는 것은 기계에게 맡겨야 합니다. AI에게 “테스트를 잘 통과했는지 확인해 줘”라고만 말하는 것보다 실제 테스트를 실행하고 실패하면 멈추는 게 강합니다. AI는 그 실패 메시지를 읽고 수정하는 데 집중해야 합니다.

긴 세션을 무조건 유지하기보다 압축하고 새로 시작할 줄 알아야 합니다. 관련 없는 도구와 파일은 컨텍스트에서 빼야 합니다. 현재 결정과 남은 과제는 반드시 파일이나 이슈로 고정해야 합니다. 그리고 무엇보다 결과를 그냥 읽고 넘어가지 말고, 테스트와 린트와 승인 같은 결정론적 문을 통과하게 해야 합니다.

이 원칙을 한 문장으로 요약하면 이렇습니다.

> **Keep Quality Left: 기계가 확인할 수 있는 것은 앞쪽의 결정론적 게이트로 옮기고, AI는 판단과 생성에 집중하게 하라.**
> 

실무에서는 이 원칙이 층으로 쌓입니다. 아래층일수록 값싸고 기계적인 제어이고, 위층으로 갈수록 더 비싸고 더 인간적인 판단입니다. 중요한 것은 위층으로 가기 전에 아래층의 문을 먼저 통과하게 만드는 것입니다.

| **계층** | **역할** |
| --- | --- |
| `Human review` | 배포, 삭제, 보안, 아키텍처처럼 비싼 결정을 최종 확인한다 |
| `E2E / test gate` | 사용자 시나리오와 회귀를 자동으로 걸러낸다 |
| `AI code generation` | 실제 생성과 수정이 일어나는 층이다 |
| `Deterministic lint / type gate` | 포맷, 타입, 정적 규칙을 기계적으로 차단한다 |
| `Dynamic tool curation` | 지금 작업에 필요한 도구만 열어 잡음을 줄인다 |
| `Isolated sandbox / VM` | 위험 명령과 외부 영향을 격리한다 |

모든 팀이 항상 여섯 층을 똑같이 둘 필요는 없습니다. 하지만 아래층이 비어 있는데 위층의 판단만 믿는 구조는 오래 버티지 못합니다.

## **SOLID도 하네스 관점으로**

하네스는 새로운 유행어지만, 기존 소프트웨어 원칙을 버리라는 뜻은 아닙니다. 오히려 오래된 원칙을 에이전트 중심으로 재해석하게 만듭니다.

예를 들어 `SOLID`를 하네스 관점에서 보면 이렇게 읽을 수 있습니다.

| **원칙** | **코드 중심 해석** | **에이전트 중심 해석** |
| --- | --- | --- |
| S: 단일 책임 | 클래스를 분리한다 | 에이전트와 도구의 역할을 분리한다 |
| O: 개방-폐쇄 | 코드는 수정 없이 확장 가능해야 한다 | 검증은 훅과 플러그인으로 확장하되 핵심 로직을 흔들지 않는다 |
| I: 인터페이스 분리 | 불필요한 메서드 의존을 줄인다 | 에이전트에게 불필요한 도구와 파일 권한을 노출하지 않는다 |
| D: 의존성 역전 | 추상화에 의존한다 | 하드코딩된 프롬프트보다 컨텍스트와 규칙을 외부화한다 |

**정리**

에이전트가 길을 잃는 이유는 모델이 멍청해서가 아닙니다. 에이전트 작업이 본질적으로 누적되는 루프이기 때문입니다. 이 루프를 통제할 구조가 없으면, 작은 오류는 조용히 커집니다.
