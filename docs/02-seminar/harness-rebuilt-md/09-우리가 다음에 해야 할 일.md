## **시작하며: FOMO 와 피로**

새 모델 소식이 나올 때마다 뒤처지는 느낌이 듭니다. 누군가는 더 빨리 자동화하고, 누군가는 새 도구를 바로 써 보고, 누군가는 이미 워크플로우를 완전히 바꾼 것처럼 보입니다. 이 감각은 `AI FoMO`라고 부를 수 있습니다.

이 감각은 생각보다 개인적인 불안이 아닙니다. 최전선에 있는 사람들도 똑같이 느낍니다. Simon Willison은 2025년에 “지식이 쌓이는 속도보다 감가상각되는 속도가 더 빠르다. 나도 지쳤다.”라고 말했습니다..

이 피로는 세 갈래로 나타납니다. 뒤처질까 두렵고, 좋은 도구를 못 쓸까 두렵고, 남들만 이득 볼까 두렵습니다. 하지만 `AI FoMO`를 가장 크게 줄이는 것은 더 많은 정보를 읽는 일이 아니라 직접 써 보는 일입니다. 최전선에 있는 사람들도 결국 더 빨리 배우는 대신, 무엇을 남기고 무엇을 버릴지 고르는 데 시간을 씁니다.

그래서 더 중요한 것은 신모델 발표를 누가 먼저 따라갔느냐가 아니라, 오래 해 온 사람들이 결국 무엇을 같이 말하느냐입니다. 50년 넘게 코드를 써 온 사람, 최전선에서 매일 도구를 쓰는 사람, 업계의 대부가 거의 같은 결론에 도달합니다.

| **사람** | **메시지** |
| --- | --- |
| Kent Beck | 프로그래밍은 여전히 프로그래밍이고, AI는 더 중요한 결정을 더 빨리 내리게 하는 도구다 |
| Simon Willison | AI는 기존 전문성을 증폭한다. 경험이 많을수록 더 빠르고 더 좋은 결과를 얻는다 |
| Martin Fowler | AI는 주니어의 산출물은 복제할 수 있어도 시니어의 경험과 판단은 복제하지 못한다 |

즉 AI 시대의 경험은 축소가 아니라 증폭입니다. `Spec`은 방향을 고정하는 원칙이고, `Test-Verification`은 결과를 검증하는 원칙이며, `Plan-Critic`은 실행과 판단을 나누는 원칙입니다. 이런 원칙은 새로 발명된 것이 아니라, 원래 잘하던 사람이 더 멀리 뻗게 만드는 레버리지입니다.

개인의 피로와 팀의 혼란은 사실 같은 문제입니다. 기준이 개인 머릿속과 채팅창 안에만 있으면 개인은 계속 새 소식을 따라가다 지치고, 팀은 각자 다른 방식으로 에이전트를 쓰면서 비용과 위험을 키웁니다. 그래서 필요한 것은 더 많은 정보가 아니라 공유된 운영 구조입니다.

## **팀 차원의 성숙도: 카오스에서 혁신으로**

조직 차원에서 차이는 모델 선택보다 성숙도에서 납니다. 카오스 단계에서는 각자가 제각각 에이전트를 써서 비용이 폭발하고, 표준화와 최적화 단계로 갈수록 템플릿, 캐시, 모델 라우팅, 운영 규칙이 붙습니다. 카카오 AI 팀 내부 공유 사례 기준으로 보면 1단계 카오스는 `월 7000달러/팀` 수준의 비용 폭발이고, 4단계 최적화는 `비용 82% 절감`, `월 1283달러/팀`까지 내려갑니다.[^09-v02]

| **단계** | **상태** | **목표** |
| --- | --- | --- |
| 1. 카오스 | 섀도우 에이전트 위험, 하네스 없음, 비용 폭발: 월 7000달러/팀 | 사고와 비용을 먼저 보이게 한다 |
| 2. 인식 | 개인별 가이드라인 도입 | 반복 실패와 공통 규칙을 모은다 |
| 3. 표준화 | 하네스 템플릿과 프로젝트별 공통 규칙 도입 | 규칙 파일, 테스트, 승인 기준을 표준화한다 |
| 4. 최적화 | 캐시, 모델 라우팅, 비용 관리, 비용 82% 절감: 월 1283달러/팀 | 품질과 비용을 함께 관리한다 |
| 5. 혁신 | 자가 조정 하네스와 조직 인프라화 | 하네스가 팀의 기본 개발 환경이 된다 |

팀 차원의 성숙도가 증가할수록 시니어 엔지니어의 판단력과 통찰의 가치는 급상승합니다. 

## **결론: Harness Engineer**

지금 더 분명해지는 역할은 직접 코드를 가장 빨리 치는 사람이 아닙니다. 에이전트가 일할 환경을 설계하는 사람입니다. 저장소의 지식을 문서와 규칙으로 정리하고, 기계적 제약을 세우고, 상태를 세션 밖에 남기고, 사람이 맡아야 할 판단을 끝까지 붙드는 사람입니다.

이 역할을 `Harness Engineer`라고 합니다.

다음 주에 바로 시작할 일은 어렵지 않습니다. 반복해서 말하는 규칙 세 가지를 파일로 빼고, 테스트나 리뷰 같은 결정론적 문 하나를 추가하고, 긴 세션의 결론을 채팅창 밖 문서에 남기면 됩니다. 그 세 가지만 해도 하네스는 추상 개념이 아니라 작업 환경의 일부가 됩니다.

소프트웨어 엔지니어링은 사라지지 않습니다. 우리의 작업 환경이 변할 뿐입니다. 우리가 다음에 해야 할 일도 분명합니다. 새 모델 소식에 휩쓸리는 것이 아니라, AI를 내 환경 안에서 먼저 현실로 만드는 일입니다.
