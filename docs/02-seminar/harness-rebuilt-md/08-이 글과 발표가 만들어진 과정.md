## **시작하며: 이 발표를 만든 방법**

기억하시나요? 이 발표를 만든 방법입니다. 저는 코드 한 줄 PPT 한 장도 만들지 않았습니다. 한 것은 하나 하네스 설계입니다. 

## **전체 파이프라인: source → prose → outline → html → pdf → notes**

이 제작 과정은 다음 순서를 기본으로 삼았습니다.

1. `source`: PDF, HTML, 외부 링크, 로컬 마크다운 자료를 입력층으로 묶는다.
2. `prose`: 발표와 글의 기준 문장을 만든다.
3. `outline`: prose를 slide 단위와 장 구조로 압축한다.
4. `html`: 승인된 outline과 design contract 안에서 HTML/CSS를 생성한다.
5. `pdf`: HTML 산출물을 변환한다.
6. `speaker notes`: 완성된 deck을 기준으로 발표 대본을 만든다.

앞에서 설명한 내용 그대로 적용하였습니다. Spec, Harness, Skills 

특별한 기술 없습니다. markdown, html, 규칙문서로만 제작되었습니다. 

| **단계** | **실제 경로** | **역할** |
| --- | --- | --- |
| `source` | `docs/01-sources` | `local-canonical`, `local-supplemental`, `maps`, `provenance`를 묶어 근거를 정리한다 |
| `seminar draft` | `docs/02-seminar/harness-rebuilt-md`, `docs/02-seminar/prose` | 초안과 서술 기준선을 세운다 |
| `html` | `docs/03-html` | `outline`, `copy`, `deck`, `slides`를 관리한다 |
| `pdf` | `docs/04-pdf` | export와 QA를 분리한다 |
| `speaker notes` | `docs/05-speaker-notes` | 발표용 notes를 최종 정리한다 |

여기서 `outline`은 별도 메모가 아닙니다. `docs/03-html/outline` 안에 들어가 있고, 바로 뒤 단계인 HTML 생성과 같은 흐름에 묶여 있습니다. 이 구조를 고정해 두면 에이전트도 지금 어느 층을 다루는지 헷갈리지 않습니다.

## **source를 수집하였습니다.**

가장 먼저 한 일은 자료 수집이었습니다. 유튜브, 링크드인, 인터넷 커뮤니티, 공식 사이트, 공개 문헌, 직접 수집한 마크다운, PDF 파일 들을 각기 다른 계층으로 두었습니다. 이 계층에 따라 서로 다른 중요도를 갖게 하였습니다. 이것을 구분하지 않으면, 나중에 검증하기 어려워집니다.

먼저 어떤 자료가 어떤 주장에 쓰이는지 정리하고, 불확실한 부분은 억지로 메우지 않았습니다. 

이 과정에서 사람이 개입한 것은 수집한 정보의 중요도 분리였습니다.

## **prose를 canonical source로 세웠다**

그 다음 prose를 만들었습니다. prose는 발표 원문이 아니라 블로그 글처럼 작성하였습니다. 이는 논리 순서와 주장 강도를 책임지고 확인하기 위한 계층이며 글입니다.

이 본문은 나중에 HTML slide, speaker note, PDF 를 구성할 때 source of truth 로 사용되었습니다. slide는 본문의 내용을 압축할 수 있지만, 본문의 내용을 벗어날 수 없습니다. notes는 본문을 구어체로 구성할 수 있지만, slide 와 본문의 구성을 벗어날 수 없습니다. 

이 단계에서 사람이 개입한 것은 다음입니다.

- 전체 논지의 흐름
- 어떤 주장이 중심이고 어떤 주장이 보조인지
- 한국어 문체와 독자 경험
- 원본 소스의 누락 방지

이는 작업 루프를 돌면서 오류를 감지하였을 때 다시 돌아와 시작할 수 있는 기준점이 되었습니다. 

## **outline과 manifest가 계획을 들고 간다**

본문이 준비되고 곧바로 HTML 을 제작하는 것이 아니라, 먼저 outline을 만듭니다. outline은 단순한 목차가 아니라 승인 게이트입니다. 어떤 slide, 어떤 source section을 참조할지, 어떤 시각 구조가 필요한지 먼저 정합니다.

그 다음 manifest가 상태를 들고 갑니다. manifest는 작업이 끝난 뒤 쓰는 회고가 아니라, 현재 산출물의 상태를 추적하는 live document입니다.

- 어떤 본문이 어떤 slide 에 담겼는가
- 어떤 항목이 완료되었고 어떤 항목이 pending인가
- 어떤 파일 경로에 산출물이 있는가
- 어떤 검증을 통과했는가
- 어떤 변경이 남았는가

이 방식은 에이전트 작업에서도 그대로 중요합니다. 상태를 대화창 안에만 두면 세션이 바뀌는 순간 기준이 사라집니다. 반대로 manifest, 이슈, PR, 진행 파일이 상태를 들고 가면 세션이 끊겨도 작업은 이어집니다.

## **design contract를 먼저 고정했다**

design contract를 고정했습니다. 디자인 계약에는 다음이 들어갑니다.

- 톤과 분위기
- 색상 팔레트
- 타이포그래피
- 여백과 레이아웃 규칙
- slide family
- placeholder 규칙
- 금지해야 할 표현과 과한 장식

디자인 규칙을 정하기 위해, MS 의 슬라이드 디자인 규칙, Google Stitch 의 Design.md, 재사용 가능한 skills 로 https://github.com/Kuneosu/make-slide 를 사용했습니다. 

## **생성과 검증을 같은 손에 쥐지 않았다**

생성자가 자기 결과를 스스로 통과시키게 두면 그럴듯한 오류가 오래 살아남습니다. 그래서 이 프로젝트는 생성과 검증을 분리했습니다.

HTML generation 이후에는 기계적 검증과 사람 검토가 따릅니다.

- slide contract 점검
- 한국어 문장 점검
- deck-level consistency 확인
- slide-level validation
- source와 주장 매핑 확인
- 디자인 계약 위반 확인

이 원칙은 코드 작업과 같습니다. 에이전트가 코드를 생성했다면 같은 에이전트의 “괜찮습니다”만 믿으면 안 됩니다. 테스트, 린트, 타입 체크, 리뷰어, 다른 모델, 사람 검토가 분리되어야 합니다.

## **speaker notes를 맨 뒤로 밀었다**

많은 발표 작업은 대본부터 길어지면서 무너집니다. 설명하고 싶은 욕심이 먼저 커지고, 아직 굳지 않은 slide 구조를 말이 덮어버리기 때문입니다.

## **이 제작 과정이 보여 주는 하네스 원리**

이 사례의 핵심은 “AI가 HTML을 만들었다”가 아닙니다. 더 중요한 것은 AI에게 맡길 층과 맡기지 않을 층을 먼저 나눴다는 점입니다.

| **층** | **사람이 붙든 것** | **에이전트에게 맡긴 것** |
| --- | --- | --- |
| Source | 입력 경계, 근거 층위, 불확실한 질문 | 자료 정리 보조 |
| Prose | 논리 순서, 주장 강도, 문체 | 초안 정리와 변형 |
| Outline | 장표 단위 주장, 배치, 누락 확인 | 구조화 보조 |
| Design | 시각 문법과 제약 | 계약 안에서 HTML/CSS 생성 |
| Validation | 통과 기준과 최종 판단 | 기계적 체크와 수정 제안 |
| Notes | 발표 흐름과 말의 순서 | 대본 초안 보조 |

## **결론: AI 를 잘 사용하는 방법: 결국 하나**

원하는 것을 글로 작성 → 규칙을 문서로 작성 → 결과를 사람이 검증 

이 발표와 글이 보여 주는 진짜 포인트는 결과물이 아닙니다. 결과물이 만들어지는 경로입니다. source boundary를 잠그고, prose spine을 세우고, outline과 manifest를 만들고, design contract를 먼저 고정하고, 생성과 검증을 분리했습니다.

새로 배워야하는 것은 없습니다. 우리가 알고 있던 내용을 AI 시대에 더 의식적으로 밖으로 꺼내두어 사용해야 한다는 것입니다.
