# 코딩은 어디로 가고 있는가

## 질문을 바꿔야 한다

AI가 코딩을 대체하느냐는 질문은 자극적이지만, 실제로는 그다지 유용하지 않습니다. 지금 더 중요한 질문은 이것입니다. 개발자가 끝까지 직접 붙들어야 하는 일이 어디로 이동하고 있는가.

손으로 코드를 한 줄씩 쓰는 시간은 분명 줄고 있습니다. 대신 규칙을 정하고, 문맥을 고르고, 작업 단위를 나누고, 결과를 검증하는 시간이 빠르게 늘고 있습니다. 이 변화는 갑자기 튀어나온 것이 아니라 소프트웨어 역사 전체에서 반복돼 온 패턴의 연장선에 있습니다.

## 우리는 늘 직접 하던 일을 줄여 왔다

소프트웨어의 역사를 한 줄로 요약하면 이렇습니다. 직접 해야 할 일은 줄고, 더 높은 수준의 일을 설계하는 책임은 늘어났습니다.

| 시대 | 핵심 전환 | 사람에게 남은 일 |
| --- | --- | --- |
| 기계어와 어셈블리의 시대 | 비트와 레지스터를 직접 다루던 작업을 언어에 위임 | 하드웨어 제어보다 프로그램 구조를 설계하기 시작함 |
| 고급 언어의 시대 | 컴파일러가 세부 구현을 더 많이 맡기 시작함 | 알고리즘과 추상화 수준이 올라감 |
| 메모리 관리의 시대 | GC가 해제 책임의 상당 부분을 가져감 | 비즈니스 로직과 시스템 설계가 더 중요해짐 |
| 프레임워크의 시대 | 보일러플레이트와 반복 패턴을 프레임워크가 흡수 | 아키텍처와 서비스 경계 설계가 핵심이 됨 |
| 클라우드의 시대 | 인프라 운영을 플랫폼에 더 많이 위임 | 시스템 구성과 운영 전략 설계가 중요해짐 |
| 에이전트의 시대 | 구현 자체의 일부를 모델에게 위임 | 에이전트가 일할 환경과 검증 구조를 설계해야 함 |

매번 비슷한 반발도 함께 있었습니다. “그걸 직접 하지 않으면 진짜 개발이 아니다”라는 말입니다. 기계어에서 고급 언어로 넘어갈 때도, 수동 메모리 관리에서 GC로 넘어갈 때도, 프레임워크와 클라우드가 들어올 때도 비슷한 얘기가 나왔습니다. 하지만 시간이 지나면 늘 같은 결론이 나왔습니다. 직접 하는 일은 줄었고, 대신 더 큰 구조를 설계하는 일이 실력의 중심으로 올라왔습니다.

## 손보다 설계가 더 무거워진다

이 흐름은 지금 AI 도구 앞에서 다시 반복되고 있습니다. 예전에는 “코드를 빨리 치는 사람”이 눈에 띄는 생산성을 보였습니다. 이제는 “AI가 잘 일할 수 있도록 규칙과 컨텍스트와 검증 구조를 잘 짜는 사람”이 결과를 더 크게 바꿉니다.

겉보기에는 코드를 적게 치는 것처럼 보일 수 있습니다. 하지만 실제 책임은 더 가벼워지지 않습니다. 오히려 더 무거워집니다. 모델은 그럴듯한 초안을 빨리 만들 수 있지만, 무엇을 만들어야 하는지와 무엇을 만들면 안 되는지를 스스로 안정적으로 정의하지는 못합니다. 이 경계를 사람이 먼저 정하지 않으면 에이전트는 빠르게 잘못된 방향으로 달립니다.

## 문서가 실행 표면이 된다

이 변화에서 가장 눈에 띄는 부분은 문서의 역할입니다. 과거에도 규칙 문서와 개발 가이드는 있었습니다. 하지만 없다고 해서 시스템이 아예 멈추지는 않았습니다. 이제는 상황이 다릅니다. `CLAUDE.md`, `AGENTS.md`, `Cursor Rules`, `copilot-instructions.md` 같은 파일은 단순한 참고자료가 아니라 에이전트의 실제 행동을 바꾸는 실행 표면이 됩니다.

이런 문서에 적는 내용도 달라집니다. 예전에는 팀 설명용 문서였다면, 이제는 에이전트가 따라야 할 운영 규칙이 됩니다. 어떤 구조를 선호하는지, 어떤 도구는 금지인지, 어떤 테스트를 통과해야 하는지, 어떤 상황에서는 사람 승인을 받아야 하는지 같은 내용이 여기에 고정됩니다. 잘 정리된 문서는 AI 시대에 실행 가능한 운영 자산이 됩니다.

## 기본기는 덜 중요해진 것이 아니라 더 중요해진다

여기서 흔히 생기는 오해가 하나 있습니다. AI가 코드를 많이 대신 써 주니 이제는 시스템 이해력이 예전보다 덜 중요해지는 것 아니냐는 생각입니다. 실제로는 정반대입니다.

데모를 만들 때는 그럴듯한 화면과 동작을 빠르게 얻을 수 있습니다. 하지만 실서비스로 넘어가면 보안, 동시성, 캐싱, 데이터 모델링, 장애 복구, 배포 전략, 운영 비용 같은 문제들이 한꺼번에 등장합니다. 이런 문제는 프롬프트 몇 줄로 안정적으로 해결되지 않습니다. 어느 부분을 의심해야 하는지, 어느 단계에 강한 검증이 필요한지, 어떤 결정이 장기적으로 위험한지를 읽는 능력은 여전히 사람 쪽에 있습니다.

AI가 강해질수록 기본기는 덜 필요해지는 것이 아니라, 더 높은 자리에서 더 무겁게 작동합니다. 손끝에서 드러나던 숙련이 이제는 설계 판단과 검증 기준으로 드러나는 것입니다.

## 개발자의 역할은 타이피스트가 아니라 지휘자에 가까워진다

이 변화를 가장 잘 설명하는 비유는 건축가나 지휘자 쪽에 가깝습니다. 건축가는 벽돌을 직접 쌓는 사람이라기보다, 어떤 구조가 안전한지와 어떤 배치가 목적에 맞는지를 결정하는 사람입니다. 지휘자는 악기를 모두 직접 연주하지 않아도 전체의 완성도를 결정합니다.

AI 시대의 개발자도 비슷해집니다. 직접 모든 코드를 작성하는 시간은 줄더라도, 에이전트가 어떤 순서로 일할지, 어떤 기준으로 멈출지, 어떤 결과만 통과시킬지 결정하는 책임은 오히려 커집니다. 그래서 앞으로 더 가치가 커지는 역할은 “코드를 가장 빨리 쓰는 사람”보다 “에이전트가 좋은 결과를 내는 환경을 설계하는 사람”입니다.

## 정리

코딩이 사라지는 것은 아닙니다. 다만 코딩의 무게중심이 이동하고 있습니다. 손으로 구현하는 비중은 줄고, 설계와 운영과 검증의 비중은 더 커집니다. 숙련이 없어지는 것이 아니라, 더 위의 추상화 수준으로 옮겨 가는 셈입니다.

이 변화는 취향의 문제가 아닙니다. 최근의 도구와 방법론, 팀 운영 방식이 모두 같은 방향으로 움직이고 있습니다. 다음 장에서는 왜 최신 도구들이 결국 하네스 중심 구조로 수렴하는지 그 흐름을 더 자세히 보겠습니다.
