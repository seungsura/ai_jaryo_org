# 에이전트는 왜 자꾸 길을 잃는가

## 실패는 예외가 아니라 기본값이다

AI 코딩 에이전트를 몇 번만 길게 돌려 봐도 알게 되는 사실이 있습니다. 문제는 한 번의 대형 사고만이 아닙니다. 더 자주 문제를 만드는 것은 그럴듯하게 지나간 작은 오류입니다.

에이전트는 읽고, 수정하고, 실행하고, 다시 판단하는 루프를 반복합니다. 이 구조에서는 초반의 작은 오해도 다음 단계의 입력으로 들어갑니다. 잘못된 가정이 계획으로 굳고, 그 계획이 코드 수정으로 이어지고, 그 수정 결과가 다시 검증 대상이 됩니다. 시간이 지날수록 어디서부터 어긋났는지 추적하기 어려워집니다.

그래서 긴 작업에서 중요한 것은 한 번의 답변 품질이 아니라, 실패가 누적되고 증폭되는 방식 전체를 이해하는 일입니다.

## 긴 작업 사슬이 특히 위험한 이유

짧은 작업에서는 모델의 즉흥적 영리함이 꽤 잘 먹힙니다. 하지만 작업이 길어질수록 기준이 달라집니다. 이제 문제는 “이번 답이 괜찮은가”가 아니라 “스무 단계 뒤에도 이 흐름이 버티는가”가 됩니다.

긴 사슬에서는 작은 오류가 비용으로 바뀝니다. 잘못된 파일을 기준으로 계획을 세우고, 이미 깨진 테스트 결과를 정상 상태로 오해하고, 오래된 문맥을 현재 상태처럼 읽기 시작합니다. 이 시점부터는 더 좋은 프롬프트보다 더 좋은 작업 분해가 중요합니다. 어디서 끊고, 어디서 검증하고, 무엇을 다음 단계로 넘기기 전에 고정할지를 먼저 결정해야 합니다.

## 컨텍스트가 길수록 항상 좋은 것은 아니다

긴 컨텍스트는 분명 강력한 기능입니다. 하지만 많은 팀이 여기서 잘못된 직관을 갖습니다. 더 많이 넣으면 더 잘할 것이라고 믿는 것입니다.

실전에서는 정반대의 장면이 자주 나옵니다. 저장소 전체를 통째로 밀어 넣고, 긴 로그와 오래된 대화까지 다 붙이면 판단력이 넓어지는 대신 흐려집니다. 중요한 것은 컨텍스트의 양이 아니라 구성입니다. 지금 단계에 필요한 정보가 앞에 있고, 끝난 시도와 잡음은 바깥으로 밀려나 있어야 합니다.

좋은 컨텍스트는 넓은 컨텍스트가 아니라 선별된 컨텍스트입니다.

## 대표 실패 패턴 네 가지

긴 세션에서 반복해서 나타나는 실패는 대체로 비슷한 얼굴을 하고 있습니다.

첫째는 `Poisoning`입니다. 오래된 가정이나 잘못된 메모가 이후 판단 전체를 오염시키는 경우입니다. 처음에 잘못 넣은 정보 하나가 루프 전체를 틀어버립니다.

둘째는 `Distraction`입니다. 관련 정보가 너무 많아 정작 핵심 신호가 묻히는 경우입니다. 에이전트는 필요한 파일을 보고도 우선순위를 잃어버립니다.

셋째는 `Confusion`입니다. 과거 결정과 현재 결정이 한 창 안에서 뒤엉켜, 무엇이 최신 기준인지 분간하지 못하는 경우입니다. 길게 이어진 세션에서 특히 자주 보입니다.

넷째는 `Clash`입니다. 서로 다른 지시와 기준이 같은 컨텍스트 안에 섞여 충돌하는 경우입니다. 예를 들어 오래된 규칙과 새 규칙이 함께 남아 있을 때 이런 일이 벌어집니다.

이 네 가지는 이름이 다를 뿐, 결국 하나의 문제를 가리킵니다. 기억과 잡음과 규칙이 한 창 안에서 썩고 있다는 뜻입니다.

## 실무에서 보이는 더 노골적인 증상들

실패는 조금 더 노골적인 형태로도 드러납니다.

`AI Slop`은 산출물이 급격히 조잡해지는 상태를 가리킵니다. 그럴듯해 보이지만 품질 기준이 빠르게 무너집니다.

`Doom Loop`는 같은 실수를 다른 표현으로 반복하는 상태입니다. 테스트를 고치지 못한 채 프롬프트만 바꾸거나, 같은 파일을 돌려 만지면서 진행이 없는 경우가 여기에 가깝습니다.

`Shadow Agent`는 지금 왜 이 작업을 하고 있는지, 어떤 기준을 따라야 하는지를 잊어버린 채 움직이는 상태입니다. 겉으로는 계속 일하고 있지만 실제로는 방향 감각이 무너진 상태라고 보는 편이 맞습니다.

이 증상들을 보면 흔히 “오늘 모델이 이상하다”고 말하고 넘어갑니다. 하지만 중요한 것은 구조입니다. 어떤 검증이 비어 있었는지, 어떤 상태가 파일로 밖에 남지 않았는지, 어디서 루프를 끊었어야 했는지를 먼저 봐야 합니다.

## Context Rot: 길어진 기억은 조용히 썩는다

긴 세션의 진짜 문제는 단순한 길이가 아닙니다. 시간이 지나면서 초반의 의도와 제약이 점점 희미해진다는 점입니다. 처음에는 분명했던 규칙이 뒤로 갈수록 약해지고, 중간에 생긴 가정이 사실처럼 굳고, 오래된 도구 결과가 아직도 유효한 정보처럼 남습니다.

이 상태를 흔히 `Context Rot`이라고 부릅니다. 세션을 오래 이어 가는 것이 곧 숙련은 아닙니다. 오히려 잘하는 팀일수록 끊을 타이밍을 더 잘 압니다. 현재 결론과 다음 행동만 남기고 새 세션으로 넘어갈 줄 알아야 합니다.

상태를 대화창 안에만 두면 결국 썩습니다. 파일, 이슈, 진행 기록, 테스트 결과, Git 이력처럼 바깥의 아티팩트에 남겨야 기억이 살아남습니다.

## 신뢰는 조율되어야 한다

AI를 믿느냐 마느냐는 감정 문제가 아닙니다. 어떤 작업에 어느 정도 자율성을 줄 것인지 조절하는 운영 문제입니다. 이것을 `Calibrated Trust`라고 볼 수 있습니다.

초안 작성, 반복적인 리팩터링, 구조 탐색처럼 비용이 낮은 구간에는 비교적 넓은 자율성을 줄 수 있습니다. 반대로 배포, 삭제, 민감 데이터 접근, 최신 API 사용, 외부 인용, 보안 관련 코드처럼 틀렸을 때 비용이 큰 구간에는 더 거친 검증이 필요합니다. 이 차이를 구조로 고정하지 않으면, 유창함을 정확성으로 착각하게 됩니다.

에이전트가 말을 잘하는 것과, 그 결과를 그대로 믿어도 된다는 것은 전혀 다른 문제입니다.

## 실패를 줄이는 실제 방법

실패를 줄이는 가장 현실적인 방법은 자동화를 더 세게 미는 것이 아닙니다. 작업 단위를 더 잘 끊고, 상태를 바깥에 남기고, 위험한 구간에 더 강한 게이트를 세우는 것입니다.

긴 세션을 무조건 유지하기보다 압축하고 새로 시작할 줄 알아야 합니다. 관련 없는 도구와 파일은 컨텍스트에서 빼야 합니다. 현재 결정과 남은 과제는 반드시 파일이나 이슈로 고정해야 합니다. 그리고 무엇보다 결과를 그냥 읽고 넘어가지 말고, 테스트와 린트와 승인 같은 결정론적 문을 통과하게 해야 합니다.

## 정리

에이전트가 길을 잃는 이유는 모델이 멍청해서가 아닙니다. 에이전트 작업이 본질적으로 누적되는 루프이기 때문입니다. 이 루프를 통제할 구조가 없으면, 작은 오류는 조용히 커집니다.

그래서 실패 패턴을 이해하는 일은 부정적인 부록이 아니라 핵심 설계 과제입니다. 다음 장에서는 이 실패를 줄이기 위해 어떤 구조가 필요한지, 좋은 하네스는 무엇으로 이루어져 있는지를 본격적으로 살펴보겠습니다.
