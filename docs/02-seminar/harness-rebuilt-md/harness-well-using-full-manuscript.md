# Harness 잘 사용하기

> 챗봇과 싸우지 않고, 에이전트가 일할 환경을 설계하는 방법

## 왜 지금 하네스를 이야기해야 하는가

AI 코딩 도구는 이제 더 이상 신기한 데모가 아닙니다. 간단한 함수 하나를 고치는 수준을 넘어, 여러 파일을 읽고 바꾸고 테스트를 돌리고 PR까지 만드는 흐름이 빠르게 일반화되고 있습니다. 문제는 여기서부터 시작됩니다. 한 번 그럴듯한 답을 내놓는 것과, 긴 작업을 끝까지 안정적으로 수행하는 것은 전혀 다른 문제이기 때문입니다.

짧은 작업에서는 모델의 말솜씨가 좋아 보입니다. 하지만 세션이 길어지고, 도구 호출이 늘고, 상태가 누적되기 시작하면 병목은 다른 곳에서 드러납니다. 무엇을 읽게 할 것인지, 무엇을 못 하게 막을 것인지, 어느 단계에서 멈춰 검증할 것인지, 세션이 바뀌어도 상태를 어떻게 남길 것인지가 결과의 품질을 갈라놓습니다. 이 바깥 구조를 여기서는 `Harness`라고 부르겠습니다.

이 글의 핵심 메시지는 단순합니다. AI 시대의 경쟁력은 더 좋은 모델을 찾는 데서 끝나지 않습니다. 더 좋은 하네스를 설계하고 운영하는 쪽으로 이동하고 있습니다.

## 이 글이 다루는 범위

이 글은 특정 제품의 사용기를 정리한 글이 아닙니다. 물론 Claude Code, Cursor, Copilot Coding Agent 같은 사례를 중간중간 언급하겠지만, 중심은 도구 이름이 아니라 구조입니다. 지금 우리가 보고 싶은 것은 왜 최근의 AI 코딩 도구와 실무 방법론이 비슷한 방향으로 수렴하는가입니다.

조금 더 구체적으로 말하면, 이 글은 네 가지 질문에 답하려고 합니다.

첫째, 개발자의 일이 정말 줄어드는지, 아니면 다른 층위로 이동하는지 살펴봅니다. 둘째, 프롬프트와 컨텍스트만으로는 왜 오래 버티지 못하는지 설명합니다. 셋째, 에이전트가 실제로 일할 수 있게 만드는 하네스의 구조를 해부합니다. 넷째, 이 원리를 실전 워크플로우와 팀 운영에 어떻게 연결할 수 있는지 정리합니다.

## 프롬프트, 컨텍스트, 하네스

초기에는 프롬프트가 전부처럼 보였습니다. 어떻게 말해야 더 좋은 답이 나오느냐가 가장 중요한 문제처럼 보였기 때문입니다. 곧 컨텍스트가 더 중요해졌습니다. 좋은 지시를 내려도 모델이 필요한 파일과 배경지식을 보지 못하면 계속 같은 실수를 반복했기 때문입니다.

하지만 실제 운영은 여기서 멈추지 않습니다. 에이전트가 한 번 답하고 끝나는 존재가 아니라, 읽고, 수정하고, 실행하고, 검증하고, 다시 시도하는 루프를 돌기 시작하면 프롬프트와 컨텍스트만으로는 부족합니다. 권한, 상태, 검증, 로깅, 복구, 사람 개입 지점까지 설계해야 합니다. 이 지점에서 하네스는 선택이 아니라 필수가 됩니다.

## 이 글의 읽는 순서

먼저 개발의 무게중심이 어떻게 이동하는지 봅니다. 그다음 최근 도구들이 왜 하네스 중심으로 수렴하는지 정리합니다. 이어서 에이전트가 길을 잃는 대표적 실패 패턴을 먼저 다룹니다. 실패를 본 뒤에야 좋은 하네스의 구조가 왜 필요한지 더 선명하게 보이기 때문입니다.

그 다음에는 구현 전에 고정해야 할 방법론을 다룹니다. `Spec-first`, `SDD`, `TDD`, `Plan-Critic-Build` 같은 원칙은 AI 때문에 낡아진 것이 아니라, 오히려 AI 때문에 더 중요해졌습니다. 이어서 멀티 에이전트 패턴과 실전 운영 모델을 정리하고, 마지막에는 이 발표 자체가 어떤 구조로 만들어졌는지를 사례로 설명합니다.

## 미리 적어 두는 결론

좋은 에이전트를 만드는 방법은 챗봇에게 더 공손하게 말하는 데 있지 않습니다. 에이전트가 일할 환경을 잘 설계하는 데 있습니다. 무엇을 볼지, 무엇을 못 할지, 무엇으로 검증할지, 무엇을 파일과 기록으로 남길지 결정하는 쪽이 결국 결과를 바꿉니다.

그래서 AI 시대에 개발자의 역할은 사라지지 않습니다. 다만 손으로 코드를 밀어 넣는 비중은 줄고, 설계와 검증과 운영의 비중은 더 커집니다. 이 글은 바로 그 이동을 설명하는 글입니다.

# 코딩은 어디로 가고 있는가

## 질문을 바꿔야 한다

AI가 코딩을 대체하느냐는 질문은 자극적이지만, 실제로는 그다지 유용하지 않습니다. 지금 더 중요한 질문은 이것입니다. 개발자가 끝까지 직접 붙들어야 하는 일이 어디로 이동하고 있는가.

손으로 코드를 한 줄씩 쓰는 시간은 분명 줄고 있습니다. 대신 규칙을 정하고, 문맥을 고르고, 작업 단위를 나누고, 결과를 검증하는 시간이 빠르게 늘고 있습니다. 이 변화는 갑자기 튀어나온 것이 아니라 소프트웨어 역사 전체에서 반복돼 온 패턴의 연장선에 있습니다.

## 우리는 늘 직접 하던 일을 줄여 왔다

소프트웨어의 역사를 한 줄로 요약하면 이렇습니다. 직접 해야 할 일은 줄고, 더 높은 수준의 일을 설계하는 책임은 늘어났습니다.

| 시대 | 핵심 전환 | 사람에게 남은 일 |
| --- | --- | --- |
| 기계어와 어셈블리의 시대 | 비트와 레지스터를 직접 다루던 작업을 언어에 위임 | 하드웨어 제어보다 프로그램 구조를 설계하기 시작함 |
| 고급 언어의 시대 | 컴파일러가 세부 구현을 더 많이 맡기 시작함 | 알고리즘과 추상화 수준이 올라감 |
| 메모리 관리의 시대 | GC가 해제 책임의 상당 부분을 가져감 | 비즈니스 로직과 시스템 설계가 더 중요해짐 |
| 프레임워크의 시대 | 보일러플레이트와 반복 패턴을 프레임워크가 흡수 | 아키텍처와 서비스 경계 설계가 핵심이 됨 |
| 클라우드의 시대 | 인프라 운영을 플랫폼에 더 많이 위임 | 시스템 구성과 운영 전략 설계가 중요해짐 |
| 에이전트의 시대 | 구현 자체의 일부를 모델에게 위임 | 에이전트가 일할 환경과 검증 구조를 설계해야 함 |

매번 비슷한 반발도 함께 있었습니다. “그걸 직접 하지 않으면 진짜 개발이 아니다”라는 말입니다. 기계어에서 고급 언어로 넘어갈 때도, 수동 메모리 관리에서 GC로 넘어갈 때도, 프레임워크와 클라우드가 들어올 때도 비슷한 얘기가 나왔습니다. 하지만 시간이 지나면 늘 같은 결론이 나왔습니다. 직접 하는 일은 줄었고, 대신 더 큰 구조를 설계하는 일이 실력의 중심으로 올라왔습니다.

## 손보다 설계가 더 무거워진다

이 흐름은 지금 AI 도구 앞에서 다시 반복되고 있습니다. 예전에는 “코드를 빨리 치는 사람”이 눈에 띄는 생산성을 보였습니다. 이제는 “AI가 잘 일할 수 있도록 규칙과 컨텍스트와 검증 구조를 잘 짜는 사람”이 결과를 더 크게 바꿉니다.

겉보기에는 코드를 적게 치는 것처럼 보일 수 있습니다. 하지만 실제 책임은 더 가벼워지지 않습니다. 오히려 더 무거워집니다. 모델은 그럴듯한 초안을 빨리 만들 수 있지만, 무엇을 만들어야 하는지와 무엇을 만들면 안 되는지를 스스로 안정적으로 정의하지는 못합니다. 이 경계를 사람이 먼저 정하지 않으면 에이전트는 빠르게 잘못된 방향으로 달립니다.

## 문서가 실행 표면이 된다

이 변화에서 가장 눈에 띄는 부분은 문서의 역할입니다. 과거에도 규칙 문서와 개발 가이드는 있었습니다. 하지만 없다고 해서 시스템이 아예 멈추지는 않았습니다. 이제는 상황이 다릅니다. `CLAUDE.md`, `AGENTS.md`, `Cursor Rules`, `copilot-instructions.md` 같은 파일은 단순한 참고자료가 아니라 에이전트의 실제 행동을 바꾸는 실행 표면이 됩니다.

이런 문서에 적는 내용도 달라집니다. 예전에는 팀 설명용 문서였다면, 이제는 에이전트가 따라야 할 운영 규칙이 됩니다. 어떤 구조를 선호하는지, 어떤 도구는 금지인지, 어떤 테스트를 통과해야 하는지, 어떤 상황에서는 사람 승인을 받아야 하는지 같은 내용이 여기에 고정됩니다. 잘 정리된 문서는 AI 시대에 실행 가능한 운영 자산이 됩니다.

## 기본기는 덜 중요해진 것이 아니라 더 중요해진다

여기서 흔히 생기는 오해가 하나 있습니다. AI가 코드를 많이 대신 써 주니 이제는 시스템 이해력이 예전보다 덜 중요해지는 것 아니냐는 생각입니다. 실제로는 정반대입니다.

데모를 만들 때는 그럴듯한 화면과 동작을 빠르게 얻을 수 있습니다. 하지만 실서비스로 넘어가면 보안, 동시성, 캐싱, 데이터 모델링, 장애 복구, 배포 전략, 운영 비용 같은 문제들이 한꺼번에 등장합니다. 이런 문제는 프롬프트 몇 줄로 안정적으로 해결되지 않습니다. 어느 부분을 의심해야 하는지, 어느 단계에 강한 검증이 필요한지, 어떤 결정이 장기적으로 위험한지를 읽는 능력은 여전히 사람 쪽에 있습니다.

AI가 강해질수록 기본기는 덜 필요해지는 것이 아니라, 더 높은 자리에서 더 무겁게 작동합니다. 손끝에서 드러나던 숙련이 이제는 설계 판단과 검증 기준으로 드러나는 것입니다.

## 개발자의 역할은 타이피스트가 아니라 지휘자에 가까워진다

이 변화를 가장 잘 설명하는 비유는 건축가나 지휘자 쪽에 가깝습니다. 건축가는 벽돌을 직접 쌓는 사람이라기보다, 어떤 구조가 안전한지와 어떤 배치가 목적에 맞는지를 결정하는 사람입니다. 지휘자는 악기를 모두 직접 연주하지 않아도 전체의 완성도를 결정합니다.

AI 시대의 개발자도 비슷해집니다. 직접 모든 코드를 작성하는 시간은 줄더라도, 에이전트가 어떤 순서로 일할지, 어떤 기준으로 멈출지, 어떤 결과만 통과시킬지 결정하는 책임은 오히려 커집니다. 그래서 앞으로 더 가치가 커지는 역할은 “코드를 가장 빨리 쓰는 사람”보다 “에이전트가 좋은 결과를 내는 환경을 설계하는 사람”입니다.

## 정리

코딩이 사라지는 것은 아닙니다. 다만 코딩의 무게중심이 이동하고 있습니다. 손으로 구현하는 비중은 줄고, 설계와 운영과 검증의 비중은 더 커집니다. 숙련이 없어지는 것이 아니라, 더 위의 추상화 수준으로 옮겨 가는 셈입니다.

이 변화는 취향의 문제가 아닙니다. 최근의 도구와 방법론, 팀 운영 방식이 모두 같은 방향으로 움직이고 있습니다. 다음 장에서는 왜 최신 도구들이 결국 하네스 중심 구조로 수렴하는지 그 흐름을 더 자세히 보겠습니다.

# 도구는 왜 하네스로 수렴하는가

## 도구 이름보다 구조를 봐야 한다

최근 몇 년 사이 AI 코딩 도구는 폭발적으로 늘어났습니다. GitHub Copilot, Cursor, Windsurf, Claude Code, Codex, Copilot Coding Agent처럼 이름도 많고 출발점도 다릅니다. 어떤 도구는 자동완성에서 시작했고, 어떤 도구는 채팅형 편집기에서 시작했고, 어떤 도구는 터미널 에이전트로 출발했습니다.

그런데 긴 작업을 맡기기 시작하면 거의 같은 문제가 드러납니다. 무엇을 읽힐 것인가, 어떤 도구를 열어 둘 것인가, 위험한 명령은 어떻게 막을 것인가, 결과를 어디서 검증할 것인가, 세션이 바뀌어도 상태를 어떻게 유지할 것인가. 결국 도구 경쟁은 모델 단품의 성능보다 그 모델을 어떤 작업 환경 위에 올려놓느냐의 경쟁으로 이동합니다.

이때 필요한 렌즈가 바로 `Harness`입니다.

## 세 번의 이동

최근의 흐름은 대략 세 단계로 정리할 수 있습니다.

첫 번째는 프롬프트 중심의 시기였습니다. 핵심 질문은 “어떻게 말해야 하나”였습니다. ChatGPT 초기 사용법, Chain-of-Thought, ReAct 같은 패턴이 이 시기의 관심사였습니다. 모델이 무엇을 하게 만들지를 자연어 지시문으로 조정하는 일이 중심이었습니다.

두 번째는 컨텍스트 중심의 시기였습니다. 질문이 “어떤 정보를 넣어야 하나”로 바뀌었습니다. 현재 파일 한 장만 보는 도구로는 프로젝트 전체를 다루기 어려웠기 때문입니다. 이 시기에 RAG, 코드베이스 검색, 시맨틱 인덱싱, 긴 컨텍스트 최적화 같은 기법들이 전면에 올라왔습니다.

세 번째가 하네스 중심의 시기입니다. 여기서 질문은 다시 바뀝니다. “어떤 시스템을 만들어야 하나”입니다. 에이전트가 여러 단계를 거쳐 오래 일하기 시작하면, 단순히 좋은 프롬프트와 좋은 컨텍스트만으로는 충분하지 않기 때문입니다.

## 왜 프롬프트만으로는 부족했는가

초기의 코딩 도구는 비교적 짧은 상호작용에 강했습니다. 코드를 설명해 달라거나, 작은 리팩터링을 제안받거나, 함수 하나를 만들어 달라는 수준에서는 프롬프트 품질이 크게 작동했습니다.

하지만 프로젝트가 커지면 한계가 분명했습니다. 프롬프트에 “기존 유틸리티를 재사용하라”고 아무리 잘 적어도, 모델이 그 유틸리티 파일을 보지 못하면 소용이 없습니다. 더 정확히 말하면 문제는 지시문이 아니라 지시문이 소비하는 정보의 범위에 있었습니다.

그래서 프롬프트의 시대는 곧 컨텍스트의 시대로 넘어갔습니다.

## 왜 컨텍스트만으로도 부족했는가

컨텍스트 엔지니어링은 큰 진전이었습니다. 전체 코드베이스에서 관련 파일을 찾아 보여 주고, 검색 결과를 요약하고, 긴 히스토리를 압축해 다음 턴으로 넘기는 기술은 실제로 품질을 크게 올렸습니다.

하지만 여기에도 벽이 있었습니다. 컨텍스트는 기본적으로 “이번 API 호출에 무엇을 넣을 것인가”에 가깝습니다. 반면 에이전트는 한 번 호출하고 끝나지 않습니다. 자료를 읽고, 파일을 바꾸고, 테스트를 돌리고, 오류를 보고, 다시 계획을 바꾸는 루프를 계속 반복합니다.

이 루프에서는 다른 종류의 문제가 생깁니다. 잘못된 상태가 누적되고, 실패가 조용히 다음 단계로 넘어가고, 도구 권한이 너무 넓어 사고 가능성이 커지고, 세션이 길어질수록 처음의 의도가 흐려집니다. 컨텍스트를 잘 넣는 것만으로는 해결되지 않는 문제들입니다.

그래서 필요한 것이 하네스입니다.

## 하네스는 에이전트 바깥의 나머지 전부다

하네스를 한 문장으로 정리하면 이렇습니다.

> `Agent = Model + Harness`

모델은 추론과 생성 엔진입니다. 하지만 실제로 에이전트의 성격을 결정하는 것은 그 엔진 바깥의 구조입니다. 무엇을 읽게 할지, 어떤 도구를 쓸 수 있을지, 어떤 단계에서 사람 승인이 필요한지, 어떤 테스트를 통과해야 하는지, 실패했을 때 어떻게 멈추고 복구할지가 모두 하네스의 영역입니다.

같은 모델을 써도 결과가 크게 달라지는 이유가 여기에 있습니다. 프롬프트를 조금 다듬는 것보다, 모델을 더 나은 작업 환경 위에 올려놓는 편이 훨씬 큰 차이를 만드는 구간이 분명히 존재합니다.

## 왜 Claude Code가 자주 예시로 등장하는가

이 글에서 Claude Code를 자주 언급하는 이유도 이 때문입니다. 특정 제품을 홍보하려는 의도가 아니라, 현재 하네스 중심 구조를 가장 노골적으로 드러내는 대표 사례이기 때문입니다.

터미널 네이티브 워크플로우, 계획 모드, 승인 구조, `CLAUDE.md`, `Skills`, `Hooks`, `MCP`, 서브에이전트, 장시간 작업을 전제로 한 루프 설계 같은 요소들이 모두 한 화면에 모여 있습니다. 다시 말해 “좋은 에이전트는 어떤 환경 위에서 굴러가야 하는가”라는 질문에 가장 직접적으로 답하는 도구 중 하나입니다.

하지만 핵심은 Claude Code 자체가 아닙니다. Cursor의 Agent Mode, Copilot Coding Agent, 여러 오픈소스 하니스 패턴도 결국 비슷한 구조로 움직입니다. 출발점은 달라도 수렴점이 비슷하다는 것이 더 중요합니다.

## 도구 경쟁의 진짜 축

예전에는 누가 더 좋은 모델을 쓰느냐가 가장 눈에 띄는 차이였습니다. 이제는 그 모델을 어떤 운영 구조 위에 얹느냐가 더 중요해지고 있습니다.

좋은 도구는 단순히 코드를 잘 생성하는 도구가 아닙니다. 컨텍스트를 선별하고, 도구를 큐레이션하고, 검증 게이트를 두고, 상태를 바깥에 남기고, 실패를 복구하고, 사람이 개입할 타이밍을 명확하게 설계한 도구입니다. 이 구조가 강할수록 모델의 확률적 특성이 더 안정적으로 다뤄집니다.

결국 요즘의 도구 경쟁은 “누가 더 똑똑한 모델을 붙였는가”보다 “누가 더 강한 하네스를 제공하는가”의 경쟁으로 바뀌고 있습니다.

## 정리

프롬프트는 여전히 중요합니다. 컨텍스트도 여전히 중요합니다. 하지만 둘만으로는 긴 작업을 안정적으로 운영할 수 없습니다. 에이전트가 루프를 돌기 시작하는 순간, 권한, 상태, 검증, 복구, 관측 같은 바깥 구조가 더 중요해집니다.

그래서 최근의 도구와 실무 방법론은 결국 하네스 중심 구조로 수렴합니다. 다음 장에서는 왜 에이전트가 그렇게 자주 길을 잃는지, 그리고 어떤 실패 패턴이 반복해서 나타나는지를 먼저 살펴보겠습니다.

# 에이전트는 왜 자꾸 길을 잃는가

## 실패는 예외가 아니라 기본값이다

AI 코딩 에이전트를 몇 번만 길게 돌려 봐도 알게 되는 사실이 있습니다. 문제는 한 번의 대형 사고만이 아닙니다. 더 자주 문제를 만드는 것은 그럴듯하게 지나간 작은 오류입니다.

에이전트는 읽고, 수정하고, 실행하고, 다시 판단하는 루프를 반복합니다. 이 구조에서는 초반의 작은 오해도 다음 단계의 입력으로 들어갑니다. 잘못된 가정이 계획으로 굳고, 그 계획이 코드 수정으로 이어지고, 그 수정 결과가 다시 검증 대상이 됩니다. 시간이 지날수록 어디서부터 어긋났는지 추적하기 어려워집니다.

그래서 긴 작업에서 중요한 것은 한 번의 답변 품질이 아니라, 실패가 누적되고 증폭되는 방식 전체를 이해하는 일입니다.

## 긴 작업 사슬이 특히 위험한 이유

짧은 작업에서는 모델의 즉흥적 영리함이 꽤 잘 먹힙니다. 하지만 작업이 길어질수록 기준이 달라집니다. 이제 문제는 “이번 답이 괜찮은가”가 아니라 “스무 단계 뒤에도 이 흐름이 버티는가”가 됩니다.

긴 사슬에서는 작은 오류가 비용으로 바뀝니다. 잘못된 파일을 기준으로 계획을 세우고, 이미 깨진 테스트 결과를 정상 상태로 오해하고, 오래된 문맥을 현재 상태처럼 읽기 시작합니다. 이 시점부터는 더 좋은 프롬프트보다 더 좋은 작업 분해가 중요합니다. 어디서 끊고, 어디서 검증하고, 무엇을 다음 단계로 넘기기 전에 고정할지를 먼저 결정해야 합니다.

## 컨텍스트가 길수록 항상 좋은 것은 아니다

긴 컨텍스트는 분명 강력한 기능입니다. 하지만 많은 팀이 여기서 잘못된 직관을 갖습니다. 더 많이 넣으면 더 잘할 것이라고 믿는 것입니다.

실전에서는 정반대의 장면이 자주 나옵니다. 저장소 전체를 통째로 밀어 넣고, 긴 로그와 오래된 대화까지 다 붙이면 판단력이 넓어지는 대신 흐려집니다. 중요한 것은 컨텍스트의 양이 아니라 구성입니다. 지금 단계에 필요한 정보가 앞에 있고, 끝난 시도와 잡음은 바깥으로 밀려나 있어야 합니다.

좋은 컨텍스트는 넓은 컨텍스트가 아니라 선별된 컨텍스트입니다.

## 대표 실패 패턴 네 가지

긴 세션에서 반복해서 나타나는 실패는 대체로 비슷한 얼굴을 하고 있습니다.

첫째는 `Poisoning`입니다. 오래된 가정이나 잘못된 메모가 이후 판단 전체를 오염시키는 경우입니다. 처음에 잘못 넣은 정보 하나가 루프 전체를 틀어버립니다.

둘째는 `Distraction`입니다. 관련 정보가 너무 많아 정작 핵심 신호가 묻히는 경우입니다. 에이전트는 필요한 파일을 보고도 우선순위를 잃어버립니다.

셋째는 `Confusion`입니다. 과거 결정과 현재 결정이 한 창 안에서 뒤엉켜, 무엇이 최신 기준인지 분간하지 못하는 경우입니다. 길게 이어진 세션에서 특히 자주 보입니다.

넷째는 `Clash`입니다. 서로 다른 지시와 기준이 같은 컨텍스트 안에 섞여 충돌하는 경우입니다. 예를 들어 오래된 규칙과 새 규칙이 함께 남아 있을 때 이런 일이 벌어집니다.

이 네 가지는 이름이 다를 뿐, 결국 하나의 문제를 가리킵니다. 기억과 잡음과 규칙이 한 창 안에서 썩고 있다는 뜻입니다.

## 실무에서 보이는 더 노골적인 증상들

실패는 조금 더 노골적인 형태로도 드러납니다.

`AI Slop`은 산출물이 급격히 조잡해지는 상태를 가리킵니다. 그럴듯해 보이지만 품질 기준이 빠르게 무너집니다.

`Doom Loop`는 같은 실수를 다른 표현으로 반복하는 상태입니다. 테스트를 고치지 못한 채 프롬프트만 바꾸거나, 같은 파일을 돌려 만지면서 진행이 없는 경우가 여기에 가깝습니다.

`Shadow Agent`는 지금 왜 이 작업을 하고 있는지, 어떤 기준을 따라야 하는지를 잊어버린 채 움직이는 상태입니다. 겉으로는 계속 일하고 있지만 실제로는 방향 감각이 무너진 상태라고 보는 편이 맞습니다.

이 증상들을 보면 흔히 “오늘 모델이 이상하다”고 말하고 넘어갑니다. 하지만 중요한 것은 구조입니다. 어떤 검증이 비어 있었는지, 어떤 상태가 파일로 밖에 남지 않았는지, 어디서 루프를 끊었어야 했는지를 먼저 봐야 합니다.

## Context Rot: 길어진 기억은 조용히 썩는다

긴 세션의 진짜 문제는 단순한 길이가 아닙니다. 시간이 지나면서 초반의 의도와 제약이 점점 희미해진다는 점입니다. 처음에는 분명했던 규칙이 뒤로 갈수록 약해지고, 중간에 생긴 가정이 사실처럼 굳고, 오래된 도구 결과가 아직도 유효한 정보처럼 남습니다.

이 상태를 흔히 `Context Rot`이라고 부릅니다. 세션을 오래 이어 가는 것이 곧 숙련은 아닙니다. 오히려 잘하는 팀일수록 끊을 타이밍을 더 잘 압니다. 현재 결론과 다음 행동만 남기고 새 세션으로 넘어갈 줄 알아야 합니다.

상태를 대화창 안에만 두면 결국 썩습니다. 파일, 이슈, 진행 기록, 테스트 결과, Git 이력처럼 바깥의 아티팩트에 남겨야 기억이 살아남습니다.

## 신뢰는 조율되어야 한다

AI를 믿느냐 마느냐는 감정 문제가 아닙니다. 어떤 작업에 어느 정도 자율성을 줄 것인지 조절하는 운영 문제입니다. 이것을 `Calibrated Trust`라고 볼 수 있습니다.

초안 작성, 반복적인 리팩터링, 구조 탐색처럼 비용이 낮은 구간에는 비교적 넓은 자율성을 줄 수 있습니다. 반대로 배포, 삭제, 민감 데이터 접근, 최신 API 사용, 외부 인용, 보안 관련 코드처럼 틀렸을 때 비용이 큰 구간에는 더 거친 검증이 필요합니다. 이 차이를 구조로 고정하지 않으면, 유창함을 정확성으로 착각하게 됩니다.

에이전트가 말을 잘하는 것과, 그 결과를 그대로 믿어도 된다는 것은 전혀 다른 문제입니다.

## 실패를 줄이는 실제 방법

실패를 줄이는 가장 현실적인 방법은 자동화를 더 세게 미는 것이 아닙니다. 작업 단위를 더 잘 끊고, 상태를 바깥에 남기고, 위험한 구간에 더 강한 게이트를 세우는 것입니다.

긴 세션을 무조건 유지하기보다 압축하고 새로 시작할 줄 알아야 합니다. 관련 없는 도구와 파일은 컨텍스트에서 빼야 합니다. 현재 결정과 남은 과제는 반드시 파일이나 이슈로 고정해야 합니다. 그리고 무엇보다 결과를 그냥 읽고 넘어가지 말고, 테스트와 린트와 승인 같은 결정론적 문을 통과하게 해야 합니다.

## 정리

에이전트가 길을 잃는 이유는 모델이 멍청해서가 아닙니다. 에이전트 작업이 본질적으로 누적되는 루프이기 때문입니다. 이 루프를 통제할 구조가 없으면, 작은 오류는 조용히 커집니다.

그래서 실패 패턴을 이해하는 일은 부정적인 부록이 아니라 핵심 설계 과제입니다. 다음 장에서는 이 실패를 줄이기 위해 어떤 구조가 필요한지, 좋은 하네스는 무엇으로 이루어져 있는지를 본격적으로 살펴보겠습니다.

# 좋은 하네스의 해부학

## 문장보다 구조가 먼저다

프롬프트 엔지니어링이 중요하지 않다는 뜻은 아닙니다. 컨텍스트 엔지니어링이 끝났다는 뜻도 아닙니다. 다만 에이전트가 긴 루프를 돌기 시작하면 둘 다 더 큰 구조 안에 들어가야 제대로 작동합니다. 좋은 하네스는 프롬프트와 컨텍스트를 버리는 것이 아니라, 제자리에 놓는 기술입니다.

하네스를 이해하는 가장 쉬운 출발점은 이 문장입니다.

> `Agent = Model + Harness`

모델은 추론과 생성을 담당합니다. 하지만 실제 에이전트의 성격을 결정하는 것은 바깥 구조입니다. 무엇을 읽게 할지, 무엇을 못 하게 할지, 어디서 멈출지, 어떤 결과를 통과시킬지, 상태를 어떻게 남길지가 모두 하네스의 문제입니다.

## 하네스를 다섯 블록으로 나누어 보기

하네스는 추상적인 태도가 아니라 운영 도면입니다. 손에 잡히게 보려면 다섯 블록으로 나누는 편이 가장 명확합니다.

### 1. Specification

무엇을 만들 것인지, 어디까지가 범위인지, 완료 기준은 무엇인지 고정하는 층입니다. 스펙 문서, 계획 파일, 이슈 설명, 규칙 파일의 목적과 제약 조건이 여기에 들어갑니다. 방향이 고정되지 않은 상태에서 자율성만 높이면 에이전트는 잘못된 목적지를 향해 더 빨리 달릴 뿐입니다.

### 2. Guardrails

위험한 행동을 막고 권한 경계를 정하는 층입니다. 삭제 명령 차단, 민감 데이터 접근 제한, 승인 없이는 배포 금지, 외부 입력과 상태 변경을 동시에 허용하지 않는 규칙 같은 것들이 여기에 속합니다. 좋은 하네스는 자유를 최대화하는 대신, 넘어가면 안 되는 경계를 먼저 분명히 합니다.

### 3. Verification

결과를 결정론적으로 걸러내는 층입니다. `lint`, `type check`, 테스트, 리뷰 게이트, 빌드 검사, 커스텀 린터, 구조 검사 등이 여기에 들어갑니다. 핵심은 단순합니다. 기계가 확인할 수 있는 것은 기계가 먼저 막아야 합니다. AI가 잘 설명했다고 해서 통과시키는 구조는 오래 버티지 못합니다.

### 4. State Management

세션이 바뀌어도 상태가 증발하지 않게 만드는 층입니다. 작업 기록, 진행 파일, 스펙 문서, 이슈, PR, Git 이력, 외부 메모가 여기에 포함됩니다. 대화창 안에만 상태를 쌓아 두면 세션이 길어질수록 기억은 썩습니다. 상태는 밖에 남겨야 합니다.

### 5. Observability

루프가 어디서 흔들리는지 보이게 만드는 층입니다. 실행 로그, 도구 호출 기록, 체크포인트, 실패 이유, 비용과 토큰 사용량, 요약된 세션 기록이 여기에 들어갑니다. 관측이 없으면 실패가 생겨도 진단할 수 없습니다. 관측은 사후 분석용이 아니라 운영용입니다.

이 다섯 블록이 모두 있어야 하네스가 도구 목록이 아니라 구조로 보이기 시작합니다.

## 기능 블록과 도구 레이어는 다르다

여기서 자주 섞이는 것이 하나 있습니다. 하네스의 기능과 그것을 구현하는 도구를 같은 것으로 생각하는 것입니다.

예를 들어 `CLAUDE.md`, `AGENTS.md`, `Cursor Rules`는 하네스 전체가 아닙니다. 그것들은 특정 규칙과 기준을 고정하는 구현 수단일 뿐입니다. `Skills`는 재사용 가능한 절차를 담는 수단이고, `Hooks`는 우회할 수 없는 기계적 게이트를 거는 수단이며, `MCP`나 플러그인은 외부 도구와 데이터를 연결하는 수단입니다.

먼저 필요한 기능 블록을 정하고, 그다음 그것을 어떤 도구 레이어에 배치할지 정해야 구조가 깔끔해집니다. 이 순서가 뒤집히면 도구는 많아지는데 구조는 오히려 흐려집니다.

## 규칙 파일은 무엇을 고정해야 하는가

규칙 파일의 역할은 모든 것을 설명하는 데 있지 않습니다. 매 세션 다시 협상하면 낭비가 큰 기준선을 고정하는 데 있습니다.

예를 들면 이런 것들입니다. 프로젝트 전반의 코딩 스타일, 금지된 명령, 테스트와 리뷰 원칙, 특정 디렉터리 구조, 보안상 건드리면 안 되는 영역, 문서 작성 기준 같은 것들입니다. 이 기준은 자주 바뀌지 않고, 팀 전체에 일관되게 적용되어야 합니다. 그래서 상시 규칙 파일에 올려두는 편이 낫습니다.

반대로 특정 작업에서만 쓰는 절차를 전부 규칙 파일에 넣으면 금세 비대해집니다. 기준선과 일회성 절차가 뒤섞이면 규칙 파일은 기준이 아니라 잡음이 됩니다.

## Skills, Hooks, MCP는 서로 다른 책임을 가진다

이 차이를 분명하게 나눠야 하네스가 또렷해집니다.

`Skills`는 필요할 때 불러오는 작업 지식입니다. 특정 종류의 업무를 할 때만 꺼내 쓰는 절차, 체크리스트, 워크플로우를 여기에 둡니다. 반복되는 설명을 대화마다 다시 쓰지 않게 만드는 층입니다.

`Hooks`는 반드시 걸려야 하는 자동화된 문입니다. 예를 들어 포맷이 맞지 않으면 저장을 막거나, 위험한 명령을 실행 전에 차단하거나, 특정 작업 후 자동으로 테스트를 돌리는 것은 Hook의 성격에 가깝습니다. Skill이 “이렇게 하는 게 좋다”라면 Hook은 “이건 반드시 통과해야 한다”에 가깝습니다.

`MCP`와 플러그인은 외부 세계와 연결되는 층입니다. 저장소, 데이터베이스, 문서 도구, 이슈 시스템, 브라우저, 디자인 툴 같은 외부 자산과 안정적으로 연결해 줍니다. 이것도 중요하지만, 연결이 곧 하네스는 아닙니다. 어떤 연결을 언제 열고 닫을지까지 포함해야 하네스가 됩니다.

## 결정론적 제어와 확률적 제어를 분리해야 한다

좋은 하네스가 가장 잘하는 일 중 하나는 결정론적 제어와 확률적 제어를 분리하는 것입니다.

기계가 확실하게 검사할 수 있는 것은 기계에 맡겨야 합니다. 테스트, 린트, 타입 검사, 빌드, 정책 검사, 권한 차단이 여기에 해당합니다. 빨간불은 무조건 멈추게 해야 합니다.

반대로 설계 대안 탐색, 우선순위 판단, 코드 초안, 설명, 개선 방향 제안은 모델이 잘할 수 있는 영역입니다. 이 영역에서 중요한 것은 자유도를 주는 것입니다. 다만 그 자유가 결정론적 게이트를 건너뛰는 자유가 되어서는 안 됩니다.

이 둘을 분리하지 않으면, 에이전트는 검증까지 자기 마음대로 해 버립니다. 그 순간 구조는 무너집니다.

## 좋은 하네스는 덜어낼수록 강하다

하네스라고 해서 규칙과 도구를 계속 덧붙이는 것이 능사는 아닙니다. 컨텍스트에 쌓여 있는 유휴 도구, 거의 쓰지 않는 플러그인, 길고 모호한 규칙 파일은 에이전트의 판단력을 넓히지 않고 흐립니다.

좋은 하네스는 필요한 것만 남깁니다. 프로젝트 전역 규칙은 짧고 분명해야 하고, 특정 업무 절차는 Skill로 분리해야 하며, 우회할 수 없는 문만 Hook으로 남겨야 합니다. 도구도 실제로 쓰는 것만 열어 두는 편이 낫습니다. 많은 팀이 도구를 더 붙이는 데서 생산성을 찾으려 하지만, 실제로는 잘 덜어낸 구조가 더 오래 버팁니다.

## 정리

하네스는 추상적인 유행어가 아닙니다. 에이전트가 어떤 정보와 어떤 권한과 어떤 검증 구조 안에서 일하는지를 설계하는 실전 운영 구조입니다. `Specification`, `Guardrails`, `Verification`, `State Management`, `Observability`라는 다섯 블록으로 나누면 구조가 선명해집니다.

다음 장에서는 이 하네스를 실제 구현 전에 어떻게 고정할 것인지, 왜 `Spec-first`, `SDD`, `TDD`, `Plan-Critic-Build` 같은 방법론이 AI 시대에 다시 중요해졌는지를 보겠습니다.

# 구현보다 먼저 고정해야 할 것들

## AI 시대에 방법론이 다시 중요해진 이유

한동안 많은 팀이 AI 도구를 속도의 문제로만 봤습니다. 얼마나 빨리 초안을 만들 수 있는지, 얼마나 적은 입력으로 많은 코드를 뽑아낼 수 있는지가 중심 관심사였습니다. 하지만 실무에서는 금방 다른 사실이 드러났습니다. 병목은 생성 속도가 아니라 방향과 검증이었습니다.

무엇을 만들어야 하는지 애매한 상태에서 속도만 올리면, 에이전트는 잘못된 방향으로 더 빠르게 달립니다. 반대로 방향과 완료 기준이 먼저 고정되어 있으면, 생성 속도는 그때부터 비로소 생산성으로 바뀝니다. 그래서 예전의 방법론이 AI 때문에 무력해진 것이 아니라, 오히려 더 중요해졌습니다.

## `Spec-first`가 다시 앞에 오는 이유

AI 시대에 가장 먼저 살아난 사고방식은 구현보다 스펙을 먼저 고정하는 태도입니다. 이름은 다양합니다. `Spec-first`, `SDD`, 요구사항 중심 개발, 계획 우선 개발. 하지만 핵심은 같습니다. “어떻게 만들 것인가”에 들어가기 전에 “무엇을 만들 것인가”와 “어떤 상태를 성공으로 볼 것인가”를 먼저 못 박는 것입니다.

이 원칙이 특히 중요한 구간은 신규 기능, 큰 구조 변경, 여러 세션에 걸친 장기 작업입니다. 이때 스펙이 없으면 매 세션마다 방향이 흔들립니다. 반대로 스펙이 파일과 이슈로 고정되어 있으면, 에이전트는 매번 새로 길을 찾는 대신 같은 북극성을 참조할 수 있습니다.

좋은 스펙 문서는 화려한 기획서가 아닙니다. 목적, 범위, 제약, 수용 기준, 열려 있는 질문을 분명하게 적어 두는 문서입니다. 애매한 지점이 있으면 그대로 표시해야 합니다. 애매함을 감춘 채 구현으로 넘어가는 것이 가장 비싼 우회로이기 때문입니다.

## `SDD`는 워터폴이 아니다

스펙을 먼저 고정한다고 하면 곧장 “그럼 다시 워터폴로 돌아가는 것 아닌가”라는 반응이 나옵니다. 겉모습만 보면 비슷해 보일 수 있습니다. 둘 다 구현 전에 문서를 만드니까요. 하지만 작동 방식은 다릅니다.

워터폴은 앞단 문서를 고정 계약서처럼 취급하고, 뒤 단계는 그것을 거의 그대로 따라가게 만드는 선형 구조에 가깝습니다. 반면 AI 시대의 `SDD`는 실행 내내 참조되는 살아 있는 기준점입니다. 구현이 진행되면서 새로운 질문이 나오면 다시 스펙으로 돌아가 범위와 기준을 조정합니다. 중요한 것은 문서를 많이 쓰는 것이 아니라, 기준을 세션 밖에 남겨 두는 것입니다.

즉, `SDD`의 목적은 경직된 절차가 아니라 방향의 흔들림을 줄이는 데 있습니다.

## `TDD`는 품질 기법이자 권한 통제다

구현 단계로 내려오면 `TDD`의 의미가 다시 커집니다. AI는 그럴듯하게 작동하는 코드를 너무 쉽게 만들어 냅니다. 문제는 그것이 진짜로 의도한 동작을 만족하는지, 회귀를 막는지, 경계 조건에서 무너지지 않는지는 전혀 다른 문제라는 점입니다.

이때 테스트는 단순한 품질 보증 수단이 아닙니다. 에이전트의 재량을 묶는 가드레일입니다. 실패하는 테스트가 먼저 있으면, 구현은 그 테스트를 통과시키는 방향으로 좁혀집니다. 테스트가 없으면 에이전트는 설명이 그럴듯한 방향으로 자유롭게 미끄러집니다.

그래서 AI 시대의 `TDD`는 예전보다 더 운영적 성격이 강합니다. 테스트를 임의로 고치지 못하게 막고, 실패를 확인하기 전 구현을 금지하고, 구조 변경과 기능 추가를 분리하고, 테스트와 구현을 같은 커밋에서 무의식적으로 섞지 않게 해야 합니다. 테스트는 귀찮은 부수 작업이 아니라 품질을 왼쪽에서 붙드는 결정론적 문입니다.

## `Plan-Critic-Build`가 중요한 이유

스펙과 테스트 사이를 이어 주는 실전 운영 패턴이 `Plan-Critic-Build`입니다. 이름 그대로 계획을 세우고, 그것을 비평하고, 합의된 뒤에 실행하는 흐름입니다.

이 구조가 중요한 이유는 단순합니다. 많은 실패가 구현 중이 아니라 구현 전에 이미 예정되어 있기 때문입니다. 계획이 부실한 상태에서 코드로 바로 들어가면, 이후의 모든 수정은 잘못된 전제를 조금씩 만지는 일이 됩니다. 반대로 계획을 먼저 파일과 문서로 드러내고, 다른 시각으로 비평한 뒤에 구현으로 넘어가면 블라인드 스폿을 상당 부분 초반에 줄일 수 있습니다.

이 패턴은 거창한 프로세스를 뜻하지 않습니다. 작은 작업에서도 충분히 쓸 수 있습니다. 문제 정의, 성공 기준, 위험 포인트, 검증 방법만 먼저 분리해 적어도 효과가 큽니다.

## 좋은 방법론은 품질을 왼쪽으로 당긴다

AI와 일할 때 계속 기억해야 할 문장이 하나 있습니다. 기계가 미리 확인할 수 있는 것은 뒤가 아니라 앞에서 확인해야 한다는 것입니다. 이것을 흔히 `Keep Quality Left`라고 부를 수 있습니다.

스펙은 구현 전에 방향을 잡고, 계획 비평은 블라인드 스폿을 초기에 드러내고, 테스트는 기능이 완성되기 전에 기대 동작을 고정하고, 린트와 타입 검사는 실행 전에 기계적 오류를 거릅니다. 이 모든 구조의 목적은 같습니다. 실패를 가능한 한 앞단에서 드러내는 것입니다.

AI 시대에는 이 원칙이 더 중요합니다. 초안 생성 속도가 워낙 빨라졌기 때문에, 잘못된 방향 역시 놀라울 만큼 빨리 증폭되기 때문입니다.

## 스펙과 테스트는 함께 가야 한다

`SDD`만 있고 `TDD`가 없으면 문서화된 의도는 남지만 코드로 증명되지 않습니다. 반대로 `TDD`만 있고 `SDD`가 없으면 로컬 최적화는 잘 되지만 전체 방향이 흔들립니다. 하나는 “무엇을 만들 것인가”를 고정하고, 다른 하나는 “정말 그렇게 동작하는가”를 걸러냅니다.

둘은 대체 관계가 아니라 보완 관계입니다. 스펙이 북극성이라면 테스트는 그 방향으로 실제로 가고 있는지 확인하는 계기판에 가깝습니다. AI 시대에는 이 두 축이 함께 있을 때만 자율성이 생산성으로 바뀝니다.

## 실무에서 바로 적용하는 최소 단위

방법론을 거창하게 도입할 필요는 없습니다. 작은 팀이나 개인도 바로 시작할 수 있습니다.

새 기능을 만들기 전에 짧은 스펙 파일 하나를 남깁니다. 목적, 범위, 수용 기준, 열린 질문만 적어도 충분합니다. 구현에 들어가기 전 계획 메모를 한 번 분리합니다. 테스트를 먼저 쓰기 어려운 작업이라면 최소한 어떤 검증으로 성공을 판단할지 명시합니다. 그리고 구현 뒤에는 반드시 자동 검증을 통과시키는 문을 둡니다.

이 네 가지만 해도 대부분의 “그때그때 잘 될 것 같아서 진행했다가 나중에 더 비싸게 돌아오는 문제”를 상당히 줄일 수 있습니다.

## 정리

AI 시대에 방법론은 장식이 아닙니다. 에이전트의 자율성을 방향과 품질 안에 묶는 운영 규율입니다. `Spec-first`와 `SDD`가 방향을 고정하고, `TDD`와 자동 검증이 결과를 걸러내며, `Plan-Critic-Build`가 구현 전후를 잇습니다.

다음 장에서는 이 원칙이 여러 에이전트로 확장될 때 어떤 패턴으로 나타나는지 살펴보겠습니다. 멀티 에이전트의 핵심도 결국 같은 질문으로 돌아옵니다. 무엇을 누가 맡고, 무엇을 메인 컨텍스트 밖으로 밀어낼 것인가입니다.

# 멀티 에이전트 활용 패턴

## 병렬화보다 분해가 먼저다

멀티 에이전트를 생각할 때 가장 먼저 버려야 할 오해가 있습니다. AI를 더 많이 붙이면 자동으로 더 강해질 것이라는 생각입니다. 실제로 중요한 것은 숫자가 아니라 분해입니다. 무엇을 메인 컨텍스트에 남기고, 무엇을 밖으로 밀어내며, 누가 무엇을 검증하게 만들 것인지를 먼저 설계해야 합니다.

멀티 에이전트는 “더 많은 AI”가 아니라 “더 좁은 역할의 AI 여럿”에 가깝습니다. 역할과 컨텍스트와 검증 책임을 나누는 기술입니다.

## 1. `Sub-agent`: 가장 자주 이기는 기본형

가장 실용적인 패턴은 `Sub-agent`입니다. 탐색, 로그 분석, 자료 수집, 비교 작업처럼 토큰은 많이 먹지만 메인 판단에는 요약만 필요한 일을 분리할 때 강합니다.

구조는 단순합니다. 메인 에이전트가 일을 던지고, 격리된 서브 에이전트가 처리한 뒤, 핵심 결과만 요약해서 돌려줍니다. 이 패턴의 장점은 속도보다 격리입니다. 서브 에이전트가 파일을 많이 읽고 컨텍스트를 길게 써도 메인 세션은 오염되지 않습니다.

이때 중요한 원칙은 최소 권한입니다. 전체 대화와 전체 저장소를 다 넘길 필요는 없습니다. 관련 파일과 필요한 배경만 좁게 넘겨야 합니다. 서브 에이전트는 많이 읽되, 넓게 오염시키지 않는 역할이어야 합니다.

## 2. `Orchestrator`: 계획자가 여러 실행자를 배치한다

큰 목표를 한 번 더 구조적으로 쪼개야 할 때는 `Orchestrator` 패턴이 맞습니다. 하나의 에이전트가 계획을 세우고, 그 계획을 여러 실행 단위로 나눠 각각 맡기고, 마지막에 결과를 다시 모읍니다.

이 패턴은 계획자 없는 병렬이 쉽게 흩어지는 문제를 줄여 줍니다. 하위 작업끼리 깊게 토론할 필요는 없지만, 전체 목표와 순서를 누군가 붙들고 있어야 할 때 유용합니다. 말하자면 `Sub-agent`가 단일 심부름이라면 `Orchestrator`는 1 대 N 구조의 분업입니다.

## 3. `Parallel`: 의존성이 낮을 때만 병렬이 빛난다

병렬 실행은 분명 매력적입니다. 하지만 항상 좋은 기본값은 아닙니다. 서로 간섭이 적고 중간 상태를 공유할 필요가 적을 때만 진가를 냅니다.

예를 들어 세 가지 다른 접근으로 구조를 비교하거나, 독립적인 조사 작업을 동시에 진행하거나, 여러 초안을 나란히 뽑아 비교하는 구간에서는 병렬이 강합니다. 반대로 중간 상태를 계속 주고받아야 하는 작업이라면 병렬은 오히려 혼란을 키웁니다.

좋은 병렬은 “다 같이 달린다”가 아니라 “서로를 더럽히지 않는다”에 가깝습니다. 충돌 없는 작업 단위와 합류 지점을 먼저 정해야 합니다.

## 4. `Generator-Evaluator`: 만드는 역할과 평가하는 역할을 분리한다

AI가 스스로 만든 결과를 자기 기준으로 통과시키게 두면, 가장 그럴듯한 오류가 가장 오래 살아남습니다. 그래서 생성과 평가는 분리하는 편이 좋습니다.

하나는 만듭니다. 다른 하나는 정해진 루브릭으로 평가합니다. 생성자는 통과시키려 하고, 평가자는 기준에 미달하면 구체적 피드백과 함께 되돌려 보냅니다. 이 구조는 self-review의 한계를 줄이는 데 특히 효과적입니다.

이 패턴은 코드뿐 아니라 문서, 디자인, 분석에서도 잘 작동합니다. 핵심은 “좋다/나쁘다”가 아니라 다시 작업할 수 있을 정도로 구체적인 피드백을 주는 평가자 계약을 두는 것입니다.

## 5. `Agent Teams`: 양방향 대화가 필요할 때만 팀을 만든다

가장 무거운 패턴은 `Agent Teams`입니다. 여러 독립 세션이 각자 역할을 가지고 직접 메시지를 주고받으며 결론을 만드는 구조입니다. 연구자, 편집자, 리뷰어, 구현자처럼 서로 다른 관점을 가진 에이전트들이 협업하는 형태를 떠올리면 됩니다.

이 패턴은 복수 관점의 충돌과 조정이 품질을 크게 바꾸는 작업에 유용합니다. 다만 항상 좋은 것은 아닙니다. 팀을 만든다는 것은 곧 조정 비용도 생긴다는 뜻입니다. 명확한 역할 분리와 공유 컨텍스트가 없으면 회의만 많아집니다. 그래서 양방향 대화 자체가 필요한 경우에만 정당화됩니다.

## 패턴보다 더 중요한 공통 원칙

패턴 이름보다 더 중요한 것은 공통 설계 원칙입니다.

첫째, 메인 에이전트는 결정과 통합에 집중해야 합니다. 탐색과 소모성 작업은 가능한 한 밖으로 밀어내야 합니다.

둘째, 각 에이전트는 자기 일에 필요한 만큼만 받아야 합니다. 전체 컨텍스트를 통째로 넘기는 습관은 멀티 에이전트에서도 그대로 실패를 재현합니다.

셋째, 생성과 검증을 분리해야 합니다. 같은 에이전트가 계획하고, 만들고, 검증하고, 승인까지 해 버리면 블라인드 스폿이 커집니다.

넷째, 합류 지점을 먼저 정해야 합니다. 결과를 어디에서 어떤 형식으로 모을지 तय하지 않으면 병렬이나 팀 구조는 곧 산만해집니다.

## 언제 멀티 에이전트를 써야 하나

모든 작업에 멀티 에이전트가 필요한 것은 아닙니다. 단일 에이전트로 충분한 작업도 많습니다. 오히려 작은 작업을 괜히 분해하면 조정 비용만 늘어납니다.

멀티 에이전트가 특히 유효한 경우는 세 가지입니다. 첫째, 메인 컨텍스트를 깨끗하게 지키고 싶을 때. 둘째, 서로 다른 시각의 검토가 중요할 때. 셋째, 병렬 탐색이나 역할 분리가 실제로 시간과 품질 모두를 개선할 때입니다.

이 기준 없이 “에이전트 여러 개가 더 멋져 보인다”는 이유로 구조를 키우면 곧 관리 비용이 더 커집니다.

## 정리

멀티 에이전트의 핵심은 병렬화가 아니라 분해입니다. 무엇을 메인 판단에서 분리할지, 어떤 컨텍스트를 좁게 줄지, 생성과 검증을 어떻게 나눌지 설계하는 일이 먼저입니다.

다음 장에서는 이런 패턴을 단발성 요령이 아니라 운영 모델로 굳히는 방법을 보겠습니다. 명령어, 게이트, 세션 격리, 이슈 기반 상태 관리 같은 실전 구조가 바로 그 다음 단계입니다.

# 실전 운영 모델

## 두 가지 막다른 길부터 피해야 한다

실전에서 가장 먼저 피해야 할 길은 둘입니다. 하나는 모든 것을 손으로 직접 세팅하느라 원리보다 손재주에 매달리는 길입니다. 다른 하나는 원리도 모른 채 남이 만든 프리셋과 플러그인만 가져다 붙이는 길입니다.

전자는 유지되지 않습니다. 후자는 설명되지 않습니다. 이상 동작이 나와도 원인을 읽을 수 없고, 팀에는 방식이 아니라 블랙박스만 남습니다. 좋은 운영 모델은 이 둘을 피합니다. 사람이 원리를 이해하되, 구현 단계에서는 그 원리가 기본값으로 작동하도록 만드는 쪽입니다.

## 좋은 워크플로우는 행동을 시스템 기본값으로 만든다

하네스의 목적은 좋은 습관을 기억에 의존하지 않게 만드는 데 있습니다. 계획을 먼저 세우는 것, 위험한 작업 전 승인받는 것, 검증을 자동으로 돌리는 것, 세션이 길어지면 상태를 요약해 바깥에 남기는 것 같은 행동이 대표적입니다.

이런 행동이 매번 사람의 의지에만 달려 있으면 결국 무너집니다. 반대로 명령어, 체크리스트, 훅, 테스트 게이트, 이슈 템플릿으로 고정되면 좋은 운영은 습관이 아니라 시스템이 됩니다.

## `Plan-Critic-Build`를 운영 문법으로 삼아라

실전에서 가장 유용한 기본 문법 중 하나는 `Plan-Critic-Build`입니다. 구현 전에 계획을 분리하고, 그 계획을 다른 시각으로 비평한 뒤, 합의된 상태에서만 빌드 단계로 들어가는 흐름입니다.

이 구조는 큰 프로젝트에만 필요한 것이 아닙니다. 작은 수정에서도 효과가 큽니다. 무엇을 바꿀지, 왜 그 방식이 맞는지, 어떤 위험이 있는지, 어떤 검증으로 완료를 판단할지 먼저 정리하는 것만으로도 많은 블라인드 스폿이 줄어듭니다.

여기서 중요한 것은 형식이 아니라 순서입니다. 탐색 없이 계획하지 말고, 비평 없이 곧바로 구현하지 말며, 검증 없이 완료 선언하지 않는 것입니다.

## 도구는 많을수록 좋은 것이 아니라, 적절할수록 좋다

AI 코딩 환경이 복잡해질수록 사람은 도구를 더 많이 붙이고 싶어집니다. 하지만 실무에서는 유휴 도구가 성능과 안정성을 동시에 깎는 경우가 많습니다. 쓰지 않는 MCP 연결, 거의 호출하지 않는 플러그인, 중복 기능을 가진 여러 인터페이스는 선택지를 늘리는 대신 판단 비용을 키웁니다.

좋은 운영은 도구를 늘리는 기술보다 도구를 큐레이션하는 기술에 가깝습니다. 지금 작업에 꼭 필요한 도구만 열어 두고, 쓰지 않는 인터페이스는 닫아 두는 편이 낫습니다. 에이전트도 사람처럼 선택지가 지나치게 많아지면 길을 잃습니다.

## 게이트를 먼저 세워라

실전 운영에서 진짜 힘을 발휘하는 것은 도구 자체보다 게이트입니다. `lint`, `type check`, 테스트, 코드 리뷰, 승인, 샌드박스, 정책 검사 같은 결정론적 문이 여기에 해당합니다.

이 문들은 건너뛸 수 없어야 합니다. 사람이 오늘은 바쁘니까 린트를 생략하고, 이번 한 번은 테스트 없이 넘어가고, 에이전트가 자신 있어 보이니 승인 없이 배포하는 순간 구조는 급격히 약해집니다.

기계가 확인할 수 있는 것은 기계가 막아야 합니다. AI는 그 문을 통과할 수 있도록 초안을 만들고 수정하는 쪽에 집중해야 합니다. 이 구분이 명확할수록 운영 품질은 안정됩니다.

## 세션은 길게 끌지 말고, 상태를 밖에 남겨라

긴 세션은 처음에는 편해 보입니다. 모든 대화와 판단이 한 곳에 모여 있으니까요. 하지만 일정 시점을 지나면 편리함이 오히려 독이 됩니다. 오래된 결론과 새로운 결론이 뒤섞이고, 초기 제약이 흐려지고, 잡음이 늘어납니다.

그래서 좋은 운영은 세션을 무조건 오래 이어 가는 대신, 적절한 타이밍에 압축하고 새로 시작합니다. 현재 결론, 남은 과제, 검증 결과, 다음 행동을 파일과 이슈와 기록으로 남기고 세션은 비우는 편이 낫습니다. 상태는 대화가 아니라 아티팩트가 들고 가야 합니다.

## `Git worktree`와 다중 세션 격리

물리적 작업 환경도 분리되어야 합니다. 탐색, 메인 구현, 검증, 이슈 정리, PR 작성이 모두 한 세션과 한 작업 디렉터리 안에 섞이면 메인 컨텍스트가 금방 오염됩니다.

이때 `Git worktree` 같은 격리 구조가 유용합니다. 각 세션이 서로 다른 작업 디렉터리에서 독립적으로 움직이게 하면 충돌이 줄고, 결과를 합칠 지점도 명확해집니다. 여기에 터미널 분할이나 세션 관리자 도구를 함께 쓰면, 메인 작업 흐름과 조사 작업을 물리적으로 분리하기 좋습니다.

핵심은 화려한 병렬화가 아니라 메인 컨텍스트 보호입니다. 중요한 판단이 이뤄지는 세션을 최대한 깨끗하게 지켜야 합니다.

## 이슈가 세션보다 오래가야 한다

장기 작업일수록 세션보다 이슈가 상태를 들고 가야 합니다. 채팅창은 금방 뜨거워지고 금방 식습니다. 반면 이슈는 목적, 범위, 증거, 완료 기준을 바깥에 남깁니다.

좋은 운영에서는 이슈가 곧 작업 지시서가 됩니다. 스펙과 수용 기준이 이슈에 남고, 구현 세션은 그것을 따라 움직이며, PR은 다시 그 이슈와 연결됩니다. 이렇게 해야 세션이 바뀌어도 작업 기준이 증발하지 않습니다.

세션은 일시적이고, 이슈와 문서와 코드 이력은 지속적입니다. 실전 운영은 후자에 무게를 둬야 합니다.

## 한 모델만 보지 말고 시야를 분산시켜라

모델마다 강점과 맹점이 다릅니다. 그래서 중요한 작업일수록 한 모델의 자신감만 믿는 것보다 다른 모델이나 다른 역할의 평가 루프를 붙이는 편이 낫습니다.

여기서 중요한 것은 “어느 모델이 더 우월한가”보다 “불일치를 어떻게 읽을 것인가”입니다. 구조 설계, 코드 리뷰, 문서 표현, 최신 정보 확인처럼 구간별로 다른 시각을 들이는 것만으로도 블라인드 스폿을 줄일 수 있습니다.

이 역시 멀티 모델을 화려하게 쓰자는 뜻이 아닙니다. 교차 검증이 필요한 구간을 먼저 정하자는 뜻입니다.

## 팀 성숙도는 단계적으로 올라간다

모든 팀이 처음부터 완성형 하네스를 가질 필요는 없습니다. 오히려 대부분의 팀은 단계적으로 성숙해집니다.

처음에는 개인 차원의 규칙 파일과 기본 테스트 정도로 시작합니다. 그다음 반복되는 절차를 스킬과 템플릿으로 정리합니다. 이후 승인 구조와 훅, 이슈 기반 워크플로우, 세션 격리, 공통 검증 게이트가 붙습니다. 마지막 단계에서는 팀 전체가 같은 운영 모델을 공유하게 됩니다.

중요한 것은 처음부터 거대한 체계를 만들겠다는 욕심보다, 지금 반복해서 겪는 실수를 구조로 막는 데 집중하는 것입니다.

## 개인과 작은 팀을 위한 최소 시작점

처음부터 모든 것을 도입할 필요는 없습니다. 개인이나 작은 팀이 바로 시작할 수 있는 최소 구성은 이 정도면 충분합니다.

짧은 규칙 파일 하나를 만들고, 새 기능마다 작은 스펙을 남기고, 테스트와 린트를 무조건 통과시키고, 세션 종료 전 현재 상태를 문서나 이슈에 남기는 것입니다. 여기에 위험한 명령에 대한 승인 한 단계만 추가해도 운영 품질은 크게 달라집니다.

규칙을 더 늘리는 것보다, 지금 자주 흔들리는 한 지점을 먼저 고정하는 편이 훨씬 효과적입니다.

## 정리

실전 운영 모델의 핵심은 도구를 많이 붙이는 데 있지 않습니다. 좋은 행동을 시스템 기본값으로 만들고, 상태를 세션 밖에 남기고, 메인 컨텍스트를 보호하고, 결정론적 게이트를 먼저 세우는 데 있습니다.

다음 장에서는 이런 원리가 실제 산출물 제작에서 어떻게 적용되는지 보기 위해, 이 발표가 어떤 구조로 만들어졌는지를 사례로 설명하겠습니다.

# 이 발표는 어떻게 만들어졌는가

## 우리는 장표부터 만들지 않았다

이 발표를 만든 방식은 앞에서 설명한 원리를 그대로 따라갑니다. 처음부터 슬라이드 툴을 열고 문장을 채우지 않았습니다. 먼저 소스를 모으고, 그 소스를 prose 형태의 본문으로 정리하고, 그 본문을 outline으로 압축한 뒤, 마지막에 HTML과 PDF를 만들었습니다. 발표 대본은 그보다 더 뒤에 왔습니다.

이 순서가 중요한 이유는 간단합니다. 결과물은 같아 보여도, 판단이 어디에서 이뤄졌는지와 무엇이 기준 문서였는지가 완전히 달라지기 때문입니다. 처음부터 장표부터 만들면 보기에는 빨라 보이지만, 나중에는 무엇이 근거이고 무엇이 즉흥적 편집 판단이었는지 구분하기 어려워집니다.

## 입력층을 먼저 잠갔다

이 프로젝트에서 가장 먼저 분리한 것은 입력층이었습니다. PDF, 발표 자료, 참고 문서, 링크 기반 자료를 먼저 source로 고정하고, 그 다음 단계부터는 그것을 해석하고 재배열한 산출물로 분리했습니다.

이 구분이 중요한 이유는 source와 해석이 한 디렉터리에 섞이는 순간부터 생깁니다. 시간이 조금만 지나도 어떤 문장이 원문 근거인지, 어떤 문장이 새롭게 추가된 해석인지 흐려집니다. 그래서 입력층은 가능한 한 좁고 분명해야 했습니다.

이 단계에서 사람이 붙든 것은 무엇을 authoritative source로 볼 것인가 하는 판단이었습니다.

## prose를 기준 문서로 세웠다

source를 모은 뒤 곧바로 슬라이드로 가지 않은 이유는, 발표의 논리 순서를 먼저 고정하고 싶었기 때문입니다. 그래서 장표보다 prose를 먼저 만들었습니다. 장표는 압축된 결과물이고, prose는 전체 논리와 문장 톤, 주장 강도를 책임지는 기준 문서입니다.

이 결정 덕분에 이후 단계는 훨씬 단순해졌습니다. outline은 prose를 요약하는 문서가 되고, 슬라이드는 outline을 시각화하는 결과물이 됩니다. 즉, 상위 레이어의 논지를 하위 레이어가 마음대로 뒤집지 못하게 구조를 만든 것입니다.

AI와 함께 일할 때 이런 기준 문서가 특히 중요합니다. 기준 문서가 없으면 매 세션마다 논지가 조금씩 흔들리고, 결과물끼리 어조와 주장 강도가 다르게 나옵니다.

## outline은 승인 게이트였다

prose 다음에는 slide 단위의 outline을 만들었습니다. 이 단계의 목적은 장표를 예쁘게 나누는 것이 아니라, 어떤 주장 하나를 어떤 슬라이드 한 장으로 압축할지를 먼저 정하는 것이었습니다.

슬라이드 제목, 핵심 메시지, 필요한 시각 요소, 어떤 소스 섹션에 대응하는지가 outline 단계에서 먼저 정해지면, 이후 제작은 훨씬 기계적으로 안정됩니다. 반대로 이 단계가 없으면 AI는 장표를 만들 때마다 새로운 서사를 발명하려고 듭니다.

그래서 outline은 단순한 초안이 아니라 승인 게이트였습니다. 이 단계를 통과해야만 실제 HTML 생성으로 넘어가도록 순서를 고정했습니다.

## 디자인도 먼저 계약으로 고정했다

많은 AI 작업이 여기서 흔들립니다. 내용 구조는 잡혀 있는데, 시각 구조를 그때그때 즉흥적으로 만들기 시작하는 순간 결과물이 들쭉날쭉해집니다.

그래서 이 프로젝트에서는 디자인도 먼저 계약으로 고정했습니다. 톤, 색상, 타이포그래피, 간격, 레이아웃 패밀리, 어떤 종류의 슬라이드가 있는지 같은 시각 문법을 먼저 정의해 두고, 그 안에서 HTML을 생성하게 했습니다. 이렇게 하면 AI는 디자인을 새로 발명하는 대신, 이미 정해진 문법 안에서 생산만 담당하게 됩니다.

핵심은 자유를 넓히는 것이 아니라, 자유가 작동할 좌표계를 먼저 만드는 데 있었습니다.

## Codex는 어디에 들어갔는가

이 구조 안에서 Codex가 맡은 역할은 명확했습니다. source를 모으는 단계도, 논지를 정하는 단계도, 승인 기준을 세우는 단계도 아니었습니다. 그보다 아래에서, 이미 고정된 prose와 outline과 디자인 계약을 바탕으로 HTML/CSS를 생성하는 역할이었습니다.

즉, Codex는 모든 층을 대신한 것이 아니라, 이미 좁혀 놓은 경계 안에서 생산을 담당했습니다. 이 구분이 중요합니다. 사람은 방향과 기준을 붙들고, 에이전트는 그 기준 안에서 빠르게 산출물을 만듭니다. 이것이 좋은 협업 구조입니다.

## 생성과 검증을 한 손에 쥐지 않았다

이 프로젝트에서 또 하나 중요했던 원칙은 생성과 검증을 같은 역할에 맡기지 않는 것이었습니다. 슬라이드 HTML을 만든 주체가 그것을 그대로 통과시키게 두면, 가장 그럴듯한 오류가 가장 오래 남습니다.

그래서 HTML 생성 뒤에는 별도의 자동 점검과 사후 검토 단계를 뒀습니다. 구조 계약을 지켰는지, 한국어 문장이 무너지지 않았는지, 장표 사이 톤이 맞는지, 레이아웃이 일관적인지를 따로 봤습니다. 이 분리는 번거로워 보이지만, 실제로는 전체 품질을 가장 크게 끌어올린 장치였습니다.

## speaker notes를 맨 뒤로 미뤘다

발표 작업에서 자주 생기는 문제는 대본이 너무 일찍 커지는 것입니다. 아직 구조가 고정되지 않았는데 설명 욕심이 먼저 커지면, 말이 슬라이드를 덮어버립니다. 그러면 장표와 본문과 대본의 주객이 뒤집힙니다.

그래서 이 프로젝트에서는 대본을 맨 마지막으로 미뤘습니다. 먼저 논지와 장표 구조를 고정하고, 그 다음에야 speaker notes를 붙였습니다. 대본은 설계의 주인이 아니라 결과의 해설자여야 한다는 원칙을 지킨 것입니다.

## 이 사례가 보여 주는 것

이 발표 제작 과정이 보여 주는 핵심은 단순합니다. 좋은 결과를 만든 것은 한 번의 멋진 프롬프트가 아니었습니다. 입력층을 좁히고, 기준 문서를 세우고, outline을 승인 게이트로 두고, 시각 문법을 먼저 고정하고, 생성과 검증을 분리한 구조였습니다.

즉, 발표 제작조차 결국 하네스 문제였습니다. 사람이 붙든 층과 AI에게 맡긴 층을 먼저 가른 뒤에야 생산성이 제대로 나왔습니다.

## 정리

이 사례는 앞 장들의 주장을 다시 확인해 줍니다. AI를 잘 쓰는 방법은 모든 것을 곧바로 맡기는 것이 아니라, 어떤 판단을 사람이 붙들고 어떤 생산을 에이전트에게 맡길지 경계를 먼저 세우는 것입니다.

다음 장에서는 이 흐름을 개인과 팀의 다음 행동으로 연결해 보겠습니다. 결국 중요한 것은 개념을 이해하는 데서 끝나지 않고, 이를 자신의 작업 환경 안에서 어떻게 현실로 만들 것인가입니다.

# 우리가 다음에 해야 할 일

## 불안의 방향부터 바로 봐야 한다

새 모델이 나올 때마다 뒤처지는 느낌을 받기 쉽습니다. 더 큰 컨텍스트, 더 좋은 벤치마크, 더 강한 에이전트 기능이 쏟아지면, 지금 내가 익힌 방식이 금방 낡아질 것처럼 보입니다.

하지만 실무 관점에서 보면 진짜 불안은 조금 다른 곳에 있습니다. 어떤 도구를 먼저 써 보느냐보다, 내 일의 통제권과 숙련이 어디에 남는가 하는 문제입니다. 모델이 강해질수록 기계적으로 복제 가능한 구간의 희소성은 줄어듭니다. 반대로 더 중요해지는 것은 범위를 정하고, 위험을 구분하고, 검증 순서를 세우고, 결과를 승인할지 말지를 판단하는 능력입니다.

그래서 앞으로의 대응도 단순히 “새 툴을 빨리 익힌다”에 머물면 얕습니다. 더 중요한 것은 숙련을 하네스 바깥으로 꺼내는 일입니다.

## 첫 번째 원칙: 암묵지를 파일과 규칙으로 외재화하라

좋은 실무자는 늘 비슷한 순간에 멈춥니다. 어떤 설계가 위험한지, 어떤 로그가 이상 신호인지, 어떤 요청이 나중에 장애로 이어질 가능성이 큰지 감으로 압니다. 문제는 이 감각이 문서와 규칙 파일로 남지 않으면 에이전트에게는 존재하지 않는 지식과 같다는 점입니다.

그래서 첫 대응은 더 세련된 프롬프트를 외우는 것이 아닙니다. 판단 기준을 밖으로 꺼내는 것입니다. 프로젝트 규칙, 설계 원칙, 리뷰 기준, 금지 패턴, 성공 기준, 자주 생기는 실수와 그 대응을 파일로 남겨야 합니다. 숙련은 머릿속에만 있으면 재사용되지 않습니다.

## 두 번째 원칙: 모델보다 하네스를 먼저 보라

최근의 발표와 도구 진화를 보면 방향은 꽤 분명합니다. 관심의 중심은 모델 단품 성능에서 실행 환경 전체로 이동하고 있습니다. 권한, 검증, 서브에이전트, 샌드박스, 훅, 장시간 실행 하니스 같은 단어가 계속 앞에 나오는 이유가 여기에 있습니다.

이 말은 곧 개인의 학습 전략도 달라져야 한다는 뜻입니다. 어느 모델이 더 똑똑한지 아는 것도 필요하지만, 그것만으로는 충분하지 않습니다. 어떤 작업에는 어떤 컨텍스트를 공급하고, 어떤 단계에서 승인과 검증을 넣고, 어떤 상태를 파일과 이슈에 남겨야 하는지를 설계하는 능력이 더 중요해집니다.

## 세 번째 원칙: 작은 규칙 파일과 검증 루프부터 시작하라

거대한 시스템을 한 번에 만들 필요는 없습니다. 오히려 대부분의 경우 작게 시작하는 편이 낫습니다.

매번 다시 설명하는 기준 세 가지를 먼저 파일로 빼십시오. 예를 들어 코드 스타일, 테스트 원칙, 위험한 명령 금지처럼 자주 반복되는 규칙이면 충분합니다. 그 다음에는 결정론적 문 하나를 세우면 됩니다. 저장 전에 린트를 강제하거나, PR 전에 테스트를 반드시 통과시키거나, 특정 명령은 승인 없이는 실행되지 않게 만드는 정도여도 좋습니다.

핵심은 추상 개념으로만 알고 있던 하네스를 실제 작업 환경의 일부로 만드는 것입니다.

## 네 번째 원칙: 긴 세션을 실력으로 착각하지 말라

많은 사람이 긴 대화를 이어 가는 것을 곧 숙련으로 착각합니다. 하지만 실제로는 적절한 시점에 요약하고 새 세션으로 넘어가는 판단이 더 중요합니다. 기억을 많이 쌓는 것보다 기억이 썩지 않게 관리하는 것이 더 중요하기 때문입니다.

현재 결론, 남은 과제, 검증 결과, 다음 행동을 파일과 이슈에 남기고 세션은 비우십시오. 세션은 작업의 운반 수단이지 영구 기억 장치가 아닙니다.

## 다섯 번째 원칙: `Build to Delete`

좋은 하네스는 오늘의 모델 약점을 덮어 주지만, 영원히 두꺼워져야 하는 구조는 아닙니다. 지금은 필요했던 규칙이 다음 모델에서는 불필요한 마찰이 될 수도 있습니다. 그래서 하네스를 설계할 때는 처음부터 삭제 조건도 함께 생각하는 편이 좋습니다.

어떤 규칙이 어떤 약점을 보완하기 위해 생겼는지 남겨 두고, 더 이상 필요 없을 때 쉽게 걷어낼 수 있어야 합니다. 그렇지 않으면 규칙은 곧 관료제가 됩니다. 잘 만든 하네스는 강하면서도 뜯어내기 쉬워야 합니다.

## 결국 더 선명해지는 역할: Harness Engineer

AI 시대에 더 또렷해지는 역할은 직접 코드를 가장 빨리 치는 사람이 아니라, 에이전트가 좋은 결과를 낼 환경을 설계하는 사람입니다. 저장소의 지식을 문서와 규칙으로 정리하고, 기계적 제약을 세우고, 상태를 세션 밖에 남기고, 사람이 붙들어야 할 판단을 끝까지 유지하는 사람입니다.

이 역할을 거창하게 부르지 않아도 됩니다. 하지만 실제로 팀 안에서 이 기능을 누군가는 맡아야 합니다. 개인 개발자든, 시니어 엔지니어든, 작은 팀의 리드든 결국 누군가는 하네스를 설계하고 관리해야 합니다.

## 이번 주에 바로 할 수 있는 일

처음부터 거대한 구조를 만들 필요는 없습니다. 이번 주에 바로 시작할 수 있는 일만 적어 보면 이렇습니다.

첫째, 반복해서 말하는 규칙 세 가지를 짧은 문서로 뺍니다. 둘째, 자동 검증 하나를 더 강하게 붙입니다. 셋째, 긴 세션 끝에는 반드시 상태를 이슈나 문서에 남깁니다. 넷째, 위험한 작업 하나에 승인 단계를 넣습니다.

이 정도만 해도 하네스는 더 이상 추상 개념이 아니라 실제 작업 환경이 됩니다.

## 맺으며

코딩이 사라지는 것은 아닙니다. 다만 숙련이 드러나는 위치가 바뀌고 있습니다. 손끝의 속도로만 드러나던 감각이 이제는 규칙과 검증과 운영 구조를 설계하는 능력으로 이동하고 있습니다.

그래서 우리가 다음에 해야 할 일도 분명합니다. 새 모델 소식에 휩쓸리는 것이 아니라, 이 이동을 내 작업 환경 안에서 먼저 현실로 만드는 일입니다. 챗봇과 더 오래 씨름하는 대신, 에이전트가 일할 환경을 설계하는 쪽으로 중심을 옮겨야 합니다.
