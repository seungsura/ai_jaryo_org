# 멀티 에이전트 활용 패턴

## 병렬화보다 분해가 먼저다

멀티 에이전트를 생각할 때 가장 먼저 버려야 할 오해가 있습니다. AI를 더 많이 붙이면 자동으로 더 강해질 것이라는 생각입니다. 실제로 중요한 것은 숫자가 아니라 분해입니다. 무엇을 메인 컨텍스트에 남기고, 무엇을 밖으로 밀어내며, 누가 무엇을 검증하게 만들 것인지를 먼저 설계해야 합니다.

멀티 에이전트는 “더 많은 AI”가 아니라 “더 좁은 역할의 AI 여럿”에 가깝습니다. 역할과 컨텍스트와 검증 책임을 나누는 기술입니다.

## 1. `Sub-agent`: 가장 자주 이기는 기본형

가장 실용적인 패턴은 `Sub-agent`입니다. 탐색, 로그 분석, 자료 수집, 비교 작업처럼 토큰은 많이 먹지만 메인 판단에는 요약만 필요한 일을 분리할 때 강합니다.

구조는 단순합니다. 메인 에이전트가 일을 던지고, 격리된 서브 에이전트가 처리한 뒤, 핵심 결과만 요약해서 돌려줍니다. 이 패턴의 장점은 속도보다 격리입니다. 서브 에이전트가 파일을 많이 읽고 컨텍스트를 길게 써도 메인 세션은 오염되지 않습니다.

이때 중요한 원칙은 최소 권한입니다. 전체 대화와 전체 저장소를 다 넘길 필요는 없습니다. 관련 파일과 필요한 배경만 좁게 넘겨야 합니다. 서브 에이전트는 많이 읽되, 넓게 오염시키지 않는 역할이어야 합니다.

## 2. `Orchestrator`: 계획자가 여러 실행자를 배치한다

큰 목표를 한 번 더 구조적으로 쪼개야 할 때는 `Orchestrator` 패턴이 맞습니다. 하나의 에이전트가 계획을 세우고, 그 계획을 여러 실행 단위로 나눠 각각 맡기고, 마지막에 결과를 다시 모읍니다.

이 패턴은 계획자 없는 병렬이 쉽게 흩어지는 문제를 줄여 줍니다. 하위 작업끼리 깊게 토론할 필요는 없지만, 전체 목표와 순서를 누군가 붙들고 있어야 할 때 유용합니다. 말하자면 `Sub-agent`가 단일 심부름이라면 `Orchestrator`는 1 대 N 구조의 분업입니다.

## 3. `Parallel`: 의존성이 낮을 때만 병렬이 빛난다

병렬 실행은 분명 매력적입니다. 하지만 항상 좋은 기본값은 아닙니다. 서로 간섭이 적고 중간 상태를 공유할 필요가 적을 때만 진가를 냅니다.

예를 들어 세 가지 다른 접근으로 구조를 비교하거나, 독립적인 조사 작업을 동시에 진행하거나, 여러 초안을 나란히 뽑아 비교하는 구간에서는 병렬이 강합니다. 반대로 중간 상태를 계속 주고받아야 하는 작업이라면 병렬은 오히려 혼란을 키웁니다.

좋은 병렬은 “다 같이 달린다”가 아니라 “서로를 더럽히지 않는다”에 가깝습니다. 충돌 없는 작업 단위와 합류 지점을 먼저 정해야 합니다.

## 4. `Generator-Evaluator`: 만드는 역할과 평가하는 역할을 분리한다

AI가 스스로 만든 결과를 자기 기준으로 통과시키게 두면, 가장 그럴듯한 오류가 가장 오래 살아남습니다. 그래서 생성과 평가는 분리하는 편이 좋습니다.

하나는 만듭니다. 다른 하나는 정해진 루브릭으로 평가합니다. 생성자는 통과시키려 하고, 평가자는 기준에 미달하면 구체적 피드백과 함께 되돌려 보냅니다. 이 구조는 self-review의 한계를 줄이는 데 특히 효과적입니다.

이 패턴은 코드뿐 아니라 문서, 디자인, 분석에서도 잘 작동합니다. 핵심은 “좋다/나쁘다”가 아니라 다시 작업할 수 있을 정도로 구체적인 피드백을 주는 평가자 계약을 두는 것입니다.

## 5. `Agent Teams`: 양방향 대화가 필요할 때만 팀을 만든다

가장 무거운 패턴은 `Agent Teams`입니다. 여러 독립 세션이 각자 역할을 가지고 직접 메시지를 주고받으며 결론을 만드는 구조입니다. 연구자, 편집자, 리뷰어, 구현자처럼 서로 다른 관점을 가진 에이전트들이 협업하는 형태를 떠올리면 됩니다.

이 패턴은 복수 관점의 충돌과 조정이 품질을 크게 바꾸는 작업에 유용합니다. 다만 항상 좋은 것은 아닙니다. 팀을 만든다는 것은 곧 조정 비용도 생긴다는 뜻입니다. 명확한 역할 분리와 공유 컨텍스트가 없으면 회의만 많아집니다. 그래서 양방향 대화 자체가 필요한 경우에만 정당화됩니다.

## 패턴보다 더 중요한 공통 원칙

패턴 이름보다 더 중요한 것은 공통 설계 원칙입니다.

첫째, 메인 에이전트는 결정과 통합에 집중해야 합니다. 탐색과 소모성 작업은 가능한 한 밖으로 밀어내야 합니다.

둘째, 각 에이전트는 자기 일에 필요한 만큼만 받아야 합니다. 전체 컨텍스트를 통째로 넘기는 습관은 멀티 에이전트에서도 그대로 실패를 재현합니다.

셋째, 생성과 검증을 분리해야 합니다. 같은 에이전트가 계획하고, 만들고, 검증하고, 승인까지 해 버리면 블라인드 스폿이 커집니다.

넷째, 합류 지점을 먼저 정해야 합니다. 결과를 어디에서 어떤 형식으로 모을지 तय하지 않으면 병렬이나 팀 구조는 곧 산만해집니다.

## 언제 멀티 에이전트를 써야 하나

모든 작업에 멀티 에이전트가 필요한 것은 아닙니다. 단일 에이전트로 충분한 작업도 많습니다. 오히려 작은 작업을 괜히 분해하면 조정 비용만 늘어납니다.

멀티 에이전트가 특히 유효한 경우는 세 가지입니다. 첫째, 메인 컨텍스트를 깨끗하게 지키고 싶을 때. 둘째, 서로 다른 시각의 검토가 중요할 때. 셋째, 병렬 탐색이나 역할 분리가 실제로 시간과 품질 모두를 개선할 때입니다.

이 기준 없이 “에이전트 여러 개가 더 멋져 보인다”는 이유로 구조를 키우면 곧 관리 비용이 더 커집니다.

## 정리

멀티 에이전트의 핵심은 병렬화가 아니라 분해입니다. 무엇을 메인 판단에서 분리할지, 어떤 컨텍스트를 좁게 줄지, 생성과 검증을 어떻게 나눌지 설계하는 일이 먼저입니다.

다음 장에서는 이런 패턴을 단발성 요령이 아니라 운영 모델로 굳히는 방법을 보겠습니다. 명령어, 게이트, 세션 격리, 이슈 기반 상태 관리 같은 실전 구조가 바로 그 다음 단계입니다.
