# 도구는 왜 하네스로 수렴하는가

## 도구 이름보다 구조를 봐야 한다

최근 몇 년 사이 AI 코딩 도구는 폭발적으로 늘어났습니다. GitHub Copilot, Cursor, Windsurf, Claude Code, Codex, Copilot Coding Agent처럼 이름도 많고 출발점도 다릅니다. 어떤 도구는 자동완성에서 시작했고, 어떤 도구는 채팅형 편집기에서 시작했고, 어떤 도구는 터미널 에이전트로 출발했습니다.

그런데 긴 작업을 맡기기 시작하면 거의 같은 문제가 드러납니다. 무엇을 읽힐 것인가, 어떤 도구를 열어 둘 것인가, 위험한 명령은 어떻게 막을 것인가, 결과를 어디서 검증할 것인가, 세션이 바뀌어도 상태를 어떻게 유지할 것인가. 결국 도구 경쟁은 모델 단품의 성능보다 그 모델을 어떤 작업 환경 위에 올려놓느냐의 경쟁으로 이동합니다.

이때 필요한 렌즈가 바로 `Harness`입니다.

## 세 번의 이동

최근의 흐름은 대략 세 단계로 정리할 수 있습니다.

첫 번째는 프롬프트 중심의 시기였습니다. 핵심 질문은 “어떻게 말해야 하나”였습니다. ChatGPT 초기 사용법, Chain-of-Thought, ReAct 같은 패턴이 이 시기의 관심사였습니다. 모델이 무엇을 하게 만들지를 자연어 지시문으로 조정하는 일이 중심이었습니다.

두 번째는 컨텍스트 중심의 시기였습니다. 질문이 “어떤 정보를 넣어야 하나”로 바뀌었습니다. 현재 파일 한 장만 보는 도구로는 프로젝트 전체를 다루기 어려웠기 때문입니다. 이 시기에 RAG, 코드베이스 검색, 시맨틱 인덱싱, 긴 컨텍스트 최적화 같은 기법들이 전면에 올라왔습니다.

세 번째가 하네스 중심의 시기입니다. 여기서 질문은 다시 바뀝니다. “어떤 시스템을 만들어야 하나”입니다. 에이전트가 여러 단계를 거쳐 오래 일하기 시작하면, 단순히 좋은 프롬프트와 좋은 컨텍스트만으로는 충분하지 않기 때문입니다.

## 왜 프롬프트만으로는 부족했는가

초기의 코딩 도구는 비교적 짧은 상호작용에 강했습니다. 코드를 설명해 달라거나, 작은 리팩터링을 제안받거나, 함수 하나를 만들어 달라는 수준에서는 프롬프트 품질이 크게 작동했습니다.

하지만 프로젝트가 커지면 한계가 분명했습니다. 프롬프트에 “기존 유틸리티를 재사용하라”고 아무리 잘 적어도, 모델이 그 유틸리티 파일을 보지 못하면 소용이 없습니다. 더 정확히 말하면 문제는 지시문이 아니라 지시문이 소비하는 정보의 범위에 있었습니다.

그래서 프롬프트의 시대는 곧 컨텍스트의 시대로 넘어갔습니다.

## 왜 컨텍스트만으로도 부족했는가

컨텍스트 엔지니어링은 큰 진전이었습니다. 전체 코드베이스에서 관련 파일을 찾아 보여 주고, 검색 결과를 요약하고, 긴 히스토리를 압축해 다음 턴으로 넘기는 기술은 실제로 품질을 크게 올렸습니다.

하지만 여기에도 벽이 있었습니다. 컨텍스트는 기본적으로 “이번 API 호출에 무엇을 넣을 것인가”에 가깝습니다. 반면 에이전트는 한 번 호출하고 끝나지 않습니다. 자료를 읽고, 파일을 바꾸고, 테스트를 돌리고, 오류를 보고, 다시 계획을 바꾸는 루프를 계속 반복합니다.

이 루프에서는 다른 종류의 문제가 생깁니다. 잘못된 상태가 누적되고, 실패가 조용히 다음 단계로 넘어가고, 도구 권한이 너무 넓어 사고 가능성이 커지고, 세션이 길어질수록 처음의 의도가 흐려집니다. 컨텍스트를 잘 넣는 것만으로는 해결되지 않는 문제들입니다.

그래서 필요한 것이 하네스입니다.

## 하네스는 에이전트 바깥의 나머지 전부다

하네스를 한 문장으로 정리하면 이렇습니다.

> `Agent = Model + Harness`

모델은 추론과 생성 엔진입니다. 하지만 실제로 에이전트의 성격을 결정하는 것은 그 엔진 바깥의 구조입니다. 무엇을 읽게 할지, 어떤 도구를 쓸 수 있을지, 어떤 단계에서 사람 승인이 필요한지, 어떤 테스트를 통과해야 하는지, 실패했을 때 어떻게 멈추고 복구할지가 모두 하네스의 영역입니다.

같은 모델을 써도 결과가 크게 달라지는 이유가 여기에 있습니다. 프롬프트를 조금 다듬는 것보다, 모델을 더 나은 작업 환경 위에 올려놓는 편이 훨씬 큰 차이를 만드는 구간이 분명히 존재합니다.

## 왜 Claude Code가 자주 예시로 등장하는가

이 글에서 Claude Code를 자주 언급하는 이유도 이 때문입니다. 특정 제품을 홍보하려는 의도가 아니라, 현재 하네스 중심 구조를 가장 노골적으로 드러내는 대표 사례이기 때문입니다.

터미널 네이티브 워크플로우, 계획 모드, 승인 구조, `CLAUDE.md`, `Skills`, `Hooks`, `MCP`, 서브에이전트, 장시간 작업을 전제로 한 루프 설계 같은 요소들이 모두 한 화면에 모여 있습니다. 다시 말해 “좋은 에이전트는 어떤 환경 위에서 굴러가야 하는가”라는 질문에 가장 직접적으로 답하는 도구 중 하나입니다.

하지만 핵심은 Claude Code 자체가 아닙니다. Cursor의 Agent Mode, Copilot Coding Agent, 여러 오픈소스 하니스 패턴도 결국 비슷한 구조로 움직입니다. 출발점은 달라도 수렴점이 비슷하다는 것이 더 중요합니다.

## 도구 경쟁의 진짜 축

예전에는 누가 더 좋은 모델을 쓰느냐가 가장 눈에 띄는 차이였습니다. 이제는 그 모델을 어떤 운영 구조 위에 얹느냐가 더 중요해지고 있습니다.

좋은 도구는 단순히 코드를 잘 생성하는 도구가 아닙니다. 컨텍스트를 선별하고, 도구를 큐레이션하고, 검증 게이트를 두고, 상태를 바깥에 남기고, 실패를 복구하고, 사람이 개입할 타이밍을 명확하게 설계한 도구입니다. 이 구조가 강할수록 모델의 확률적 특성이 더 안정적으로 다뤄집니다.

결국 요즘의 도구 경쟁은 “누가 더 똑똑한 모델을 붙였는가”보다 “누가 더 강한 하네스를 제공하는가”의 경쟁으로 바뀌고 있습니다.

## 정리

프롬프트는 여전히 중요합니다. 컨텍스트도 여전히 중요합니다. 하지만 둘만으로는 긴 작업을 안정적으로 운영할 수 없습니다. 에이전트가 루프를 돌기 시작하는 순간, 권한, 상태, 검증, 복구, 관측 같은 바깥 구조가 더 중요해집니다.

그래서 최근의 도구와 실무 방법론은 결국 하네스 중심 구조로 수렴합니다. 다음 장에서는 왜 에이전트가 그렇게 자주 길을 잃는지, 그리고 어떤 실패 패턴이 반복해서 나타나는지를 먼저 살펴보겠습니다.
