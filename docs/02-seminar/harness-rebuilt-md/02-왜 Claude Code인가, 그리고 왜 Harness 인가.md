**시작하며: 특정 제품 리뷰가 아닙니다.**

“AI 코딩은 무엇인가?” 최근 AI 코딩 도구가 어떤 구조로 진화했는지 살펴보겠습니다.

“왜 `Claude Code`인가”는 특정 제품 리뷰를 하려는 이야기가 아닙니다.

Claude Code는 최근 AI 코딩 도구의 흐름을 가장 선명하게 보여주는 대표 사례에 가깝습니다. `Best Practice`, 공개 자료, 생태계가 모두 빠르게 축적됐고, 성능과 기능 면에서도 지금 시점의 기준선을 또렷하게 드러냅니다. 그래서 이후 예시는 주로 Claude Code를 기준으로 잡겠습니다.

최근 유출 사고 이후에는 역설적으로 오픈소스 진영의 분석 자료도 더 풍부해졌습니다. "Claude Code가 오픈소스 발전에 가장 많이 기여했다"는 농담이 나올 정도로, 뜯어보고 배울 재료가 많아졌습니다.

`Agent = Model + Harness`

“모델이 아닌 것은 전부 하네스입니다.” - LangChain, VivekTrivedy

`Harness`는 원래 말을 다루는 마구를 뜻합니다. 고삐, 안장, 재갈을 따로 부르는 말이 아니라, 힘이 센 말을 사람이 원하는 방향으로 몰고 멈추게 만드는 장비 전체를 가리킵니다. LLM도 마찬가지입니다. 모델 자체는 힘의 원천이지만, 무엇을 보게 할지, 어떤 도구를 쓰게 할지, 언제 멈출지, 실패했을 때 어떻게 복구할지는 모델 바깥 장치가 결정합니다.

## 에이전틱 코딩의 실제 성과 ##

카카오 AI 팀 내부 공유 사례에서는 CTO가 4~8개월로 추정한 프로젝트를 에이전틱 코딩으로 완료한 시간이 2주였습니다. 또한 신규 개발자 온보딩은 기존 수주 ~ 수개월 걸리던 것에서 1~2일로 단축되었습니다. 모든 내용이 문서로 존재하기 때문에 가능한 일입니다. 이로인해 첫 날부터 의미있는 기여가 가능해졌습니다. 또한 불가능하던 작업을 해낸다던지, 기술 부채를 해결하고 피드백을 즉각적으로 받는 등의 성과를 얻을 수 있었습니다.


## **1막: Copilot과 ChatGPT, 프롬프트의 시대**

AI 코딩 도구의 출발점에는 `GitHub Copilot`이 있었습니다. 초창기 Copilot은 현재 파일의 코드를 바탕으로 다음 줄을 제안하는 스마트 자동완성에 가까웠습니다. 현재 열려 있는 파일이 곧 암묵적 프롬프트였고, 모델은 그 프롬프트의 다음 토큰을 예측했습니다.

몇 달 뒤 `ChatGPT`가 등장했습니다. “AI가 코드를 추천해 준다”에서 “AI에게 어떻게 말해야 원하는 코드를 얻을 수 있는가”로 이동했습니다. 개발자들은 프롬프트의 길이, 역할 부여, 예시, 단계별 지시를 실험했습니다. 이 시기에는 자연어 지시문이 새로운 프로그래밍 인터페이스처럼 보였습니다.

2022 ~ 2024 년도는 프롬프트의 시대입니다. 학계와 업계에서도 비슷한 기법이 쏟아졌습니다. `Chain-of-Thought`는 모델이 중간 추론 단계를 쓰도록 유도했고, `ReAct`는 추론과 행동을 번갈아 수행하도록 만들었습니다. `Tree-of-Thought`는 하나의 추론 경로만 따라가지 않고 여러 경로를 탐색하게 했습니다. 

이 흐름을 나란히 놓고 보면 변화 방향이 더 분명합니다.[^02-v01]

| **기법** | **프롬프트가 추가로 하게 만든 일** | **의미** |
| --- | --- | --- |
| `Chain-of-Thought` | 중간 추론 단계를 쓰게 한다 | 답만 내던 모델을 reasoning path가 있는 모델처럼 다루기 시작했다 |
| `ReAct` | 추론과 행동을 번갈아 수행한다 | 검색과 도구 호출이 루프 안으로 들어왔다 |
| `Tree-of-Thought` | 여러 추론 경로를 동시에 탐색한다 | 하나의 답이 아니라 후보 경로를 비교하기 시작했다 |
| `Self-Refine` / `Reflexion` | 자기 출력을 다시 비판하고 수정한다 | 출력 이후의 피드백 루프가 생겼다 |

그러다 2024년 Andrew Ng이 정리한 네 가지 에이전틱 패턴이 나옵니다.[^02-v02]

Andrew Ng의 정리는 이 흩어진 기법을 실무 패턴으로 압축한 순간이었습니다. 중요한 것은 이름보다 축입니다. 자기 수정, 도구 호출, 계획 분해, 역할 분리가 한 장 안에 모이면서, 모델 바깥의 구조가 성능을 좌우한다는 사실이 더 선명해졌습니다.

- `Reflection`: 모델이 자기 결과를 검토하고 수정한다.
- `Tool Use`: 검색, 계산기, 코드 실행 같은 외부 도구를 호출한다.
- `Planning`: 큰 목표를 하위 단계로 나눈다.
- `Multi-Agent Collaboration`: 여러 에이전트가 역할을 나눠 협업한다.

| **패턴** | **모델이 하게 되는 일** | **핵심 효과** |
| --- | --- | --- |
| `Reflection` | 한 번 낸 답을 다시 보고 고친다 | 자기 수정 루프가 생긴다 |
| `Tool Use` | 검색, 계산, 실행을 바깥 도구에 맡긴다 | 모델 바깥 세계와 연결된다 |
| `Planning` | 목표를 단계로 쪼갠다 | 긴 작업을 밀어붙일 수 있다 |
| `Multi-Agent Collaboration` | 역할을 나눠 서로 다른 관점으로 본다 | 생성과 검증을 분리할 수 있다 |

이 발견의 중요한 점은 “모델 그 자체”보다 “모델을 감싸는 패턴”이 성능을 크게 바꿀 수 있다는 사실이었습니다. 이미 이때부터 하네스의 씨앗이 보였습니다. 다만 당시의 중심 언어는 여전히 프롬프트였습니다.

## **프롬프트 시대의 벽: 모델은 보지 못한 것을 알 수 없다**

프롬프트는 강력했지만, 곧 한계가 드러났습니다. 아무리 좋은 지시를 내려도 모델이 프로젝트의 다른 파일, 기존 설계 결정, 내부 유틸리티, 팀 컨벤션을 보지 못하면 제대로 활용할 수 없습니다.

예를 들어 “기존 코드를 재사용하라”고 프롬프트에 아무리 써도, 관련 유틸리티 파일이 컨텍스트에 없으면 에이전트는 그 존재를 알지 못합니다. 문제는 지시문이 부족한 것이 아니라 모델이 소비하는 정보가 부족한 것입니다.

이 지점에서 `Blind Prompting`이 문제가 됩니다. 팀이 프롬프트를 조금씩 고치고, 결과를 눈으로 보고, “이번엔 괜찮은 것 같다”고 판단하는 방식은 프로덕션 엔지니어링이라기보다 시행착오에 가깝습니다. 비결정론적 모델을 다루면서 측정과 검증 없이 문장만 만지는 접근은 오래 버티기 어렵습니다.

## **2막: Cursor와 컨텍스트의 시대**

프롬프트의 벽을 가장 먼저 크게 밀어낸 도구가 `Cursor`였습니다. Copilot이 현재 파일 중심이었다면 Cursor는 전체 코드베이스를 인덱싱하고, 필요한 파일과 심볼을 검색해 컨텍스트에 넣는 방식으로 질문 자체를 바꿨습니다.

| **차원** | **초기 Copilot** | **Cursor 계열 도구** |
| --- | --- | --- |
| 컨텍스트 범위 | 현재 파일 중심 | 전체 코드베이스 |
| 검색 방식 | 없음 또는 제한적 | RAG, AST, 시맨틱 검색 |
| 편집 단위 | 한 줄 또는 한 블록 | 여러 파일 동시 수정 |
| 사용자 인터페이스 | 자동완성 중심 | 채팅, Composer, Agent Mode |
| 핵심 질문 | 다음 줄은 무엇인가 | 이 프로젝트에서 필요한 변경은 무엇인가 |

이때부터 Tools 라는 기능을 적극적으로 사용하기 시작합니다. 

Cursor 아키텍처를 보면 전환점이 더 또렷합니다. 저장소 전체를 인덱싱하고, `AST`와 시맨틱 검색으로 필요한 파일과 심볼을 찾아 컨텍스트에 넣고, 채팅·Composer·Agent Mode가 그 결과를 바로 편집과 실행으로 연결합니다. 즉 모델 앞에 붙은 것은 더 긴 프롬프트가 아니라 코드베이스 검색기, 멀티파일 편집기, 실행 루프였습니다.[^02-v04]

```text
사용자 요청
   ↓
코드베이스 인덱싱
(파일, 심볼, AST, 문서)
   ↓
관련 파일 / 규칙 / 히스토리 검색
   ↓
컨텍스트 조립
   ↓
Composer / Agent Mode
   ↓
멀티파일 수정 + 명령 실행
   ↓
테스트 / 린트 / 로그 확인
```

컨텍스트 시대의 핵심 질문은 “어떤 말을 해야 하나”가 아니라 “어떤 정보를 넣어야 하나”였습니다. 2025년 중반부터 `Context Engineering`이라는 표현이 널리 쓰이기 시작했습니다.  컨텍스트 윈도우에 다음 단계에 필요한 정보를 선별해 넣는 기술이 중요해진 것입니다.

이때 RAG 파이프라인, 코드베이스 인덱싱, 문서 검색, KV-cache 최적화, 안정 접두어와 동적 접미어 같은 개념이 전면으로 올라왔습니다. 컨텍스트 엔지니어링은 진짜 진보였습니다. 모델이 볼 수 있는 정보의 질이 결과를 바꾼다는 점을 업계가 받아들였기 때문입니다.

## **컨텍스트 시대의 벽: 좋은 입력만으로는 루프를 통제할 수 없다**

하지만 컨텍스트만으로도 충분하지 않았습니다. 이유는 간단합니다. 코딩 에이전트는 한 번 답하고 끝나는 챗봇이 아니라, 루프를 도는 실행 주체이기 때문입니다.

대부분의 에이전트형 코딩 도구는 다음 루프를 반복합니다.

1. 컨텍스트를 수집한다.
2. 파일을 수정하거나 명령어를 실행한다.
3. 테스트, 린트, 타입 체크, 로그로 결과를 확인한다.
4. 실패하면 다시 수정한다.
5. 충분하다고 판단하면 보고하거나 PR을 만든다.

이 구조를 네 단계로 줄이면 `gather context -> take action -> verify -> repeat`입니다. 하네스는 이 네 지점 각각에 개입합니다. 무엇을 보여줄지, 어떤 행동을 허용할지, 어떤 검증에서 멈출지, 다음 루프에 무엇만 남길지를 설계하는 일이 곧 하네스 엔지니어링입니다.

이때 문제는 도구 호출이 실패할 수 있고, 테스트를 잘못 해석할 수 있고, 비용이 폭주할 수 있고, 위험한 명령을 실행할 수 있고, 보안 경계를 넘을 수 있고, 장시간 작업 중 목표를 잊을 수 있습니다. 한 번 잘못된 결과나 응답이 들어가게 되면 컨텍스트가 오염되고, 실행 권한이 느슨하면 사고 가능성이 커지고, 검증이 잘못되면 빠른 코딩 능력과 함께 잘못된 방향으로 전력질주 하게 됩니다. 

즉, “무엇을 보여줄 것인가”만으로는 부족합니다. “무엇을 하게 할 것인가”, “무엇을 못 하게 막을 것인가”, “어디서 멈출 것인가”, “어떻게 검증할 것인가”가 함께 필요해집니다. 이 지점에서 하네스가 본격적으로 등장합니다.

## **3막: Claude Code, Codex, OpenCode 와 하네스의 시대**

2026년 하네스 시대의 대표 장면은 코딩 도구가 단순한 자동완성이나 채팅을 넘어 작업 환경 전체를 품기 시작한 것입니다.

Claude Code는 터미널 네이티브 워크플로우를 중심에 둡니다. 파일을 읽고, 셸을 실행하고, 테스트 결과를 보고, 다시 수정합니다. 여기에 `Plan Mode`, 승인 체계, `CLAUDE.md`, `Skills`, `Hooks`, `MCP`, `Plugins`, `Subagents` 같은 요소가 붙습니다. 이 요소들은 각각 독립 기능처럼 보이지만, 한데 묶이면 에이전트의 실행 환경을 구성합니다.


즉, `Agent = Model + Harness`

가장 좋은 AI 도구는 내 환경에 딱 맞는 도구입니다. 하네스는 그런 환경을 구성하는데 필요한 요소 전체를 일컫습니다. 더 좋은 도구와 더 좋은 모델은 시간이 지나면 따라잡을 수 있습니다. 하지만 우리의 환경에 알맞은 도구가 나올 수 없습니다. 플러그인, 스킬, 노하우, 교육자료와 같이 복제할 수 없는 자산을 구성해두고 문제가 생겼을 때 대응할 수 있도록 하는 것이 실무 도입의 핵심 조건입니다. 

하네스가 결정하는 것 - 무엇을 보는지 (컨텍스트) * 무엇을 할 수 있는지 (도구, 권한) * 언제 멈추는지 (제약) * 잘못되었을 때 (복구) ….[^02-v05]

| **시대 구분** | **시기** | **핵심 질문** | **대표 도구** | **주안점** |
| --- | --- | --- | --- | --- |
| **Prompt Engineering** | 2022~2024 | "어떤 말을 해야 하나?" | ChatGPT, 초기 Copilot | 자연어 지시문의 품질과 기법 (CoT 등) |
| **Context Engineering** | 2025 중반~ | "어떤 정보를 넣어야 하나?" | Cursor Composer, RAG 파이프라인, KV-cache 최적화 | 전체 코드베이스 및 구조화된 정보의 적절한 주입 |
| **Harness Engineering** | 2026~ | "어떤 시스템을 만들어야 하나?" | Claude Code, Copilot Coding Agent | 권한 제어, 피드백 루프, 검증 자동화, 서브에이전트 |

이 세 시대는 서로를 대체하지 않습니다. 하네스 안에 컨텍스트가 있고, 컨텍스트 안에 프롬프트가 있습니다. 좋은 하네스는 좋은 프롬프트와 좋은 컨텍스트를 여전히 필요로 합니다. 좀 더 넓은 의미에서 우리가 개발하는 환경 자체를 구성하는 것입니다.
