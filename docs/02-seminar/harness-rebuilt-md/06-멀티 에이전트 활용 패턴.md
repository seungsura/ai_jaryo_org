## **시작하며: 병렬화보다 분해가 먼저다**

멀티 에이전트는 AI를 더 많이 붙이는 묘기가 아닙니다. 더 많은 모델을 동시에 돌린다고 자동으로 품질이 좋아지지는 않습니다. 오히려 역할과 컨텍스트가 정리되지 않은 상태에서 에이전트 수만 늘리면 비용과 혼란이 함께 커집니다.

멀티 에이전트의 본질은 병렬화가 아니라 분해입니다. 컨텍스트를 쪼개고, 역할을 쪼개고, 검증 책임을 쪼개는 설계입니다. 질문은 하나입니다.

> **메인 컨텍스트에 무엇을 남기고, 무엇을 밖으로 밀어내며, 누가 누구를 검증하게 만들 것인가?**
> 

이 장에서는 여섯 가지 패턴을 봅니다. `Sub-Agent`, `Orchestrator`, `Parallel`, `GAN-Style`, `Agent Teams`, `Advisor`입니다. 이름은 여섯 개지만, 바닥의 원리는 같습니다. 역할 경계를 분명히 하고, 중간 작업이 메인 판단을 오염시키지 않게 하며, 생성과 검증을 같은 손에 쥐지 않는 것입니다.

이 흐름을 가장 선명하게 보여 주는 예가 `Planner / Generator / Evaluator` 3역할 구조입니다. `Planner`가 목표를 제품 수준의 스펙으로 풀고, `Generator`가 한 번에 하나의 기능을 구현하고, `Evaluator`가 테스트와 브라우저 자동화 기준으로 다시 판정합니다. 핵심은 셋으로 나눴다는 사실보다, 생성과 평가를 같은 손에 쥐지 않았다는 데 있습니다. 이 장의 여러 패턴은 결국 이 원칙을 다른 방식으로 구현한 변형들입니다.

## **왜 하나의 에이전트만으로는 부족한가**

하나의 에이전트에게 모든 것을 맡기면 세 가지 문제가 생깁니다.

첫째, 컨텍스트가 비대해집니다. 탐색, 구현, 로그 분석, 리뷰, 문서 작성이 모두 같은 창에 쌓이면 핵심 신호가 흐려집니다. 많이 읽을수록 똑똑해지는 것이 아니라, 무관한 정보가 판단을 오염시킬 수 있습니다.

둘째, 역할이 섞입니다. 계획하는 에이전트가 구현까지 하고, 구현한 에이전트가 자기 결과를 리뷰하고, 리뷰한 에이전트가 다시 테스트 해석까지 하면 blind spot이 남습니다. 자기 답을 자기가 채점하는 구조가 됩니다.

셋째, 신뢰 경계가 흐려집니다. 탐색만 해야 하는 작업에 쓰기 권한이 열려 있거나, 민감 데이터를 볼 필요가 없는 작업이 전체 컨텍스트를 받으면 사고 가능성이 커집니다.

멀티 에이전트는 이 문제를 줄이기 위한 구조입니다.

## **1. Sub-Agent: 중간 작업을 격리하는 기본형**

가장 자주 쓰는 기본형은 `Sub-Agent`입니다. 메인 에이전트가 특정 작업을 던지고, 격리된 워커가 처리한 뒤, 요약만 돌려주는 구조입니다.

대표적인 사용처는 다음과 같습니다.

- 특정 디렉터리의 파일을 대량으로 읽고 요약하기
- 긴 로그에서 원인 후보 찾기
- 테스트 실패 원인 분석하기
- 문서 묶음에서 관련 규칙 찾기
- 여러 API 사용 예시 비교하기

Sub-Agent의 강점은 성능보다 격리에 있습니다. 서브 에이전트가 파일을 많이 읽고 수만 토큰을 써도 메인 컨텍스트는 오염되지 않습니다. 메인은 결과 요약과 결론만 받습니다.

이때 중요한 원칙은 `least privilege`입니다. 서브 에이전트에게 전체 대화와 전체 저장소를 통째로 넘기지 않습니다. 그 작업에 필요한 파일, 질문, 출력 형식, 권한만 줍니다. 가능하면 읽기 전용 권한으로 시작하고, 쓰기 권한은 필요한 순간에만 열어야 합니다.

좋은 Sub-Agent 지시는 다음 요소를 포함합니다.

- 작업 목적
- 읽어야 할 범위
- 읽지 말아야 할 범위
- 출력 형식
- 확신도 또는 근거 표시 방식
- 금지 행동

Sub-Agent는 단순하지만 가장 자주 이기는 패턴입니다. 메인 컨텍스트를 지키는 데 특히 강합니다.

## **2. Orchestrator: 계획자 하나가 여러 실행자를 배치한다**

`Orchestrator`는 큰 목표를 하위 작업으로 나누고 여러 실행자에게 배치하는 패턴입니다. Sub-Agent가 단일 심부름이라면, Orchestrator는 1:N 위임 구조입니다.

흐름은 보통 이렇습니다.

1. 계획자가 목표를 해석한다.
2. 하위 작업으로 분해한다.
3. 각 작업을 실행자에게 배정한다.
4. 실행 결과를 수집한다.
5. 전체 결과를 통합한다.
6. 필요하면 재작업을 지시한다.

이 패턴은 하위 작업끼리 깊게 토론할 필요는 없지만, 큰 목표를 여러 실행 단위로 안정적으로 나눠야 할 때 유용합니다. 예를 들어 대규모 리팩터링, 문서 전체 개편, 여러 모듈에 걸친 테스트 보강, 마이그레이션 준비처럼 범위가 넓은 작업에 맞습니다.

주의할 점도 있습니다. 계획자가 약하면 실행자는 흩어집니다. 반대로 실행자가 없는 계획은 문서로만 끝납니다. Orchestrator의 품질은 분해의 품질과 수렴 지점의 품질에 달려 있습니다.

## **3. Parallel: 같은 목표를 평면으로 벌리고 나중에 합친다**

`Parallel`은 이름 그대로 여러 작업을 동시에 돌리는 패턴입니다. 하지만 핵심은 속도보다 간섭 제어입니다. 서로 의존성이 낮은 탐색, 비교, 초안 생산을 나란히 벌려 놓고 나중에 합칩니다.

적합한 작업은 다음과 같습니다.

- 세 가지 라이브러리 대안 비교하기
- 같은 요구사항에 대해 여러 설계안 만들기
- 서로 다른 파일 묶음을 독립적으로 점검하기
- 문서 초안을 여러 스타일로 생성하기
- UI 개선안을 여러 방향으로 탐색하기

좋은 병렬은 “다 같이 달린다”가 아니라 “서로를 더럽히지 않는다”에 가깝습니다. 중간 상태가 서로를 오염시키지 않게 해야 합니다. 이를 위해 `Git Worktree`, 별도 브랜치, 별도 작업 디렉터리, 명확한 출력 계약이 필요합니다.

Parallel은 언제나 좋은 기본값이 아닙니다. 서로 의존성이 강한 작업을 억지로 병렬화하면 충돌과 통합 비용이 커집니다. 병렬은 서로가 서로의 중간 상태를 몰라도 되는 작업에서만 강합니다.

## **4. GAN-Style: 생성자와 평가자를 분리한다**

`GAN-Style` 패턴은 생성과 평가를 같은 주체에 묶지 않는 방식입니다. 한쪽은 만들고, 다른 한쪽은 루브릭으로 평가합니다. 중요한 것은 감정적인 대립이 아니라 역할의 분리입니다.

Self-review에는 blind spot이 남습니다. 에이전트는 자기 결과를 “완료”라고 선언하는 데 능숙하지만, 그 결과가 실제로 요구사항을 만족하는지는 별도의 관점에서 봐야 합니다. 그래서 `Generator`와 `Evaluator`를 분리합니다.

실전형 구조는 다음과 같습니다.

- `Planner`: 사용자 요청을 제품 스펙과 작업 범위로 확장한다.
- `Generator`: 한 번에 하나의 기능 또는 스프린트를 구현한다.
- `Evaluator`: 테스트, 브라우저 자동화, 루브릭, 코드 리뷰 기준으로 평가한다.

이 패턴이 강한 이유는 평가자가 구체적인 피드백을 다시 생성자에게 돌려보낼 수 있기 때문입니다. “별로다”가 아니라 “로그인 버튼이 모바일에서 겹치고, API 응답이 스펙의 필드명을 만족하지 않으며, 테스트가 빈 케이스를 검증하지 않는다”처럼 다시 작업 가능한 피드백이어야 합니다.

이 구조는 비용이 듭니다. 단일 에이전트보다 훨씬 많은 토큰과 시간이 필요할 수 있습니다. 하지만 긴 작업에서 사람이 나중에 고치는 비용을 줄이고 싶다면, 평가 루프를 시스템 안에 넣는 편이 낫습니다.

## **5. Agent Teams: 양방향 대화가 가능한 팀을 만든다**

`Agent Teams`는 여러 에이전트가 각자 별도 컨텍스트를 가진 채 직접 메시지를 주고받는 구조입니다. `Sub-Agent`가 “시켜 놓고 받는” 구조라면, Agent Teams는 함께 토론하고 합의하는 구조입니다.

예를 들어 다음과 같은 역할을 둘 수 있습니다.

- `Researcher`: 외부 문서와 사례를 조사한다.
- `Architect`: 설계 대안을 비교한다.
- `Reviewer`: 위험과 누락을 지적한다.
- `Editor`: 최종 문서나 코드 품질을 다듬는다.
- `Coordinator`: 논의를 수렴시키고 결론을 정리한다.

이 패턴은 복수 관점의 충돌과 조정이 결과 품질을 좌우할 때 유용합니다. 연구, 전략 문서, 대규모 아키텍처 검토, 제품 기획처럼 하나의 정답보다 관점의 균형이 중요한 작업에 맞습니다.

다만 팀이라고 늘 좋은 것은 아닙니다. 에이전트 수가 늘면 대기 시간, 통합 비용, 충돌 관리 비용이 증가합니다. 보통 3명 안팎에서는 이득이 나지만, 너무 커지면 “AI 회의”만 길어지고 결과는 흐려질 수 있습니다.

## **Sub-Agent와 Agent Team은 다르다**

둘은 자주 혼동되지만 목적이 다릅니다.

| **구분** | **Sub-Agent** | **Agent Team** |
| --- | --- | --- |
| 구조 | 메인이 작업을 던지고 결과를 받음 | 여러 에이전트가 서로 대화하며 수렴 |
| 컨텍스트 | 제한된 범위, 요약 반환 | 각자 별도 컨텍스트와 공유 맥락 |
| 통신 | 주로 메인과 단방향 | 에이전트 사이 양방향 |
| 적합한 작업 | 로그 파싱, 파일 탐색, 단일 검증 | 관점 충돌, 기획, 아키텍처 토론 |
| 위험 | 요약 품질이 낮으면 메인이 잘못 판단 | 토론 비용과 수렴 실패 |

단순한 심부름은 Sub-Agent로 충분합니다. 양방향 토론이 필요한 경우에만 Agent Team을 고려하는 편이 낫습니다.

## **6. Advisor 전략: 작은 실행자, 큰 자문**

`Advisor` 전략은 팀을 하나 더 꾸리는 방식이 아닙니다. 메인 실행자는 그대로 두고, 막히는 순간에만 더 강한 판단을 빌려오는 방식입니다. 작은 `executor`가 파일 읽기, 도구 호출, 수정, 검증까지 실제 일을 계속 맡고, 어려운 갈림길에서만 `advisor`에게 묻습니다.

여기서 `advisor`는 별도 worker가 아닙니다. shared context를 읽고 `다음 계획`, `방향 수정`, `여기서 멈춰라` 같은 판단만 돌려줍니다. 도구를 직접 잡지도 않고, 결과물을 대신 만들지도 않습니다. 실행은 계속 `executor`가 합니다.

복잡한 아키텍처 분기, 막힌 디버깅, 위험한 계획 수정처럼 지금 판단만 한 단계 높여야 할 때 특히 잘 맞습니다. 반대로 작업을 병렬로 나누거나 역할별 토론이 필요하면 `Orchestrator`나 `Agent Team` 쪽이 더 낫습니다.

## **설계 원칙: 패턴보다 경계가 중요하다**

패턴 이름보다 중요한 것은 경계 설계입니다. 어떤 패턴을 쓰든 다음 원칙은 같습니다.

**1. 각 에이전트는 필요한 것만 받아야 한다**

컨텍스트를 많이 주는 것은 친절이 아닙니다. 불필요한 파일과 기록은 판단을 흐립니다. 작업에 필요한 정보만 주고, 나머지는 차단해야 합니다.

**2. 역할과 금지 행동을 먼저 정해야 한다**

“조사해 줘”보다 “이 디렉터리 안에서 관련 테스트 파일만 찾아 요약하고, 코드는 수정하지 마라”가 강합니다. 역할과 금지 행동이 있어야 결과를 통제할 수 있습니다.

**3. 출력 계약을 구조화해야 한다**

자유 형식 보고서는 읽기 좋지만 파싱하기 어렵습니다. 에이전트 간 통신이나 자동화된 통합에는 JSON, 체크리스트, 표, 고정 섹션 같은 구조화된 출력이 유리합니다.

**4. 생성자와 검증자는 갈라야 한다**

자기 결과를 자기가 통과시키는 구조는 약합니다. 최소한 중요한 단계에서는 다른 에이전트, 다른 모델, 결정론적 테스트, 또는 사람이 검증해야 합니다.

**5. 중단 조건과 수렴 조건이 있어야 한다**

멀티 에이전트는 쉽게 길어집니다. 몇 번까지 재시도할지, 무엇을 완료로 볼지, 어떤 경우 사람에게 넘길지를 미리 정해야 합니다.

## **멀티 모델은 멀티 에이전트와 다르다**

Claude, Codex, Gemini 같은 여러 모델을 교차 사용하는 것은 유용합니다. 하지만 그것만으로 곧바로 Agent Team이 되는 것은 아닙니다. 대부분의 경우 이는 독립 평가자 풀을 두는 방식에 가깝습니다.

예를 들어 Claude가 구현 계획을 만들고, Codex가 아키텍처 리스크를 검토하고, Gemini가 UI나 문서 가독성을 피드백할 수 있습니다. 이때 핵심은 어느 모델이 더 우월한지 겨루는 것이 아닙니다. 합의, 불일치, 불확실을 분리해 판단의 해상도를 높이는 것입니다.

다만, 하나의 컨텍스트 즉, 하나의 세션에서 모델을 바꿔가며 물어보는 것은 성능 저하와 비용 증가로 이어질 수 있습니다. 이 점을 유의해주세요. 

## **결론: 더 많은 AI가 아니라 더 좁은 역할의 AI 여럿**

멀티 에이전트가 강력한 이유는 처리량이 늘어서만이 아닙니다. 메인 컨텍스트를 비우고, 판단 책임을 분리하고, 검증 관점을 독립시킬 수 있기 때문입니다.

좋은 멀티 에이전트 설계는 “AI를 많이 붙인다”가 아닙니다. “더 좁은 역할의 AI 여럿을, 더 명확한 경계 안에서 일하게 한다”입니다. 다음 장에서는 이 원칙을 실제 워크플로우와 도구 세팅으로 고정하는 방법을 봅니다.
