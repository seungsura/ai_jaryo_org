아래는 1장부터 9장까지 전체 소제목 지도입니다.
아직 원문을 바꾸는 것이 아니라, 각 장이 발표 안에서 어떤 역할을 하는지, 현재 소제목 흐름이 자연스러운지, 어디를 줄이거나 옮길 후보로 볼지 정리한 것입니다.

|  장 | 분류                   | 발표상 역할                               | 핵심 질문                      |
| -: | -------------------- | ------------------------------------ | -------------------------- |
|  1 | 시대 변화 / 문제 제기        | 개발자의 역할이 어디로 이동하는지 설명                | 코딩은 정말 사라지는가               |
|  2 | 도구 흐름 / Harness로의 수렴 | Claude Code를 통해 AI 코딩 도구의 변화 방향을 보여줌 | 왜 도구들이 비슷한 구조로 모이는가        |
|  3 | 방법론의 재등장             | AI가 빨라질수록 방향 고정과 검증이 중요해졌음을 설명       | 왜 SDD와 TDD가 다시 중요해졌는가      |
|  4 | 개념 본론                | Harness의 정의와 구성 요소를 설명               | Harness는 정확히 무엇인가          |
|  5 | 실패 패턴                | Harness가 없을 때 어디서 망가지는지 보여줌          | 긴 작업과 긴 컨텍스트는 왜 위험한가       |
|  6 | 역할 분해 / 멀티 에이전트      | 컨텍스트와 책임을 나누는 방법을 설명                 | 역할과 검증 책임을 어떻게 나눌 것인가      |
|  7 | 실전 운영                | 원칙을 워크플로우와 도구 세팅으로 고정                | 현장에서 어떻게 굴릴 것인가            |
|  8 | 실제 적용 사례             | 이 발표 자체가 Harness 방식으로 만들어졌음을 보여줌     | 이 원칙이 실제 제작 과정에 어떻게 적용되었는가 |
|  9 | 결론 / 행동 제안           | 개인과 팀이 다음에 해야 할 일을 제시                | 무엇부터 바꿀 것인가                |

전체 흐름은 다음처럼 잡는 것이 가장 자연스럽습니다.

```
1장: 개발자의 역할이 변하고 있다
  ↓
2장: AI 코딩 도구는 Harness 구조로 수렴하고 있다
  ↓
3장: 그래서 방향 고정과 검증 방법론이 다시 중요해졌다
  ↓
4장: Harness는 그 방법론과 도구를 담는 실행 환경이다
  ↓
5장: Harness가 없으면 긴 작업에서 실패가 누적된다
  ↓
6장: 그래서 역할과 컨텍스트를 나눠야 한다
  ↓
7장: 이를 실제 워크플로우와 도구 세팅으로 고정한다
  ↓
8장: 이 발표 제작 과정이 그 사례다
  ↓
9장: 이제 개인과 팀은 Harness Engineer의 방향으로 움직여야 한다
```

1장. 코딩은 사라지는가
분류: 시대 변화 / 개발자 역할 전환 / 문제 제기
역할: 청중의 불안과 저항을 역사적 패턴 안에 넣어 해석하는 장입니다. 
현재 소제목 흐름
```
## 시작하며: 언어의 발전
## 추상화의 역사: 직접 하던 일은 계속 줄어들었다
## 숫자로 보는 변화
## 문서가 코드다.
## 개발자의 새로운 역할: 컨텍스트를 설계하는 능력
## 그래도 기초가 중요하다.
```
| 소제목                        | 역할                                                            | 평가                                           |
| -------------------------- | ------------------------------------------------------------- | -------------------------------------------- |
| 시작하며: 언어의 발전               | 기계어 → 어셈블리 → C/Pascal → Java → Python → AI 개발로 이어지는 역사적 전환 제시 | 도입으로 좋음. “AI도 또 하나의 추상화 상승”이라는 메시지가 자연스럽게 열림 |
| 추상화의 역사: 직접 하던 일은 계속 줄어들었다 | 각 시대마다 무엇을 위임했는지 정리                                           | 1장의 중심 주장. 반드시 유지                            |
| 숫자로 보는 변화                  | AI 코딩이 이미 현실화되고 있다는 근거 제시                                     | 설득력은 강하지만, 수치 출처와 표현 강도는 나중에 점검 필요           |
| 문서가 코드다.                   | AI 시대에 문서가 실행 규칙이 된다는 전환 설명                                   | 2장, 4장과 연결되는 중요한 다리                          |
| 개발자의 새로운 역할: 컨텍스트를 설계하는 능력 | 개발자의 역할이 구현자에서 환경 설계자로 이동한다는 주장                               | 1장의 핵심 결론에 가까움                               |
| 그래도 기초가 중요하다.              | “AI가 코드를 쓰면 기본기가 덜 중요해진다”는 오해 반박                              | 결론부로 좋음. 청중의 불안을 안정시키는 역할                    |

권장 흐름
1장은 현재 순서가 대체로 좋습니다.
```
언어 발전의 역사
  ↓
추상화의 역사
  ↓
실제 변화의 근거
  ↓
문서가 실행 규칙이 되는 변화
  ↓
개발자의 새 역할
  ↓
그래도 기초가 중요하다는 결론
```
조정 후보
1. “숫자로 보는 변화”는 근거 장으로 짧고 강하게
이 부분은 청중을 설득하는 힘이 있지만, 발표에서는 길게 설명하기보다 “이미 이 변화는 시작됐다” 정도로 쓰는 것이 좋습니다. 너무 오래 머물면 1장의 중심인 “역할 변화”보다 사례 자체가 더 커질 수 있습니다.
2. “문서가 코드다”는 4장으로 가는 중요한 다리
이 소제목은 반드시 유지하는 것이 좋습니다.
왜냐하면 4장에서 CLAUDE.md, AGENTS.md, Skills, Hooks, Memory 같은 이야기가 나오기 전에, 1장에서 먼저 문서가 더 이상 사후 정리가 아니라 실행 환경의 일부라는 감각을 만들어 주기 때문입니다.
3. 마지막 질문은 2장으로 넘겨야 함
1장의 마지막은 현재 “코딩은 사라지는가?”가 아니라 “AI 코딩은 무엇인가?”로 질문을 바꿉니다. 이 흐름은 좋습니다. 이 질문이 2장으로 넘어가며 “그럼 최근 AI 코딩 도구는 어떤 구조로 진화했는가?”로 이어집니다.
1장 평가

| 기준          | 평가             |
| ----------- | -------------- |
| 장의 역할       | 명확함            |
| 흐름          | 좋음             |
| 발표 도입 이후 연결 | 좋음             |
| 중복 위험       | 낮음             |
| 조정 필요       | 수치 근거 표현 강도 점검 |
| 결론성         | 좋음             |

1장은 큰 구조를 바꾸지 않는 것이 좋습니다.
이 장의 핵심 문장은 다음으로 볼 수 있습니다.

> 직접 하는 일은 줄고, 설계하는 일은 늘어난다.


2장. 왜 Claude Code인가, 그리고 왜 Harness인가
분류: 도구 흐름 / Prompt → Context → Harness 전환 / 대표 사례
역할: 특정 제품 리뷰가 아니라, Claude Code를 통해 AI 코딩 도구가 어떤 방향으로 수렴하는지 보여주는 장입니다. 
현재 소제목 흐름

```
시작하며: 특정 제품 리뷰가 아닙니다.
## 에이전틱 코딩의 실제 성과
## 1막: Copilot과 ChatGPT, 프롬프트의 시대 (2022~2024)
## 프롬프트 시대의 벽: 모델은 보지 못한 것을 알 수 없다
## 2막: Cursor와 컨텍스트의 시대
## 컨텍스트 시대의 벽: 좋은 입력만으로는 루프를 통제할 수 없다
## 3막: Claude Code, Codex, OpenCode 와 하네스의 시대
```

소제목 지도

| 소제목                                        | 역할                                                                 | 평가                                           |
| ------------------------------------------- | ------------------------------------------------------------------ | -------------------------------------------- |
| 시작하며: 특정 제품 리뷰가 아닙니다.              | Claude Code를 대표 사례로 쓰는 이유 설명                                       | 좋음. 제품 홍보나 리뷰가 아니라는 선 긋기가 필요함 |
| 에이전틱 코딩의 실제 성과                         | 왜 이 흐름이 중요한지 성과로 보여줌                                             | 위치는 좋지만, 수치와 사례의 출처 점검 필요           |
| 1막: Copilot과 ChatGPT, 프롬프트의 시대          | 프롬프트가 중심이던 시기 설명                                                | 좋음                                           |

| 2막: Cursor와 컨텍스트의 시대                    | 코드베이스 인덱싱, RAG, 시맨틱 검색 등 컨텍스트 중심 전환 설명                               | 좋음                                           |
| 컨텍스트 시대의 벽: 좋은 입력만으로는 루프를 통제할 수 없다 | 좋은 입력만으로는 실행 루프를 통제할 수 없다는 한계 제시                               | 매우 중요. 3막으로 넘어가는 전환점                 |
| 3막: Claude Code, Codex, OpenCode와 하네스의 시대 | 실행 환경 전체를 품는 도구로의 수렴 설명                                             | 2장의 결론 역할                                |

권장 흐름
2장은 3막 구조를 유지하는 것이 좋습니다.
```
특정 제품 리뷰가 아니다
  ↓
에이전틱 코딩의 성과가 이미 나오고 있다
  ↓
1막: Prompt Engineering
  ↓
Prompt 시대의 벽
  ↓
2막: Context Engineering
  ↓
Context 시대의 벽
  ↓
3막: Harness Engineering
  ↓
Prompt ⊂ Context ⊂ Harness
```
2장과 4장의 경계
이 장에서 가장 중요한 것은 4장과 겹치지 않는 것입니다.

| 구분 | 2장 | 4장 |
| --- | --- | --- |
| 중심 질문 | 왜 Harness로 왔는가 | Harness는 무엇인가 |
| 설명 방식 | 역사, 흐름, 대표 도구 | 개념, 구성 요소, 설계 원리 |
| Claude Code | 대표 사례 | 도구 레이어 예시 |
| Skills, Hooks, MCP | 하네스 시대의 특징으로 언급 | 각각의 역할과 기능 설명 |
| 결론 | 도구들이 Harness로 수렴한다 | Harness는 프롬프트와 컨텍스트를 포함하는 실행 환경이다 |

조정 후보
1. Agent = Model + Harness는 2장에서 짧게
2장 초반에 Agent = Model + Harness를 제시하는 것은 좋습니다. 다만 여기서 너무 깊게 설명하면 4장과 겹칩니다.
2장에서는 이 정도 역할이면 충분합니다.
```
최근 AI 코딩 도구의 핵심은 모델 자체가 아니라,
모델을 감싸는 실행 환경 전체로 이동하고 있다.
```
구성 요소의 상세 설명은 4장으로 넘기는 것이 좋습니다.
2. 에이전틱 코딩 성과는 도입부의 동기 부여로 사용
“4~8개월 프로젝트를 2주에 완료”, “온보딩 단축” 같은 내용은 강력한 훅입니다. 다만 발표에서는 이 부분이 너무 커지면 도구 흐름 설명이 늦어집니다.
따라서 이 소제목은 짧고 강한 동기 부여로 두는 것이 좋습니다.
3. 2장의 결론은 “하네스의 정의”가 아니라 “하네스로 수렴”
2장의 마지막 메시지는 다음 정도가 좋습니다.

> 프롬프트 시대에는 “어떻게 말할까”가 중요했고, 컨텍스트 시대에는 “무엇을 보여줄까”가 중요했습니다. 이제 하네스 시대에는 “어떤 실행 환경을 만들까”가 중요해졌습니다.

2장 평가

| 기준          | 평가             |
| ----------- | -------------- |
| 장의 역할       | 명확함            |
| 1장과 연결      | 좋음             |
| 4장과 중복 위험   | 있음             |
| 조정 필요       | Harness 구성 요소 설명을 4장으로 넘기기 |
| 발표 흐름       | 좋음             |

2장은 도구 리뷰가 아니라 시대 전환 장입니다.

3장. AI 시대의 개발 방법론
분류: 방법론의 재등장 / 방향 고정과 검증 / SDD → TDD
역할: AI가 빨라질수록 왜 오히려 방법론이 필요해지는지 설명하는 장입니다. 
현재 소제목 흐름

```
왜 지금 방법론
## TDD: AI 가드레일
## SDD: 스펙이 진실의 원천
## Waterfall과는 무엇이 다른가
```

확정 방향
사용자 결정에 따라 3장은 다음 방향으로 정리합니다.
AI가 매우 빠르기 때문에 오히려 방향 고정과 검증이 중요해졌다.SDD는 방향을 고정하고, TDD는 결과를 검증한다.
그리고 흐름은 SDD → TDD로 갑니다.
권장 소제목 흐름

```
## 왜 지금 방법론
## SDD: 스펙이 진실의 원천
## TDD: AI 가드레일
## Waterfall과는 무엇이 다른가
## SDD + TDD가 Harness로 이어지는 이유
```

마지막 소제목은 현재 원문에 독립 제목으로 있지는 않지만, 결론부의 역할을 분명히 하기 위한 구조 제안입니다.
소제목 지도

| 소제목 | 역할 | 평가 |
| --- | --- | --- |
| 왜 지금 방법론 | AI 코딩이 빨라지면서 병목이 생성에서 방향과 검증으로 이동했음을 설명 | 매우 중요. 여기에 새 핵심 문장을 넣는 것이 좋음 |
| SDD: 스펙이 진실의 원천 | 구현 전에 무엇을 만들지 고정 | TDD보다 먼저 오는 것이 발표 흐름상 자연스러움 |
| TDD: AI 가드레일 | 구현 결과가 요구사항을 만족하는지 검증 | SDD 뒤에 오면 역할이 더 선명해짐 |
| Waterfall과는 무엇이 다른가 | 스펙을 먼저 쓰는 것이 옛날 Waterfall 회귀가 아님을 설명 | 반론 차단용으로 필요 |
| SDD + TDD가 Harness로 이어지는 이유 | 사람이 매번 붙드는 방법론을 파일, 규칙, Skills, Hooks로 고정하면 Harness가 됨 | 4장으로 넘어가는 다리 |

추가할 핵심 문장 위치
현재 도입부는 다음 흐름입니다.
```
이전의 병목이 `코드의 생성`이었다면 지금의 병목은 `AI에게 무엇을 시킬지, 어떻게 검증할 것인지`입니다.
```
이 바로 다음에 아래 문장을 넣는 것이 가장 자연스럽습니다.
```
AI가 매우 빠르기 때문에 오히려 방향 고정과 검증이 중요해졌습니다.
```
조금 더 설명형으로 쓰면 다음도 가능합니다.
```
AI가 매우 빠르게 코드를 만들 수 있기 때문에, 오히려 방향을 먼저 고정하고 결과를 검증하는 일이 더 중요해졌습니다.
```
하지만 사용자께서 제시한 원문 감각을 살리려면 첫 번째 문장이 더 좋습니다.
SDD를 앞으로 옮길 때 주의할 점
현재 SDD 소제목의 첫 문장은 TDD가 이미 앞에서 설명되었다는 전제로 시작합니다.
```
구현 단계에서 `TDD`가 에이전트의 일탈을 막아낸다면, 그 이전 단계에서 방향을 통제하는 것은 `SDD`입니다.
```
SDD를 TDD보다 앞으로 옮기면 이 연결문은 문맥상 맞지 않게 됩니다.
따라서 실제 수정 단계에서는 이 부분만 연결문 조정이 필요합니다.
방향은 다음처럼 가야 합니다.
```
구현에 들어가기 전에 방향을 통제하는 것은 SDD입니다.
```
아직 확정 수정은 아니고, SDD를 앞으로 옮길 경우 필요한 문맥 조정 후보입니다.
Spec-first 처리
Spec-first는 독립 축으로 키우지 않고, SDD의 작동 방식으로 흡수하는 것이 좋습니다.
현재 3장의 메시지를 다음처럼 정리하면 선명합니다.

방법론 | 역할
--- | ---
SDD | 무엇을 만들지 먼저 고정
TDD | 만든 것이 맞는지 검증
Spec-first | SDD 안에서 구현 전에 스펙을 먼저 잠그는 작동 방식

3장 평가

| 기준 | 평가 |
| --- | --- |
| 장의 역할 | 매우 중요 |
| 현재 순서 | 조정 필요 |
| 권장 순서 | SDD → TDD |
| 줄일 부분 | Spec-first 독립 설명 |
| 추가할 문장 | “AI가 매우 빠르기 때문에 오히려 방향 고정과 검증이 중요해졌습니다.” |
| 4장 연결 | 좋음 |

3장의 핵심 문장은 이렇게 잡는 것이 좋습니다.

> AI가 매우 빠르기 때문에 오히려 방향 고정과 검증이 중요해졌습니다. SDD는 방향을 고정하고, TDD는 결과를 검증합니다.

연결 논리는 다음입니다.
```
개인은 새 모델을 쫓느라 피로해지고,팀은 각자 다른 방식으로 AI를 쓰면서 혼란과 비용을 만든다.그래서 필요한 것은 더 많은 정보가 아니라 운영 성숙도다.
```
2. Spec-first 용어 정합성
9장에는 Spec-first, TDD, Plan-Critic, 검증 분리 같은 원칙이 언급됩니다.
3장에서 Spec-first를 독립 축으로 키우지 않기로 했으므로, 9장에서는 이 표현을 쓰더라도 SDD의 일부로 받아들여지게 정합성을 맞추는 것이 좋습니다.
즉 9장에서의 의미는 다음이어야 합니다.
```
Spec-first는 별도 장르가 아니라,SDD처럼 구현 전에 방향을 먼저 고정하는 태도다.
```
3. 마지막 행동 제안은 매우 좋음
현재 9장의 마지막 행동 제안은 발표 결론으로 좋습니다.
```
반복해서 말하는 규칙 세 가지를 파일로 빼고,
테스트나 리뷰 같은 결정론적 문 하나를 추가하고,
긴 세션의 결론을 채팅창 밖 문서에 남기면 됩니다.
```
이 부분은 청중이 바로 가져갈 수 있는 문장입니다.
9장 평가
| 기준       | 평가                                  |
| -------- | ----------------------------------- |
| 결론 역할    | 명확함                                 |
| 개인과 팀 연결 | 좋음                                  |

9장은 요약 장이 아니라 행동 장입니다.

전체 조정 우선순위
이제 1장부터 9장까지 봤을 때, 구조적으로 가장 먼저 정리해야 할 부분은 다음 순서입니다.
1순위. 3장 순서 조정
현재:
TDD → SDD
권장:
SDD → TDD
그리고 도입부에 다음 문장을 추가합니다.
AI가 매우 빠르기 때문에 오히려 방향 고정과 검증이 중요해졌습니다.
이 조정은 3장의 메시지를 크게 선명하게 만듭니다.

2순위. 4장 내부 구조 재정렬
4장은 내용이 가장 많기 때문에, 순서가 중요합니다.
권장 원칙:
정의 → 실행 루프 → 기능 블록 → 도구 레이어 → 운영 원리 → 결론
특히 다음 소제목은 앞으로 당기는 것이 좋습니다.
기능 블록과 도구 레이어는 다르다하네스 엔지니어링의 5대 기능 블록
이 둘이 먼저 나와야 CLAUDE.md, Skills, Hooks, MCP 같은 도구들이 단순 기능 나열처럼 보이지 않습니다.

3순위. 5장 결론의 힘 조정
5장은 Keep Quality Left가 핵심입니다.
기계가 확인할 수 있는 것은 앞쪽의 결정론적 게이트로 옮기고,AI는 판단과 생성에 집중하게 하라.
이 메시지가 5장의 정점으로 느껴지게 해야 합니다.
SOLID도 하네스 관점으로는 좋은 내용이지만, 발표에서는 비중과 위치를 조절할 필요가 있습니다.

4순위. 6장 패턴 비중 조절
6장은 모든 패턴을 같은 비중으로 다루면 길어집니다.
발표에서는 다음 세 가지를 중심으로 두는 것이 좋습니다.
Sub-AgentGAN-StyleOrchestrator / Parallel
Agent Teams, Advisor는 고급 패턴으로 짧게 다뤄도 됩니다.

5순위. 8장은 디테일보다 적용 사례
8장은 디렉터리 구조 설명이 아니라, 다음 원리를 보여주는 장입니다.
source boundaryprose spineoutline / manifestdesign contractgeneration / validation separationnotes last
즉 “어떤 파일이 어디 있었는가”보다 “각 층이 어떤 책임을 맡았는가”가 중심입니다.

전체 평가
현재 문서는 큰 흐름이 좋습니다.
가장 강한 점은 이론에서 끝나지 않고 실패, 운영, 실제 제작 사례, 다음 행동까지 내려온다는 점입니다.
다만 발표 순서에 맞게 더 선명하게 만들려면 다음 세 가지를 계속 기준으로 삼으면 됩니다.
1. 이 장은 앞 장의 어떤 질문에 답하는가?2. 이 장은 다음 장으로 어떤 질문을 넘기는가?3. 이 장에서만 해야 하는 말은 무엇인가?
이 기준으로 보면 전체 문서는 다음 구조로 정리됩니다.
1장: 코딩은 사라지는가→ 개발자의 역할은 사라지는 것이 아니라 올라간다.2장: 왜 Claude Code인가, 그리고 왜 Harness인가→ AI 코딩 도구는 Prompt → Context → Harness로 수렴한다.3장: AI 시대의 개발 방법론→ AI가 빠르기 때문에 SDD로 방향을 고정하고 TDD로 검증해야 한다.4장: 프롬프트를 넘어서: Harness→ Harness는 프롬프트와 컨텍스트를 포함하는 실행 환경이다.5장: 이렇게 하면 망한다→ 긴 루프와 긴 컨텍스트는 실패를 누적시키므로 결정론적 게이트가 필요하다.6장: 멀티 에이전트 활용 패턴→ 역할과 컨텍스트를 나누고 생성과 검증을 분리해야 한다.7장: 실전 워크플로우와 도구 세팅→ 원칙을 명령어, 게이트, 승인, Worktree, 이슈로 고정해야 한다.8장: 이 글과 발표가 만들어진 과정→ 이 발표 자체가 Harness 방식으로 만들어진 사례다.9장: 우리가 다음에 해야 할 일→ 개인과 팀은 Harness Engineer의 방향으로 움직여야 한다.
이제 구조 논의의 다음 단계는 각 장별 실제 소제목 순서 확정입니다. 가장 먼저 손대야 할 장은 3장, 그다음은 4장입니다.
