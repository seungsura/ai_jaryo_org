# 01-코딩은 사라지는가

## **시작하며: 언어의 발전**

- **기계어 → 어셈블리**:

```jsx
10111000 00000001 00000000 00000000 00000000  // "레지스터(EAX)에 1을 넣어라"
```

```jsx
MOV EAX, 1 // "레지스터(EAX)에 1을 넣어라"
```

- "기계어를 직접 다루지 않으면 진정한 개발이 아니다!" — 1950년대 초기 프로그래머들은 '컴파일러가 만든 코드가 사람보다 효율적일 리 없다'며 고급 언어 도입을 거부했습니다. 이는 단순한 기술적 주장이 아니라 "하드웨어를 직접 제어하는 것이 진짜 전문성"이라는 직업적 자부심에서 비롯된 저항이었습니다.
- **어셈블리 → C/Pascal**:
    
    ```jsx
    section .data
      msg db "Hello World"
    
    section .text
      global_start
    
    _start:
    	mov rax, 1
    	mov rdi, 1
    	mov rsi, msg
    	mov rdx, 12
    	syscall
    	mov rax, 60
    	mov rdi, 0
    	syscall
    ```
    
    ```jsx
    #include <stdio.h>
    
    int main() {
        printf("Hello\n");
        return 0;
    }
    ```
    
    - "진짜 프로그래머는 파스칼 같은 걸 쓰지 않는다!" — 1983년 Ed Post의 에세이 *"Real Programmers Don't Use Pascal"*은 당시 분위기를 잘 보여줍니다. 고급 언어를 쓰는 개발자는 '퀴슈나 먹는 사람들(quiche eaters)'이라는 비하가 유행했고, 하드웨어를 직접 제어하지 않으면 실력 없는 개발자로 취급받았습니다.
- **C → Java**:
    
    ```jsx
    #include <stdio.h>
    #include <stdlib.h>
    
    ...
    
    int main() {
        Point* p = (Point*)malloc(sizeof(Point));
        p->x = 10;
        p->y = 20;
        printf("Point: %d, %d\n", p->x, p->y);
        free(p); 
        return 0;
    }
    ```
    
    ```jsx
            Point p = new Point(10, 20);
            System.out.println("Point: " + p.x + ", " + p.y);
            p = null;
    ```
    
    - "메모리를 직접 관리하지 않으면 진짜 개발자가 아니다!" — GC가 등장하자 숙련된 C 개발자들은 "내가 포인터를 직접 다룰 줄 알아야 진짜지"라고 반발했습니다. 하지만 GC는 결국 메모리 해제 버그를 획기적으로 줄이며 대세가 되었습니다.
- **Java → Python**:
    
    ```jsx
    public class HelloWorld {
        public static void main(String[] args) {
            System.out.println("Hello, World!");
        }
    }
    ```
    
    ```jsx
    print("Hello, World!")
    ```
    
    - "장난감 언어로 무슨 개발을 하냐!" — Python이 처음 등장했을 때, 정적 타입 없이 인터프리터로 동작하는 언어는 '실업무에 쓸 수 없는 스크립트' 수준으로 여겨졌습니다. 오늘날 Python은 AI·데이터 분야의 사실상 표준 언어입니다.
- **AI 개발:**
    
    ```jsx
    이 함수를 리펙토링하고 테스트 코드를 작성해줘.
    ```
    
    - “자연어로 시키는 건 진짜 개발이 아니다!” 하지만 역사는 매번 비슷한 방향으로 움직였습니다. 직접 하는 일은 줄었고, 더 큰 구조를 설계하는 일이 그 자리를 채웠습니다. 이번에도 핵심은 “개발이 사라진다”가 아니라 “개발의 추상화 수준이 올라간다”입니다.

## **추상화의 역사: 직접 하던 일은 계속 줄어들었다**

| **시대** | **전환** | **위임한 것** |
| --- | --- | --- |
| 1950s | 기계어 → 어셈블리 | 이진 코드를 직접 작성하는 부담 |
| 1960s | 어셈블리 → C | 레지스터와 명령어 최적화를 컴파일러에 위임 |
| 1990s | C/C++ → Java | 메모리 해제를 가비지 컬렉터에 위임 |
| 2000s | 언어 → 프레임워크 | 반복되는 보일러플레이트를 프레임워크에 위임 |
| 2010s | 온프레미스 → 클라우드 | 서버 운영과 확장을 클라우드에 위임 |
| 2020s | 직접 코딩 → AI 에이전트 | 코드 작성의 상당 부분을 위임하고, 작업 환경을 설계 |

AI 코딩 도구가 빠르게 좋아질수록 사람들은 같은 질문을 반복합니다. “이제 코딩은 사라지는가?” 하지만 이 질문은 조금 빗나가 있습니다. 더 정확한 질문은 이것입니다.

> **개발자가 끝까지 직접 붙들어야 하는 일은 어디로 옮겨 가는가?**
> 

매번 비슷한 반발도 함께 있었습니다. “그걸 직접 하지 않으면 진짜 개발이 아니다”라는 말입니다. 기계어에서 고급 언어로 넘어갈 때도, 수동 메모리 관리에서 GC로 넘어갈 때도, 프레임워크와 클라우드가 들어올 때도 비슷한 얘기가 나왔습니다. 하지만 시간이 지나면 늘 같은 결론이 나왔습니다. 직접 하는 일은 줄었고, 대신 더 큰 구조를 설계하는 일이 실력의 중심으로 올라왔습니다.

## 숫자로 보는 변화

YC W25(Y Combinator 2025 참가팀)의 통계에 따르면, 약 25%의 스타트업이 코드베이스의 95% 이상을 AI로 만들었습니다. OpenAI Codex 팀은 5개월간 수동 코드를 한 줄도 직접 작성하지 않고 100만 줄 규모의 프로덕트를 완성했으며, Meta 엔지니어는 직접 코드를 타이핑 하지 않은지 3개월이 넘었다고 밝혔습니다. 

## **문서가 코드다.**

AI와 함께 일하는 환경에서 문서의 역할도 바뀌었습니다. 과거의 문서는 대개 사람이 읽는 참고 자료였습니다. README, 컨벤션 문서, 아키텍처 노트는 중요했지만, 실제 실행 경로와는 떨어져 있는 경우가 많았습니다.

AI 코딩 도구에서는 이야기가 달라집니다. `CLAUDE.md`, `AGENTS.md`, `Cursor Rules`, `copilot-instructions.md` 이름은 모두 다르지만 역할은 동일합니다. 에이전트가 매 세션 읽고 따라야 할 운영 규칙입니다. 코딩 스타일, 금지 명령, 리뷰 기준, 테스트 규칙, 문서 어투, 승인 조건 같은 내용이 이 파일에 들어갑니다. 

과거에는 “문서화”가 사후 정리처럼 취급되었습니다. 이제는 문서가 곧 개발의 일부가 됩니다.

## **개발자의 새로운 역할: 컨텍스트를 설계하는 능력**

과거의 개발자는 코드를 잘 작성하고 버그를 잘 고치는 사람이었습니다. 앞으로의 개발자는 에이전트가 코드를 잘 쓸 수 있도록 환경을 설계하고, 결과를 판단하는 사람에 가까워집니다.

건축가는 직접 벽돌을 모두 쌓지 않습니다. 하지만 설계도, 구조, 재료, 안전 기준, 시공 순서를 정의합니다. 오케스트라 지휘자도 모든 악기를 직접 연주하지 않습니다. 대신 어느 악기가 언제 들어오고, 어디서 멈추고, 전체가 어떤 소리를 내야 하는지 조율합니다.

AI 시대의 개발자도 비슷한 방향으로 이동합니다. 직접 구현하는 시간은 줄어들 수 있지만, 다음 능력의 중요성은 더 커집니다.

- 문제를 올바른 단위로 자르는 능력
- 에이전트에게 보여 줄 컨텍스트를 고르는 능력
- 요구사항과 수용 기준을 명확히 쓰는 능력
- 테스트와 검증 게이트를 설계하는 능력
- 위험한 권한과 안전한 권한을 나누는 능력
- 에이전트의 결과를 의심하고 판단하는 능력
- 반복되는 판단 기준을 파일과 규칙으로 외재화하는 능력

이것이 이 글에서 말하는 `Harness Engineer`의 출발점입니다.

## **그래도 기초가 중요하다.**

AI가 코드를 더 많이 써 줄수록 기본기가 덜 중요해질 것처럼 보입니다. 실제로는 반대입니다. AI는 빠르게 초안을 만들 수 있지만, 그 초안이 안전한지, 유지보수 가능한지, 기존 시스템과 잘 맞는지 판단하려면 여전히 깊은 기본기가 필요합니다.

간단한 데모 앱은 매우 쉽게 만들 수 있습니다. 하지만 실제 서비스에서 곧 문제가 터집니다. 인증과 권한, 동시성, 데이터베이스 스키마, 캐싱, 관측 가능성, 장애 복구, 보안, 비용, 배포 파이프라인 같은 문제가 뒤따릅니다. 이런 구간에서 “AI 가 시켰어요” 라고 탓할 수는 없습니다.

오히려 AI가 코드를 많이 만들어 줄수록, 어느 부분을 의심해야 하는지 아는 능력이 더 중요해집니다. 모델이 제안한 아키텍처가 왜 위험한지, 테스트가 무엇을 놓쳤는지, API 설계가 나중에 어떤 병목을 만들지, 보안 경계가 어디서 무너지는지를 파악할 수 있어야 합니다.

줄어든 것은 타이핑의 비중이지, 시스템을 이해하는 능력의 가치가 아닙니다. 개발자의 기본기는 사라지는 것이 아니라 더욱 더 중요하게 여겨집니다.

코딩은 끝났다 라는 말은 틀렸습니다. 하지만 손으로 한 줄씩 치는 작성하는 비중은 줄고 있습니다. 대신 규칙을 세우고, 컨텍스트를 고르고, 결과를 검증하고, 상태를 남기는 비중이 점점 늘고 있습니다. 

그래서 질문을 바꿔야 합니다. “코딩은 사라지는가?”가 아니라 “AI 코딩은 무엇인가?”입니다. 

> **코딩은 에이전트가 일할 환경을 설계하고 검증하는 영역으로 이동하고 있습니다.**
>
