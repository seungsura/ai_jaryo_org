# 01-코딩은 사라지는가

## **시작하며: 언어의 발전**

- **기계어 → 어셈블리**:

```jsx
10111000 00000001 00000000 00000000 00000000  // "레지스터(EAX)에 1을 넣어라"
```

```jsx
MOV EAX, 1 // "레지스터(EAX)에 1을 넣어라"
```

- "컴파일러가 만든 코드가 사람보다 효율적일 리 없다!" — 1950년대 초기 프로그래머들은 '컴파일러가 만든 코드가 사람보다 효율적일 리 없다'며 고급 언어 도입을 거부했습니다. 이는 단순한 기술적 주장이 아니라 "하드웨어를 직접 제어하는 것이 진짜 전문성"이라는 직업적 자부심에서 비롯된 저항이었습니다.
- **어셈블리 → C/Pascal**:

    ```jsx
    section .data
      msg db "Hello World"

    section .text
      global_start

    _start:
    	mov rax, 1
    	mov rdi, 1
    	mov rsi, msg
    	mov rdx, 12
    	syscall
    	mov rax, 60
    	mov rdi, 0
    	syscall
    ```

    ```jsx
    #include <stdio.h>

    int main() {
        printf("Hello\n");
        return 0;
    }
    ```

    - "진짜 프로그래머는 파스칼 같은 걸 쓰지 않는다!" — 1983년 Ed Post의 에세이 *"Real Programmers Don't Use Pascal"*은 당시 분위기를 잘 보여줍니다. 고급 언어를 쓰는 개발자는 '퀴슈나 먹는 사람들(quiche eaters)'이라는 비하가 유행했고, 하드웨어를 직접 제어하지 않으면 실력 없는 개발자로 취급받았습니다.
- **C → Java**:

    ```jsx
    #include <stdio.h>
    #include <stdlib.h>

    ...

    int main() {
        Point* p = (Point*)malloc(sizeof(Point));
        p->x = 10;
        p->y = 20;
        printf("Point: %d, %d\n", p->x, p->y);
        free(p);
        return 0;
    }
    ```

    ```jsx
            Point p = new Point(10, 20);
            System.out.println("Point: " + p.x + ", " + p.y);
            p = null;
    ```

    - "메모리를 직접 관리하지 않으면 진짜 개발자가 아니다!" — GC가 등장하자 숙련된 C 개발자들은 "내가 포인터를 직접 다룰 줄 알아야 진짜지"라고 반발했습니다. 하지만 GC는 결국 메모리 해제 버그를 획기적으로 줄이며 대세가 되었습니다.
- **Java → Python**:

    ```jsx
    public class HelloWorld {
        public static void main(String[] args) {
            System.out.println("Hello, World!");
        }
    }
    ```

    ```jsx
    print("Hello, World!")
    ```

    - "장난감 언어로 무슨 개발을 하냐!" — Python이 처음 등장했을 때, 정적 타입 없이 인터프리터로 동작하는 언어는 '실업무에 쓸 수 없는 스크립트' 수준으로 여겨졌습니다. 오늘날 Python은 AI·데이터 분야의 사실상 표준 언어입니다.
- **AI 개발:**

    ```jsx
    이 함수를 리펙토링하고 테스트 코드를 작성해줘.
    ```

    - “자연어로 시키는 건 진짜 개발이 아니다!” 하지만 역사는 매번 비슷한 방향으로 움직였습니다. 직접 하는 일은 줄었고, 더 큰 구조를 설계하는 일이 그 자리를 채웠습니다. 이번에도 핵심은 “개발이 사라진다”가 아니라 “개발의 추상화 수준이 올라간다”입니다.

## **추상화의 역사: 직접 하던 일은 계속 줄어들었다**[^01-v10]

| **시대** | **전환** | **위임한 것** |
| --- | --- | --- |
| 1950s | 기계어 → 어셈블리 | 이진 코드를 직접 작성하는 부담 |
| 1960s | 어셈블리 → C | 레지스터와 명령어 최적화를 컴파일러에 위임 |
| 1990s | C/C++ → Java | 메모리 해제를 가비지 컬렉터에 위임 |
| 2000s | 언어 → 프레임워크 | 반복되는 보일러플레이트를 프레임워크에 위임 |
| 2010s | 온프레미스 → 클라우드 | 서버 운영과 확장을 클라우드에 위임 |
| 2020s | 직접 코딩 → AI 에이전트 | 코드 작성의 상당 부분을 위임하고, 작업 환경을 설계 |

AI 코딩 도구가 빠르게 좋아질수록 사람들은 같은 질문을 반복합니다. “이제 코딩은 사라지는가?” 하지만 이 질문은 조금 빗나가 있습니다. 더 정확한 질문은 이것입니다.

> **직접하는 일은 줄고, 설계하는 일은 늘어난다**
>

매번 비슷한 반발도 함께 있었습니다. “그걸 직접 하지 않으면 진짜 개발이 아니다”라는 말입니다. 기계어에서 고급 언어로 넘어갈 때도, 수동 메모리 관리에서 GC로 넘어갈 때도, 프레임워크와 클라우드가 들어올 때도 비슷한 얘기가 나왔습니다. 하지만 시간이 지나면 늘 같은 결론이 나왔습니다. 직접 하는 일은 줄었고, 대신 더 큰 구조를 설계하는 일이 실력의 중심으로 올라왔습니다.

## 숫자로 보는 변화

이 변화는 먼 미래 이야기가 아닙니다.
YC W25(Y Combinator 2025 참가팀) 중 약 25%의 스타트업이 코드베이스의 95% 이상을 AI로 만들었습니다. OpenAI Codex 팀은 5개월간 수동 코드를 한 줄도 직접 작성하지 않고 100만 줄 규모의 프로덕트를 완성했으며, Meta 엔지니어는 실무에서 직접 코드를 타이핑 하지 않은지 3개월이 넘었다고 밝혔습니다.
이런 변화들은 단순히 “개발이 편해졌다”는 수준을 넘어섭니다. 개발의 본질이 바뀌고 있음을 보여줍니다.

## AI와 어떻게 함께 일할 것인가?

이제 개발의 무게중심이 바뀌었습니다. 직접 구현하는 일은 줄었지만, 방향을 정하고, 기준을 세우고, 결과를 의심하고, 검증하는 일은 여전히 중요합니다.
