# 2. 왜 Claude Code인가, 그리고 코딩 도구는 왜 Harness로 수렴하는가

## 시작하며: 특정 제품 리뷰가 아니라 진화 방향을 보려는 것이다

이 장의 출발점은 “왜 `Claude Code`인가”입니다. 하지만 목적은 특정 제품을 칭찬하는 리뷰가 아닙니다. Claude Code는 최근 AI 코딩 도구들이 어디로 진화하는지 보여 주는 대표 사례에 가깝습니다.

초기의 코딩 AI는 자동완성에 가까웠습니다. 그다음에는 대화형 코드 도우미가 되었고, 이제는 터미널에서 파일을 읽고, 명령어를 실행하고, 테스트를 돌리고, 실패하면 다시 수정하는 에이전트가 되었습니다. 이 과정에서 도구의 출발점은 달랐습니다. 어떤 도구는 IDE 플러그인에서 시작했고, 어떤 도구는 AI 에디터에서 시작했고, 어떤 도구는 터미널 CLI에서 출발했습니다. 그런데 긴 작업을 맡기기 시작하면 결국 비슷한 문제를 만납니다.

- 무엇을 읽게 할 것인가?
- 어떤 도구를 쓰게 할 것인가?
- 어떤 권한은 막을 것인가?
- 결과를 어디서 검증할 것인가?
- 실패하면 어떻게 복구할 것인가?
- 상태를 세션 밖에 어떻게 남길 것인가?

이 질문에 대한 답이 `Harness`입니다. 그래서 Claude Code를 보는 일은 Claude Code만 보는 일이 아니라, AI 코딩 도구가 하네스 중심 구조로 수렴하는 과정을 보는 일이기도 합니다.

## 세 시대: Prompt → Context → Harness

지난 몇 년의 흐름을 하나의 문장으로 압축하면 이렇습니다.

| 시대 | 핵심 질문 | 대표 장면 | 한계 |
| --- | --- | --- | --- |
| `Prompt Engineering` | 어떻게 말할 것인가 | ChatGPT, 초기 Copilot, CoT/ReAct | 모델이 보지 못하는 정보는 쓸 수 없다 |
| `Context Engineering` | 무엇을 보여줄 것인가 | Cursor, RAG, 전체 코드베이스 인덱싱 | 좋은 컨텍스트만으로 실행 루프를 통제할 수 없다 |
| `Harness Engineering` | 어떤 시스템 위에서 일하게 할 것인가 | Claude Code, Codex, Copilot Coding Agent | 하네스 자체가 복잡해지고 보안·운영 문제가 생긴다 |

이 세 시대는 서로를 대체하지 않습니다. 하네스 안에 컨텍스트가 있고, 컨텍스트 안에 프롬프트가 있습니다. 좋은 하네스는 좋은 프롬프트와 좋은 컨텍스트를 여전히 필요로 합니다. 다만 그것들을 단독 요령으로 다루지 않고, 운영 구조의 일부로 다룹니다.

## 1막: Copilot과 ChatGPT, 프롬프트의 시대

AI 코딩 도구의 대중적 출발점에는 `GitHub Copilot`이 있었습니다. 초창기 Copilot은 현재 파일의 코드를 바탕으로 다음 줄을 제안하는 스마트 자동완성에 가까웠습니다. 현재 열려 있는 파일이 곧 암묵적 프롬프트였고, 모델은 그 프롬프트의 다음 토큰을 예측했습니다.

몇 달 뒤 `ChatGPT`가 등장하면서 질문은 바뀌었습니다. “AI가 코드를 추천해 준다”에서 “AI에게 어떻게 말해야 원하는 코드를 얻을 수 있는가”로 이동했습니다. 개발자들은 프롬프트의 길이, 역할 부여, 예시, 단계별 지시를 실험했습니다. 이 시기에는 자연어 지시문이 새로운 프로그래밍 인터페이스처럼 보였습니다.

학계와 업계에서도 같은 방향의 기법이 쏟아졌습니다. `Chain-of-Thought`는 모델이 중간 추론 단계를 쓰도록 유도했고, `ReAct`는 추론과 행동을 번갈아 수행하도록 만들었습니다. `Tree-of-Thought`는 하나의 추론 경로만 따라가지 않고 여러 경로를 탐색하게 했습니다. `Self-Refine`과 `Reflexion`은 모델이 자기 출력을 비판하고 개선하게 만들었습니다.

2024년 Andrew Ng이 정리한 네 가지 에이전틱 패턴도 이 흐름의 정점에 있었습니다.

- `Reflection`: 모델이 자기 결과를 검토하고 수정한다.
- `Tool Use`: 검색, 계산기, 코드 실행 같은 외부 도구를 호출한다.
- `Planning`: 큰 목표를 하위 단계로 나눈다.
- `Multi-Agent Collaboration`: 여러 에이전트가 역할을 나눠 협업한다.

이 발견의 중요한 점은 “모델 그 자체”보다 “모델을 감싸는 패턴”이 성능을 크게 바꿀 수 있다는 사실이었습니다. 이미 이때부터 하네스의 씨앗이 보였습니다. 다만 당시의 중심 언어는 여전히 프롬프트였습니다.

## 프롬프트 시대의 벽: 모델은 보지 못한 것을 알 수 없다

프롬프트는 강력했지만, 곧 한계가 드러났습니다. 아무리 좋은 지시를 내려도 모델이 프로젝트의 다른 파일, 기존 설계 결정, 내부 유틸리티, 팀 컨벤션을 보지 못하면 제대로 활용할 수 없습니다.

예를 들어 “기존 코드를 재사용하라”고 프롬프트에 아무리 써도, 관련 유틸리티 파일이 컨텍스트에 없으면 에이전트는 그 존재를 알지 못합니다. 문제는 지시문이 부족한 것이 아니라 모델이 소비하는 정보가 부족한 것입니다.

이 지점에서 `Blind Prompting`이 문제가 됩니다. 팀이 프롬프트를 조금씩 고치고, 결과를 눈으로 보고, “이번엔 괜찮은 것 같다”고 판단하는 방식은 프로덕션 엔지니어링이라기보다 시행착오에 가깝습니다. 비결정론적 모델을 다루면서 측정과 검증 없이 문장만 만지는 접근은 오래 버티기 어렵습니다.

## 2막: Cursor와 컨텍스트의 시대

프롬프트의 벽을 가장 먼저 크게 밀어낸 도구가 `Cursor`였습니다. Copilot이 현재 파일 중심이었다면 Cursor는 전체 코드베이스를 인덱싱하고, 필요한 파일과 심볼을 검색해 컨텍스트에 넣는 방식으로 질문 자체를 바꿨습니다.

Cursor가 보여 준 핵심은 명확했습니다.

| 차원 | 초기 Copilot | Cursor 계열 도구 |
| --- | --- | --- |
| 컨텍스트 범위 | 현재 파일 중심 | 전체 코드베이스 |
| 검색 방식 | 없음 또는 제한적 | RAG, AST, 시맨틱 검색 |
| 편집 단위 | 한 줄 또는 한 블록 | 여러 파일 동시 수정 |
| 사용자 인터페이스 | 자동완성 중심 | 채팅, Composer, Agent Mode |
| 핵심 질문 | 다음 줄은 무엇인가 | 이 프로젝트에서 필요한 변경은 무엇인가 |

컨텍스트 시대의 핵심 질문은 “어떤 말을 해야 하나”가 아니라 “어떤 정보를 넣어야 하나”였습니다. 2025년 중반 `Context Engineering`이라는 표현이 널리 쓰이기 시작한 것도 이 흐름과 맞닿아 있습니다. 컨텍스트 윈도우에 다음 단계에 필요한 정보를 선별해 넣는 기술이 중요해진 것입니다.

이때 RAG 파이프라인, 코드베이스 인덱싱, 문서 검색, KV-cache 최적화, 안정 접두어와 동적 접미어 같은 개념이 전면으로 올라왔습니다. 컨텍스트 엔지니어링은 진짜 진보였습니다. 모델이 볼 수 있는 정보의 질이 결과를 바꾼다는 점을 업계가 받아들였기 때문입니다.

## 컨텍스트 시대의 벽: 좋은 입력만으로는 루프를 통제할 수 없다

하지만 컨텍스트만으로도 충분하지 않았습니다. 이유는 간단합니다. 코딩 에이전트는 한 번 답하고 끝나는 챗봇이 아니라, 루프를 도는 실행 주체이기 때문입니다.

대부분의 에이전트형 코딩 도구는 다음 루프를 반복합니다.

1. 컨텍스트를 수집한다.
2. 파일을 수정하거나 명령어를 실행한다.
3. 테스트, 린트, 타입 체크, 로그로 결과를 확인한다.
4. 실패하면 다시 수정한다.
5. 충분하다고 판단하면 보고하거나 PR을 만든다.

이 루프에서는 컨텍스트만 좋아도 문제가 생깁니다. 도구 호출이 실패할 수 있고, 테스트를 잘못 해석할 수 있고, 비용이 폭주할 수 있고, 위험한 명령을 실행할 수 있고, 보안 경계를 넘을 수 있고, 장시간 작업 중 목표를 잊을 수 있습니다.

즉, “무엇을 보여줄 것인가”만으로는 부족합니다. “무엇을 하게 할 것인가”, “무엇을 못 하게 막을 것인가”, “어디서 멈출 것인가”, “어떻게 검증할 것인가”가 함께 필요해집니다. 이 지점에서 하네스가 본격적으로 등장합니다.

## 3막: Claude Code, Codex, Copilot Coding Agent와 하네스의 시대

하네스 시대의 대표 장면은 코딩 도구가 단순한 자동완성이나 채팅을 넘어 작업 환경 전체를 품기 시작한 것입니다. `Claude Code`가 대표 사례로 자주 언급되는 이유도 여기에 있습니다.

Claude Code는 터미널 네이티브 워크플로우를 중심에 둡니다. 파일을 읽고, 셸을 실행하고, 테스트 결과를 보고, 다시 수정합니다. 여기에 `Plan Mode`, 승인 체계, `CLAUDE.md`, `Skills`, `Hooks`, `MCP`, `Plugins`, `Subagents` 같은 요소가 붙습니다. 이 요소들은 각각 독립 기능처럼 보이지만, 한데 묶이면 에이전트의 실행 환경을 구성합니다.

- `CLAUDE.md`: 프로젝트의 상시 규칙과 기준선
- `Skills`: 필요할 때 불러오는 재사용 작업 지식
- `Hooks`: 우회할 수 없는 이벤트 기반 검증과 차단
- `MCP`: 외부 도구와 데이터 소스를 연결하는 표준 통로
- `Subagents`: 역할과 컨텍스트를 분리하는 위임 단위
- 승인/샌드박스: 위험한 행동을 멈추게 하는 권한 경계

이런 구조를 보면 하네스가 무엇인지 더 선명해집니다. 하네스는 모델이 출력하는 문장이 아니라, 모델이 어떤 세계를 보고 어떤 행동을 하며 어떤 문을 통과해야 하는지 정하는 운영 환경입니다.

## 도구 경쟁의 축도 바뀐다

AI 코딩 도구를 비교할 때 과거에는 “어느 모델이 코드를 더 잘 쓰는가”가 중심이었습니다. 이제 그 질문만으로는 부족합니다. 더 중요한 비교 항목은 다음으로 이동합니다.

- 코드베이스 컨텍스트를 얼마나 잘 읽는가?
- 규칙 파일과 문서 계층을 어떻게 다루는가?
- 외부 도구를 어떤 표준으로 연결하는가?
- 위험한 명령을 어떻게 승인하거나 차단하는가?
- 테스트와 린트를 자동 루프에 어떻게 넣는가?
- 서브에이전트와 멀티 에이전트 분업을 어떻게 지원하는가?
- 긴 작업의 상태를 어디에 남기는가?
- 실패와 비용을 어떻게 관측하고 복구하는가?

Claude Code, Codex, Cursor, Copilot Coding Agent, OpenCode 계열 도구는 출발점은 달라도 점점 이 질문들로 모입니다. 그래서 “Claude Code가 좋은가”보다 더 중요한 질문은 “내 작업 환경의 하네스를 어느 정도까지 도구가 지원하는가”입니다.

## Vibe Coding의 숙취가 남긴 교훈

이 전환을 이해하려면 `Vibe Coding`도 짚어야 합니다. Vibe Coding은 코드를 깊게 읽지 않고, AI가 제안한 변경을 빠르게 수락하며, 에러 메시지를 다시 붙여 넣어 돌아갈 때까지 밀어붙이는 방식으로 유명해졌습니다. 주말 프로젝트나 프로토타입에서는 강력할 수 있습니다. 문제는 이 방식이 프로덕션으로 넘어갈 때입니다.

AI가 만든 코드가 빠르게 커졌지만, 아무도 그 코드를 제대로 읽지 않았고, 설계 결정이 기록되지 않았고, 테스트와 검증이 약했다면 유지보수는 곧 어려워집니다. 코드를 누가 썼는지가 문제가 아닙니다. 엄밀함이 어디에 있는지가 문제입니다.

LLM이 모든 코드를 작성했더라도, 사람이 리뷰하고 테스트하고 배포 기준을 통과시켰다면 그것은 무책임한 Vibe Coding이 아닙니다. 반대로 사람이 직접 조금 고쳤더라도 검증 없이 밀어 넣으면 위험합니다. 핵심은 생성 주체가 아니라 하네스입니다.

## 결론: Claude Code는 목적지가 아니라 렌즈다

Claude Code를 보는 이유는 그것이 모든 상황의 정답이기 때문이 아닙니다. Claude Code는 최근 코딩 도구가 하네스 중심으로 이동하는 방향을 선명하게 보여 주는 렌즈입니다.

초기 AI 코딩 도구는 프롬프트를 잘 쓰는 문제처럼 보였습니다. 그다음에는 컨텍스트를 잘 넣는 문제처럼 보였습니다. 하지만 에이전트가 실제 프로젝트에서 오래 일하려면 더 넓은 질문이 필요합니다.

> **어떤 도구가 내 개발 문맥, 권한, 검증 루프, 상태 기록을 하나의 하네스로 묶어 주는가?**

앞으로의 도구 경쟁은 이 질문에 더 잘 답하는 쪽으로 움직일 것입니다.
