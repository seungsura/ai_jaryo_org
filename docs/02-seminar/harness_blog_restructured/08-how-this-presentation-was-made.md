# 8. 이 글과 발표가 만들어진 과정: Codex를 어디에 넣었는가

## 시작하며: 장표부터 만들지 않았다

이 글과 발표를 만든 방식도 앞에서 설명한 원리를 따릅니다. 처음부터 슬라이드 도구를 열고 문장을 채운 것이 아닙니다. 먼저 source를 잠그고, 그 source를 prose로 정리하고, prose를 outline으로 압축한 뒤, 마지막에 에이전트가 HTML/CSS를 만들게 했습니다. PDF와 speaker notes는 그 다음이었습니다.

순서를 바꾸면 결과물은 더 빨리 나올 수 있습니다. 하지만 어떤 판단이 어디서 이뤄졌는지, 무엇을 근거로 다음 단계로 넘겼는지는 훨씬 흐려집니다. 이 프로젝트에서 먼저 만든 것은 개별 slide가 아니라, slide와 글이 생겨나는 경로였습니다.

중요한 질문은 “AI가 대신 만들었는가”가 아닙니다.

> **어떤 층을 사람이 붙들고, 어떤 층을 에이전트에게 맡겼는가?**

## 전체 파이프라인: source → prose → outline → html → pdf → notes

이 제작 과정은 다음 순서를 기본으로 삼았습니다.

1. `source`: PDF, HTML, 외부 링크, 로컬 마크다운 자료를 입력층으로 묶는다.
2. `prose`: 발표와 글의 기준 문장을 만든다.
3. `outline`: prose를 slide 단위와 장 구조로 압축한다.
4. `html`: 승인된 outline과 design contract 안에서 HTML/CSS를 생성한다.
5. `pdf`: HTML 산출물을 변환한다.
6. `speaker notes`: 완성된 deck을 기준으로 발표 대본을 만든다.

이 순서의 장점은 각 단계의 책임이 분명하다는 점입니다. source는 근거이고, prose는 해석과 논리이며, outline은 발표 구조이고, html은 표현이며, notes는 말로 풀어내는 층입니다. 층이 섞이면 나중에 무엇이 근거이고 무엇이 편집 판단인지 구분하기 어렵습니다.

## source를 입력층으로 가뒀다

가장 먼저 한 일은 입력층을 좁히는 것이었습니다. 모든 자료가 같은 권위를 갖는 것은 아닙니다. PDF와 HTML 원문, 공개 문헌, 로컬 마크다운 아카이브, 내부 확인 정보는 서로 다른 층위입니다. 이 층위를 섞으면 원고가 빠르게 그럴듯해지지만, 나중에 검증하기 어려워집니다.

그래서 source는 입력층으로만 다뤘습니다. source 자체를 바로 slide나 최종 문장으로 사용하지 않았습니다. 먼저 어떤 자료가 어떤 주장에 쓰이는지 정리하고, 불확실한 부분은 억지로 메우지 않았습니다. 이 단계에서 사람이 붙든 것은 입력의 경계였습니다.

이 원칙은 하네스 설계와 같습니다. 에이전트에게 모든 자료를 던져 주고 “좋은 발표 만들어 줘”라고 하지 않습니다. 먼저 무엇을 source로 인정할지, 무엇을 보조 자료로만 볼지, 무엇을 추가 확인이 필요한 질문으로 남길지 정합니다.

## prose를 canonical source로 세웠다

source를 모았다고 곧바로 장표로 가지 않았습니다. 먼저 prose를 만들었습니다. prose는 발표 bullet이 아니라 논리 순서와 주장 강도를 책임지는 본문입니다.

이 결정이 중요합니다. HTML slide도, speaker notes도, 나중의 PDF도 모두 이 prose를 따라가야 합니다. slide는 prose를 압축할 수 있지만 논지 바깥으로 뛰어넘으면 안 됩니다. notes는 prose를 말로 풀 수 있지만 새로운 사실 주장을 추가하면 안 됩니다.

이 단계에서 사람이 붙든 것은 다음입니다.

- 전체 논지의 흐름
- 장과 절의 순서
- 어떤 주장이 중심이고 어떤 주장이 보조인지
- 한국어 문체와 독자 경험
- 원본 소스의 누락 방지

에이전트에게 맡기기 전에 canonical prose를 세운 이유는, 이후 산출물이 흔들릴 때 돌아올 기준점이 필요했기 때문입니다.

## outline과 manifest가 계획을 들고 간다

prose가 준비되면 곧바로 HTML을 만들지 않습니다. 먼저 outline을 만듭니다. outline은 단순한 목차가 아니라 승인 게이트입니다. 어떤 slide 또는 어떤 절이 어떤 주장을 담을지, 어떤 source section을 참조할지, 어떤 시각 구조가 필요한지 먼저 정합니다.

그 다음 manifest가 상태를 들고 갑니다. manifest는 작업이 끝난 뒤 쓰는 회고가 아니라, 현재 산출물의 상태를 추적하는 live document입니다.

- 어떤 prose가 어떤 slide 또는 장으로 풀렸는가
- 어떤 항목이 완료되었고 어떤 항목이 pending인가
- 어떤 파일 경로에 산출물이 있는가
- 어떤 검증을 통과했는가
- 어떤 변경이 남았는가

이 방식은 에이전트 작업에서도 그대로 중요합니다. 상태를 대화창 안에만 두면 세션이 바뀌는 순간 기준이 사라집니다. 반대로 manifest, 이슈, PR, 진행 파일이 상태를 들고 가면 세션이 끊겨도 작업은 이어집니다.

## design contract를 먼저 고정했다

이 프로젝트에서 에이전트가 맡은 핵심 일은 HTML/CSS generation이었습니다. 하지만 에이전트에게 “예쁘게 만들어 줘”라고 맡기지 않았습니다. 먼저 design contract를 고정했습니다.

디자인 계약에는 다음이 들어갑니다.

- 톤과 분위기
- 색상 팔레트
- 타이포그래피
- 여백과 레이아웃 규칙
- slide family
- placeholder 규칙
- 금지해야 할 표현과 과한 장식

이 계약이 있어야 에이전트의 자유가 작동할 좌표계가 생깁니다. 자유를 최대화한다고 좋은 결과가 나오는 것이 아닙니다. 좋은 결과는 대개 적절한 제약 안에서 나옵니다.

따라서 Codex가 맡은 일은 디자인을 즉흥적으로 발명하는 것이 아니라, 이미 잠긴 문법 안에서 `slide-XX.html`을 생산하는 일이었습니다. 이것이 좋은 위임입니다. 사람은 기준과 좌표계를 잡고, 에이전트는 그 안에서 반복 생산을 맡습니다.

## 생성과 검증을 같은 손에 쥐지 않았다

생성자가 자기 결과를 스스로 통과시키게 두면 그럴듯한 오류가 오래 살아남습니다. 그래서 이 프로젝트는 생성과 검증을 분리했습니다.

HTML generation 이후에는 기계적 검증과 사람 검토가 따릅니다.

- slide contract 점검
- 한국어 문장 점검
- deck-level consistency 확인
- slide-level validation
- source와 주장 매핑 확인
- 디자인 계약 위반 확인

이 원칙은 코드 작업과 같습니다. 에이전트가 코드를 생성했다면 같은 에이전트의 “괜찮습니다”만 믿으면 안 됩니다. 테스트, 린트, 타입 체크, 리뷰어, 다른 모델, 사람 검토가 분리되어야 합니다.

## speaker notes를 맨 뒤로 밀었다

많은 발표 작업은 대본부터 길어지면서 무너집니다. 설명하고 싶은 욕심이 먼저 커지고, 아직 굳지 않은 slide 구조를 말이 덮어버리기 때문입니다.

이 프로젝트에서는 notes를 마지막 단계로 밀었습니다. 먼저 prose가 주장하고, outline이 구조화하고, slide가 압축하고, 마지막에 notes가 말로 풀어냅니다. notes는 canonical prose를 대체하지 않습니다. 새로운 factual claim도 추가하지 않습니다. slide order가 바뀌면 먼저 mapping을 고치고, 그다음 대본을 손봅니다.

이 순서 덕분에 발표 대본은 설계의 주인이 아니라 결과의 해설자가 됩니다. 발표는 말로 끝나지만, 말이 먼저 앞서 나가면 구조는 무너집니다.

## 이 제작 과정이 보여 주는 하네스 원리

이 사례의 핵심은 “AI가 HTML을 만들었다”가 아닙니다. 더 중요한 것은 AI에게 맡길 층과 맡기지 않을 층을 먼저 나눴다는 점입니다.

| 층 | 사람이 붙든 것 | 에이전트에게 맡긴 것 |
| --- | --- | --- |
| Source | 입력 경계, 근거 층위, 불확실한 질문 | 자료 정리 보조 |
| Prose | 논리 순서, 주장 강도, 문체 | 초안 정리와 변형 |
| Outline | 장표 단위 주장, 배치, 누락 확인 | 구조화 보조 |
| Design | 시각 문법과 제약 | 계약 안에서 HTML/CSS 생성 |
| Validation | 통과 기준과 최종 판단 | 기계적 체크와 수정 제안 |
| Notes | 발표 흐름과 말의 순서 | 대본 초안 보조 |

이 표가 바로 하네스의 작은 예입니다. 에이전트의 능력을 제한하려는 것이 아니라, 에이전트가 잘할 수 있는 구간과 사람이 책임져야 할 구간을 나누는 것입니다.

## 글쓰기에도 같은 원칙이 적용된다

코드가 아니어도 원리는 같습니다. 긴 글을 AI와 함께 쓸 때도 다음 순서가 안전합니다.

1. source를 고정한다.
2. 주장 목록을 만든다.
3. 목차를 정한다.
4. 각 장의 역할을 정한다.
5. 초안을 생성한다.
6. 장 간 중복과 누락을 검토한다.
7. 문체를 통일한다.
8. 최종 검증과 출처 정리를 한다.

처음부터 “긴 글 써 줘”라고 하면 빠르지만 구조가 흔들립니다. 반대로 source, prose, outline, validation이 분리되어 있으면 긴 글도 추적 가능합니다.

## 결론: 우리가 만든 것은 slide보다 파이프라인이다

이 발표와 글이 보여 주는 진짜 포인트는 결과물이 아닙니다. 결과물이 만들어지는 경로입니다. source boundary를 잠그고, prose spine을 세우고, outline과 manifest를 만들고, design contract를 먼저 고정하고, 생성과 검증을 분리했습니다.

Codex가 들어간 정확한 위치도 분명합니다. prose와 outline, design contract가 잠긴 뒤 HTML/CSS generation을 맡는 위치입니다. Codex는 모든 층을 대신한 것이 아니라, 이미 잠근 경계 안에서 생산을 담당했습니다.

이것이 하네스의 실전적 의미입니다. 좋은 결과는 한 번의 프롬프트가 아니라, 입력층을 좁히고, 기준 문장을 세우고, 계획을 파일에 남기고, 생성과 검증을 분리한 구조에서 나옵니다.
