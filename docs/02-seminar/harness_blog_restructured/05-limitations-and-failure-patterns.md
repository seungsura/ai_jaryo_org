# 5. 이렇게 하면 망한다: 코딩 에이전트 한계와 실패 패턴

## 시작하며: 실패는 피할 수 없는 핵심 설계 과제다

AI 코딩 에이전트의 한계를 사소한 부작용처럼 다루면 대응은 늘 늦습니다. 문제가 터진 뒤 로그를 뒤지고, 왜 이런 일이 벌어졌는지 뒤늦게 추적하는 방식으로는 오래 버티기 어렵습니다.

짧은 데모에서는 모든 것이 매끈해 보입니다. 한 번의 응답이 그럴듯하면 시스템 전체도 잘 굴러갈 것처럼 보입니다. 하지만 실제 작업은 다릅니다. 대화는 길어지고, 코드 수정은 누적되고, 검증은 여러 단계를 거칩니다. 이 과정에서는 작은 불일치 하나도 다음 단계의 입력으로 다시 들어갑니다.

그래서 질문도 달라져야 합니다. 중요한 것은 모델이 어느 순간 영리한 답을 냈는지가 아닙니다. 우리가 만든 구조 안에서 오류가 어디에 쌓이고, 어느 지점에서 증폭되며, 무엇이 그것을 막지 못하는지를 봐야 합니다.

> **실패는 예외가 아니라 하네스 설계의 입력입니다.**

## 긴 작업 사슬: 한 번의 그럴듯함보다 무서운 누적 비용

긴 작업에서 문제를 만드는 것은 대개 한 번의 대형 오답이 아닙니다. 더 위험한 것은 그럴듯하게 지나간 작은 오류입니다. 에이전트는 읽고, 수정하고, 실행하고, 다시 판단하는 사슬을 오래 이어갑니다. 이 사슬 안에서는 초반의 작은 착오도 다음 판단의 전제가 됩니다.

예를 들어 에이전트가 초반에 요구사항을 조금 잘못 이해했다고 해 봅시다. 그 오해가 계획으로 굳고, 계획이 코드 수정으로 이어지고, 수정된 코드가 다시 다음 검증의 대상이 됩니다. 시간이 지나면 실패의 기원을 추적하기가 어려워집니다. 작업이 길수록 오류는 단순한 실수가 아니라 누적 비용이 됩니다.

이 지점에서 필요한 것은 모델을 더 믿는 일이 아닙니다. 작업을 더 잘 자르는 일입니다. 어디까지를 한 단위로 볼지, 어디서 멈춰 검증할지, 무엇을 다음 단계로 넘기기 전에 고정할지 정해야 합니다. 긴 작업의 품질은 한 번의 응답 품질보다 작업 단위를 어떻게 끊고 넘기느냐에 더 크게 좌우됩니다.

## Long Context의 역설: 많이 담는 것에 집착하면 무너진다

`Long Context`는 강력한 기능입니다. 하지만 컨텍스트 창이 커졌다고 해서 문제도 함께 사라지는 것은 아닙니다. 저장소 전체를 한꺼번에 밀어 넣으면 더 잘할 것이라는 기대는 자주 틀립니다.

문제는 길이 자체보다 구성입니다. 어떤 규칙이 들어가고, 어떤 로그가 섞이고, 어떤 오래된 메모가 남아 있는지가 더 중요합니다. 여기서 여러 실패 모드가 나옵니다.

| 실패 모드 | 의미 | 전형적인 증상 |
| --- | --- | --- |
| `Poisoning` | 잘못된 정보가 컨텍스트를 오염시킨다 | 초반 오류가 계속 사실처럼 재사용된다 |
| `Distraction` | 무관한 정보가 주의를 빼앗는다 | 핵심 파일보다 주변 문서에 반응한다 |
| `Confusion` | 유사하지만 다른 정보가 섞인다 | 오래된 규칙과 최신 규칙을 혼동한다 |
| `Clash` | 지시가 서로 충돌한다 | 한 지시를 따르며 다른 지시를 어긴다 |

좋은 컨텍스트는 넓은 컨텍스트가 아닙니다. 선별된 컨텍스트입니다. 지금 단계와 무관한 파일, 이미 끝난 시도의 흔적, 낡은 지시, 중복 로그를 덜어내야 핵심 신호가 살아납니다.

## Context Rot: 길어진 맥락은 결국 흐려진다

`Context Rot`은 긴 입력과 긴 세션이 품질을 어떻게 무너뜨리는지를 보여 주는 이름입니다. 컨텍스트가 길어질수록 성능이 고르게 유지되지 않는 기술적 문제도 있고, 긴 대화가 이어지면서 초반의 제약과 의도가 서서히 흐려지는 운영 문제도 있습니다. 실전에서는 이 둘이 자주 겹칩니다.

처음에는 분명했던 기준이 세션 뒤쪽으로 갈수록 흐려집니다. 초반에 고정한 제약이 약해지고, 중간에 나온 가정이 사실처럼 눌어붙고, 오래된 지시와 새 지시가 한 창 안에서 뒤엉킵니다. 세션을 오래 끄는 일이 곧 숙련은 아닙니다. 오히려 길게 유지할수록 drift를 관리하는 비용이 커집니다.

그래서 상태는 대화창 안에만 두면 안 됩니다. 현재 결정, 남은 과제, 검증 결과, 다음 단계는 파일과 기록으로 밖에 남겨야 합니다. 요약한 뒤 새 세션으로 넘어가는 판단도 필요합니다. 많이 기억하는 것보다, 기억이 썩지 않게 관리하는 편이 더 중요합니다.

## 현장의 실패 신호: 하네스 공백은 이렇게 드러난다

실무에서 자주 마주치는 실패 패턴은 낯설지 않습니다.

### AI Slop

겉으로는 그럴듯하지만 품질이 낮은 산출물이 쌓이는 상태입니다. 코드 스타일은 맞는 것처럼 보이지만 예외 처리가 허술하고, 테스트는 있지만 핵심 경계를 검증하지 않고, 문서는 길지만 판단 기준은 비어 있습니다.

### Doom Loop

에이전트가 같은 실패를 반복합니다. 테스트가 실패하고, 수정하고, 다시 같은 테스트가 실패하고, 또 다른 방식으로 비슷한 수정을 합니다. 중단 조건이나 다른 전략으로 전환하는 장치가 없으면 루프가 계속됩니다.

### Shadow Agent

에이전트가 지금 무엇을 하고 있는지, 왜 이 파일을 만지는지, 어떤 목표를 향해 가는지 흐려지는 상태입니다. 사용자는 작업이 진행 중이라고 믿지만, 실제로는 목표와 무관한 탐색이나 수정이 이어집니다.

### Silent Failure

실패가 조용히 묻힙니다. 테스트를 실행하지 않았거나, 실패 로그를 무시했거나, 위험한 경고를 지나쳤는데도 에이전트는 “완료”라고 말합니다. 가장 위험한 실패는 크게 터지는 실패가 아니라 통과한 것처럼 보이는 실패입니다.

이런 장면을 보고 “오늘 모델 컨디션이 안 좋다”고 넘기면 얻는 것이 없습니다. 더 중요한 질문은 따로 있습니다.

- 어떤 가드레일이 비어 있었는가?
- 어떤 검증이 빠졌는가?
- 어떤 상태가 기록되지 않았는가?
- 어떤 실행 흔적이 남지 않았는가?
- 어디서 사람이 개입했어야 하는가?

실패 패턴은 대개 하네스의 빈틈을 가리킵니다.

## Calibrated Trust: 신뢰는 감정이 아니라 설정값이다

`Calibrated Trust`는 AI를 믿느냐 마느냐의 감정 문제가 아닙니다. 어떤 작업에는 자율성을 넓게 주고, 어떤 작업에는 더 강한 검증을 붙일지 조절하는 운영 원칙입니다.

예를 들어 다음 작업에는 비교적 넓은 자율성을 줄 수 있습니다.

- 반복적인 리팩터링 초안
- 문서 초안 작성
- 테스트 실패 원인 탐색
- 로그 요약
- 대안 목록 작성

반대로 다음 작업은 더 강한 검증이 필요합니다.

- 보안, 인증, 권한 변경
- 데이터 삭제나 마이그레이션
- 금전 처리와 결제
- 최신 API 동작과 외부 서비스 연동
- 숫자, 날짜, 인용, URL 같은 사실 검증
- 배포와 인프라 변경

핵심은 유창함을 정확성과 혼동하지 않는 일입니다. 그럴듯하게 말하는 능력과 사실을 보증하는 능력은 다릅니다. 에이전트를 잘 쓰는 팀은 이 차이를 감으로 처리하지 않습니다. 어떤 단계에서 무엇을 반드시 다시 확인할지 구조로 고정합니다.

## 보안 실패: Lethal Trifecta

에이전트가 강해질수록 보안은 선택이 아니라 필수입니다. 특히 다음 세 가지가 동시에 열리면 위험이 급격히 커집니다.

1. 신뢰할 수 없는 입력을 처리한다. 예: 외부 웹, 이메일, 사용자 업로드, 임의 문서
2. 민감한 시스템이나 데이터에 접근한다. 예: 개인정보, 내부 DB, API 키, 사내 문서
3. 상태를 변경할 수 있다. 예: 이메일 발송, 파일 삭제, PR 생성, 배포, API 호출

이 셋이 결합되면 프롬프트 인젝션이나 악의적 입력이 민감 정보 유출과 상태 변경으로 이어질 수 있습니다. “모델이 잘 판단하겠지”로 막을 수 있는 문제가 아닙니다. 권한 구조로 막아야 합니다.

실천 원칙은 `Rule of Two`로 정리할 수 있습니다. 위 세 가지 능력 중 동시에 최대 두 가지만 허용하고, 세 가지가 모두 필요한 경우에는 사람 승인이나 별도 샌드박스를 요구합니다.

- 외부 입력 + 민감 정보 접근 → 상태 변경은 막는다.
- 외부 입력 + 상태 변경 → 민감 정보 접근은 막는다.
- 민감 정보 접근 + 상태 변경 → 외부 입력은 차단한다.

이 규칙은 기능을 제한하려는 것이 아니라 신뢰 가능한 자동화를 만들기 위한 최소 조건입니다.

## 실패 복구도 하네스의 일부다

좋은 하네스는 실패를 완전히 없애지 않습니다. 대신 실패를 빨리 드러내고, 비용이 커지기 전에 멈추고, 다음 시도를 더 낫게 만듭니다.

실전에서 필요한 복구 장치는 다음과 같습니다.

- **중단 조건**: 같은 테스트가 일정 횟수 이상 실패하면 루프를 멈춘다.
- **전략 전환**: 같은 방식의 수정이 반복되면 다른 접근을 요구한다.
- **비용 예산**: 토큰, 시간, API 비용, 실행 횟수에 제한을 둔다.
- **체크포인트**: 일정 단계마다 상태를 파일이나 Git에 남긴다.
- **롤백 경로**: 잘못된 변경을 쉽게 되돌릴 수 있게 한다.
- **사람 승인**: 위험 작업, 대규모 변경, 배포 전에는 사람이 개입한다.

실패 복구는 나중에 붙이는 부가 기능이 아닙니다. 에이전트 루프를 설계할 때 처음부터 들어가야 합니다.

## 결론: 자동화를 더 세게 미는 것이 답은 아니다

실패를 줄이는 길은 자동화를 더 세게 미는 데 있지 않습니다. 컨텍스트를 줄이고, 작업을 끊고, 상태를 밖에 남기고, 검증 강도를 구간마다 다르게 주는 구조를 세우는 데 있습니다.

`Long Context`를 쓴다고 판단이 저절로 또렷해지지 않습니다. 하네스가 비어 있으면 긴 세션은 곧 오염된 세션이 됩니다. 반대로 작업 단위를 자르고, 기록을 남기고, 새 세션으로 전환하고, 위험한 경계에 더 강한 검증을 세우면 같은 모델로도 훨씬 오래 버틸 수 있습니다.

문제는 모델의 기분이 아니라 운영의 설계입니다. 그래서 다음 질문도 자연스럽게 이어집니다. 하나의 에이전트에게 모든 것을 맡길 것인가, 아니면 역할과 컨텍스트와 검증을 나눌 것인가. 이제 멀티 에이전트 패턴으로 넘어가 보겠습니다.
