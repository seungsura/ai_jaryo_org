# 1. 코딩은 어디로 가고 있는가

## 시작하며: “코딩이 사라지는가”는 질문이 잘못됐다

AI 코딩 도구가 빠르게 좋아질수록 사람들은 같은 질문을 반복합니다. “이제 코딩은 사라지는가?” 하지만 이 질문은 조금 빗나가 있습니다. 더 정확한 질문은 이것입니다.

> **개발자가 끝까지 직접 붙들어야 하는 일은 어디로 옮겨 가는가?**

손으로 코드를 한 줄씩 입력하던 일의 비중은 줄고 있습니다. 대신 무엇을 만들지 정하고, 어떤 정보를 보여 줄지 고르고, 결과를 어떻게 검증할지 설계하는 일이 앞줄로 올라옵니다. 이 장은 그 이동의 방향을 먼저 짚습니다.

## 추상화의 역사: 직접 하던 일은 계속 줄어들었다

소프트웨어의 역사는 “직접 하던 일을 줄이고, 더 높은 수준의 설계로 올라가는 과정”이었습니다. 지금 일어나는 변화도 그 연장선에 있습니다.

| 시대 | 전환 | 위임한 것 |
| --- | --- | --- |
| 1950s | 기계어 → 어셈블리 | 이진 코드를 직접 작성하는 부담 |
| 1960s | 어셈블리 → 고급 언어 | 레지스터와 명령어 최적화를 컴파일러에 위임 |
| 1990s | C/C++ → Java | 메모리 해제를 가비지 컬렉터에 위임 |
| 2000s | 언어 → 프레임워크 | 반복되는 보일러플레이트를 프레임워크에 위임 |
| 2010s | 온프레미스 → 클라우드 | 서버 운영과 확장을 클라우드에 위임 |
| 2020s | 직접 코딩 → AI 에이전트 | 코드 작성의 상당 부분을 위임하고, 작업 환경을 설계 |

전환이 일어날 때마다 반발도 있었습니다. 고급 언어가 처음 등장했을 때는 “기계어를 직접 다루지 않으면 진짜 개발이 아니다”라는 말이 나왔습니다. 가비지 컬렉터가 등장했을 때는 “메모리를 직접 관리하지 않으면 실력이 아니다”라는 반발이 있었습니다. Python 같은 동적 언어가 커질 때도 “장난감 언어로 무슨 개발을 하냐”는 말이 따라붙었습니다.

지금도 비슷한 반응이 나옵니다. “자연어로 시키는 건 진짜 개발이 아니다.” 하지만 역사는 매번 비슷한 방향으로 움직였습니다. 직접 하는 일은 줄었고, 더 큰 구조를 설계하는 일이 그 자리를 채웠습니다. 이번에도 핵심은 “개발이 사라진다”가 아니라 “개발의 추상화 수준이 올라간다”입니다.

## 타이핑은 더 이상 실력을 설명하지 못한다

AI 코딩 도구가 만든 가장 큰 변화는 타이핑의 희소성을 낮춘 것입니다. 과거에는 코드를 빠르게 작성하고, 문법을 기억하고, 반복 작업을 능숙하게 처리하는 능력이 생산성의 큰 부분을 차지했습니다. 이제 그 구간은 빠르게 자동화되고 있습니다.

중요한 변화는 단순히 “AI가 코드를 많이 쓴다”가 아닙니다. 개발자의 시간이 어디로 이동하는가입니다. 에이전트가 초안을 빠르게 만들수록 사람은 더 자주 다음 질문 앞에 섭니다.

- 이 요구사항은 정확한가?
- 이 구현 방향은 유지보수 가능한가?
- 기존 아키텍처와 충돌하지 않는가?
- 테스트로 검증할 수 있는가?
- 위험한 권한을 열어 둔 것은 아닌가?
- 이 판단을 다음 세션에서도 재사용할 수 있게 남겼는가?

타이핑이 줄어든다고 해서 판단이 줄어들지는 않습니다. 오히려 초안의 속도가 빨라질수록 잘못된 판단도 빠르게 증폭됩니다. 그래서 개발자의 숙련은 손끝에서 사라지는 것이 아니라, 의도를 명세하고 결과를 검증하는 곳으로 이동합니다.

## 문서가 결과를 바꾸는 세계

AI와 함께 일하는 환경에서 문서의 역할도 바뀌었습니다. 과거의 문서는 대개 사람이 읽는 참고 자료였습니다. README, 컨벤션 문서, 아키텍처 노트는 중요했지만, 실제 실행 경로와는 떨어져 있는 경우가 많았습니다.

AI 코딩 도구에서는 이야기가 달라집니다. `CLAUDE.md`, `AGENTS.md`, `Cursor Rules`, `copilot-instructions.md` 같은 파일은 더 이상 장식용 문서가 아닙니다. 에이전트가 매 세션 읽고 따라야 할 운영 규칙입니다. 코딩 스타일, 금지 명령, 리뷰 기준, 테스트 규칙, 문서 어투, 승인 조건 같은 내용이 이 파일에 들어가면 실제 산출물의 방향이 바뀝니다.

이 변화의 의미는 큽니다. 과거에는 “문서화”가 사후 정리처럼 취급되기 쉬웠습니다. 이제는 문서가 에이전트의 행동을 바꾸는 입력입니다. 문서가 곧 실행 환경의 일부가 됩니다.

따라서 좋은 개발자는 이제 문서를 “예쁘게 쓰는 사람”이 아니라, 에이전트가 따라야 할 기준을 기계가 사용할 수 있는 형태로 정리하는 사람입니다. 반복해서 설명하는 규칙이 있다면 그것은 채팅창이 아니라 파일로 내려가야 합니다. 매번 사람이 막아야 하는 실수가 있다면 그것은 구두 지시가 아니라 테스트, 린트, 훅, 승인 게이트로 올라가야 합니다.

## 개발자의 새로운 역할: 코더에서 환경 설계자로

이 변화는 개발자의 역할을 없애는 것이 아니라 재배치합니다. 과거의 개발자는 직접 코드를 작성하고 버그를 수정하는 사람에 가까웠습니다. 앞으로의 개발자는 에이전트가 코드를 잘 쓸 수 있도록 환경을 설계하고, 결과의 맛을 판단하는 사람에 가까워집니다.

건축가의 비유가 도움이 됩니다. 건축가는 직접 벽돌을 모두 쌓지 않습니다. 하지만 설계도, 구조, 재료, 안전 기준, 시공 순서를 정의합니다. 오케스트라 지휘자도 모든 악기를 직접 연주하지 않습니다. 대신 어느 악기가 언제 들어오고, 어디서 멈추고, 전체가 어떤 소리를 내야 하는지 조율합니다.

AI 시대의 개발자도 비슷한 방향으로 이동합니다. 직접 구현하는 시간은 줄어들 수 있지만, 다음 능력의 중요성은 더 커집니다.

- 문제를 올바른 단위로 자르는 능력
- 에이전트에게 보여 줄 컨텍스트를 고르는 능력
- 요구사항과 수용 기준을 명확히 쓰는 능력
- 테스트와 검증 게이트를 설계하는 능력
- 위험한 권한과 안전한 권한을 나누는 능력
- 에이전트의 결과를 의심하고 판단하는 능력
- 반복되는 판단 기준을 파일과 규칙으로 외재화하는 능력

이것이 이 글에서 말하는 `Harness Engineer`의 출발점입니다.

## 그래도 기본기는 더 무거워진다

AI가 코드를 더 많이 써 줄수록 기본기가 덜 중요해질 것처럼 보입니다. 실제로는 반대입니다. AI는 빠르게 초안을 만들 수 있지만, 그 초안이 안전한지, 유지보수 가능한지, 기존 시스템과 잘 맞는지 판단하려면 여전히 깊은 기본기가 필요합니다.

간단한 데모 앱은 매우 쉽게 만들 수 있습니다. 하지만 실제 서비스는 곧 복잡한 문제를 드러냅니다. 인증과 권한, 동시성, 데이터베이스 스키마, 캐싱, 관측 가능성, 장애 복구, 보안, 비용, 배포 파이프라인 같은 문제가 뒤따릅니다. 이런 구간에서 “AI가 그럴듯하게 말한다”는 사실은 충분한 보증이 되지 않습니다.

오히려 AI가 코드를 많이 만들어 줄수록, 어느 부분을 의심해야 하는지 아는 능력이 더 중요해집니다. 모델이 제안한 아키텍처가 왜 위험한지, 테스트가 무엇을 놓쳤는지, API 설계가 나중에 어떤 병목을 만들지, 보안 경계가 어디서 무너지는지를 읽을 수 있어야 합니다.

줄어든 것은 타이핑의 비중이지, 시스템을 이해하는 능력의 가치가 아닙니다. 개발자의 기본기는 사라지는 것이 아니라 더 높은 레이어에서 요구됩니다.

## “코드를 잘 짜는 능력”의 의미가 바뀐다

앞으로도 코드를 읽고 쓸 줄 아는 능력은 필요합니다. 다만 “코드를 잘 짠다”는 말의 의미가 조금 달라집니다. 과거에는 직접 구현하는 능력이 중심이었다면, 이제는 다음 능력까지 포함해야 합니다.

첫째, 에이전트가 이해할 수 있는 작업 단위를 정의하는 능력입니다. “이 기능을 만들어 줘”보다 “이 수용 기준을 만족하도록 이 범위 안에서 수정하고, 다음 테스트를 통과시켜라”가 훨씬 강합니다.

둘째, 결과를 자동으로 걸러낼 수 있는 기준을 만드는 능력입니다. 사람이 매번 눈으로 확인해야 하는 기준은 쉽게 누락됩니다. 기계가 확인할 수 있는 것은 기계에게 맡겨야 합니다.

셋째, 지식을 세션 밖에 남기는 능력입니다. 채팅창에서 한 번 성공한 지시가 팀의 자산이 되려면 파일, 명령어, 템플릿, 훅, 테스트, 이슈 기록으로 바뀌어야 합니다.

넷째, 삭제 가능한 하네스를 만드는 능력입니다. 오늘 필요한 제약이 내일은 불필요한 마찰이 될 수 있습니다. 좋은 환경 설계자는 규칙을 만드는 사람인 동시에, 더 이상 필요 없는 규칙을 제거할 수 있는 사람입니다.

## 결론: 숙련은 손끝에서 시스템 설계로 이동한다

코딩이라는 행위가 끝난다는 말은 틀렸습니다. 하지만 손으로 한 줄씩 치는 비중은 줄고 있습니다. 대신 규칙을 세우고, 컨텍스트를 고르고, 결과를 검증하고, 상태를 남기는 책임이 훨씬 무거워지고 있습니다.

그래서 질문을 바꿔야 합니다. “AI가 코딩을 대체하는가?”가 아니라 “개발자의 숙련은 어디에서 드러나는가?”입니다. 지금 보이는 대답은 분명합니다.

> **코딩의 중심은 손가락 끝에서, 에이전트가 일할 환경을 설계하고 검증하는 영역으로 이동하고 있습니다.**
