# 7. 실전 워크플로우와 도구 세팅

## 시작하며: 두 가지 막다른 길

실전에서 먼저 피해야 할 막다른 길이 두 가지 있습니다.

첫째는 모든 것을 직접 세팅하느라 구조보다 손기술에 매달리는 길입니다. 명령어를 매번 손으로 조합하고, 검증 루프를 기억에 의존하고, 세션마다 비슷한 지시를 다시 입력합니다. 초반에는 통제하는 느낌이 들지만 오래 유지되지 않습니다.

둘째는 원리도 모른 채 남이 만든 `Skills`, 플러그인, 프리셋만 주워 붙이는 길입니다. 겉보기에는 편하지만 이상 동작이 나오면 원인을 읽지 못합니다. 시간이 지나면 팀에는 방식이 아니라 블랙박스만 남습니다.

그래서 실전 워크플로우의 출발점은 새 도구를 더 붙이는 일이 아닙니다. 어떤 원리를 시스템이 대신 강제하게 만들 것인가입니다. 좋은 워크플로우는 손재주보다 운영 구조를 먼저 세웁니다.

## Preset pack을 볼 때도 핵심은 명령어와 게이트다

`OMC(Oh My Claude Code)` 같은 preset pack을 예로 들 수 있습니다. 여기서 중요한 것은 특정 도구 이름이 아닙니다. 사람이 매번 원리를 외워 재현하는 대신, 명령어와 라우팅이 그 원리를 먼저 강제하게 만든다는 점입니다.

예를 들어 이런 요소가 한 묶음으로 등장할 수 있습니다.

- 역할 분리된 여러 에이전트
- 모델 라우팅
- `Plan-Critic-Build` 명령
- QA와 검증 명령
- 병렬 worker
- `Git Worktree` 기반 격리
- 이슈와 PR 연동

핵심은 사용자가 작업의 성격만 말하면 시스템이 먼저 모드, 역할, 검증 순서를 잡아 준다는 것입니다. 실전에서 품질을 무너뜨리는 것은 대개 모델의 지능 부족이 아니라 운영 일관성의 붕괴입니다. 어떤 날은 계획을 보고, 어떤 날은 바로 실행하고, 검증은 가끔만 돌리는 식이면 세션마다 기준이 흔들립니다. 반대로 원리가 명령어로 내려가면 좋은 습관은 기억이 아니라 기본값이 됩니다.

## 구현 전에 계획을 분리한다: Plan-Critic-Build

실전에서 가장 먼저 고정할 문법은 `Plan-Critic-Build`입니다.

1. `Plan`: 구현 전에 계획을 쓴다.
2. `Critic`: 계획의 빈틈을 비판한다.
3. `Build`: 합의된 뒤에만 실행한다.

이 순서는 간단하지만 효과가 큽니다. 에이전트가 가장 많이 실패하는 지점은 “나쁜 코드”가 아니라 “나쁜 전제”입니다. 전제가 잘못된 상태에서 실행을 시작하면 파일 수정, 테스트 실행, 로그 해석이 모두 그 전제를 따라 오염됩니다.

실무에서는 `Explore → Plan → Code → Commit` 흐름으로 풀어 쓸 수 있습니다.

- `Explore`: 프로젝트 구조와 관련 파일을 읽는다.
- `Plan`: 변경 계획과 위험을 쓴다.
- `Code`: 승인된 계획만 구현한다.
- `Commit`: 검증 결과와 함께 작은 단위로 커밋한다.

여기서 빠지면 안 되는 것이 `success criteria`입니다. 무엇을 끝으로 볼 것인지, 어떤 테스트와 어떤 검증을 통과해야 하는지 먼저 못 박아야 합니다. 그래야 실행 단계가 속도가 아니라 기준을 향해 움직입니다.

## 좋은 워크플로우는 도구보다 게이트를 설계한다

좋은 운영은 도구를 늘리는 기술보다 도구를 덜어내는 기술에 가깝습니다. 컨텍스트 창에 유휴 `MCP`와 쓰지 않는 인터페이스를 쌓아 두면 판단력이 넓어지는 것이 아니라 흐려집니다. 필요한 도구만 남기는 `tool curation`은 취향이 아니라 성능 문제입니다.

Stripe 사례처럼 내부 도구가 수백 개 있어도 실제 태스크에서 오케스트레이터가 고르는 도구는 소수일 수 있습니다. Vercel 계열 실험에서도 도구를 줄였을 때 정확도와 속도가 함께 개선되는 사례가 언급됩니다. 중요한 것은 “많이 연결했다”가 아니라 “필요한 도구만 정확히 열었다”입니다.

도구 큐레이션과 함께 더 중요한 것이 게이트입니다.

- `lint`
- `type check`
- 테스트
- 리뷰
- 승인
- 샌드박스
- `PreToolUse` / `PostToolUse` 훅
- 위험 명령 차단
- 배포 전 human-in-the-loop

기계가 확인할 수 있는 것은 기계가 닫아야 합니다. AI는 그 문을 통과할 답을 내놓는 데 집중해야 합니다.

## Approval, Auto-accept, Plan Mode를 위험도에 맞춰 조절한다

모든 작업에 같은 자율성을 주면 안 됩니다. 위험도에 따라 모드를 나눠야 합니다.

| 작업 유형 | 권장 모드 | 이유 |
| --- | --- | --- |
| 문서 초안, 로그 요약 | 비교적 넓은 자율성 | 실패 비용이 낮고 되돌리기 쉬움 |
| 테스트 보강, 작은 리팩터링 | 제한적 auto-accept + 검증 | 반복 작업이지만 검증 필요 |
| 아키텍처 변경, DB 마이그레이션 | Plan Mode + 승인 | 잘못되면 비용이 큼 |
| 삭제, 배포, 권한 변경 | 수동 승인 필수 | 상태 변경과 사고 위험이 큼 |

좋은 워크플로우는 에이전트를 믿거나 믿지 않는 이분법이 아닙니다. 작업별로 신뢰 수준을 조절하는 구조입니다.

## 컨텍스트를 줄이고 반복을 자산화한다

긴 세션이 망가지는 이유는 정보가 부족해서가 아니라, 오래 쌓인 기억이 잡음으로 바뀌기 때문입니다. 그래서 `/context`, `/compact`, `/clear` 같은 명령은 편의 기능이 아니라 운영 명령입니다.

- `/context`: 지금 무엇이 얼마나 쌓였는지 본다.
- `/compact`: 이어갈 결정과 상태만 압축한다.
- `/clear`: 다른 작업으로 넘어갈 때 세션을 비운다.

세션을 비우지 못하면 작업은 계속 이어지는 것이 아니라 계속 끌려갑니다.

반복되는 지침도 대화 안에 묻어 두면 안 됩니다. 매번 같은 설명을 타이핑하고 있다면 이미 시스템 바깥에 있어야 할 지식이라는 뜻입니다.

- 반복 지시는 `Skills`로 옮긴다.
- 우회하면 안 되는 규칙은 `Hooks`로 고정한다.
- 자주 쓰는 루프는 `Custom Commands`로 묶는다.
- 프로젝트 공통 기준은 `CLAUDE.md`나 `AGENTS.md`에 둔다.
- 특정 태스크의 상태는 이슈나 진행 파일에 남긴다.

개인 메모에 머물면 요령으로 끝나지만, 리포지토리와 팀 설정으로 올라가면 표준이 됩니다.

## 한 모델에만 기대지 않는다: Claude, Codex, Gemini

역할을 분해했다면 모델의 시야도 분산시킬 수 있습니다. 단일 모델에 모든 판단을 몰아주지 않고 Claude, Codex, Gemini 같은 모델을 교차 투입하는 방식입니다. 이유는 단순합니다. 모델마다 훈련 분포와 blind spot이 다릅니다.

실전에서는 다음처럼 사용할 수 있습니다.

- Claude: 긴 맥락과 구현 루프 중심
- Codex: 코드 구조, 리팩터링, 터미널 중심 검토
- Gemini: UI, 가독성, 대안적 시각, 문서 피드백

중요한 것은 모델 경쟁이 아닙니다. 여러 모델의 응답을 `합의`, `불일치`, `불확실`로 나누어 판단의 해상도를 높이는 것입니다. 불일치가 생기면 오히려 좋은 신호일 수 있습니다. 그 지점이 사람이 봐야 할 위험 구간이기 때문입니다.

## 메인 컨텍스트를 지키는 물리적 운영: cmux와 Git Worktree

논리적 경계만으로는 부족합니다. 실전에서는 물리적 작업 환경도 함께 나눠야 합니다.

하나의 창에 탐색, 구현, 로그, 이슈 관리, PR 생성까지 모두 밀어 넣으면 메인 컨텍스트는 금방 흐려집니다. 반대로 창과 작업 디렉터리를 나누면 탐색은 옆에서 소모되고, 결과만 메인으로 돌아옵니다.

`cmux`나 터미널 멀티플렉서 계열 도구는 여러 세션을 나란히 운영하는 데 도움이 됩니다. `Git Worktree`는 이 분리를 파일 시스템 수준에서 고정합니다. 창마다 독립 작업 디렉터리를 두면 충돌 없이 여러 세션을 병렬로 굴릴 수 있고, 메인 흐름도 오염되지 않습니다.

예를 들어 다음처럼 나눌 수 있습니다.

- 창 1: 메인 구현 세션
- 창 2: 조사와 탐색 전용
- 창 3: 테스트와 감시 로그
- 창 4: 이슈와 PR 정리

핵심은 병렬성 자체보다 경계입니다. 탐색은 탐색 자리에서 끝내고, 메인 세션은 결정과 통합과 검증에 집중하게 만드는 것입니다.

## 세션이 아니라 이슈가 상태를 들고 간다

이 구조의 마지막 고정점은 이슈 기반 워크플로우입니다. 여기서 중요한 것은 특정 트래커 이름이 아니라, 이슈가 세션보다 오래가는 상태 저장소라는 사실입니다.

채팅은 빨리 달아오르지만 빨리 휘발합니다. 반대로 이슈는 작업의 목적, 기준, 증거를 붙잡아 둡니다. 그래서 `이슈 1개 = 에이전트 세션 1개`라는 규칙이 유용합니다. 이슈가 작업 지시서가 되고, Worktree 생성과 세션 시작, PR 생성과 검증 완료가 하나의 선으로 이어집니다.

흐름은 이렇게 만들 수 있습니다.

1. Linear/PMS/GitHub Issue에 작업 목적과 수용 기준을 적는다.
2. 이슈에서 Worktree를 생성한다.
3. 에이전트 세션은 해당 이슈 범위만 다룬다.
4. 변경은 작은 커밋으로 남긴다.
5. 검증 결과를 이슈와 PR에 붙인다.
6. 완료 판단은 수용 기준과 검증 증거로 한다.

이슈 생성, 증거 첨부, 검증, 완료 판정 같은 체인은 에이전트를 더 똑똑하게 만드는 장치가 아닙니다. 완료 기준과 검증 흔적을 바깥에 남겨 컨텍스트 드리프트를 막는 장치입니다.

## 첫 주에 바로 세울 수 있는 최소 워크플로우

거창하게 시작할 필요는 없습니다. 다음 정도만 해도 하네스가 생기기 시작합니다.

1. `CLAUDE.md` 또는 `AGENTS.md`에 반복 지시 세 가지를 적는다.
2. 위험 명령 하나를 훅이나 승인 규칙으로 막는다.
3. `lint` 또는 `type check` 하나를 필수 게이트로 둔다.
4. 긴 작업은 이슈 하나와 Worktree 하나로 묶는다.
5. 구현 전에 계획을 쓰고, 승인 후 수정한다.
6. 세션 종료 시 결정과 남은 일을 파일이나 이슈에 남긴다.

이 여섯 가지는 도구 과시가 아닙니다. 대화창에 떠다니는 기준을 작업 환경의 일부로 만드는 첫 단계입니다.

## 결론: 실전 워크플로우의 중심은 운영 구조다

실전 워크플로우의 중심은 더 많은 도구를 붙이는 데 있지 않습니다. 원리를 명령어로 고정하고, 계획과 실행을 분리하고, 결정론적 게이트를 먼저 세우고, 여러 모델의 시야를 교차시키고, `cmux`와 `Git Worktree`로 세션을 격리하고, 외부 아티팩트에 상태와 기준을 남기는 데 있습니다.

에이전트가 강해질수록 더 중요한 것은 생성 능력 자체가 아닙니다. 무엇을 어디서 검증하고, 어떤 흔적 위에 다음 판단을 올릴 것인가를 설계하는 운영 구조입니다.
