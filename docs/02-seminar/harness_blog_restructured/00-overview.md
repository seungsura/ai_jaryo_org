# Harness 잘 사용하기

> 챗봇과 싸우지 않고, 에이전트가 일할 환경을 설계하는 방법

## 시작하며: 이제는 프롬프트가 아니라 작업 환경의 문제다

AI 코딩 도구는 빠르게 좋아졌습니다. 처음에는 자동완성이었고, 다음에는 채팅이었고, 이제는 파일을 읽고, 셸을 실행하고, 테스트를 돌리고, 실패하면 다시 고치는 에이전트가 되었습니다. 이 변화가 커질수록 개발자가 붙들어야 하는 일도 달라집니다. 코드를 한 줄씩 입력하는 시간은 줄고, 에이전트가 어떤 규칙과 권한과 검증 루프 안에서 일해야 하는지를 설계하는 시간이 늘어납니다.

이 글의 제목을 **“Harness 잘 사용하기”**로 잡은 이유도 여기에 있습니다. 우리가 더 이상 챗봇에게 더 좋은 문장을 던지는 일만으로는 충분하지 않기 때문입니다. 좋은 결과는 한 번의 멋진 프롬프트에서 나오지 않습니다. 반복 가능한 규칙, 선별된 컨텍스트, 위험을 막는 권한 경계, 결정론적 검증, 세션 밖에 남는 상태 기록이 함께 작동할 때 나옵니다. 그 전체 구조를 여기서는 `Harness`라고 부릅니다.

한 문장으로 말하면 이렇습니다.

> **이제 개발자의 핵심 역량은 “AI에게 무엇을 말할까”에서 “AI가 안전하게 일할 환경을 어떻게 만들까”로 이동하고 있습니다.**

## 이 글의 핵심 주장

첫째, 코딩은 사라지는 것이 아니라 추상화되고 있습니다. 소프트웨어의 역사는 직접 하던 일을 기계와 도구에 위임해 온 역사였습니다. 컴파일러는 기계어를 위임받았고, 가비지 컬렉터는 메모리 해제를 위임받았고, 클라우드는 서버 운영을 위임받았습니다. AI 에이전트 시대에는 코드 작성의 상당 부분이 위임되고, 대신 개발자는 에이전트가 일할 환경을 설계하는 쪽으로 이동합니다.

둘째, `Prompt → Context → Harness`는 서로를 대체하는 유행어가 아닙니다. 프롬프트는 “어떻게 말할 것인가”의 문제이고, 컨텍스트는 “무엇을 보여줄 것인가”의 문제이며, 하네스는 “무엇을 통제하고 방지할 것인가”의 문제입니다. 하네스는 프롬프트와 컨텍스트를 버리는 개념이 아니라, 둘을 더 큰 운영 구조 안에 제자리에 놓는 개념입니다.

셋째, AI 시대에는 오래된 개발 방법론이 다시 중요해집니다. `TDD`, `Spec-first`, `SDD`, `Plan-Critic-Build` 같은 원칙은 낡은 절차가 아니라 에이전트의 속도를 품질로 바꾸는 통제 장치입니다. 생성 속도가 빨라질수록 잘못된 방향도 더 빨리 증폭되기 때문에, 무엇을 먼저 고정하고 어디서 검증할지를 더 분명히 해야 합니다.

넷째, 실패는 예외가 아니라 설계 입력입니다. `AI Slop`, `Doom Loop`, `Context Rot`, `Shadow Agent`, 보안 사고 같은 문제는 모델이 가끔 이상해서 생기는 일이 아닙니다. 대개 컨텍스트, 권한, 검증, 상태 관리, 관측 가능성 중 어딘가가 비어 있을 때 반복됩니다. 좋은 팀은 실패를 에이전트 탓으로 끝내지 않고, 같은 실패가 다시 발생하지 않도록 하네스를 고칩니다.

다섯째, 앞으로 더 분명해질 역할은 `Harness Engineer`입니다. 이 역할은 코드를 가장 빨리 치는 사람이 아니라, 에이전트가 일할 기준과 환경을 설계하는 사람입니다. 규칙을 문서화하고, 반복 절차를 명령어로 만들고, 위험한 경계를 막고, 검증 루프를 자동화하고, 세션 밖에 상태를 남기는 사람입니다.

## 이 글의 구성

이 원고는 하나의 긴 기술 블로그 시리즈처럼 읽히도록 재구성했습니다. 원본의 10개 마크다운 파일 흐름은 유지하되, 중복되는 주제는 합치고 약한 섹션은 보강했습니다.

| 장 | 제목 | 다루는 질문 |
| --- | --- | --- |
| 1 | 코딩은 어디로 가고 있는가 | 개발자의 숙련은 어디로 이동하는가 |
| 2 | 왜 Claude Code인가, 그리고 도구는 왜 Harness로 수렴하는가 | 최근 AI 코딩 도구는 왜 비슷한 구조로 모이는가 |
| 3 | AI 시대의 개발 방법론 | `TDD`, `SDD`, `Spec-first`는 왜 다시 중요해졌는가 |
| 4 | 프롬프트를 넘어서: 에이전트를 움직이는 기술, Harness | 하네스는 무엇이고, 컨텍스트 엔지니어링과 어떻게 연결되는가 |
| 5 | 이렇게 하면 망한다: 한계와 실패 패턴 | 긴 작업, 긴 컨텍스트, 보안, 신뢰 문제는 어디서 터지는가 |
| 6 | 멀티 에이전트 활용 패턴 | 역할과 컨텍스트를 어떻게 쪼개야 하는가 |
| 7 | 실전 워크플로우와 도구 세팅 | 원칙을 명령어, 게이트, 워크트리, 이슈 시스템으로 어떻게 고정하는가 |
| 8 | 이 글과 발표가 만들어진 과정 | 하네스 원리를 실제 제작 파이프라인에 어떻게 적용했는가 |
| 9 | 우리가 다음에 해야 할 일 | 개인과 팀은 무엇부터 바꿔야 하는가 |
| 부록 | 출처와 후속 읽기 가이드 | 근거층과 source-to-section map |

## 읽는 순서

처음 읽는다면 1장부터 9장까지 순서대로 읽는 편이 가장 좋습니다. 1~2장은 왜 이 변화가 생겼는지를 설명하고, 3~4장은 그 변화에 대응하는 개발 방법론과 하네스의 구조를 설명합니다. 5장은 실패를 통해 하네스가 왜 필요한지 보여 주고, 6~7장은 실제 운영 패턴으로 내려갑니다. 8장은 이 원고와 발표를 만든 과정을 사례로 보여 주고, 9장은 바로 실행할 수 있는 다음 행동을 정리합니다.

이미 Claude Code나 Codex 같은 코딩 에이전트를 쓰고 있다면 4장, 5장, 7장부터 읽어도 됩니다. 당장 도움이 되는 내용은 `CLAUDE.md`/`AGENTS.md` 같은 규칙 파일, `Skills`, `Hooks`, `MCP`, `Git Worktree`, 이슈 기반 워크플로우, `Plan-Critic-Build` 같은 운영 문법에 모여 있습니다.

## 이 글에서 말하는 Harness

여기서 말하는 `Harness`는 특정 제품 기능 하나가 아닙니다. `CLAUDE.md`도 하네스의 일부일 수 있고, `Hooks`도 일부일 수 있고, `lint`, `type check`, 테스트, 리뷰 게이트, 샌드박스, 승인 정책, 진행 기록, 이슈 템플릿, `Git Worktree`도 모두 하네스의 일부가 될 수 있습니다. 중요한 것은 도구 이름이 아니라 역할입니다.

하네스는 에이전트가 다음 네 가지를 반복하도록 만드는 운영 구조입니다.

1. 필요한 컨텍스트를 읽는다.
2. 제한된 권한 안에서 행동한다.
3. 결정론적 검증을 통과한다.
4. 상태와 증거를 세션 밖에 남긴다.

이 네 가지가 없으면 에이전트는 데모에서는 빛나지만 실제 프로젝트에서는 쉽게 길을 잃습니다. 반대로 이 네 가지가 갖춰지면 같은 모델을 쓰더라도 훨씬 안정적으로 오래 일할 수 있습니다.

## 결론: 챗봇과 싸우지 말고, 환경을 설계하자

AI 코딩 도구를 잘 쓰는 일은 더 이상 “프롬프트를 잘 쓰는 요령”으로 충분히 설명되지 않습니다. 물론 프롬프트는 여전히 중요합니다. 컨텍스트도 중요합니다. 하지만 긴 작업과 실제 코드베이스에서는 프롬프트와 컨텍스트가 작동할 구조가 더 중요합니다.

그래서 이 글의 방향은 분명합니다. 챗봇과 말싸움하듯 매번 지시를 고치기보다, 에이전트가 같은 실수를 반복하지 못하게 환경을 바꿔야 합니다. 규칙을 파일로 빼고, 검증을 자동화하고, 상태를 외부 아티팩트로 남기고, 위험한 권한을 닫고, 사람의 판단이 필요한 지점을 명시해야 합니다.

그것이 이 글에서 말하는 **Harness 잘 사용하기**입니다.
