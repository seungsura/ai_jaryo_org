# Korean Writing Rules

이 문서는 프로젝트에서 글을 작성하거나 고칠 때 적용하는 한국어 규칙이다. 적용 범위는 `docs/` 아래의 문서, `docs/prose/`, 사용자에게 돌려주는 한국어 질문과 검토 의견이다.

슬라이드에 직접 보이는 문구는 이 문서가 아니라 `docs/03-html/shared/korean-slide-copy-rules.md`를 따른다. prose가 구어체에 가까워져도 slide copy는 발표 문장 전체를 옮기지 않는다.

## 기본 원칙

- 새 문서와 수정 문서는 한국어를 기본으로 쓴다.
- 기술적 정확도가 떨어지는 번역은 하지 않는다. term, technical vocabulary, product name, service name, library/framework name, API name, command, file path, protocol, method name은 English를 유지한다.
- English term을 유지하더라도 문장 전체는 한국어 어순과 호흡으로 쓴다.
- 필요한 경우 첫 언급에서만 짧은 한국어 설명을 덧붙인다. canonical English term 자체를 억지로 바꾸지 않는다.
- 주장, source boundary, 기술적 정밀도를 먼저 지킨 뒤 문체를 다듬는다.
- source에 없는 사실, 비교 축, 예시, 해설은 만들지 않는다. 부족한 내용은 open question으로 남긴다.

## 목표 문체

- 실제 발표자가 말할 수 있는 한국어를 기준으로 쓴다.
- 다만 수다스럽게 풀지 않고, 밀도 있는 기술 블로그 에세이처럼 읽히게 쓴다.
- 보고서, 안내문, 제품 문서, 강의안, bland summary 말투로 미끄러지지 않는다.
- 문장은 직접적이고 단정적으로 쓴다.
- 문단은 주장과 대비가 앞으로 밀고 가는 리듬을 가져야 한다.
- 강한 문장은 source로 뒷받침될 때 허용한다.
- 밈체, 허세체, 과장된 인터넷체, 장식적인 문예체는 쓰지 않는다.

## 문장 규칙

- 중요한 행위자는 동사 가까이에 둔다.
- 살아 있는 동사를 우선한다. 명사구만 쌓아서 문장을 만들지 않는다.
- 한 문장에는 주장을 하나만 싣는 것을 기본으로 한다. 대비가 필요할 때만 길게 묶는다.
- 수식어는 꾸미는 말 가까이에 둔다.
- `A의 B의 C`처럼 `의`가 겹치는 명사 사슬은 절로 풀거나 동사를 세운다.
- bullet을 줄글로 옮긴 듯한 문장은 다시 쓴다.
- 설명을 시작하기 전 목을 푸는 표현은 지운다.
- 반복이 논지를 붙들어 줄 때는 핵심 term을 반복해도 된다. 괜히 대명사로 바꿔 흐리지 않는다.

## 구어체 기준

- 소리 내어 읽었을 때 자연스러워야 한다.
- 구어체는 느슨한 잡담이 아니다. `그러니까`, `뭔가`, `사실은`이 문장을 풀어지게 만들면 지운다.
- 긴 설명 뒤에는 짧고 단정적인 문장으로 압력을 만든다.
- 문단 길이와 리듬을 모두 같게 만들지 않는다.
- 너무 일찍 요약하지 않는다. 문제, 결과, 답으로 독자를 밀고 간다.
- 발표 흐름은 `이 챕터에서`, `앞 챕터`, `다음 챕터`를 기본 표현으로 쓴다.

## English Term 규칙

- `Harness`, `Prompt`, `Context`, `Spec`, `Agent`, `RAG`, `MCP`, `TDD`, `SDD`처럼 정확한 English 표기가 중요한 term은 유지한다.
- product name, organization name, library/framework/API/command/path/protocol은 English를 유지한다.
- English term 주변의 조사는 자연스럽게 붙인다.
- 모든 English term을 순우리말로 바꾸려 하지 않는다. 이 프로젝트의 자연스러운 한국어는 정확한 English term을 한국어 문장 안에 제대로 놓는 것이다.
- English 개념을 그대로 직역한 추상명사만 늘어놓지 않는다. 누가 무엇을 어떻게 바꾸는지 풀어 쓴다.

## 금지 표현

다음 표현은 기본적으로 쓰지 않는다. source quote나 반드시 필요한 기술 맥락이 있으면 예외로 둘 수 있지만, 그 경우에도 문장 전체가 자연스러운지 다시 본다.

| 금지/주의 표현 | 바꾸는 방향 |
| --- | --- |
| `장표` | `슬라이드`, `발표 자료` |
| `이 장에서`, `앞 장`, `다음 장` | `이 챕터에서`, `앞 챕터`, `다음 챕터` |
| `상류`, `하류` | `앞단`, `뒤쪽`, `먼저`, `나중에` |
| `강하게 호출된다` | `자주 필요해진다`, `여기서 중요해진다`, `여기서 쓰인다` |
| `핵심은 ~ 데 있다` | `중요한 것은 ~이다`, `결국 ~해야 한다` |
| `~의 측면에서` | `~으로 보면`, 또는 해당 frame을 지우고 주장 직접 서술 |
| `~라고 볼 수 있다` | 근거가 있으면 직접 말한다 |
| `~에 의해` | 행위자를 세운다 |
| `되어진`, `되어질`, `되어져` | 피동을 줄이고 자연스러운 동사로 바꾼다 |
| `수행된다`, `수행되었다`, `수행될` | `한다`, `돈다`, `붙인다`, `실행한다` 등 실제 동사 |
| `~을 통해서` 남발 | 실제 동사를 쓴다. 예: `검증으로 막는다`, `파일에 남긴다` |
| `기반으로 하여` | `바탕으로`, 또는 실제 동사로 축약 |
| `칼끝 후보`, `논지의 스파인`, `게이트를 통과한다` 같은 작업 은유 | 의미를 한국어로 풀어 쓴다 |

## 번역투 점검

- 영어식 passive order가 남아 있지 않은지 본다.
  - 나쁨: `AI에 의해 생성된 코드`
  - 좋음: `AI가 만든 코드`
- actor가 사라진 stiff passive를 줄인다.
  - 나쁨: `검증이 수행된다`
  - 좋음: `검증한다`, `검증이 돈다`, `검증을 붙인다`
- 추상 frame이 주장을 숨기면 제거한다.
  - 나쁨: `생산성의 측면에서 중요하다고 볼 수 있다`
  - 좋음: `생산성을 바꾸는 지점이다`
- connector가 실제 동사를 대신하면 동사를 복구한다.
  - 나쁨: `테스트를 통해 품질을 확보한다`
  - 좋음: `테스트로 잘못된 변경을 막는다`

## 피해야 할 문장 냄새

- 맞기는 한데 말맛이 없는 문장
- 영어 개념을 한국어 단어로만 치환한 문장
- 명사구가 계속 포개져 주어와 동사가 흐린 문장
- 설명과 부연이 주장보다 길어진 문장
- 모든 문단이 같은 길이와 리듬으로 이어지는 문장
- 사내 보고서처럼 안전하고 무해하게만 정리된 문장
- source에 없는 은유나 멋을 덧댄 문장

## 허용되는 강도

- source-backed claim은 단정적으로 말해도 된다.
- 약한 근거를 강한 문장으로 바꾸지 않는다.
- 확실하지 않은 내용은 `~일 수 있다`로 뭉개지 말고, open question이나 확인 필요로 남긴다.
- 공손하고 무해한 요약보다 정확한 주장과 대비를 우선한다.

## 검토 절차

1. source claim과 기술 term이 보존됐는지 본다.
2. English word order와 passive translationese를 걷어낸다.
3. 명사구 사슬을 동사 중심 문장으로 바꾼다.
4. 보고서식 connector와 padded summary를 줄인다.
5. 소리 내어 읽었을 때 발표자가 실제로 말할 수 있는지 본다.
6. 필요한 경우 scanner를 돌린다.

```bash
python3 .codex/skills/local/natural-korean-prose/scripts/check_natural_korean.py docs/02-seminar/prose
```

scanner 결과는 자동 실패가 아니라 review candidate다. 문맥이 최종 판단한다.
