## S004 코딩은 사라지는가
- intent: 01장의 질문을 챕터 단위로 고정
- opening: 개발의 추상화 수준 상승
- bridge: 코딩이 사라지는가보다 개발자가 직접 붙들 일이 어디로 이동하는가를 묻는다.

## S005 기계어 → 어셈블리
- intent: 언어 발전의 첫 추상화 사례
- opening: 이진 코드 직접 작성에서 명령어 기호로
- bridge: 기계어의 이진 표현이 어셈블리 명령어로 올라오며 직접 다루는 부담이 줄어든다.

## S006 어셈블리 → C/Pascal
- intent: 어셈블리와 C/Pascal의 표현량 차이
- opening: 레지스터와 syscall에서 함수와 컴파일러로
- bridge: Hello World 수준에서도 어셈블리와 C의 표현량 차이가 명확하다.

## S007 C → Java
- intent: 메모리 관리 책임의 이동
- opening: 수동 메모리 관리에서 GC로
- bridge: C의 malloc/free와 Java의 객체 생성 및 GC를 비교한다.

## S008 Java → Python
- intent: Java와 Python의 boilerplate 차이
- opening: 보일러플레이트 축소와 표현 밀도 상승
- bridge: 같은 Hello World라도 Java와 Python의 표현 밀도는 크게 다르다.

## S009 AI 개발
- intent: 언어 발전 사례의 현재형
- opening: 자연어 지시에서 작업 위임으로
- bridge: AI 개발은 개발의 종말이 아니라 추상화 수준 상승으로 읽어야 한다.

## S010 직접 하던 일의 감소
- intent: 시대별 위임 표와 질문 반전
- opening: 직접 작성 부담의 지속적 위임
- bridge: 시대별 전환을 표로 압축하고, 개발자가 끝까지 붙들 일이 어디로 이동하는지 묻는다.

## S011 숫자로 보는 변화
- intent: 숫자 근거를 대형 카드와 표로 압축
- opening: 타이핑 비중 감소의 현장 신호
- bridge: YC W25, OpenAI Codex 팀, Meta 엔지니어 사례를 큰 숫자 중심으로 보여 준다.

## S012 문서의 코드화
- intent: 문서 역할 변화와 규칙 파일 예시
- opening: 문서의 실행 경로 편입
- bridge: AI 코딩 도구에서 문서는 사람이 읽는 참고 자료를 넘어 에이전트가 따르는 운영 규칙이 된다.

## S013 컨텍스트를 설계하는 능력
- intent: Harness Engineer의 핵심 역량
- opening: 개발자의 역할은 환경 설계와 결과 판단으로 이동
- bridge: 문제 분해, 컨텍스트 선택, 수용 기준, 검증, 권한 경계가 새로운 핵심 역량이 된다.
