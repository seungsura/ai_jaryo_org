실전 워크플로우와 도구 세팅

운영 구조가 손기술을 이긴다

도구보다 원리

좋은 워크플로우는 손재주가 아니라
시스템이 원리를 강제하게 만드는 것입니다.

기억 의존

매번 지시를 다시 입력하고
검증을 기억에 맡김 (지속 불가)

시스템 강제

명령어와 라우팅이
검증 순서를 잡아줌 (표준화)

Workflow

Plan-Critic-Build 실전

Explore: 구조와 관련 파일 파악
Plan: 변경 계획과 위험 명시
Code: 승인된 계획만 구현
Commit: 검증 결과와 함께 커밋

도구 큐레이션 (Tool Curation)

많이 연결하는 것보다 필요한 것만 남기는 것이 성능입니다.

"유휴 인터페이스와 소음은
에이전트의 판단력을 흐립니다."

게이트 설계

  • Lint / Type Check
  • Unit / Integration Test
  • Code Review Gate
  • Pre/Post Tool Use Hooks
  • 위험 명령 차단 규칙
  • Human-in-the-loop

기계가 확인할 수 있는 것은 기계가 닫아야 합니다.

위험도에 따른 모드 조절

작업 유형 권장 모드
문서 초안, 로그 요약High Autonomy
테스트 보강, 리팩터링Auto-accept + Verify
아키텍처 변경, DBPlan Mode + Approval
삭제, 배포, 권한Manual Approval Only

/clear

/save

Issue → Worktree → Session → PR → Verify → Close

채팅창에 떠다니는 기준을 작업 환경의 일부로!

최소 워크플로우 (MVW)

  • CLAUDE.md에 반복 지시 3개 적기
  • 위험 명령 하나를 훅으로 막기
  • Lint/Type Check를 필수 게이트로 두기
  • 구현 전 계획 쓰고 승인 받기
  • 세션 종료 시 결정 사항 파일로 남기
