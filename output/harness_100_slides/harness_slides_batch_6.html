Chapter 06

멀티 에이전트 활용 패턴

병렬화보다 분해가 먼저다

분해의 미학

에이전트 수만 늘린다고 품질이 좋아지지 않습니다.
역할과 컨텍스트를 쪼개는 설계가 핵심입니다.

"메인 컨텍스트에 무엇을 남기고,
무엇을 밖으로 밀어낼 것인가?"

단일 에이전트의 한계

  • 컨텍스트 비대화: 핵심 신호가 소음에 묻힘
  • 역할 혼동: 자기 답을 자기가 채점하는 구조
  • 신뢰 경계 모호: 불필요한 권한 노출로 인한 사고 위험
Pattern 01

Sub-Agent

중간 작업을 격리하는 기본형

Isolation

대량 파일 요약, 로그 분석 등
메인 컨텍스트 오염 방지

Least Privilege

필요한 파일과 권한만 부여
(주로 읽기 전용)

Pattern 02

Orchestrator

1:N 위임 구조

Planner (목표 해석 및 분해)
Worker A
Worker B
Worker C
Aggregator (결과 통합)
Pattern 03

Parallel

속도보다 간섭 제어

의존성 낮은 작업

대안 비교, 독립 모듈 점검,
다양한 스타일 초안 생성

격리 도구

Git Worktree, 별도 브랜치,
명확한 출력 계약

Pattern 04

GAN-Style

생성과 평가의 분리

Generator

기능 구현 및 스프린트 수행

Evaluator

테스트, 루브릭, 리뷰 기준 평가
구체적 피드백 제공

Pattern 05

Agent Teams

양방향 토론과 합의

Researcher Architect Reviewer Coordinator

관점의 균형이 중요한 기획, 아키텍처 검토에 적합

Sub-Agent vs Agent Team

구분 Sub-Agent Agent Team
구조단방향 심부름양방향 토론
컨텍스트제한적, 요약 반환각자 별도 맥락 유지
적합 작업로그 파싱, 탐색기획, 아키텍처 토론

멀티 에이전트 설계 원칙

  • 필요한 것만 준다: 불필요한 정보는 소음이다
  • 역할과 금지를 정한다: 행동 반경을 명확히 통제
  • 출력 계약을 구조화한다: JSON, 표 등 파싱 가능한 형식
  • 생성과 검증을 가른다: 자기 채점 방지

멀티 모델 활용

모델의 특성에 따른 역할 분담

Claude
구현 계획 및 로직 작성

Codex
아키텍처 리스크 검토

Gemini
UI 및 문서 가독성 피드백

DeepSeek
복잡한 알고리즘 최적화

6장 요약

  • 멀티 에이전트는 병렬화가 아니라 분해다
  • 메인 컨텍스트를 비우는 것이 성능의 핵심이다
  • 생성과 검증의 분리가 품질을 보장한다
  • 패턴보다 명확한 역할 경계가 중요하다
